# Task 5: ErrorLog — on-device ERROR-only writer

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Реализовать Hilt-singleton `ErrorLog` — локальный писатель ERROR-событий в файл `context.filesDir/logs/errors.log`. Это единственный канал логирования ошибок в Phase 1: все failure-ветки в `DownloadRepository` и `LlmModelHelper`-обёртках (Task 6) будут звать `ErrorLog.e(component, description, cause)`.

Формат и правила — строго по `patterns.md § Error logging conventions`: одна строка на событие, без переносов, без эмодзи, без декоративных разделителей, шаблон `ERROR [component] description :: cause.message`. Перед записью `description` и `cause?.message` санитизируются (schape control whitespace → space + truncate до `MAX_FIELD_LEN = 500`). После записи — проверка размера файла и ротация при превышении `2 * 1024 * 1024` байт (переименование `errors.log` → `errors.log.1` с перезаписью предыдущей ротированной копии).

В Phase 1 легальный whitelist компонентов фиксирован: `download`, `inference-init`, `inference`, `inference-cleanup` (user-spec D11). Whitelist документируется в KDoc класса `ErrorLog` — в коде не enforced (строка `component` передаётся as-is), но вызовы с иными значениями будут отловлены code-reviewer'ом в последующих задачах.

`ErrorLog` регистрируется в Hilt через constructor-injection: класс аннотируется `@Singleton` + `@Inject constructor(@ApplicationContext context: Context)` — Hilt автоматически генерирует binding, явный `@Provides` в `CoreRuntimeModule` НЕ нужен (и вызвал бы duplicate binding error). Внутри держит `Mutex` для сериализации записей из параллельных корутин (инференс + download могут писать одновременно).

## What to do

1. Создать файл `core-runtime/src/main/kotlin/app/sanctum/machina/core/log/ErrorLog.kt`:
   - Публичный класс `ErrorLog` с конструктором `@Inject constructor(@ApplicationContext context: Context)`.
   - Публичный suspend-метод `suspend fun e(component: String, description: String, cause: Throwable? = null)`.
   - Приватные константы: `MAX_FIELD_LEN = 500`, `MAX_LOG_BYTES = 2 * 1024 * 1024`.
   - Приватный `Mutex` для сериализации записей.
   - Приватная функция sanitize (strip `\n`, `\r`, `\t` → space + truncate до `MAX_FIELD_LEN`).
   - Логика append: собрать строку `ERROR [component] sanitized-description :: sanitized-cause-message` (если `cause == null` — отбросить суффикс ` :: ...`), открыть target файл (создать родительскую директорию `filesDir/logs/` если её нет), дописать строку + `\n`.
   - Логика ротации после записи: если `file.length() > MAX_LOG_BYTES` — `File(file.parentFile, "errors.log.1").delete()` (если есть), затем `file.renameTo(File(file.parentFile, "errors.log.1"))`. Путь ВСЕГДА резолвится относительно `filesDir/logs/`, не bare-name (иначе резолв пошёл бы от cwd и упал).
   - KDoc в заголовке класса: описать whitelist Phase 1 компонентов (`download`, `inference-init`, `inference`, `inference-cleanup`), формат строки, поведение ротации, ссылку на `patterns.md § Error logging conventions`.
2. НЕ добавлять `@Provides provideErrorLog` в `CoreRuntimeModule`. `ErrorLog` аннотирован `@Singleton @Inject constructor(@ApplicationContext ctx: Context)` — Hilt автоматически генерирует binding; ручной `@Provides` вызвал бы duplicate-binding error.  `CoreRuntimeModule.kt` создаётся в Task 4 (skeleton); Task 5 его не трогает.

## Details

**Files:**
- `core-runtime/src/main/kotlin/app/sanctum/machina/core/log/ErrorLog.kt` — новый файл. Hilt-singleton, `@Inject constructor(@ApplicationContext context: Context)`. Приватный `Mutex`, приватные константы `MAX_FIELD_LEN=500` и `MAX_LOG_BYTES = 2 * 1024 * 1024`. Единственный публичный метод — `suspend fun e(component, description, cause)`.
- `core-runtime/src/main/kotlin/app/sanctum/machina/core/di/CoreRuntimeModule.kt` — создаётся Task 4 и не модифицируется Task 5. ErrorLog провайдится Hilt-ом автоматически через `@Inject constructor`.

**Dependencies:**
- Task 2 (Gradle skeleton + `:core-runtime` с Hilt, `CoreRuntimeModule` скелетом, `@ApplicationContext` qualifier доступен).
- Runtime libs уже в `:core-runtime`: `hilt-android`, `kotlinx.coroutines` (для `Mutex`), `androidx.core.ktx`.

**Edge cases:**
- Директория `filesDir/logs/` может не существовать при первом вызове — `parentFile?.mkdirs()` перед open.
- `cause == null` — НЕ добавлять суффикс ` :: null` в строку; писать только `ERROR [component] description`.
- `description` пустой / только whitespace — записывать как есть после sanitize (семантика не меняется, агент-писатель отвечает за осмысленность).
- Sanitize применяется к `description` и `cause?.message` независимо; `component` не санитизируется (он всегда literal из whitelist).
- Параллельный вызов `e()` из двух корутин (инференс + download) — `Mutex.withLock` гарантирует атомарность write+rotation пары.
- Файл `errors.log.1` уже существует — перед `renameTo` его надо явно `.delete()` (иначе `renameTo` на Android вернёт `false`).
- Размер строки > `MAX_LOG_BYTES` (маловероятно при MAX_FIELD_LEN=500) — после записи ротация сработает и одна строка станет единственной в `errors.log.1`.
- Исключения при I/O (например, `filesDir` недоступен — теоретически невозможно в норме) — swallow внутри `withLock`, никогда не пробрасывать вверх (логгер не должен валить вызывающий код).

**Implementation hints:**
- `Mutex` из `kotlinx.coroutines.sync.Mutex`; метод суспенд именно из-за `mutex.withLock`.
- File I/O блокирующий — завернуть тело `withLock` в `withContext(Dispatchers.IO)` (стандартный паттерн для работы с файлами из suspend-метода).
- Sanitize — одна regex-замена `Regex("[\\n\\r\\t]")` → `" "`, потом `take(MAX_FIELD_LEN)`.
- Сборка строки — обычный `buildString` или простая интерполяция; ветка `if (cause != null)` добавляет `" :: " + sanitize(cause.message ?: cause::class.simpleName.orEmpty())`.
- Append в файл — `File.appendText(line + "\n", Charsets.UTF_8)`.
- Ротация — один блок сразу после append, внутри того же `withLock`, чтобы чтение `file.length()` соответствовало последней записи.
- KDoc шаблон:
  - Краткое описание «ERROR-only on-device writer per patterns.md § Error logging conventions».
  - Секция `Components (Phase 1 whitelist):` с четырьмя значениями.
  - Секция `Format:` с примером одной строки.
  - Секция `Rotation:` — 2 MB, `errors.log.1`, overwrite.
- НЕ добавлять `Log.e(...)` в Logcat параллельно с файлом — по patterns.md файловый лог — единственный канал.
- НЕ экранировать `[` или `]` в формате — оставить как в шаблоне.
