# Task 9: ChatScreen + ChatViewModel (streaming, TTFT, reset, stop, init-error UI)

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Реализовать чат-экран поверх `ModelRegistry` для активной модели. ViewModel получает `modelName` через `SavedStateHandle` (из nav-аргумента), инициализирует engine асинхронно, выставляет `ChatUiState` (Loading / Ready / Failed) и оркестрирует streaming-диалог, сброс и остановку.

Scope покрывает три user-spec критерия: AC-8 (streaming + TTFT footer), AC-10 (reset без повторной загрузки engine), AC-11 (stop ≤500 мс + «(прервано)») и AC-15 (graceful init-failure UI для битого `.litertlm`). Все механики фиксированы в Decisions T10 (Stop через готовый `LlmChatModelHelper.stopResponse`) и T11 (формат футера `"TTFT {N}ms · {X.X}s total"`, без tok/s по D9).

ChatViewModel (@HiltViewModel):
- Зависимости в конструкторе: `SavedStateHandle` (для `modelName`), `ModelRegistry` (init / cleanup / resetConversation — lifecycle-операции под `lifecycleMutex`), `LlmChatModelHelper` (stateless helper для `runInference` / `stopResponse` — вызывается напрямую, в обход registry, чтобы длинный стриминг НЕ блокировал `lifecycleMutex`), `ErrorLog` (для логирования ошибок inference). Rationale: `runInference` — долгая стриминговая операция; registry mutex'ом защищены только init/cleanup/reset.
- В `init` запускает корутину `registry.initialize(modelName)`, по результату переводит UI в Ready либо Failed. В `Failed(rawCause)` кладёт `result.exceptionOrNull()?.message` напрямую — сообщение уже sanitized через `cleanUpMediapipeTaskErrorMessage` внутри `LlmChatModelHelper.initialize.onDone(...)`, повторно оборачивать НЕ нужно.
- Держит `StateFlow<ChatUiState>` и `StateFlow<List<Message>>`.
- `send(text)`: добавляет user-message, затем placeholder-assistant (`streaming=true`), получает `model: Model` через `registry.getModel(modelName)` (новый метод registry после Task 6 fix, возвращает `Model?` — null если неизвестна или ещё не инициализирована; после успешного `registry.initialize(modelName)` всегда не-null, поэтому `val model: Model = registry.getModel(modelName) ?: error("Model not initialized")`), вызывает `LlmChatModelHelper.runInference(model=model, input=text, resultListener, cleanUpListener, onError, images=emptyList(), audioClips=emptyList(), coroutineScope=viewModelScope, extraContext=null)`. На первом непустом `partial` фиксирует TTFT, accumulate токенов в текст сообщения; на `done=true` — считает `totalMs` и прикладывает футер через `stringResource`/`context.getString(R.string.ttft_footer_format, ttftMs, totalSec)` (два аргумента: `Long` ms и `Double` seconds); `onError(msg)` — помечает сообщение interrupted и логирует через `errorLog.e("inference", msg)` (registry НЕ перехватывает ошибки runInference, т.к. вызов идёт напрямую через helper).
- `stop()`: `helper.stopResponse(model)` (принимает `Model`, не `LlmModelInstance`) + помечает последнее assistant-message `interrupted=true` (UI-суффикс `chat_message_interrupted_suffix` из `strings.xml`). Перед вызовом получить `model` через `registry.getModel(modelName)`.
- `reset()`: `registry.resetConversation(modelName)`; очищает список сообщений; engine НЕ пересоздаётся (research §13 — conversation заменяется на том же engine, подтверждает AC-10 «без повторной задержки»).
- `onCleared()`: освобождает native-память через **отдельный background-scope**, т.к. `viewModelScope` уже отменён к моменту вызова `onCleared()`, а `GlobalScope` дискуражен lint'ом. Паттерн: `CoroutineScope(SupervisorJob() + Dispatchers.IO).launch { registry.cleanup(modelName) }`. Альтернатива: выставить `registry.cleanupBlocking(modelName)` как fire-and-forget wrapper (внутри делает тот же launch на собственном scope).

UI (ChatScreen):
- **Loading**: центрованный `CircularProgressIndicator` + текст «Загрузка модели…» (D12: Back во время loading возвращает в список моделей — стандартный `popBackStack`).
- **Failed** (AC-15): центрованный column с `Icon(Icons.Default.Error)`, заголовок `stringResource(R.string.chat_load_failed_title)` = «Не удалось загрузить модель. Скачайте заново.», под ним сообщение в моноширинном small caption (raw cause — уже sanitized через `cleanUpMediapipeTaskErrorMessage` в `LlmChatModelHelper.initialize`, повторно НЕ оборачивать), `TextButton` с текстом `stringResource(R.string.btn_back)` → `popBackStack`. Retry-кнопки НЕТ (битый `.litertlm` нужно перекачать).
- **Ready**: `TopAppBar` с именем модели + `IconButton(Icons.Default.Refresh)` справа → `reset()`; `LazyColumn` с автоскроллом к последнему токену; message-bubble по роли (user / assistant), футер под assistant-текстом, суффикс `stringResource(R.string.chat_message_interrupted_suffix)` = " (прервано)" если `interrupted=true`; нижняя строка — `OutlinedTextField` с placeholder `stringResource(R.string.chat_input_placeholder)` = "Сообщение" + `IconButton` переключается Send ↔ Stop (content description — `stringResource(R.string.btn_send)` / `stringResource(R.string.btn_stop)`) в зависимости от `isGenerating`.
- Все строки — из `res/values/strings.xml` (создан в Task 7), файл ТОЛЬКО читается, не модифицируется. Канонический набор ключей для этого экрана: `chat_loading_model`, `chat_load_failed_title`, `chat_input_placeholder`, `chat_message_interrupted_suffix`, `ttft_footer_format`, `btn_send`, `btn_stop`, `btn_reset`, `btn_back`.

## What to do

1. Создать `app/src/main/kotlin/app/sanctum/machina/ui/chat/Message.kt`:
   - `enum class MessageRole { USER, ASSISTANT }`
   - `data class Message(val role: MessageRole, val text: String, val streaming: Boolean = false, val interrupted: Boolean = false, val footer: String? = null)`

2. Создать `app/src/main/kotlin/app/sanctum/machina/ui/chat/ChatViewModel.kt`:
   - `@HiltViewModel class ChatViewModel @Inject constructor(savedStateHandle: SavedStateHandle, private val registry: ModelRegistry, private val helper: LlmChatModelHelper, private val errorLog: ErrorLog) : ViewModel()`.
     - `registry` — только lifecycle: `initialize`, `cleanup`, `resetConversation`, `getModel(modelName): Model?` (новый accessor, заменил `getInstance` в Task 6 fix; возвращает `Model?` — null если неизвестна или ещё не инициализирована).
     - `helper` — stateless инференс: `runInference`, `stopResponse` — принимают `Model` (не `LlmModelInstance`); вызываются напрямую, минуя registry's `lifecycleMutex`.
     - `errorLog` — для логирования ошибок inference (runInference НЕ обёрнут в registry, поэтому логируем здесь).
   - `modelName` читается из `savedStateHandle["modelName"]` (nav arg, см. Task 10).
   - `sealed class ChatUiState { object Loading; data class Ready(val isGenerating: Boolean); data class Failed(val rawCause: String) }` — либо отдельным файлом внутри пакета, либо вложенным в ChatViewModel.
   - `StateFlow<ChatUiState>` — initial `Loading`.
   - `StateFlow<List<Message>>` — initial empty.
   - В `init { viewModelScope.launch { registry.initialize(modelName).fold(onSuccess = { _uiState.value = Ready(false) }, onFailure = { e -> _uiState.value = Failed(e.message ?: e::class.simpleName.orEmpty()) }) } }`. Сообщение `e.message` уже sanitized через `cleanUpMediapipeTaskErrorMessage` внутри `LlmChatModelHelper.initialize.onDone(...)` — повторно НЕ оборачиваем.
   - `fun send(text: String)`: добавляет user-Message, placeholder-assistant (`streaming=true`), ставит `Ready(isGenerating=true)`, фиксирует `startMs = System.currentTimeMillis()`, получает `val model: Model = registry.getModel(modelName) ?: error("Model not initialized")` (после успешного init в Ready-состоянии всегда не-null), вызывает с **именованными параметрами** (чтобы не зависеть от порядка 9 аргументов):
     ```
     helper.runInference(
       model = model,
       input = text,
       resultListener = { partial, done, _partialThinkingResult -> /* accumulate, TTFT, footer on done */ },
       cleanUpListener = { /* no-op / debug */ },
       onError = { msg -> errorLog.e("inference", msg); /* mark interrupted, Ready(isGenerating=false) */ },
       images = emptyList(),
       audioClips = emptyList(),
       coroutineScope = viewModelScope,
       extraContext = null
     )
     ```
     **Полная сигнатура helper (9 параметров):**
     ```
     LlmChatModelHelper.runInference(
       model: Model,                                                                                   // ← Model, не LlmModelInstance
       input: String,
       resultListener: (partialResult: String, done: Boolean, partialThinkingResult: String?) -> Unit,
       cleanUpListener: CleanUpListener,
       onError: (String) -> Unit,
       images: List<Bitmap> = emptyList(),
       audioClips: List<ByteArray> = emptyList(),                                                      // ← ByteArray, не AudioClip
       coroutineScope: CoroutineScope,
       extraContext: Map<String, String>? = null                                                       // ← 9-й параметр
     )
     ```
     В Phase 1 все опциональные параметры пустые/null: `images = emptyList()`, `audioClips = emptyList()`, `extraContext = null`. **Обязательно использовать именованные параметры** — иначе легко перепутать порядок (9 аргументов).
     resultListener: первый непустой `partial` → `firstTokenMs = now`; accumulate `partial` в текст последнего сообщения; `done=true` → `totalMs = now - startMs`, footer через `context.getString(R.string.ttft_footer_format, (firstTokenMs - startMs), totalMs / 1000.0)` (два аргумента: `Long` ms и `Double` seconds — совпадает с плейсхолдерами `%1$dms · %2$.1fs` в strings.xml), `streaming=false`, `Ready(isGenerating=false)`.
   - `fun stop()`: достаёт `val model: Model = registry.getModel(modelName) ?: return`, вызывает `helper.stopResponse(model)` (принимает `Model`), помечает последнее assistant-message `interrupted=true`, `Ready(isGenerating=false)`.
   - `fun reset()`: `viewModelScope.launch { registry.resetConversation(modelName); _messages.value = emptyList() }`. engine не пересоздаётся.
   - `onCleared()` — background-cleanup в отдельном scope, т.к. `viewModelScope` уже cancel'нут, а `GlobalScope` discouraged by lint:
     ```
     override fun onCleared() {
       super.onCleared()
       CoroutineScope(SupervisorJob() + Dispatchers.IO).launch {
         registry.cleanup(modelName)
       }
     }
     ```
     `registry.cleanupBlocking(...)` — такого метода в Task 6 canonical interface НЕТ; есть только `suspend fun cleanup(name)`. Использовать SupervisorJob-паттерн выше. НЕ использовать `viewModelScope` (мёртв) и НЕ использовать `GlobalScope.launch` напрямую (lint warning).

3. Создать `app/src/main/kotlin/app/sanctum/machina/ui/chat/ChatScreen.kt`:
   - `@Composable fun ChatScreen(modelName: String, onBack: () -> Unit, viewModel: ChatViewModel = hiltViewModel())`. NavController в ChatScreen НЕ передаётся — back обрабатывается через callback `onBack`, который NavHost (Task 10) связывает с `navController.popBackStack()`. `modelName` приходит из nav-args (Task 10 parsing) — ChatViewModel также читает его через SavedStateHandle.
   - Собрать `uiState` и `messages` через `collectAsStateWithLifecycle()`.
   - `when (state) { Loading -> LoadingContent(); Failed -> FailedContent(rawCause, onBack = onBack); Ready -> ReadyContent(messages, isGenerating, onSend, onStop, onReset, onBack) }`.
   - `LoadingContent`: `Box(Modifier.fillMaxSize(), Alignment.Center) { Column { CircularProgressIndicator(); Text(stringResource(R.string.chat_loading_model)) } }`.
   - `FailedContent`: центровая Column с `Icon(Icons.Default.Error)`, title (`stringResource(R.string.chat_load_failed_title)`), monospace small caption с `rawCause`, `TextButton(onBack) { Text(stringResource(R.string.btn_back)) }`.
   - `ReadyContent`: `Scaffold(topBar = { TopAppBar(title = { Text(modelName) }, actions = { IconButton(onReset) { Icon(Icons.Default.Refresh, contentDescription = stringResource(R.string.btn_reset)) } }) }) { padding -> Column(Modifier.padding(padding)) { LazyColumn(...) { items(messages) { MessageBubble(it) } }; ChatInputRow(isGenerating, onSend, onStop) } }`.

   - `MessageBubble`: по роли — выравнивание + фон; для assistant рендерит текст + (если `interrupted`) суффикс `stringResource(R.string.chat_message_interrupted_suffix)` = " (прервано)" (встроенный ведущий пробел) + (если `footer != null`) footer строкой ниже в caption-стиле.

**Edge cases:**
- Пустой `text` в `send()` — игнорировать (не отправлять), чтобы не создавать пустые bubble'ы.
- `registry.getModel(modelName)` вернёт `null` если engine ещё не инициализирован — в Ready такого не должно быть, но на всякий случай проверить: в `send()` — fail fast (`error("Model not initialized")`) или мягко перевести в Failed с log; в `stop()` — просто `return` (нет смысла останавливать несуществующий стрим).
- Stop нажат до первого токена (TTFT ещё не зафиксирован) — `interrupted=true`, footer не выставляется; assistant-message остаётся с пустым текстом или показывает только частичный. UX: пусть bubble виден с «(прервано)» без футера.
- Reset во время стрима — сначала вызвать `stopResponse`, дождаться остановки, потом `resetConversation` и очистку списка. Либо проще: UI отключает кнопку Refresh когда `isGenerating=true` (рекомендуется).
- Back во время стрима → `onCleared` → cleanup. `viewModelScope` умирает, inference должен быть отменён (coroutineScope=viewModelScope в runInference обеспечит отмену).
- Configuration change (rotation) — `ChatViewModel` переживёт, engine — тоже (registry @Singleton). StateFlow восстановится из ViewModel. Но при rotation во время Loading — корутина `registry.initialize` уже запущена, повторный init не должен вызываться (init-блок ViewModel выполняется ОДИН раз — это правильно).
- `onError` от inference с пустым message — использовать `"(no message)"` как fallback в UI-сообщении.
- `cleanUpListener` — просто колбек о завершении (как в Gallery); в нашем случае used for debug/no-op. Не путать с `onError`.

**Implementation hints:**
- **Типы из helper API (актуально для Phase 1):** `LlmChatModelHelper.runInference` / `stopResponse` / `resetConversation` / `cleanUp` принимают **`Model`**, НЕ `LlmModelInstance`. Получать `Model` через `registry.getModel(modelName)` (новый accessor после Task 6 fix, заменил старый `getInstance`). `runInference` имеет **9 параметров**, `audioClips: List<ByteArray>` (не `List<AudioClip>`), последний параметр — `extraContext: Map<String, String>? = null`. Phase 1 передаёт `images = emptyList()`, `audioClips = emptyList()`, `extraContext = null`. **Обязательно использовать именованные аргументы** в вызове `helper.runInference(...)` — при 9 параметрах порядковый вызов легко сломать.
- Stop-кнопка — отдельная иконка `Icons.Default.Stop`; Send — `Icons.Default.Send`; переключение через `if (isGenerating) ... else ...` в `IconButton.onClick` и `Icon.imageVector`. contentDescription переключается через `stringResource(if (isGenerating) R.string.btn_stop else R.string.btn_send)`.
- Для TTFT фиксируй `firstTokenMs` только ПЕРВЫЙ раз (проверяй `if (firstTokenMs == 0L && partial.isNotEmpty()) firstTokenMs = System.currentTimeMillis()`).
- Форматирование footer'а: `context.getString(R.string.ttft_footer_format, ttftMs.toInt(), totalMs / 1000.0)`. Строка в strings.xml = `"TTFT %1$dms · %2$.1fs total"`. Аргумент №1 — `Int` (ms, TTFT безопасно влезает; `ttftMs` — Long от `System.currentTimeMillis()` разности, кастим через `.toInt()` — TTFT всегда < 60_000 ms на живой модели), аргумент №2 — `Double` (seconds). Android `getString(int, Object...)` форматирует однозначно для `%d` / `%.1f` (не locale-зависимые). **НЕ** использовать Kotlin `"...".format(...)` — default locale поломает десятичный разделитель.
- Accumulate токенов: держи `StringBuilder` локально в `send()` через closure, на каждом `partial` делай `sb.append(partial)` и копируй в messages через `.copy(text = sb.toString())` на последнем элементе. StateFlow update: `_messages.update { list -> list.toMutableList().also { it[it.lastIndex] = it.last().copy(text = sb.toString()) } }`.
- Автоскролл — `rememberLazyListState()` + `LaunchedEffect(messages.size, messages.lastOrNull()?.text?.length) { listState.animateScrollToItem(messages.lastIndex.coerceAtLeast(0)) }`.
- `onCleared()` cleanup — `viewModelScope` уже cancel'ится к моменту вызова `onCleared()`, поэтому launch'ить на нём нельзя. Использовать отдельный scope: `CoroutineScope(SupervisorJob() + Dispatchers.IO).launch { registry.cleanup(modelName) }`. НЕ использовать `GlobalScope.launch` — discouraged by lint. Cleanup должен пережить смерть ViewModel'и. Метода `registry.cleanupBlocking` НЕТ — только `suspend fun cleanup(name)` в canonical interface Task 6.
- **НЕ** оборачивать raw-cause повторно через `cleanUpMediapipeTaskErrorMessage`: сообщение из `Result.Failed(msg)` в `registry.initialize(...)` уже прошло через этот sanitizer внутри `LlmChatModelHelper.initialize.onDone(cleanUpMediapipeTaskErrorMessage(e.message ?: ""))`. Двойная обёртка = мусор в UI.
- НЕ делай retry-кнопку в Failed state (user-spec AC-15 — только «Назад»).
- Логирование ошибок inference происходит в ChatViewModel через инжектированный `errorLog.e("inference", msg)` в коллбеке `onError` у `helper.runInference`. Registry НЕ перехватывает эти ошибки (вызов идёт напрямую через helper, минуя registry wrapper). Ошибки init/cleanup/reset логирует registry — их повторно НЕ логируем.
- Моноширинный шрифт для raw-cause: `fontFamily = FontFamily.Monospace` на `Text`.
- Disable поля ввода (или игнорируй tap) в состоянии Loading / Failed — TextField рендерится только в Ready.
- `SavedStateHandle` ключ — `"modelName"` (должен совпадать с nav-arg в Task 10 `chat/{modelName}`).
