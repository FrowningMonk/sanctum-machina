# Task 14: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](../../../.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Финальная приёмка Phase 1 перед передачей APK пользователю. Агент автоматически выполняет всё, что поддаётся автоматизации (сборка, unit-тест, grep-проверки TAC-1…TAC-15), собирает APK, готовит инструкцию для ручной проверки на Honor 200 (user-spec § «Пользователь проверяет» — 15 шагов + back-to-back с Gallery по AC-14 protocol), вместе с пользователем выполняет 6-замерный протокол TTFT, считает медианы, применяет гейт `median(Phase 1) / median(Gallery) ≤ 1.5` и фиксирует финальный verdict Phase 1 в `logs/qa-report.json`.

Phase 1 не имеет CI/CD и MCP-инструментов для Android — post-deploy verification не применима. Pre-deploy QA + ручная проверка пользователем = единственные финальные гейты фазы.

## What to do

1. **Агентом-автоматически (блокирующая часть):**
   - Запустить `./gradlew build` из корня проекта. Убедиться, что команда завершается `BUILD SUCCESSFUL` (exit code 0). Затем запустить `./gradlew lintDebug` и проверить отчёты `app/build/reports/lint-results-debug.xml` и `core-runtime/build/reports/lint-results-debug.xml`: в XML не должно быть ни одного атрибута `severity="Error"` (`grep -c 'severity="Error"' <report>` → 0 на каждом).
   - Запустить `./gradlew :core-runtime:test`. Убедиться: `BUILD SUCCESSFUL`, ровно 1 тест пройден (`AllowlistLoaderTest`).
   - Выполнить полный набор grep/jq-проверок TAC-1 … TAC-15 (см. Details → список команд). Для каждого TAC зафиксировать: команда, фактический результат, verdict (pass/fail).
   - Убедиться что APK существует по пути `app/build/outputs/apk/debug/app-debug.apk`. Зафиксировать в отчёте: путь, размер в байтах, sha256-хэш.
   - Если хотя бы один TAC-1…TAC-15 fail — остановиться, записать частичный отчёт с failed-verdict, вернуть управление пользователю для разбора. APK пользователю не отдаём.

2. **Подготовить инструкцию для пользователя:**
   - Сформировать понятный блок «Что сделать на Honor 200» с 15 шагами из user-spec § «Пользователь проверяет» (1. установка APK → 15. corrupted model). Инструкция включает команды `adb install -r app-debug.apk`, `adb shell dumpsys meminfo app.sanctum.machina`, `adb push` и т.п.
   - Включить отдельным разделом AC-14 protocol: 3 прогона Gallery (одинаковый промпт средней длины, ответ ~200-500 токенов, Reset conversation между прогонами, секундомер от тапа Send до первого символа) → `G1, G2, G3` → `median(G)`; затем 3 прогона Phase 1 APK на том же промпте с Reset между (TTFT читаем из футера каждого ответа) → `S1, S2, S3` → `median(S)`.
   - Дать инструкцию: после завершения пользователь возвращает агенту 6 замеров + промпт, который использовал.

3. **Совместно с пользователем (после его ручных шагов):**
   - Получить от пользователя результаты 15 шагов (pass/fail по каждому) + 6 TTFT замеров.
   - Посчитать `median(G)`, `median(S)`, отношение `median(S) / median(G)`. Применить гейт `≤ 1.5`.
   - Если гейт пройден и все 15 пользовательских шагов прошли — Phase 1 verdict = `PASS`. Если любой из 15 шагов fail ИЛИ гейт > 1.5 — Phase 1 verdict = `FAIL` + зафиксировать причину.

4. **Записать финальный QA-отчёт** в `work/phase-1-foundation/logs/qa-report.json` со всеми полями из Details → структура отчёта.

## Verification Steps

### Automated

- `./gradlew build` → exit code 0, `BUILD SUCCESSFUL`.
- `./gradlew lintDebug` → `grep -c 'severity="Error"' app/build/reports/lint-results-debug.xml` = 0 и то же для `core-runtime/build/reports/lint-results-debug.xml`.
- `./gradlew :core-runtime:test` → 1 passed (`AllowlistLoaderTest`).
- Все grep/jq команды из Details → «TAC checks» возвращают ожидаемый результат.
- `qa-report.json` валиден (парсится как JSON) и содержит все обязательные поля.

### User

Пользователь на Honor 200 выполняет 15 шагов (user-spec § «Пользователь проверяет»):

1. Установка APK — иконка Sanctum, MainActivity запускается без краша.
2. POST_NOTIFICATIONS → Allow.
3. Экран моделей: Gemma-4-E2B-it (~2.4 ГБ) + Gemma-4-E4B-it (~4.3 ГБ), оба «Не скачано».
4. Скачивание E2B: 0% → 100%, foreground-нотификация, скорость.
5. Resume после kill: начать E4B, дождаться 20%, убить приложение, перезапустить — продолжает с ~20%, HTTP Range в logcat.
6. Отмена скачивания: тап «Отмена» → статус «Не скачано», `.gallerytmp` сохраняется, дотапать.
7. Загрузка модели: тап «Загрузить» на E2B → ChatScreen, `Preferred backend: GPU` в logcat.
8. Инференс: промпт → стрим токенов, футер `TTFT YYYms · NNs total`.
9. Stop mid-generation: длинный промпт → Stop в ~500мс, «(прервано)», следующий промпт работает.
10. Multi-turn: 3 пары, контекст сохраняется, нет OOM.
11. Сброс конверсации: без задержки на re-initialize.
12. Cleanup при выходе: PSS падает до базового.
13. Переключение модели: E2B → E4B без OOM.
14. Оффлайн: Wi-Fi/data off → инференс работает.
15. Corrupted model (опц.): `adb push` мусора в `.litertlm` → читаемая ошибка, не native crash.

**AC-14 protocol (TAC-16):**
- В Gallery загрузить Gemma-4-E2B-it. Выбрать один промпт (ответ ~200-500 токенов). 3 раза подряд: Reset conversation → Send → засечь секундомером от тапа Send до появления первого символа. Три значения: `G1, G2, G3`.
- В Phase 1 APK: тот же промпт 3 раза с Reset между прогонами. Из футера каждого ответа взять TTFT: `S1, S2, S3`.
- Вернуть агенту: промпт, 6 замеров (`G1..G3, S1..S3`), результаты пунктов 1-15.

Агент считает медианы, применяет гейт, фиксирует verdict.

- Path: `app/build/outputs/apk/debug/app-debug.apk`
- Size: `wc -c < app/build/outputs/apk/debug/app-debug.apk` (портативно, работает в git-bash на Windows и Linux). Альтернатива: `python -c "import os; print(os.path.getsize('app/build/outputs/apk/debug/app-debug.apk'))"`. НЕ использовать `stat -c %s` — Linux-only, не работает в git-bash на Windows.

### Edge cases

- Первый `./gradlew build` может быть холодным (3-10 мин). Не считать за ошибку долгое время сборки.
- Lint-отчёты создают имена с суффиксом `-debug`, `-release`; используем именно `lint-results-debug.xml` (из `./gradlew lintDebug`). Gradle плагин android всегда пишет XML рядом с HTML — полагаемся на XML.
- Если пользователь предоставил только 2 замера вместо 3 — попросить третий, не считать медиану от двух.
- Если `median(G) == 0` (секундомер некорректен) — просить перезамерить.
- Если один из 15 user-шагов показал warning, но не блокер — фиксировать как `pass` с `notes`, решение по допуску принимает пользователь.

### Implementation hints

- Первичный pass/fail для TAC-1 — exit code `./gradlew build` (0 = pass). XML lint-отчёт — дополнительная проверка: `grep -c 'severity="Error"' app/build/reports/lint-results-debug.xml` → ожидаем 0, и то же для `core-runtime/build/reports/lint-results-debug.xml`. НЕ использовать HTML-отчёты (`class="error"` не гарантирован) — только XML.
- Пути lint-отчётов — строго `.xml`, не `.html`: `app/build/reports/lint-results-debug.xml` и `core-runtime/build/reports/lint-results-debug.xml`.
- Использовать Grep tool вместо shell `grep` там, где возможно — он работает одинаково на Windows/Linux.
- JSON-отчёт писать через Python one-liner или прямой `json.dumps` — проверить валидность через `jq . qa-report.json` или `python -c "import json; json.load(open(...))"`.
- Перед выдачей инструкции пользователю — коротко озвучить в чате: «Автоматика пройдена, APK размером X байт, SHA256=…; жду 6 замеров + отчёт по 15 шагам».
- TAC-9 protocol: после `grep -rE "ErrorLog\.e\(" ...` пройти по каждому совпадению вручную, выписать первый аргумент (component-string) — должен быть строго один из `"download" | "inference-init" | "inference" | "inference-cleanup"`. Любое другое значение (например `"allowlist"`, `"inference-run"`, `"download-cancel"`) — TAC-9 fail с указанием файла:строки.
- TAC-2 protocol: проверить, что в `AllowlistLoaderTest` есть ассерт вида `assertEquals("gpu", model.accelerators.split(",")[0])` (или эквивалент); поле — CSV String "gpu,cpu", не массив. Ассерт `accelerators[0] == 'g'` (по символу) или `accelerators[0] == "gpu"` (как массив) — fail.
