## Description

Финальная сшивка Phase 1: создаём `SanctumApp.kt` с `NavHost` и двумя route'ами (`model_manager` как start, `chat/{modelName}` с аргументом), затем заменяем stub в `MainActivity.setContent` (оставленный в Task 7) на `SanctumTheme { SanctumApp() }`. NavHost связывает готовые экраны из Task 8 (`ModelManagerScreen`) и Task 9 (`ChatScreen`) — это последний кусок, после которого APK проходит полный сценарий user-spec § Основной сценарий пп. 1-10 на Honor 200.

Важно: companion-FQN плумбинг (`DefaultDownloadRepository` FQN в `SanctumApplication.onCreate` per Decision T6) НЕ относится к этой задаче — он уже сделан в Task 7. Task 10 трогает в `MainActivity` ТОЛЬКО блок `setContent { ... }`.

Навигация без type-safe routes (per User-Spec Deviations — в Phase 1 не тянем `kotlinx-serialization`). Восстановление после process death обрабатывается дефолтным поведением NavHost: backstack восстанавливается системой, либо при полной очистке граф стартует с `model_manager` — никакой дополнительной сериализации state не добавляем.

**Dependencies:**
- Task 7 (MainActivity stub + SanctumTheme + POST_NOTIFICATIONS + `SanctumApplication` с FQN-плумбингом `DefaultDownloadRepository` per Decision T6 + strings.xml).
- Task 8 (`ModelManagerScreen` + `ModelManagerViewModel` с `onLoad: (String) -> Unit` callback).
- Task 9 (`ChatScreen` + `ChatViewModel` принимающий `modelName: String` + `onBack: () -> Unit` callback).
- Gradle-артефакт `androidx.navigation:navigation-compose` уже в libs.versions.toml (Task 2) — проверить перед началом, если отсутствует — добавить через Task 2 или уточнить у пользователя.
- `androidx.hilt:hilt-navigation-compose` для `hiltViewModel()` — аналогично, должен быть добавлен в Wave 2.

**Edge cases:**
- Process death: при восстановлении Android сам ресторит `NavController` backstack; если очищен полностью — граф стартует заново с `model_manager`. Никакой extra saved-state логики не добавлять.
- `modelName` в route содержит символы, требующие URL-encoding (пробелы, точки, слэши): имена моделей из `ModelAllowlist` — ASCII-safe (litert-community/gemma-3n-*), но для безопасности при `navigate("chat/$modelName")` — полагаемся на дефолтное NavHost кодирование (Compose Navigation делает encode StringType автоматически).
- Системный back на `ChatScreen` обрабатывается дефолтным поведением — `popBackStack()` вызывается navigation framework'ом. Button «Назад» в Failed-state из Task 9 должен явно вызывать `onBack()` коллбэк, который в SanctumApp → `navController.popBackStack()`.
- Deep linking НЕ поддерживаем в Phase 1 — никаких `deepLinks = listOf(...)` в route'ах.

**Implementation hints:**
- Hoist `hiltViewModel()` ВНУТРИ `composable(...)` лямбды, НЕ снаружи (ViewModel должна быть scoped к NavBackStackEntry, не к `MainActivity`).
- Для чтения аргумента: `composable("chat/{modelName}", arguments = listOf(navArgument("modelName") { type = NavType.StringType })) { entry -> val modelName = entry.arguments?.getString("modelName") ?: return@composable; ChatScreen(modelName = modelName, ...) }`. Null-branch `return@composable` вместо throw — защищает от теоретического race-случая.
- `SanctumApp()` — no-args composable, без модификаторов. Всю тему оборачивает `MainActivity` через `SanctumTheme { SanctumApp() }`.
- Проверить, что `SanctumTheme` доступен (создан в Task 7 по пути `app/src/main/kotlin/app/sanctum/machina/ui/theme/Theme.kt`).
