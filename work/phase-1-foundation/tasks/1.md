# Task 1: Git init + .gitignore

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:infrastructure-setup` — [skills/infrastructure-setup/SKILL.md](~/.claude/skills/infrastructure-setup/SKILL.md)

## Description

Инициализировать git-репозиторий в корне проекта `C:\AI-WORK\PhoneWrap\`, заложить `.gitignore` и `.gitattributes` так, чтобы с первого же коммита в историю не попали ни артефакты сборки Android, ни секреты (HF-токены, keystore-файлы, `.env`), ни вендорный снимок `gallery-source/`. Это стартовая задача всей фазы — от неё зависят все остальные: без чистого репо и рабочей ветки ни одна последующая таска не сможет коммититься по правилам из `patterns.md § Git Workflow`.

Ключевые особенности:
- **Tag `master`** (не ветка) на первом коммите — immutable zero-state marker. Никогда не двигается, используется как «ядерный» escape hatch для возврата проекта в начальное состояние.
- **Рабочая ветка `phase/1-foundation`** создаётся от первого коммита (в Phase 1 `main` ещё не существует как отдельная ветка — репозиторий ещё пустой, а `master` это тег). Вся работа фазы пойдёт в `phase/1-foundation`.
- **CRLF для Windows** через `.gitattributes` — проект разрабатывается на Windows (Honor 200 — только deploy target), поэтому end-of-line policy должна быть зафиксирована явно, иначе Gradle/Kotlin файлы будут пересохраняться с разными line endings на разных машинах.

После этой задачи репозиторий готов принимать коммиты от Task 2 (Gradle infrastructure) и далее.

## What to do

1. Выполнить `git init` в корне `C:\AI-WORK\PhoneWrap\` (если `.git/` ещё нет — проверить сначала).
2. Создать `.gitignore` со списком из `patterns.md § .gitignore essentials` и дополнениями из tech-spec Task 1 — включить как минимум:
   - `gallery-source/` (вендорный снимок Gallery, читается по ходу фазы, но в историю не пишется)
   - `.idea/`, `*.iml` (IntelliJ / Android Studio)
   - `build/`, `**/build/` (Gradle-артефакты всех модулей)
   - `local.properties` (SDK path, абсолютный путь машины разработчика)
   - `.env` (env-переменные)
   - `*.keystore`, `*.jks`, `release-keys/` (подписные ключи APK)
   - `gradle.properties.local` (локальные оверрайды Gradle-настроек)
   - `app/schemas/*.local.json` (локальные схемы Room, не должны шариться)
   - log output files (`*.log`, каталоги логов)
3. Создать `.gitattributes` с политикой line endings для Windows — `* text=auto eol=lf` как базовое, и явные правила для Kotlin/Gradle/скриптов (где `eol=lf`) и `.bat`/`.cmd` (где `eol=crlf`). Бинарники (PNG, JKS, PDF) помечаются `binary`.
4. Сделать первый коммит (conventional: `chore: initialize repository with .gitignore and .gitattributes`) и навесить на него tag `master` — это immutable zero-state marker согласно `patterns.md § Tags`.
5. Создать ветку `phase/1-foundation` от этого коммита и переключиться на неё.
6. Проверить, что `git status` clean и что в staging не оказалось ничего из игнор-листа (случайно закоммиченный `.idea/` или `build/`).

### Smoke

Агент выполняет последовательно из `C:\AI-WORK\PhoneWrap\`:

```bash
git log --oneline          # → ровно 1 строка (один коммит)
git tag                    # → содержит "master"
git rev-parse master       # → совпадает с HEAD
git branch                 # → содержит "phase/1-foundation", звёздочка рядом с ней

git status                 # → "nothing to commit, working tree clean"
```

Дополнительно — проверка, что ignore-шаблоны действительно срабатывают:
```bash

```
Должны вернуться все шесть путей с указанием строки в `.gitignore`.

## Details

**Files:**
- `C:\AI-WORK\PhoneWrap\.gitignore` — создать с нуля. Полный список паттернов из `patterns.md § .gitignore essentials` + `gallery-source/`. Группировать по секциям (IDE / Gradle / Android / Secrets / Logs) с комментариями-разделителями для читаемости.
- `C:\AI-WORK\PhoneWrap\.gitattributes` — создать с нуля. Базовое правило `* text=auto eol=lf`, override `*.bat text eol=crlf`, `*.cmd text eol=crlf`, binary-пометки для `*.jks`, `*.keystore`, `*.png`, `*.jpg`, `*.webp`, `*.pdf`, `*.tflite`, `*.task`.

**Dependencies:** никаких — это стартовая задача Wave 0.

**Edge cases:**
- Если `.git/` уже существует (случайная инициализация раньше) — НЕ делать `rm -rf .git` автоматически. Остановиться, сообщить пользователю, спросить, что делать.
- Перед `git add .` убедиться, что `gallery-source/` уже в `.gitignore` — иначе при первом `git add .` мегабайты gallery-исходников попадут в staging.
- На Windows в bash: пути к репо писать в Unix-стиле (`/c/AI-WORK/PhoneWrap` или форвард-слэши) — обратные слэши в git-командах ведут себя непредсказуемо.
- Tag `master` — именно аннотированный tag (`git tag -a master -m "zero state"`), чтобы его нельзя было спутать с веткой и чтобы он нёс сообщение.

**Implementation hints:**
- Порядок операций: `git init` → написать `.gitignore` и `.gitattributes` ДО первого `git add` → затем `git add .gitignore .gitattributes` (точечно, не `git add .`, пока не уверены, что игнор работает) → `git commit` → `git tag -a master -m "initial zero state"` → `git branch phase/1-foundation` → `git checkout phase/1-foundation`.
- Для `.gitignore` брать паттерны директорий с `/` на конце (`build/`, не `build`) — иначе исключатся и файлы с таким именем.
- `**/build/` нужно явно, иначе `build/` в корне проекта игнорится, а `:app/build/` и `:core-runtime/build/` — нет (хотя git обычно игнорирует рекурсивно; проверить на конкретной версии git, лучше добавить `**/build/` для надёжности).
- После создания веток — прогнать полный smoke-блок сверху, зафиксировать вывод каждой команды для передачи в review.
- Pre-commit hook с gitleaks (упомянутый в `patterns.md § Security & Quality Gates`) — НЕ в этой задаче. Он настраивается в Task 3 (pre-commit infrastructure). Здесь только файлы и структура.
