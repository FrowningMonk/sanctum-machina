# Task 11: Code Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-reviewing` — [skills/code-reviewing/SKILL.md](~/.claude/skills/code-reviewing/SKILL.md)

## Description

Full-feature code quality audit всей Phase 1 — единственный момент, когда один ревьюер читает весь созданный код целиком (между-задачные ревью видят только diff своей задачи). Задача — холистически оценить качество кода, единство архитектурных паттернов и соответствие user-spec / tech-spec decisions, а результат записать в JSON-отчёт.

Аудитор **не фиксит** найденные проблемы — только собирает findings. Если отчёт содержит issues, оркестратор (главный агент) создаст отдельную фикс-задачу с `reviewers: [code-reviewer]`, которая займётся устранением.

Специфика Phase 1: это валидационный спайк — минимум 1911 LOC портирования + ~1000 LOC новой обёртки. Холистика важна, потому что в Phase 2 этот код будет либо расширяться, либо переписываться; проблемы module-boundary / error-handling / anti-pattern proliferation сейчас дешёвые, а в Phase 2 — дорогие.

## Verification Steps

### Automated

Аудит-таска не пишет код и не запускает тесты. Валидация — самим JSON-отчётом: оркестратор при чтении отчёта проверяет, что схема валидна и все 7 измерений покрыты.

## Details

**Files:**
- `work/phase-1-foundation/logs/audit/code-review.json` — создать с нуля. Формат — см. Report schema ниже. Родительскую директорию (`logs/audit/`) создать если её нет.

**Review dimensions (7 измерений — обязательно покрыть каждое):**

1. **Module boundary compliance (`:core-runtime` purity).**
   - `grep -rE "androidx\\.compose\\.|androidx\\.activity\\.|androidx\\.lifecycle\\.ViewModel" core-runtime/src/main/` → ожидается **пусто**.
   - Любой hit — `critical` finding (нарушение архитектурного контракта, блокирует Phase 2).

2. **Unified error handling через `ErrorLog`.**
   - Все catch/failure-ветки в `DefaultDownloadRepository`, `DownloadWorker`, `LlmChatModelHelper`, `DefaultModelRegistry` вызывают `ErrorLog.e(component, description, cause)`.
   - `component` — из whitelist Phase 1: `download`, `inference-init`, `inference`, `inference-cleanup` (user-spec D11).
   - Альтернативные каналы (`Log.e`, `println`, `throw` без предварительного лога) — `high` finding.

3. **Log.i/w/d absence (production code).**
   - `grep -rE "android\\.util\\.Log\\.(i|w|d)|Log\\.i\\(|Log\\.w\\(|Log\\.d\\(" app/src/main/ core-runtime/src/main/` → ожидается **пусто** (patterns.md: только `Log.e` разрешён как stdout-канал, и то не должен использоваться вместо `ErrorLog`).
   - Hit в продакшн-коде — `medium` finding. Hit в тестах — OK (игнорировать).

4. **`Model.instance: Any?` anti-pattern proliferation.**
   - D6 откладывает полный рефакторинг, но НЕ разрешает протаскивать анти-паттерн в **новый** код.
   - Проверка: в `ModelRegistry`, `ModelEntry`, VM и любых новых data-классах поле `instance: Any?` / `var instance` должно **отсутствовать**. Runtime-state живёт в `LlmModelInstance` (Gallery as-is) либо в `ModelRuntimeHandle` если он введён.
   - Появление `instance: Any?` в новом коде — `high` finding.

5. **Shared resources compliance.**
   - `LlmModelInstance` single-active: `DefaultModelRegistry` использует `Mutex` (T9) для сериализации load/unload, `initialize` никогда не вызывается, пока предыдущий `cleanUp` не отработал.
   - Hilt singletons: `DownloadRepository`, `LlmModelHelper`, `ModelRegistry`, `ErrorLog` — все `@Provides @Singleton` в `CoreRuntimeModule`.
   - Отсутствие `Mutex` для single-active — `critical`. Отсутствие `@Singleton` на одном из 4 провайдеров — `high`.

6. **User-spec D1-D12 + tech-spec T1-T11 compliance.**
   - По каждому пункту решения — явный вердикт: `implemented` (с указанием file:line либо task-reference), `deviation` (со ссылкой на User-Spec Deviations секцию tech-spec), либо `finding` (отсутствует и не задокументировано).
   - Особое внимание: D6 (ModelDefinition/ModelRuntimeHandle — документирован как deferred?), D8 (единственный unit-тест — AllowlistLoader), D9 (только TTFT+total, без tok/s), D11 (whitelist компонентов ErrorLog), T3 (Hilt через KSP), T4 (plain `CoroutineWorker`), T6 (FQN MainActivity + package-prefix guard), T8 (GPU + CPU fallback hardcoded), T10 (`stopResponse`), T11 (формат футера `TTFT {N}ms · {X.X}s total`).

7. **Kotlin idioms.**
   - `val` предпочтительнее `var` где возможно (особенно в data-классах и конструкторах).
   - Expression body для коротких функций (`fun foo() = bar()`).
   - `.also` / `.apply` / `.let` использованы идиоматично, без злоупотребления (не цепочка из 5 `.let`).
   - Нет platform-type leaks: все Java-interop значения явно типизированы (`String?` или `String!` не утекает дальше).
   - Nullability: `!!` избегать, `?:` с дефолтом или ранним возвратом.
   - Нарушения — `low`/`medium` в зависимости от масштаба.

**Report schema (`logs/audit/code-review.json`):**

```json
{
  "status": "pass | issues_found",
  "findings": [
    {
      "location": "path/to/file.kt:42",
      "severity": "critical | high | medium | low",
      "category": "module-boundary | error-handling | logging | anti-pattern | shared-resources | spec-compliance | kotlin-idiom",
      "description": "Что именно не так и почему это проблема.",
      "recommendation": "Конкретный fix — что изменить и куда."
    }
  ],
  "summary": "1-3 предложения: холистическая оценка качества + сколько findings по severity (N critical, M high, K medium, L low). Для status=pass объяснить, что критичных проблем нет."
}
```

**Status resolution rule:**
- `status: pass` — 0 critical + 0 high findings. Medium/low допустимы и не блокируют Phase 1.
- `status: issues_found` — ≥1 critical или ≥1 high finding. Оркестратор на основе этого создаст fixer-таску.

**Dependencies:**
- Task 10 (Navigation + финальная wiring MainActivity) — после неё весь Phase 1 код существует в репо.
- Отчёты between-task ревью из Tasks 1-10 (`work/phase-1-foundation/logs/working/task-*/`) — опциональный контекст: если промежуточные ревьюеры уже флагали проблемы, аудит должен подтвердить, что они исправлены (либо открыть finding, если нет).

**Edge cases:**
- Некоторые файлы (Gallery-extracted, Task 3) — копии с минимальной чисткой. Для них критерий compliance мягче: оценивается, что чистка (удаление HF-token-plumbing, замена package) сделана, а не стиль оригинального Gallery-кода. Явно помечать `note: "Gallery-extracted, минимальный scope"` в finding'ах по таким файлам.
- Отсутствие файла из списка (Task не дошёл до него) — НЕ finding аудита, а сигнал оркестратору; в summary упомянуть, но не как issue.
- Файл пустой / плейсхолдер — `high` finding (нарушение DoD конкретной таски).
- Аудит выполняется после Task 10, но до Task 12 (Security Audit) и Task 13 (Test Audit) — не дублировать security-проверки (OWASP, permissions) и test-quality (assertions, coverage), это scope других двух аудит-тасок. Фокус Task 11 — именно code quality.

**Implementation hints:**
- Структурировать прохождение по 7 измерениям: сначала собрать все `grep`-based evidence (measurements 1, 3), потом read-based evidence (measurements 2, 4, 5, 6, 7). Это экономит context — grep'ы параллельны и дают быстрые факты.
- Для measurement 6 (D1-D12 + T1-T11) удобно использовать табличный формат в `summary` либо как `info`-severity entries в findings (если скилл code-reviewing их поддерживает) — иначе просто описать в summary.
- Findings писать максимально конкретно: не «где-то в registry плохой lock», а `core-runtime/.../DefaultModelRegistry.kt:57 — load() вызывается без Mutex-обёртки, параллельный load+unload может оставить LlmModelInstance в inconsistent state`.
- `recommendation` — actionable: `оборачивать load/unload в mutex.withLock { ... }` — не «пересмотреть locking strategy».
- Report file path — абсолютный относительно репозитория: `work/phase-1-foundation/logs/audit/code-review.json`. Родительскую директорию создать, если её нет.
- В summary обязательно явно указать `status` одной из двух строк и total counts по severity — это машинно-читаемый сигнал для оркестратора.
