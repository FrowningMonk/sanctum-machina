# Task 7: `:app` bootstrap — SanctumApplication, AndroidManifest, MainActivity-stub, Theme, strings.xml (весь Phase 1)

## Required Skills

## Description

Задача поднимает каркас `:app` модуля: `SanctumApplication` как Hilt entry-point, `MainActivity` c runtime-запросом `POST_NOTIFICATIONS`, голая Material 3 тема, полный `AndroidManifest.xml` со всеми permissions и security-hardening флагами, и **полный набор строк `strings.xml` для всего Phase 1** — Tasks 8 (ModelManagerScreen) и 9 (ChatScreen) будут только читать эти строки, не модифицировать.

Ключевой плумбинг — Decision T6: companion `@Volatile var mainActivityFqn: String?` объявлен в `DefaultDownloadRepository` внутри `:core-runtime` (создаётся Task 3). Task 7 его не объявляет — только **устанавливает** значение: в `SanctumApplication.onCreate()`, первой строкой после `super.onCreate()`, пишет `DefaultDownloadRepository.mainActivityFqn = "app.sanctum.machina.MainActivity"` (требует импорта `app.sanctum.machina.core.data.DefaultDownloadRepository`). Это закрывает TAC-14 со стороны `:app`.

`MainActivity.setContent` сейчас — stub (`Box` c «Sanctum loading…»). В Task 10 stub заменится на `SanctumApp()` с NavHost — эта задача только подготавливает каркас, чтобы APK собирался и запускался.

## Details

**Files:**
- `app/src/main/kotlin/app/sanctum/machina/SanctumApplication.kt` — создать новый. `@HiltAndroidApp class SanctumApplication : Application()`. В `onCreate()` первой строкой после `super.onCreate()` — присваивание `DefaultDownloadRepository.mainActivityFqn` (import `app.sanctum.machina.core.data.DefaultDownloadRepository`). Именно через companion (Decision T6), не через BuildConfig и не через Hilt-модуль — иначе сломаешь модульную границу и добавишь лишние DI-слои.
- `app/src/main/kotlin/app/sanctum/machina/MainActivity.kt` — создать новый. `@AndroidEntryPoint class MainActivity : ComponentActivity()`. В `onCreate()`: сначала регистрируем `registerForActivityResult(ActivityResultContracts.RequestPermission()) { /* no-op — granted/denied обрабатывается системно */ }`, затем проверяем `Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU` и `ContextCompat.checkSelfPermission(this, POST_NOTIFICATIONS) != PERMISSION_GRANTED` — если оба true, `launch(POST_NOTIFICATIONS)`. Далее `setContent { SanctumTheme { Box(Modifier.fillMaxSize(), contentAlignment = Alignment.Center) { Text("Sanctum loading…") } } }`. Важно: launcher регистрируется **до** setContent (требование lifecycle — до STARTED).
- `app/src/main/kotlin/app/sanctum/machina/ui/theme/Theme.kt` — создать новый. Composable `fun SanctumTheme(content: @Composable () -> Unit)` — обёртка `MaterialTheme(content = content)`. Динамические цвета опционально: если хочешь, на SDK 31+ — `dynamicLightColorScheme(LocalContext.current)`. Без кастомных палитр.
- `app/src/main/AndroidManifest.xml` — полный пакет. Добавить `xmlns:tools="http://schemas.android.com/tools"`. Все permissions списком. `uses-native-library` — внутри `<application>`. `<service>` с `tools:node="merge"` нужен чтобы смерджиться поверх WorkManager-манифеста.
- `app/src/main/res/values/strings.xml` — создать/переписать со всеми **22 каноническими ключами**, блок целиком скопировать из «What to do» шаг 6. Значения дословные, не перефразировать (batch3 reality-checker зафиксировал эти строки как контракт для Task 8/9): `app_name`="Sanctum", `model_manager_title`="Модели", `model_status_not_downloaded`="Не скачано", `model_status_downloading`="Скачивание", `model_status_downloaded`="Скачано", `model_status_failed`="Ошибка", `model_size_gb_format`="%1$.1f ГБ", `model_download_progress_format`="%1$d / %2$d МБ", `model_error_prefix`="Ошибка: %1$s", `btn_download`="Скачать", `btn_cancel`="Отмена", `btn_load`="Загрузить", `btn_retry`="Повторить", `btn_send`="Отправить", `btn_stop`="Стоп", `btn_reset`="Сбросить", `btn_back`="Назад", `chat_loading_model`="Загрузка модели…", `chat_load_failed_title`="Не удалось загрузить модель. Скачайте заново.", `chat_input_placeholder`="Сообщение", `chat_message_interrupted_suffix`=" (прервано)", `ttft_footer_format`="TTFT %1$dms · %2$.1fs total". Никакие другие ключи (`screen_model_manager_title`, `ttft_footer_template`, `progress_bytes_template`, `chat_load_failed_download_again` и т.п.) в файле появляться не должны — они сломают Task 8/9.
- `app/src/main/res/values/themes.xml` — если в `:app/build.gradle.kts` ещё нет темы, создать `<style name="Theme.Sanctum" parent="Theme.Material3.DayNight.NoActionBar"/>` (Compose app обычно только манифестный theme handoff нужен до инфляции Compose tree).

**Dependencies:**
- Task 6 (Wave 2): `DefaultDownloadRepository` с companion `mainActivityFqn` уже существует в `:core-runtime` — Task 7 только его «втыкает». Если Task 6 не выполнен — SanctumApplication не скомпилится.
- Task 2 (Wave 0): `:app/build.gradle.kts` должен иметь `applicationId = "app.sanctum.machina"`, `namespace = "app.sanctum.machina"`, `buildConfigField MAIN_ACTIVITY_CLASS_NAME`, Hilt plugin, Compose BOM.

**Edge cases:**
- SDK < 33: пропускаем runtime-запрос `POST_NOTIFICATIONS` (permission gets granted by install на API < 33).
- `DefaultDownloadRepository.mainActivityFqn` присваивается до того, как любой WorkManager worker стартует (SanctumApplication.onCreate() запускается раньше любой активности и до возобновления WM-очереди).
- Если пользователь откажет в `POST_NOTIFICATIONS` — приложение должно работать без падений (notifications просто не будут показываться). В Phase 1 явной обработки denied не нужно.
- `tools:node="merge"` в `<service>` обязателен — иначе столкновение с WorkManager-library манифестом даст manifest-merger error.
- Strings с `%1$.1f`/`%1$d`/`%2$d`/`%1$s` — экранировать `%` НЕ нужно (Android сам парсит). Для `model_size_gb_format` формат `%1$.1f` (float с 1 знаком после точки), для `model_download_progress_format` — `%1$d / %2$d` (int/int), для `ttft_footer_format` — `%1$dms · %2$.1fs total` (int мс + float сек). Знак `…` (U+2026) писать символом, не `...`, по ux-guidelines. Точка `·` (U+00B7) в `ttft_footer_format` — тоже unicode-символ, не `*` и не `.`.

**Implementation hints:**
- В `SanctumApplication` — один-единственный `override fun onCreate()`. Ничего не инициализируем вручную (Hilt сам, WorkManager сам через manifest-provider, DataStore отсутствует).
- В `MainActivity` — launcher как `private val notificationPermissionLauncher = registerForActivityResult(...)`. Это field, не локальная переменная — иначе lifecycle-требования нарушишь.
- Избегай `ActivityCompat.requestPermissions` — это legacy API. Используй ActivityResultContracts (современный путь).
- Material 3 Compose нужен — проверь что в `:app/build.gradle.kts` есть `implementation(libs.androidx.compose.material3)` (должно быть по результатам Task 2).
- Hardening-атрибуты (`allowBackup="false"`, `fullBackupContent="false"`, `usesCleartextTraffic="false"`) критичны для security-auditor — именно они закрывают TAC-12, не пропускай ни один.
- `uses-native-library` — внутри `<application>`, не снаружи. `required="false"` обязателен — иначе на устройствах без этих .so установка упадёт.
