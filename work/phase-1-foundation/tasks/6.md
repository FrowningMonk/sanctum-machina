# Task 6: DefaultModelRegistry — lifecycle координатор

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Реализовать централизованный координатор жизненного цикла моделей `ModelRegistry` (interface) + `DefaultModelRegistry` (Hilt `@Singleton`) в модуле `:core-runtime`. Это сердце фазы — именно через эту точку `ChatViewModel` и `ModelManagerViewModel` из Task 8–9 будут получать список моделей, запускать скачивания, инициализировать/освобождать native-движок LiteRT-LM и сбрасывать контекст разговора.

Ключевые инженерные гарантии, которые задача должна обеспечить:

1. **Single-active-model через `Mutex`.** `private val lifecycleMutex = Mutex()` владеется регистри; все операции жизненного цикла (`initialize`, `cleanup`, `resetConversation`, `delete`) идут через `lifecycleMutex.withLock { ... }`. Это заменяет антипаттерн Gallery из пары флагов `Model.initializing` + `cleanUpAfterInit` (см. `Model.kt:267,269`) на корректную синхронизацию — user-spec Risks R3 мандатит эту замену (Decision T9).
2. **Stale-instance guard при cleanup.** Перед вызовом `LlmChatModelHelper.cleanUp(model, onDone = { })` сравнить `currentInstance === model.instance` (by reference). Если instance был подменён в промежутке — abort. Это защита от двойного close native engine → SIGSEGV (research §4, §12.3).
3. **Безусловный GPU→CPU fallback в `initialize()`** без парсинга текста ошибки (Decision T8, research §14). При первой ошибке GPU-инициализации — `errorLog.e("inference-init", ...)` + `cleanUp(model, onDone = { })` (гард от native-leak) + патч ключа `ConfigKeys.ACCELERATOR.label` в `configValues` на значение `Accelerator.CPU.label` + повторная попытка. При ошибке CPU — `Result.failure`.
4. **Старт: scan + resume.** В `init { }` корутина выполняет `refreshAllowlist()` → `scanLocalFiles()` (`File(Model.getPath(ctx)).exists()` для каждой модели, обновить status в `StateFlow`) → `resumePartialDownloads()` (re-enqueue всех найденных `*.gallerytmp` через `DownloadRepository.downloadModel(model)`).
5. **Data-типы.** Сопутствующие `ModelEntry` (data class: `Model`, `downloadStatus`, `initStatus`) и `ModelInitStatus` (sealed: `Idle`, `Initializing`, `Ready`, `Failed(msg)`) — это Phase 1 view-тип над Gallery `Model` (не рефакторим `Model.instance: Any?` — D6 отложен в Phase 2, см. tech-spec User-Spec Deviations). Runtime-мутации `model.instance = null` и `model.configValues = ...` допустимы — оба поля в Gallery `Model.kt` объявлены как `var` и мутируются под `lifecycleMutex`, это state-мутация, не структурный рефакторинг.
6. **DI-модуль.** Завершить `CoreRuntimeModule` — provides всех компонентов рантайма: `DownloadRepository` (как `DefaultDownloadRepository` из Task 3), `LlmModelHelper` (как Kotlin-`object` `LlmChatModelHelper`), `ModelRegistry` (как `DefaultModelRegistry`), плюс уже существующие `AllowlistLoader` (Task 4) и `ErrorLog` (Task 5).

### Канонический интерфейс `ModelRegistry`

Это единственное место, где перечислены методы интерфейса. В разделах What to do / Acceptance Criteria / Details не дублируем сигнатуры — только ссылаемся на этот список.

| Метод | Контракт |
|---|---|
| `val models: StateFlow<List<ModelEntry>>` | Hot stream текущего состояния всех моделей. Обновляется из `refreshAllowlist`, `scanLocalFiles`, download-ом, `initialize`, `cleanup`, `delete`. |
| `suspend fun refreshAllowlist(): Result<Unit>` | Через `AllowlistLoader` (Task 4): fetch JSON → parse → update `_models.value`. Ошибки сетевого/JSON → `errorLog.e("download", ...)` (тэг `"download"` — парсинг allowlist — часть download-discovery пути) + `Result.failure`. |
| `fun download(model: Model): Flow<ModelDownloadStatus>` | Мост callback-API → Flow через `callbackFlow`: `DownloadRepository.downloadModel(model, onStatusUpdated)` (Task 3) — callback-API, не Flow-returning. Внутри `callbackFlow { downloadRepository.downloadModel(model) { _, status -> trySend(status) }; awaitClose { /* cancellation через cancelDownload — WorkManager external */ } }`. На каждое событие — обновить `downloadStatus` соответствующего `ModelEntry` в `_models`. |
| `fun cancelDownload(modelName: String)` | Находит `Model` по имени в `_models.value` и делегирует в `DownloadRepository.cancelDownloadModel(model)` (реальная Gallery-сигнатура — не `cancel(String)`). |
| `suspend fun delete(modelName: String)` | `lifecycleMutex.withLock { if (initStatus != Idle) cleanup(modelName); DownloadRepository.deleteModel(model); update StateFlow }`. |
| `suspend fun initialize(modelName: String): Result<Unit>` | GPU→CPU fallback (см. Decision T8 ниже). Под `lifecycleMutex.withLock`. При успехе — обновить `initStatus = Ready` в `_models`. |
| `suspend fun cleanup(modelName: String)` | Stale-instance guard (см. Decision T9 ниже). Под `lifecycleMutex.withLock`. После — `initStatus = Idle`, `model.instance = null`. |
| `suspend fun resetConversation(modelName: String, systemPrompt: String? = null)` | Под `lifecycleMutex.withLock`. Конвертирует `systemPrompt: String?` во внутренний тип `Contents?` LiteRT-LM (см. ниже) и вызывает `LlmChatModelHelper.resetConversation(model, systemInstruction = contents)`. Engine НЕ пересоздаётся — только `conversation` мутируется (см. `LlmChatModelHelper.kt:152-207`). |
| `fun getModel(modelName: String): Model?` | Read-only accessor возвращает Gallery `Model` (с внутренним полем `model.instance: LlmModelInstance?`). Возвращает `Model` только если `initStatus == Ready` (иначе `null`). Используется `ChatViewModel` (Task 9): `LlmChatModelHelper.runInference(model, ...)` принимает именно `Model`, не `LlmModelInstance`. Если modelName неизвестен или модель не инициализирована — `null`. |

**Важно — реальные сигнатуры Gallery API (верифицировано):**

- `LlmChatModelHelper.cleanUp(model: Model, onDone: () -> Unit)` — `onDone` обязателен, без default. Все вызовы в регистри передают лямбду (как правило `{ }` — мы не ждём завершения, либо wrap через `suspendCoroutine` если нужно).
- `LlmChatModelHelper.resetConversation(model: Model, supportImage: Boolean = false, supportAudio: Boolean = false, systemInstruction: Contents? = null, tools: List<ToolProvider> = listOf(), enableConversationConstrainedDecoding: Boolean = false)` — `systemInstruction` имеет тип `Contents?` (LiteRT-LM), НЕ `String?`. Наш `ModelRegistry.resetConversation` принимает `systemPrompt: String?` для удобства вызывающих и конвертирует внутри: `systemPrompt?.let { Contents.of(listOf(Content.Text(it))) }`. На Phase 1 реально система-промпт из UI не задаётся (Phase 1 минимальный чат), так что типичный вызов — с `systemPrompt = null`.
- `DownloadRepository.downloadModel(model: Model, onStatusUpdated: (Model, ModelDownloadStatus) -> Unit)` — callback-based API (не Flow-returning, не `enqueue`). Task 3 удаляет параметр `task: Task?` из Gallery-сигнатуры (в Phase 1 нет multi-task dispatch — только `llm_chat`). Чтобы отдавать Flow наружу из регистри, оборачиваем через `callbackFlow` — см. каноническую таблицу, строка `download(model)`.
- `DownloadRepository.cancelDownloadModel(model: Model)` — для отмены (не `cancel(String)`), принимает `Model`, не имя.

После этой задачи слой `:core-runtime` функционально закрыт: из него можно управлять download-ом, инициализацией и жизненным циклом моделей. Следующая волна задач (UI + Application class) подцепит регистри через Hilt и потребит его `StateFlow`.

### Decision T8: GPU→CPU fallback в `initialize()`

Алгоритм (строго, без парсинга текста ошибки):

1. `lifecycleMutex.withLock { ... }`.
2. Найти `Model` по `modelName` в `_models.value`. Если нет — `Result.failure(IllegalArgumentException(...))`.
3. Обновить `initStatus = Initializing` в `_models`.
4. **Попытка 1 (GPU).** В `model.configValues` ключ `ConfigKeys.ACCELERATOR.label` (= строка `"Accelerator"` — это KEY-имя ConfigKey) должен иметь значение `Accelerator.GPU.label` (= строка `"GPU"`) после `Model.preProcess()` (из-за `defaultConfig.accelerators = "gpu,cpu"`, где первый — GPU). Не путать: `ConfigKeys.ACCELERATOR.label` — имя ключа в map, `Accelerator.CPU.label` / `Accelerator.GPU.label` — значения в этом map. Оба поля называются `.label`, но относятся к разным объектам. Вызвать `LlmChatModelHelper.initialize(context, model, onDone = { err -> continuation.resume(err) })` через `suspendCancellableCoroutine`. Успех: `err == ""` И `model.instance != null`.
5. **Fallback на CPU:**
   - `errorLog.e("inference-init", "GPU init failed: $err1", cause = null)`.
   - `LlmChatModelHelper.cleanUp(model, onDone = { })` — защита от native-leak (Engine может быть частично создан). `onDone`-лямбда пустая: нам не нужно ждать завершения, это fire-and-forget очистка перед retry.
   - Мутировать карту `configValues`: `model.configValues = model.configValues + (ConfigKeys.ACCELERATOR.label to Accelerator.CPU.label)` — ключ `"Accelerator"`, значение меняется с `"GPU"` на `"CPU"`.
   - Повторить `initialize` через `suspendCancellableCoroutine`.
6. Успех CPU: `initStatus = Ready`, `Result.success(Unit)`. Провал CPU: `errorLog.e("inference-init", "CPU init failed: $err2")`, `initStatus = Failed(err2)`, `Result.failure(RuntimeException("GPU+CPU init failed: $err2"))`.

Важно: НЕТ парсинга `err1` на предмет «это GPU-specific ошибка или нет» — fallback срабатывает на любую непустую ошибку первой попытки (research §14).

### Decision T9: Stale-instance guard в `cleanup()`

Алгоритм:

1. `lifecycleMutex.withLock { ... }`.
2. Найти `Model` по `modelName`. `currentInstance = model.instance as? LlmModelInstance`.
3. Если `currentInstance == null` — no-op, `initStatus = Idle`, return.
4. Перед вызовом `LlmChatModelHelper.cleanUp(model, onDone = { })` — проверить `currentInstance === model.instance` (by reference). Если `false` (instance подменили через switch между моделями) — abort cleanup (другой вызов уже отработал или переподнял движок).
5. Если `initStatus == Initializing` или идёт активный inference — сначала `LlmChatModelHelper.stopResponse(model)`, затем `cleanUp(model, onDone = { })`.
6. `model.instance = null`, `initStatus = Idle`, обновить `_models`.

**Edge cases:**

- **Race: `initialize(A)` и одновременный `initialize(B)`.** Mutex последовательно выполнит обе. После первой — A в `Ready`; при второй мы МОЖЕМ параллельно держать два engine. Phase 1: интерфейс допускает only-one-active, так что B должен сперва `cleanup(A)`. Документировать это в KDoc `initialize` («caller обязан cleanup текущей активной модели перед initialize новой»). Tech-spec §3 уже это описывает.
- **Race: `initialize` в прогрессе, параллельно `cleanup` того же `modelName`.** Mutex сериализует. Cleanup выполнится после `initialize`. Stale-instance guard не нужен (instance будет тот же), но `stopResponse` вызвать можно (safe no-op если не-inference).
- **Двойной `cleanup`.** Первый отработает (instance → null), второй увидит `currentInstance == null` → no-op. Без падений.
- **Cleanup во время активного inference (`runInference` в другой корутине).** Перед `LlmChatModelHelper.cleanUp(model, onDone = { })` — обязательный `LlmChatModelHelper.stopResponse(model)`. Без этого native `Conversation.close()` на лету → риск SIGSEGV (research §4, строка 286).
- **Instance подменили между capture и cleanUp.** Stale-instance guard: `currentInstance === model.instance` — если нет, abort. Реальный сценарий: cleanup вызвали ДВАЖДЫ для той же модели, между первым cleanup и вторым cleanup кто-то уже re-initialize → `model.instance` указывает на новый engine → guard aborts, чтобы не закрыть чужой engine.
- **GPU-init частично создал Engine, упал на createConversation.** `model.instance` может быть уже `!= null` частично. Поэтому перед CPU retry — `LlmChatModelHelper.cleanUp(model, onDone = { })` безусловно (research §14).
- **`scanLocalFiles` падает на одной модели (например, битый path).** Не валить весь старт — `runCatching` per-model + `errorLog.e("download", "scan $modelName failed", it)`, продолжаем остальные.
- **`refreshAllowlist` упал (оффлайн на старте).** Регистри остаётся с пустым `_models.value = emptyList()` — UI покажет empty-state. Не блокировать старт приложения.

**Implementation hints (НЕ псевдокод):**

- `SupervisorJob` (не `Job`) в scope — чтобы падение одной корутины (например, download listener) не убивало весь scope регистри.
- `LlmChatModelHelper` — Kotlin `object`, не class. Интерфейс `LlmModelHelper` существует в `:core-runtime` (extraction из Gallery); object реализует этот интерфейс. В DI провайдим именно object'ом: `@Provides fun provideLlmModelHelper(): LlmModelHelper = LlmChatModelHelper`.
- `LlmChatModelHelper.cleanUp(model, onDone = { })` — `onDone` обязателен (нет default). Лямбда `{ }` используется, когда fire-and-forget достаточно (наш стандартный случай, поскольку мы уже под `lifecycleMutex`). Если нужно дождаться завершения cleanup до релиза Mutex — wrap в `suspendCancellableCoroutine { cont -> llmModelHelper.cleanUp(model) { cont.resume(Unit) } }`.
- `StateFlow.update { list -> ... }` (из kotlinx-coroutines-core) — атомарное обновление без race. Для замены одного элемента: `update { list -> list.map { if (it.model.name == name) it.copy(...) else it } }`.
- `suspendCancellableCoroutine` (не `suspendCoroutine`) — поддерживает отмену корутины. На cancel-hook вызвать `LlmChatModelHelper.cleanUp(model, onDone = { })` best-effort.
- `Model.configValues` в Gallery — `MutableMap<String, Any>` или `Map<String, Any>` в зависимости от копии. Мутация под Mutex: `model.configValues = model.configValues + (ConfigKeys.ACCELERATOR.label to Accelerator.CPU.label)`. Ключ = строка `"Accelerator"` (имя KEY). Значение = строка `"CPU"` (human-label enum-варианта). Не путать два разных `.label`-поля: одно принадлежит ConfigKey (имя ключа), другое — enum `Accelerator` (значение).
- Поле `model.instance: Any?` в Gallery `Model.kt` — `var`, runtime-мутация допустима без структурного рефакторинга (D6 deferred в Phase 2).
- Конвертация `systemPrompt: String? → Contents?`: если `null` → передаём `null` (не вызываем конвертер). Если не-null → `Contents.of(listOf(Content.Text(systemPrompt)))` (Contents — data-класс из LiteRT-LM, `of` — factory). Типы импортируются из пакета, в котором они лежат после extraction в Task 3.
- **`download()` — callback → Flow мост через `callbackFlow`.** Gallery `DownloadRepository.downloadModel(model, onStatusUpdated: (Model, ModelDownloadStatus) -> Unit)` — callback-based (Task 3 удаляет параметр `task: Task?` — в Phase 1 только `llm_chat`). Регистри возвращает `Flow<ModelDownloadStatus>`, поэтому оборачиваем:
  ```kotlin
  override fun download(model: Model): Flow<ModelDownloadStatus> = callbackFlow {
      downloadRepository.downloadModel(model) { _, status -> trySend(status) }
      awaitClose { /* WorkManager cancellation идёт через cancelDownload — здесь explicit cleanup не нужен */ }
  }
  ```
  Параллельно с `trySend` нужно обновлять `_models` (через `onEach { update { ... } }` в вызывающей стороне, либо внутри callback перед `trySend`).
- **`getModel(modelName)` — возвращает `Model?`, не `LlmModelInstance?`.** Gallery `LlmChatModelHelper.runInference(model, ...)` принимает именно `Model` (instance внутри, `model.instance: Any?`). Реализация: найти `ModelEntry` по имени в `_models.value`; если `entry.initStatus == Ready` → `entry.model`, иначе `null`. Так Task 9 получает готовый к inference объект одним вызовом.
- KDoc на classes + публичные методы — обязательно (patterns.md §Code Style требует KDoc для публичного API).
- Component-теги для ErrorLog — строго из D11 whitelist (user-spec): `"download"`, `"inference-init"`, `"inference"`, `"inference-cleanup"`. Использование любых других тэгов (включая `"inference-run"` и `"allowlist"`, которых в whitelist нет) — ошибка. QA будет grep'ать. Ошибка при парсинге allowlist логируется с тэгом `"download"` — это часть download-discovery пути (сначала узнаём какие модели доступны, потом качаем).
