# User Spec: phase-1-foundation

## Что делаем

Собираем фундамент проекта Sanctum Machina: создаём пустой gradle-проект с двумя модулями (`:app` и `:core-runtime`), копируем из Google AI Edge Gallery (`gallery-source/`) ядро инференса и скачивания моделей в `:core-runtime`, пишем поверх него минимальный текстовый .apk с одним чатом «вопрос-ответ». Никакой истории, проектов, мультимодальности, настроек, тем, дизайна. Задача — доказать, что после извлечения ядра Gemma-4-E2B-it на Honor 200 работает так же быстро, как в оригинальном Gallery.

## Зачем

В `work/research/gallery-analysis.md` принято решение идти **вариантом B** — тащить из Gallery только ядро, писать свою оболочку. Фазы 2-5 рассчитаны на 2-3 месяца работы по UI, persistence, projects, RAG, полировке. Всё это имеет смысл только если ядро после извлечения работает **не хуже**, чем в Gallery.

Если на Honor 200 извлечённая Gemma-4-E2B-it отвечает так же быстро, как в Gallery — вариант B подтверждён и Phase 2 стартует. Если медленнее — переоцениваем стратегию до того, как вложились в UI и базы данных. Это чисто валидационная веха, не продуктовая. Архитектуру Phase 1 (имена сервисов, раскладка Hilt-модулей) мы оставляем себе право переделать в Phase 2 — цель Phase 1 не «правильная архитектура», а «факт работоспособности».

Пользователь Phase 1 — я сам (solo dev) на своём Honor 200. Внешних пользователей нет.

## Как должно работать

### Основной сценарий (happy path)

1. Я собираю debug-APK командой `./gradlew :app:assembleDebug` и переношу его на Honor 200. Ставлю через Android Studio (Run) или `adb install -r`.
2. Первый запуск — система показывает диалог разрешения `POST_NOTIFICATIONS` (нужно для foreground-нотификации при скачивании). Разрешаю.
3. Открывается **Экран моделей** (`ModelManagerScreen`) — список из двух моделей из встроенного allowlist'а: `Gemma-4-E2B-it` (~2.4 ГБ) и `Gemma-4-E4B-it` (~4.3 ГБ). Обе в статусе «Не скачано» (перед тестом я удалил копию из Gallery).
4. Тапаю «Скачать» на Gemma-4-E2B. Появляется системная нотификация с прогрессом, карточка модели показывает прогресс-бар, полученные мегабайты и скорость. На Wi-Fi — примерно 10-20 минут.
5. По завершении статус меняется на «Скачано», кнопка — на «Загрузить».
6. Тапаю «Загрузить». Показывается индикатор «Загрузка модели…» 5-30 секунд, пока LiteRT-LM инициализирует движок на backend=GPU. Затем открывается **Экран чата** (`ChatScreen`).
7. В чате: верхняя плашка с именем модели и иконкой «Сбросить», пустая лента, снизу поле ввода и кнопка Send. Пишу промпт, отправляю. Ответ стримится токен за токеном. Под каждым завершённым ответом модели — мелким шрифтом: `TTFT YYYms · NNs total`.
8. Отправляю второй промпт — модель помнит контекст из первого (multi-turn память работает).
9. Тапаю «Сбросить» в TopAppBar. Лента в UI очищается, на движке вызывается `resetConversation` (не полная переинициализация). Следующий ход стартует с нуля без задержки на загрузку Engine.
10. Жму back — возвращаюсь в список моделей, Engine очищается, native-память освобождается.

### Сценарий валидации (почему всё это нужно)

11. На том же телефоне у меня рядом стоит оригинальный Google AI Edge Gallery с той же моделью Gemma-4-E2B-it. Модели лежат в разных директориях и не шарятся между приложениями — это нормально и даже требуется.
12. Запускаю один и тот же промпт (на моё усмотрение) в Phase 1 APK и в Gallery, смотрю глазами на скорость стрима и сравниваю TTFT из футера Phase 1 с субъективным ощущением от Gallery. Если примерно одинаково — Phase 1 пройдена.

### Сценарии, которые должны работать, но не являются «основным путём»

- **Отмена скачивания.** Если во время загрузки жму «Отмена» — прогресс замирает, карточка возвращается в статус «Не скачано», файл `*.gallerytmp` остаётся на диске. Повторное нажатие «Скачать» продолжает с того же места через HTTP Range (этот механизм унаследован из Gallery as-is).
- **Kill приложения во время скачивания.** Если убиваю приложение свайпом во время скачивания — тот же результат: `*.gallerytmp` сохраняется, при следующем запуске повторный тап «Скачать» возобновляет загрузку.
- **Прерывание ответа.** Во время стрима тапаю Stop — стрим останавливается в пределах 500 мс, частичный ответ получает суффикс «(прервано)» приглушённым серым. Следующий промпт работает штатно.
- **Переключение модели.** Вернувшись в список моделей, тапаю «Загрузить» на другой модели (E4B). Предыдущий Engine выгружается, новый загружается. Обе модели работают в одной сессии приложения.
- **Оффлайн после скачивания.** Выключаю Wi-Fi и мобильный интернет, возвращаюсь в чат с уже скачанной моделью — инференс работает. Это подтверждает, что всё локально, без облачных зависимостей (манифест).
- **Повреждённый файл модели.** Если я вручную (через `adb push`) перезапишу `.litertlm` мусором — «Загрузить» покажет читаемое сообщение «Не удалось загрузить модель. Скачайте заново.» + кнопку возврата в список моделей. Не native crash.

## Критерии приёмки

### Обязательные (блокирующие)

- [ ] **AC-1.** Debug-APK собирается (`./gradlew :app:assembleDebug` → BUILD SUCCESSFUL), устанавливается на Honor 200, открывает MainActivity без краша.
- [ ] **AC-2.** При первом запуске на Android 13+ запрашивается разрешение `POST_NOTIFICATIONS`. После grant'а приложение продолжает работу без ошибок.
- [ ] **AC-3.** Экран моделей показывает ровно две модели из встроенного allowlist'а: `Gemma-4-E2B-it` (~2.4 ГБ) и `Gemma-4-E4B-it` (~4.3 ГБ). Каждая карточка содержит имя модели, размер в ГБ и текущий статус.
- [ ] **AC-4.** Скачивание работает end-to-end: тап «Скачать» → появляется системная foreground-нотификация → прогресс обновляется в карточке → по завершении статус меняется на «Скачано», файл лежит по пути `/sdcard/Android/data/app.sanctum.machina/files/{normalizedModelName}/{commitHash}/{fileName}.litertlm`, размер совпадает с `sizeInBytes` из allowlist.
- [ ] **AC-5.** Скачивание поддерживает возобновление: если убить приложение свайпом на 30% скачивания, при следующем запуске повторный тап «Скачать» продолжает с того же места (через `.gallerytmp` + HTTP Range header). Механизм унаследован из Gallery и должен продолжать работать после переноса.
- [ ] **AC-6.** Скачивание поддерживает отмену: во время `IN_PROGRESS` тап «Отмена» возвращает карточку в статус «Не скачано», файл `*.gallerytmp` сохраняется, следующий «Скачать» продолжает, а не начинает заново.
- [ ] **AC-7.** Загрузка модели (init Engine) работает: тап «Загрузить» → индикатор «Загрузка модели…» → в пределах 5-30 секунд открывается экран чата без крашей, в logcat нет native-исключений. Backend — GPU (fallback на CPU только если GPU init падает).
- [ ] **AC-8.** Одиночный ход чата: отправляю «Hello», получаю непустой стримящийся ответ, стрим завершается. Под ответом — футер `TTFT YYYms · NNs total`. **Только эти две метрики — без tok/s** (без настоящего tokenizer'а цифра tok/s была бы недостоверной).
- [ ] **AC-9.** Multi-turn чат: три последовательных промпт-ответ пары в одном чате. В третьем ответе модель демонстрирует память предыдущего контекста. Нет OOM, нет крашей. PSS приложения не превышает 4 ГБ для Gemma-4-E2B.
- [ ] **AC-10.** Сброс конверсации: тап «Сбросить» в TopAppBar очищает UI-историю и вызывает `resetConversation` на LiteRT-LM (**не** полную переинициализацию). Следующий ход стартует без памяти прошлых, но без повторной задержки на загрузку движка.
- [ ] **AC-11.** Прерывание стрима: во время длинного ответа тап Stop останавливает стрим за ~500 мс. Частичный текст помечен «(прервано)». Следующий ход работает штатно.
- [ ] **AC-12.** Переключение между моделями: вернулся в список, тапнул «Загрузить» на другой модели. Предыдущий Engine выгрузился (`cleanUp`), новый загрузился. Нет OOM. Обе модели можно использовать в одной сессии.
- [ ] **AC-13.** Оффлайн-инференс: после успешного скачивания, при выключенных Wi-Fi и мобильном интернете, я могу открыть чат и вести несколько ходов. Это подтверждает ноль сетевой зависимости во время инференса (манифест).
- [ ] **AC-14.** **Паритет скорости с Gallery (главный гейт Phase 1):** субъективно стрим Gemma-4-E2B-it в Phase 1 APK на Honor 200 при back-to-back прогоне с Gallery на промпте моего выбора — «примерно такой же». **Объективный backstop:** TTFT из футера Phase 1 не должен превышать 1.5× TTFT Gallery на том же промпте и том же устройстве. TTFT Gallery замеряется секундомером (от тапа Send до появления первого символа ответа) — одного прогона достаточно. Если TTFT Phase 1 > 1.5× TTFT Gallery — критерий провален **независимо от субъективного ощущения**. Этот численный порог нужен, чтобы исключить мотивированное додумывание «ну вроде одинаково» после недель работы над извлечением. Если AC-14 провален — останавливаемся, ищем, где потеряли скорость (backend-настройки, sampler config, per-model defaults из allowlist).

### Желательные (не блокируют выпуск Phase 1)

- [ ] **AC-15.** Обработка повреждённого файла: если я вручную перезапишу `.litertlm` мусором через `adb push` и потом тапну «Загрузить» — в чат-экране появится читаемое сообщение «Не удалось загрузить модель. Скачайте заново.» и кнопка возврата в список моделей. Не native SIGSEGV.
- [ ] **AC-16.** On-device лог ошибок по пути `context.filesDir/logs/errors.log` (internal storage, **не** externalFilesDir) содержит ERROR-записи минимум для трёх компонентов: `download`, `inference-init`, `inference`. Формат по `patterns.md`: `ERROR [component] short description :: cause`. Ротация при достижении 2 МБ (старый файл → `errors.log.1`). Экспорт лога в Phase 1 — ручной, через debug-ориентированный `adb shell run-as app.sanctum.machina cat files/logs/errors.log` (internal storage требует `run-as` на debug-сборке). In-app share-sheet не делаем (это Phase 2+).

## Ограничения

### В скоупе Phase 1

- Только текстовый ввод/вывод. Никаких картинок, аудио, камеры, микрофона.
- Ровно одна платформа — Android, целевое устройство — Honor 200 (Android 14+, 12 ГБ RAM, Mediatek Dimensity без LiteRT NPU-backend). Эмуляторы не цель.
- Две модели в allowlist: `litert-community/gemma-4-E2B-it-litert-lm` и `litert-community/gemma-4-E4B-it-litert-lm`. Обе скачиваются анонимно с HuggingFace (community-репо, OAuth не требуется).
- Allowlist — встроенный asset (форк `1_0_11.json` из Gallery, обрезанный до двух моделей). Сетевой подгрузки allowlist'а нет.
- Inference-настройки зашиты из allowlist defaults: `topK=64, topP=0.95, temperature=1.0, maxTokens=4000, accelerator=GPU`. UI для per-model override — отсутствует.
- Пакет: `app.sanctum.machina` (приложение), `app.sanctum.machina.core` (ядро).
- Хранение моделей: `context.getExternalFilesDir(null)/{normalizedModelName}/{commitHash}/{fileName}.litertlm`.
- Хранение логов: `context.filesDir/logs/errors.log` (+ ротация `errors.log.1` при > 2 МБ).
- Пакеты минимум для `:core-runtime`: `androidx-core-ktx`, `litertlm-android:0.10.0`, `work-runtime-ktx:2.10.0`, `gson`, `hilt-android`.
- Пакеты минимум для `:app`: всё из `:core-runtime` плюс `androidx-activity-compose`, `compose-bom:2026.03.00` (material3, ui), `navigation-compose`, `hilt-navigation-compose`.

### Вне скоупа Phase 1 (откладывается в последующие фазы)

- **Хранение истории чатов** — нет Room, нет БД, нет таблицы сообщений. Всё в памяти.
- **Проекты** — нет понятия Project, нет system prompt'ов на группу чатов.
- **Мультимодальность** — нет CameraX, нет MediaRecorder, нет `Content.ImageBytes` / `AudioBytes`.
- **Полированный UI** — чистый Material 3 без темизации, без accent-цветов, без анимаций, без иконок (кроме системных Material Symbols для send/stop/reset), без онбординга, без About-экрана, без манифеста.
- **Экран настроек** — нет переключателя темы, нет переопределения inference-параметров, нет экспорта лога через share sheet, нет кнопки HuggingFace login.
- **HuggingFace OAuth** — `net.openid:appauth` в зависимости не добавляется. Работаем только с community-репо.
- **AICore / `mlkit-genai-prompt`** — закрытый системный сервис Google, противоречит манифесту, не портируется.
- **Firebase / `google-services` / FCM** — по манифесту, никогда.
- **Рефакторинг god-object `ModelManagerViewModel`** (1411 строк) — отложен на Phase 2. В Phase 1 пишется тонкий `ModelRegistry` (~100 строк), покрывающий только list/download/load/unload.
- **Customtasks Gallery** — Agent Skills, Mobile Actions, Tiny Garden, Prompt Lab, Benchmark — не портируются.
- **Автоматические тесты** (unit, integration, instrumentation, Compose UI) — **в Phase 1 почти не пишутся**, кроме одного узкого исключения (см. D8 и секцию «Тестирование» ниже): один unit-тест на парсер встроенного allowlist-JSON. Всё остальное покрытие — исключение из стандартной практики `patterns.md`: Phase 1 — валидационный спайк, его архитектура может быть переделана в Phase 2, покрытие тестами текущей архитектуры было бы преждевременным. Инфраструктура JUnit 5 + MockK появляется в Phase 2.

### Прочие ограничения

- APK — debug-сборка, не signed release. Signing keystore появляется только в Phase 5 (`deployment.md`).
- CI/CD нет. Сборка APK — ручная с dev-машины, `./gradlew :app:assembleDebug`, передача файла на телефон — ручная.
- Безопасность: никаких секретов в коде, `.env`, `*.key`, `*.keystore` в `.gitignore`. HF-токена нет (работаем анонимно), поэтому хранить нечего. PII не собирается (манифест).

## Риски

- **R1 — Хардкод `Class.forName("com.google.ai.edge.gallery.MainActivity")` в `DownloadWorker.kt:342`.** Если его не пропатчить при переносе, первый тап на notification во время скачивания уронит приложение. **Митигация:** Место патча зафиксировано в `work/phase-1-foundation/code-research.md § 1`; в extraction-процессе прокидываем Activity class через `BuildConfig` константу либо инъекцию из `:app`.
- **R2 — Honor 200 не поддерживает NPU через LiteRT-LM.** Попытка `accelerator=NPU` уронит Engine.initialize. **Митигация:** UI вообще не предлагает выбор акселератора; в Phase 1 backend захардкожен `GPU` с fallback на `CPU`, если init GPU падает.
- **R3 — Cleanup Engine во время активного inference может вызвать native SIGSEGV.** В Gallery нет встроенной защиты. **Митигация:** `ModelRegistry` обязан вызывать `stopResponse()` перед `cleanUp()`, когда пользователь навигирует из чата с активным стримом. Точная реализация — решение tech-spec (Mutex/AtomicBoolean).
- **R4 — Баг `renameTo(.gallerytmp → .litertlm)` в `DownloadWorker.kt:258` без проверки return value.** При редких ошибках файловой системы файл останется под именем `.gallerytmp`, а UI покажет SUCCEEDED. **Митигация:** Осознанно **не чиним** в Phase 1 (вне скоупа Foundation). Если сработает — init упадёт с читаемой ошибкой, пользователь перекачает. Фикс — в Phase 2.
- **R5 — Отсутствие sha256-верификации скачанного файла.** Gallery не проверяет целостность. Битый файл проявится только на `Engine.initialize`. **Митигация:** ошибка пробрасывается через `cleanUpMediapipeTaskErrorMessage` как читаемое сообщение (AC-15), пользователь перекачивает.
- **R6 — Метрика tok/s может быть добавлена позже и оказаться недостоверной.** Без настоящего tokenizer'а любое значение будет эвристикой (`chars/4` или `words/sec`). Если её ввести и показать, а потом пользователь перестанет ей доверять — потеря доверия к метрикам в целом. **Митигация:** Phase 1 показывает только TTFT и total duration (оба измеряются точно через `System.currentTimeMillis`). `tok/s` добавим только если LiteRT-LM 0.10.0 выдаст настоящий счётчик через `Message.channels["stats"]` или подобное (проверить в Phase 2).
- **R7 — Зависший инференс (stream stall) без onError и без onMessage.** В LiteRT-LM нет watchdog'а. Пользователь видит «генерация…» бесконечно. **Митигация:** Осознанно принимаем в Phase 1. Обходные пути для пользователя — кнопка Stop, при неэффективности — swipe приложения из recents. Watchdog-таймер — решение для Phase 2+, если проблема реально повторяется на Honor 200.
- **R8 — Архитектура `:core-runtime` Phase 1 может потребовать переделки в Phase 2.** Имена сервисов (`ModelRegistry`, `DownloadService`, `ModelInitService`), раскладка Hilt-модулей, подписи функций — всё может измениться. **Митигация:** Явно принято — Phase 1 это спайк, не финальная архитектура. Phase 2 может рефакторить свободно.
- **R9 — Throwaway UI.** Compose-код Phase 1 (`ModelManagerScreen` и `ChatScreen`, ~250 строк суммарно) будет удалён в Phase 2 и переписан под дизайн `ux-guidelines.md`. **Митигация:** код намеренно минималистичный — удаление будет дешёвое, не жалко.
- **R10 — Схема allowlist Gallery может измениться в upstream.** `1_0_11.json` — снапшот. **Митигация:** Форкаем как bundled asset, никаких runtime-фетчей. Мы не зависим от upstream-изменений.

## Технические решения

- **D1.** Идём вариантом B из `work/research/gallery-analysis.md` — тащим из Gallery только ядро, оболочку пишем сами. Причина: долгоиграющий инструмент, структурно несовместим с Gallery (проекты → чаты вместо задачи → модели), нужна чистая архитектура.
- **D2.** Из `gallery-source/` в `:core-runtime` копируются 13 Kotlin-файлов + 1 JSON-asset (точный список — `work/phase-1-foundation/code-research.md § 2`). Всего ~1900-2100 строк после чистки.
- **D3.** Патчим три вещи в копируемом коде: (a) удаляем ~11 строк плumbing'а HF-токена по 4 файлам (`DownloadWorker.kt:32,105,134-137`, `DownloadRepository.kt:123-125`, `Consts.kt:33`, `Model.kt:273`); (b) удаляем вызовы `firebaseAnalytics` из `DownloadRepository`; (c) заменяем хардкод `Class.forName("com.google.ai.edge.gallery.MainActivity")` в `DownloadWorker.kt:342` на `BuildConfig`-константу или инъекцию класса. Причина: работаем только с community-репо (OAuth не нужен), Firebase противоречит манифесту, пакет у нас другой.
- **D4.** Allowlist bundled как asset `:core-runtime/src/main/assets/model_allowlist.json`, содержит только Gemma-4-E2B-it и Gemma-4-E4B-it. Причина: сетевой подгрузки не нужно для Phase 1, runtime-фетч добавляет сложность без пользы.
- **D5.** Никакого Room, Proto DataStore, AppAuth в Phase 1. State живёт в `StateFlow` (memory) + файлы на диске (модели в `externalFilesDir`). Причина: persistence добавляется в Phase 3, настройки — в Phase 2.
- **D6.** Рефакторим анти-паттерн `Model.instance: Any?` в `ModelDefinition` (immutable) + `ModelRuntimeHandle` (mutable) — это единственный non-copy-paste рефакторинг в Phase 1. Причина: протащить анти-паттерн дальше Phase 1 заблокирует архитектуру Phase 2 (`patterns.md`).
- **D7.** God-object `ModelManagerViewModel` (1411 строк) **не** портируется. Его заменяет тонкий `ModelRegistry` в `:core-runtime` (~100 строк), экспонирующий list/download/load/unload. Причина: 1411 строк god-object выходят за скоуп валидационного спайка; полный рефакторинг на честные сервисы — Phase 2.
- **D8.** Автотестов в Phase 1 нет **кроме одного** — unit-тест на парсер встроенного allowlist-JSON (plain JUnit4 + Gson, без MockK, ~20 строк). Причина: 99% кода Phase 1 — throwaway или спайк, тесты преждевременны. Но парсер allowlist'а исполняется на каждом старте, и его беззвучная поломка делает `ModelManagerScreen` пустым (AC-3 провалится невидимо). Это единственное место с асимметричным payoff: копеечный тест — высокая страховка. JUnit 5 + MockK инфраструктура добавляется в Phase 2.
- **D9.** Из метрик в футере ответа — только `TTFT YYYms · NNs total`. **Без tok/s.** Причина: настоящего tokenizer'а в руках нет, эвристика `chars/4` неточна на 15-25%. Пользователь отверг недостоверные метрики — если цифра хоть сколько-то неточная, её не показываем.
- **D10.** UI чисто функциональный Material 3, без темизации и полировки. Строки в `res/values/strings.xml`, русский, тон по `ux-guidelines.md`. Причина: визуал — задача Phase 2-5, Phase 1 UI всё равно выбрасывается.
- **D11.** On-device error log ставится в Phase 1 (`context.filesDir/logs/errors.log`, ERROR-only, ротация при 2 МБ). Компоненты: `download`, `inference-init`, `inference`, `inference-cleanup`. Остальные компоненты (`history-load`, `history-write`, `attachment-*`, `auth`, `room-migration`) — в последующих фазах, когда соответствующая инфраструктура появится. Причина: `patterns.md` мандат, дешёво сделать сразу, полезно для дебага.
- **D12.** Back во время «Загрузка модели…» — сразу возвращает в список моделей (вариант **a**). Native `Engine.initialize` не имеет API отмены, поэтому доработает в фоне и затем cleanUp-нется. Повторный Load до завершения старого сериализуется через Mutex в `ModelRegistry`. Причина: UX-интуитивность важнее, чем «чистое» блокирование back; race — решаем на уровне tech-spec.

## Тестирование

**Unit-тесты: почти ничего в Phase 1 — осознанное исключение из стандартной практики, кроме одного теста.** Обоснование: Phase 1 — валидационный спайк для подтверждения варианта B из `gallery-analysis.md`. Архитектура `:core-runtime` (имена сервисов, подписи функций, раскладка Hilt-модулей) может быть переделана в Phase 2 после того, как мы увидим, что реально нужно. Писать тесты под архитектуру, которую мы сами планируем переделать — преждевременная фиксация контракта. В Phase 2 добавится инфраструктура JUnit 5 + MockK (объявленная в `architecture.md`), и тогда же пишем unit-тесты на устоявшиеся компоненты `ModelRegistry`, `DownloadService`.

**Единственное исключение — unit-тест на парсер allowlist'а.** Встроенный `model_allowlist.json` парсится на каждом запуске приложения. Если парсер тихо сломается, `ModelManagerScreen` покажется пустым, и AC-3 провалится беззвучно — без крашей, без логов, просто «моделей нет». Стоимость теста — ~20 строк на plain JUnit4 + Gson (инфра MockK не нужна): загрузить ассет из test-resources, распарсить, проверить что моделей ровно две и что у каждой заполнены `name`, `modelId`, `modelFile`, `commitHash`, `sizeInBytes`, `defaultConfig`. Асимметричный payoff: копеечный код ловит поломку одного гарантированно-исполняемого на старте пути.

**Интеграционные тесты:** не делаем. В Phase 1 Room не используется → DAO-тестов нет. Инфраструктура `androidx.test` + `room-testing` вводится в Phase 3 вместе с Room.

**E2E тесты:** не делаем. Compose UI-тесты в проекте не планируются в принципе (`patterns.md § Test Infrastructure`: «Compose UI tests are explicitly not in scope»). E2E с реальным LiteRT-LM на CI не выполнимо (нет модели в CI, нужен NPU/GPU).

**Верификация Phase 1 — ручная на устройстве.** Чеклист в секции «Как проверить» ниже.

### Пользователь проверяет (на Honor 200)

Выполняется последовательно, каждый пункт — отдельный шаг с наблюдаемым результатом. Адаптация 14-шагового чеклиста из `work/phase-1-foundation/code-research.md § 8` под Phase 1 allowlist (где в чеклисте упомянут Gemma3-1B — читать как Gemma-4-E2B; где «big model» — читать как Gemma-4-E4B).

1. **Установка APK.** `adb install -r app-debug.apk` или Android Studio → Run. Иконка «Sanctum» появляется в лаунчере; тап запускает MainActivity без крашей.
2. **POST_NOTIFICATIONS.** Первый запуск → системный диалог → Allow. В logcat нет AndroidRuntime-исключений.
3. **Экран моделей.** Показаны две модели: `Gemma-4-E2B-it` (~2.4 ГБ), `Gemma-4-E4B-it` (~4.3 ГБ). Обе в статусе «Не скачано».
4. **Скачивание малой модели.** Тап «Скачать» на `Gemma-4-E2B-it`. Прогресс 0% → 100%, foreground-нотификация видна, скорость отображается. По завершении — статус «Скачано», файл на месте.
5. **Resume после kill.** Начать скачивание `Gemma-4-E4B-it`, дождаться 20%, свайпнуть приложение из recents → ждать 10 сек → запустить снова → тапнуть «Скачать». Загрузка продолжается с ~20%, в logcat виден HTTP Range запрос.
6. **Отмена скачивания.** Во время скачивания E4B тап «Отмена». Прогресс замирает, статус возвращается в «Не скачано», `.gallerytmp` сохраняется. Дотапать скачивание до конца (через повторный «Скачать»).
7. **Загрузка модели.** Тап «Загрузить» на E2B. Появляется «Загрузка модели…» на 5-30 секунд, затем открывается ChatScreen. В logcat нет native-ошибок, видно `Preferred backend: GPU`.
8. **Инференс.** Ввести «Hello, what can you do?», Send. Стрим токенов виден за 1-5 сек TTFT. Под ответом футер `TTFT YYYms · NNs total`.
9. **Stop mid-generation.** Задать длинный промпт («write a 500-word essay about quantum physics»), во время стрима нажать Stop. Стрим останавливается за ~500 мс. Ответ помечен «(прервано)». Следующий промпт отрабатывает.
10. **Multi-turn контекст.** Три последовательные пары промпт-ответ. В третьем ответе модель помнит предыдущий контекст. Нет OOM.
11. **Сброс конверсации.** Тап «Сбросить» в TopAppBar → лента очищается. Следующий промпт стартует с чистой истории, но **без** задержки на re-initialize (это проверяет, что используется `resetConversation`, а не полная переинициализация).
12. **Cleanup при выходе из чата.** Back → возврат в список моделей. PSS в `adb shell dumpsys meminfo app.sanctum.machina` падает до базового (native-память освобождена).
13. **Переключение модели.** В списке тап «Загрузить» на E4B. Первая модель корректно unload'ится, вторая загружается, без OOM, отвечает.
14. **Оффлайн после download.** Выключить Wi-Fi + mobile data → открыть чат с E2B → инференс работает полностью.
15. **Corrupted model (опциональный).** `adb push` случайных 1 КБ в файл E2B `.litertlm` → в приложении тап «Загрузить» → в ChatScreen появляется читаемое сообщение об ошибке + кнопка назад. Не native crash.

### Главный критерий Phase 1 — back-to-back сравнение с Gallery

После прохождения пунктов 1-14 выше — **обязательный финальный шаг:**

- На том же Honor 200 в Gallery загрузить Gemma-4-E2B-it (если была удалена для теста — предварительно скачать в Gallery).
- Выбрать один промпт по своему усмотрению (рекомендуется промпт средней длины, который даст ответ на ~200-500 токенов).
- Запустить промпт в Gallery, посмотреть на субъективную скорость стрима и время до первого токена.
- Запустить тот же промпт в Phase 1 APK, посмотреть на субъективную скорость и прочитать TTFT/total из футера.
- **Вердикт Phase 1:** субъективно скорость «примерно такая же»? **Объективный backstop** (см. AC-14): TTFT Phase 1 ≤ 1.5× TTFT Gallery на том же промпте (Gallery замеряется секундомером). Если оба условия выполнены — Phase 1 пройдена, идём в Phase 2. Если хоть одно нарушено — останавливаемся, исследуем, где потеряли скорость (backend настройки, sampler, per-model config из allowlist).

### Проверка лога ошибок

Для подтверждения AC-16: вызвать хотя бы одну ошибку каждого типа (отключить Wi-Fi во время скачивания для `download`; `adb push` мусора в модель и «Загрузить» для `inference-init`; для `inference` — сложнее на Honor 200 провоцировать, допустимо не воспроизводить), затем вытянуть файл из внутреннего хранилища debug-сборки:

```
adb shell run-as app.sanctum.machina cat files/logs/errors.log > errors.log
```

(Команда `run-as` работает только для debug-APK — это ок для Phase 1; в release-сборке Phase 5 внутреннее хранилище будет недоступно извне без root, поэтому в Phase 2+ нужен in-app share-sheet.)

Открыть сохранённый `errors.log`. Должны быть строки формата `ERROR [download] ... :: ...` и `ERROR [inference-init] ... :: ...`. Каждая — одна строка без переносов, без эмодзи, без декоративных разделителей.
