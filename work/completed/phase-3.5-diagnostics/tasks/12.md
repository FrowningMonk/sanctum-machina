# Task 12: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Финальное приёмочное тестирование Phase 3.5 перед раздачей APK. Задача прогоняет полный набор автоматических проверок (`:core-runtime:test`, `:app:test`, `:app:lintDebug`, `:app:assembleDebug --configuration-cache`), верифицирует все AC из user-spec (G1-G7, V1-V4, L1-L3, H1-H4, D1-D10, A1-A3, T1-T9) и все технические AC из tech-spec, прогоняет полный Honor 200 manual smoke, собирает скриншоты от внешних тестеров (S20 FE 4G / SM-G780F / realme RMX3085) и компилирует AVP-отчёт `logs/qa/pre-deploy-qa.md`.

QA — финальный gate для фазы. Согласно Decision 13 (Final Wave = Pre-deploy QA only) post-deploy stage отсутствует: distribution Sanctum Machina = manual APK transfer на Honor 200, live-environment не существует, MCP-инструменты неприменимы. Все user-facing проверки исполняются здесь же на физическом устройстве.

Особое внимание — AC-L3 (cap-flooding риск из Risks-таблицы): после переезда LogcatReader argv с `*:E` на `*:W` (Task 3) нужно эмпирически проверить, что `MAX_LOGCAT_BYTES` cap не вытесняет ERROR-записи на Honor 200 при реальной нагрузке.

## What to do

1. **Автоматический gradle-прогон.**
   - Запустить `./gradlew :core-runtime:test :app:test :app:lintDebug :app:assembleDebug --configuration-cache`.
   - Зафиксировать в отчёте: zero new lint warning'ов (baseline не сдвинут), zero config-cache serialization warning'ов, все unit-тесты зелёные.
   - Если что-то падает — задача не завершена, фиксить в соответствующей упавшей задаче (1-8).

2. **Tech-spec Acceptance Criteria — 12 пунктов.**
   - Module-boundary grep: `grep -rEn "app\.sanctum\.machina\.diagnostics" core-runtime/src/main` → ноль матчей (interface seam).
   - `grep -rEn "androidx.compose|androidx.activity" core-runtime/src/main` → ноль (TAC-7).
   - `ErrorLog.ALLOWED_COMPONENTS` — список из 14 компонентов из `patterns.md` без изменений (grep + diff).
   - `BuildConfig.VERSION_NAME` через `aapt dump badging app/build/outputs/apk/debug/app-debug.apk | grep versionName` совпадает с `git describe --tags --always --dirty=-dev`.
   - `minDeviceMemoryInGb` присутствует у каждой модели в `model_allowlist.json`; парсер-тест с null/missing проходит; range-check (1..64) отвергает out-of-range.
   - `DiagnosticsModule.kt` имеет `@InstallIn(SingletonComponent::class)` и `@Binds InitDiagnostics: DiagnosticsState`.
   - `formatRamShortage` fixture-table тест покрывает ≥4 кейса (точное равенство порогу, edge под 4 ГБ, реальные 5.3/11.5/10.7).
   - `RecordingInitDiagnostics` test-fake существует в `:core-runtime/src/test/`, используется ≥1 тестом `DefaultModelRegistryTest` с positive call-order assertion'ом.
   - `DiagnosticsViewModel` 4-й вариант `Outcome.InProgress` (Decision 12 — расширение AC-D6) покрыт тестом: содержит маркер «инициализация», `freeRamBytes`, `atEpochMs`, `modelName`, **не** содержит «ok»/«ошибка».

3. **User-spec Acceptance Criteria — построчно.**
   Пройти каждый пункт: G1-G7 (memory gate), V1-V4 (versionName), L1-L3 (logcat argv), H1-H4 (header), D1-D10 (DiagnosticsScreen + state + drawer), A1-A3 (AboutScreen refactor), T1-T9 (тесты). Для каждого — указать в отчёте либо ссылку на тест/файл (auto), либо ссылку на manual-smoke шаг.

4. **Honor 200 manual smoke (полный чеклист user-spec § «Пользователь проверяет»).**
   - Drawer footer: пины «Модели» / «Диагностика» / «О приложении» в указанном порядке.
   - Тап «Диагностика» → экран открыт, TopAppBar с back-arrow, две секции «RAM» и «Логи».
   - Секция RAM: «Последняя инициализация: X.X ГБ RAM · HH:mm · Gemma-4-E4B · ok» (или E2B), вторая строка «Сейчас свободно RAM: X.X ГБ» обновляется примерно раз в секунду — заметить визуально, что цифра меняется.
   - «Сохранить лог» → SAF-picker → файл `.txt` создан.
   - Открыть `.txt`: header содержит `version: v0.3.5-diagnostics` (или git-описание текущего HEAD), строка `memory:` имеет `threshold=` и `lowMemory=`, строка `process:` со значениями ≠ `n/a`, строка `last init:` с актуальной моделью и `ok`. Логкат-блок не пустой.
   - **AC-L3 cap-flooding эмпирическая проверка:** на Honor 200 сразу после warmup сохранить `.txt` → проверить, что в logcat-блоке присутствуют **и** WARN-, **и** ERROR-записи (cap не вытеснил ERROR при `*:W` фильтре). Зафиксировать byte-size logcat-блока в отчёте — для отслеживания запаса под cap'ом.
   - About: манифест из `about.md` рендерится как раньше; секции «Диагностика» нет; footer показывает новый `versionName`.
   - 7 быстрых тапов на version-label в About → `dev_crash_dialog` появляется → подтверждение → краш → `:crash` recovery activity открывается с `RestartCrashBanner` (регрессия R-5 из interview).
   - Локальная проверка гейта: править `core-runtime/src/main/assets/model_allowlist.json` — для `gemma-4-E4B-it-litert-lm` поставить `minDeviceMemoryInGb: 99` → `./gradlew :app:assembleDebug` → переустановить → Drawer → Модели → строка E4B показывает disabled «Скачать» + secondary-text «Недостаточно RAM (X.X ГБ устройство, нужно 99 ГБ)» (точное X — `floor(11.5)` или ровно `11.5` в зависимости от реального `MemoryInfo.totalMem`). Откатить правку (`git checkout`).
   - Финальный `git status` — только наши изменения, allowlist-правка откачена.

5. **Внешние тестеры (S20 FE 4G 5.3GB / SM-G780F / realme RMX3085).**
   - Раздать debug APK тестерам.
   - Получить скриншоты Drawer → Модели: строки Gemma-4-E2B и Gemma-4-E4B с disabled «Скачать» и текстом «Недостаточно RAM (X.X ГБ устройство, нужно Y ГБ)».
   - Сохранить скриншоты в `work/phase-3.5-diagnostics/logs/qa/external-screenshots/` (или вложить путём в отчёт).
   - Это закрывает SC-1 — основная success criteria фазы (тестеры не могут запустить обречённое скачивание).
   - Опционально: попросить тестера прислать `.txt` от «Сохранить лог» — подтвердит формат header'а на не-Honor-устройствах.

6. **AVP-отчёт `work/phase-3.5-diagnostics/logs/qa/pre-deploy-qa.md`.**
   - Per-AC pass/fail table: каждый AC из user-spec и tech-spec — pass/fail/deferred с ссылкой на evidence (test name, screenshot path, command transcript, manual-smoke шаг).
   - Раздел AC-L3 cap-flooding: byte-size logcat-блока в выгрузке Honor 200, наличие/отсутствие ERROR-записей.
   - Раздел SC-1: ссылки на скриншоты от внешних тестеров.
   - Раздел Risks vs reality: пройтись по Risks-таблице tech-spec'а (7 рисков) — какие материализовались, какие нет.
   - Финальный verdict: PASS / FAIL.
   - Если что-то не прошло — explicit deviation entry в `work/phase-3.5-diagnostics/decisions.md` с описанием что именно отклоняется и почему (намеренное deferred vs реальный fail).

**Edge cases:**
- Если auto-тест падает → задача 12 не завершена, фиксить в соответствующей задаче 1-8 и повторить QA.
- Если lint выдаёт новые warning'и (не baselined) → блокер, не deferred.
- Если config-cache warning'и появились после Task 2 (gradle git-describe plugin) → блокер AC-V1, фиксить в Task 2.
- Если на Honor 200 cap-flooding (AC-L3) подтверждается — ERROR вытесняются `*:W`-флудом — блокер, поднимать `MAX_LOGCAT_BYTES` в `LogExportManager` (риск 3 из Risks).
- Если внешний тестер не доступен → SC-1 deferred (явная entry в `decisions.md`); фаза остаётся в `released-pending-sc1` до получения скриншота. Не блокер для merge.
- Локальная проверка гейта (шаг 4 в user-spec smoke) обязательна, даже если все unit-тесты на гейте зелёные — это проверяет реальный rendering Compose-кода с реальным `DeviceInfoProvider`.

**Implementation hints:**
- AVP-отчёт — markdown-таблицы с колонками `AC | Verdict | Evidence`. Для evidence — относительные пути от корня репо (`app/src/test/kotlin/...::testName`) или ссылки на screenshot-файлы.
- AC-L3 measurement: после warmup на Honor 200 сделать SAF-export, открыть `.txt`, найти границы logcat-блока (`--- logcat ---` маркеры из `LogExportManager`), посчитать byte-size и `grep -c "^[0-9]" | grep " W "` vs `grep " E "` для соотношения WARN/ERROR.
- Если все AC прошли — `decisions.md` всё равно получает entry от Task 12 (Summary, Reviews=N/A, Verification=ссылка на pre-deploy-qa.md). Шаблон в начале `decisions.md`.
- Внешние тестеры — раздача APK через прямой transfer (Telegram / email с .apk-файлом). Скриншоты могут прийти асинхронно — допустимо завершить Task 12 со статусом `released-pending-sc1` и задним числом обновить отчёт.
