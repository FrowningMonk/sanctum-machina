# Task 9: Code Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-reviewing` — [skills/code-reviewing/SKILL.md](~/.claude/skills/code-reviewing/SKILL.md)

## Description

Полный code-quality аудит фичи Phase 3.5 (diagnostics). Аудитор читает весь код, изменённый/созданный в Tasks 1-8, и пишет single-pass отчёт с явным pass/fail по кросс-компонентным проверкам. Это **Audit Wave** — аудитор сам по себе является ревью; per-task ревьюеров нет, поэтому фокус audit'а — именно те риски, которые per-task code-reviewer не видит: межмодульная архитектурная целостность, согласованность подходов между задачами, дублирование инициализации тяжёлых ресурсов, соответствие фактического кода Shared Resources таблице из Architecture.

Задача — не повторить per-task ревью, а валидировать фичу как единое целое. Вход — финальное состояние веток после Tasks 1-8 (Wave 3 завершена). Выход — markdown-отчёт `logs/audit/code-audit.md` с итогом «pass»/«fail» и список findings, если есть.

## What to do

1. Прочитать всю фактическую базу изменений Tasks 1-8 — список файлов брать из tech-spec'а («Files to modify» в каждой задаче) плюс заметки в `decisions.md` (Deviations, добавленные/удалённые файлы).
2. Выполнить кросс-компонентные проверки (детали в Verification Steps). Для каждой проверки зафиксировать **pass / fail** и доказательство (grep-output, цитата из кода, ссылка на файл:строку).
3. Применить методологию из `code-reviewing` SKILL.md (11 review dimensions) ко всему набору изменений как к одному changeset'у. Особое внимание — dimensions, которые редко покрываются per-task ревью: модульность/слои, дублирование, согласованность, ownership ресурсов.
4. Найти inconsistent approaches между задачами — например, если одни тесты используют `RecordingInitDiagnostics`, а другие inline-фейк `InitDiagnostics`; или если в Task 5 и Task 8 одинаковые форматтеры реализованы дважды.
5. Найти случаи дублирования инициализации тяжёлых ресурсов (`ActivityManager`, `MemoryInfo`, `DiagnosticsState`, `DeviceInfoProvider`) — должны соответствовать таблице Shared Resources из tech-spec.
6. Записать итоговый отчёт в `work/phase-3.5-diagnostics/logs/audit/code-audit.md` со структурой, описанной в Details.

## Verification Steps

### Smoke

Все 6 проверок ниже — обязательны. Выполнять из корня репо. Каждая → pass/fail в отчёте.

1. **Audit-report file produced**
   - `ls work/phase-3.5-diagnostics/logs/audit/code-audit.md` → файл существует, размер > 0.
   - Шапка отчёта содержит `Result: pass` или `Result: fail`.

2. **Interface seam — `:core-runtime` не импортирует `:app`-пакеты diagnostics**
   - `grep -rEn "app\.sanctum\.machina\.diagnostics" core-runtime/src/main` → 0 матчей.
   - Любой матч = blocker (нарушение модульной границы из Decision 6).

3. **Никакого Compose / Activity в `:core-runtime`**
   - `grep -rEn "androidx.compose|androidx.activity" core-runtime/src/main` → 0 матчей.
   - Любой матч = blocker (TAC-7 из Acceptance Criteria).

4. **`ErrorLog.ALLOWED_COMPONENTS` whitelist не изменён**
   - Найти определение `ALLOWED_COMPONENTS` в коде, сверить со списком из `patterns.md` (14 компонентов).
   - Любое расширение / урезание / переименование без записи в `decisions.md` = major.

5. **Shared resources ownership соответствует таблице Architecture § Shared resources**
   - `DiagnosticsState`: единственный owner — `DiagnosticsModule.@Binds`. Нет других `@Provides`/`new DiagnosticsState()`.
   - `DeviceInfoProvider`: owner — `LogExportModule` (main process) + secondary ctor для `:crash`. Нет дополнительных конструкторов.
   - `ActivityManager`: только system service lookup, без Hilt-обёрток.
   - Каждый consumer (writer/reader) из таблицы реально объявляет depends_on (constructor injection или explicit param).
   - Несоответствие = major (или blocker, если duplicate singleton).

6. **Нет дублирования инициализации тяжёлых ресурсов**
   - Поиск повторных `(context.getSystemService(Context.ACTIVITY_SERVICE) as ActivityManager)` — должно быть централизовано в `AndroidDeviceInfoProvider` / `DefaultModelRegistry` snapshot site.
   - Поиск повторного создания `DiagnosticsState` инстансов — должен быть один (Hilt singleton).
   - Дубль = major.

Дополнительно (sweep по консистентности, не отдельный grep — отметить в отчёте):

- Согласованность тестовых фейков: `InitDiagnostics` фейкуется через `RecordingInitDiagnostics` (для positive-assertion) или `NoOpInitDiagnostics` (для изоляции). Inline-анонимные `: InitDiagnostics by NoOp...` без причины = minor.
- Согласованность форматтеров: `formatRamShortage` (Task 5) и любые форматтеры в `DeviceInfoCollector` / `DiagnosticsScreen` — нет дубликатов с расходящимся форматом.
