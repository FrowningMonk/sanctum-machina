# Task 13: Test Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:test-master` — [skills/test-master/SKILL.md](~/.claude/skills/test-master/SKILL.md)

## Description

Full-feature аудит качества тестов Phase 1. По осознанному решению user-spec D8 вся фаза содержит **ровно один unit-тест** — `AllowlistLoaderTest` в `:core-runtime/src/test/`. Остальное покрытие намеренно отложено в Phase 2+ (инфраструктура JUnit 5 + MockK, concurrency-тесты на `Mutex`, state-machine GPU→CPU fallback, ротация `ErrorLog`). Основание — Phase 1 это валидационный спайк с throwaway-архитектурой; тесты под архитектуру, которую сами же переделываем в Phase 2, — преждевременная фиксация контракта.

Задача аудита — не «написать больше тестов» (это прямое нарушение скоупа D8), а проверить, что:
1. Единственный существующий тест действительно стоит своих ~20 строк: asserts осмысленные, покрывают критичный путь, payoff действительно асимметричный (как заявлено в D8 — парсер исполняется на каждом старте, silent break → пустой `ModelManagerScreen` → AC-3 проваливается невидимо).
2. Fixture в test-resources идентична production-ассету — иначе тест проходит, а prod ломается.
3. Тест покрывает защиту критичных инвариантов: TAC-15 (схема-guard `modelId` → `litert-community/`, URL → `https://huggingface.co/`, `sizeInBytes 1..10GB`), Decision T8 (`accelerators[0] == "gpu"`, хардкод без UI-выбора).
4. Scope-lock соблюдён: в `:core-runtime/src/test/` ровно один тестовый класс, никто не добавил «заодно» тесты на `DownloadRepository`, `ModelRegistry`, `ErrorLog` и прочее (это отложено в Phase 2 осознанно, см. D8 и user-spec § «Вне скоупа Phase 1»).
5. Debt для Phase 2 явно задокументирован — реальные source-of-risk места (concurrency `Mutex` в `DefaultModelRegistry`, stale-instance guard, GPU→CPU fallback state machine, `ErrorLog` rotation at 2 MB) должны быть записаны где-то, чтобы в Phase 2 их не забыли покрыть.

Результат — audit report в `work/phase-1-foundation/logs/audit/test-review.json`. Report содержит verdict (pass / issues) + findings с severity + ссылки на файлы/строки. Отчёт читают Task 14 (pre-deploy QA) и команда Phase 2 (как входную точку для testing-backlog'а).

## What to do

1. Убедиться, что в `:core-runtime/src/test/` существует **ровно один** тестовый класс — `AllowlistLoaderTest.kt`. Найти все файлы с суффиксом `Test.kt` / `Tests.kt` / `Spec.kt` под `core-runtime/src/test/` и `core-runtime/src/androidTest/` (последняя не должна существовать в Phase 1 в принципе), плюс под `app/src/test/` и `app/src/androidTest/`. Любой дополнительный тест — finding severity `high` (нарушение скоупа D8).

2. Прочитать `AllowlistLoaderTest.kt` целиком и оценить качество каждого теста по чеклисту:
   - Assertions осмысленные: не просто `assertNotNull(result)` / «не кинул исключение», а проверяют конкретные поля (`size == 2`, непустые `name`, `modelId`, `modelFile`, `commitHash`, `sizeInBytes > 0`, `defaultConfig.topK != null`, `defaultConfig.temperature != null` — по tech-spec § Testing Strategy).
   - Покрыт happy path (Scenario 1 — success, list.size == 2).
   - Покрыта per-entry валидация полей (Scenario 2 — каждый `Model` имеет заполненные обязательные поля).
   - Покрыт сценарий failure: битый JSON → `Result.failure` с читаемым сообщением (Scenario 3, опциональный по tech-spec, но желательный).
   - Покрыт TAC-15 схема-guard: есть хотя бы один негативный сценарий, где валидация ломается (например, `modelId` без префикса `litert-community/` → `Result.failure`, `sizeInBytes == 0` или `> 10 GB` → `Result.failure`, URL не с `https://huggingface.co/` → `Result.failure`). Если сценария нет — finding severity `medium` (TAC-15 заявляет «проверяется кодом + `AllowlistLoaderTest` расширенным сценарием»).
   - Покрыт Decision T8 invariant: `accelerators[0] == "gpu"` для всех моделей (либо в тесте явная проверка `model.defaultConfig.accelerators.firstOrNull() == "gpu"`, либо Scenario 2 это подразумевает через assert на полноту полей). Silent drift allowlist'а на `cpu` в первом элементе делает GPU→CPU fallback бессмысленным (runtime всегда стартует с CPU).

3. Прочитать fixture `core-runtime/src/test/resources/model_allowlist_fixture.json` и production-ассет `core-runtime/src/main/assets/model_allowlist.json`. Сравнить побайтно (или через структурное сравнение распарсенного JSON — смотря как проще). Tech-spec явно требует: «Fixture: копия финального `assets/model_allowlist.json`». Если расходятся — finding severity `high` (тест проходит, а реальный allowlist сломан). Допустимое расхождение — только дополнительный невалидный fixture для Scenario 3 (битый JSON) — он должен лежать под другим именем (например, `model_allowlist_invalid.json`).

4. Проверить соответствие fixture обязательным инвариантам:
   - `.models | length == 2` (AC-3, TAC-6).
   - Имена моделей: `Gemma-4-E2B-it` и `Gemma-4-E4B-it` (user-spec § Ограничения, D4).
   - Каждая модель имеет непустые `name`, `modelId`, `modelFile`, `commitHash`, `sizeInBytes` (TAC-6).
   - `defaultConfig.accelerators[0] == "gpu"` для каждой модели (Decision T8).
   - `modelId` начинается с `litert-community/` (TAC-15).
   - `sizeInBytes` в диапазоне `1..10_737_418_240` (10 GB, TAC-15).

5. Оценить обоснование D8 на пост-фактум:
   - Подтверждает ли фактический тест формулировку «копеечный код ловит поломку одного гарантированно-исполняемого на старте пути» (user-spec D8). Если тест покрывает только happy-path без валидации полей — обоснование «silent break = AC-3 провалится невидимо» не выполнено: парсер может вернуть `Result.success` с пустым / невалидным списком, а тест этого не поймает.
   - Проверить, что `AllowlistLoader` действительно использует fixture-механизм загрузки, совместимый с prod-путём: тест читает `javaClass.getResourceAsStream("/model_allowlist_fixture.json")`, а prod-код — `context.assets.open("model_allowlist.json")`. Оба пути должны идти через **один и тот же метод парсинга** (например, `AllowlistLoader.parseJson(inputStream): Result<List<Model>>`), иначе тест проверяет параллельный код, а не production-путь. Finding severity `medium` если parser-путь раздвоен.

6. Проверить, что testing-debt для Phase 2 задокументирован. Ожидаемые места документирования (достаточно одного):
   - `work/phase-1-foundation/tech-spec.md` § Risks / § Testing Strategy — упоминание отложенного покрытия.
   - `work/phase-1-foundation/user-spec.md` § «Вне скоупа Phase 1» — уже содержит общую формулировку «автоматические тесты… в Phase 1 почти не пишутся… инфраструктура JUnit 5 + MockK появляется в Phase 2».
   - `work/phase-1-foundation/decisions.md` — если команда писала туда инкрементальные заметки.

   Конкретные точки testing-debt, которые должны быть упомянуты явно (или явно покрываться формулировкой «инфраструктура JUnit 5 + MockK + вся внутренняя `:core-runtime` проверка вводятся в Phase 2»):
   - **Mutex concurrency** в `DefaultModelRegistry` (Decision T9) — race между `initialize`, `cleanUp`, `resetConversation`, stale-instance guard `currentInstance === model.instance`.
   - **GPU→CPU fallback state machine** (Decision T8) — последовательность init-попыток, пробрасывание ошибок GPU-инициализации в `ErrorLog`, повторная попытка с `accelerator=CPU`.
   - **ErrorLog ротация** (Decision T11, TAC-9) — триггер при > 2 МБ, переименование `errors.log` → `errors.log.1`, перезапись старого `.1`.
   - **DownloadWorker** (рефакторинг `renameTo` без проверки return — R4) — намеренно не чиним в Phase 1, но finding severity `low` должен подсветить эту область как приоритет Phase 2.

   Если ни одна из этих точек нигде не задокументирована — finding severity `medium` («Phase 2 testing debt not documented»).

8. Важное ограничение: **не писать новые тесты**, даже если обнаруживается пробел. Аудит только фиксирует дефекты; решение о добавлении тестов в Phase 1 требует пересмотра D8 пользователем. Finding + suggestion — достаточно. Исключение: если fixture расходится с prod или тест явно не работает (не запускается, не компилируется) — это не «пробел в покрытии», а баг существующего кода, и его разбирает Task 14 pre-deploy QA по своему контракту.

**Dependencies:**
- Task 10 (Navigation + final wiring) — формально, чтобы весь Phase 1 код существовал до аудита. Фактически Task 13 читает только то, что создано в Task 4 (тест + fixture + loader + asset). Но по Audit Wave порядку он идёт после всех Code/Impl-задач и параллельно с Task 11 (Code Audit) и Task 12 (Security Audit).
- Параллелизм: Task 11, 12, 13 параллельны друг другу (разные reviewers, разный фокус, общий read-only доступ к исходникам).

**Edge cases:**
- `AllowlistLoaderTest.kt` не существует вообще (баг Task 4) → finding severity `critical`, verdict `issues`, report всё равно записывается.
- Fixture отсутствует → finding severity `critical`, verdict `issues`.
- Более одного test-класса → finding severity `high` на каждый «лишний» класс, verdict `issues`. Не предлагать удалить — предложить перенос в Phase 2 backlog (решение за пользователем).
- Тесты игнорируются (`@Ignore`, `assumeTrue(false)`) → finding severity `high`, фактически покрытие == 0.
- В fixture случайно появились поля, которых нет в prod (расхождение наружу) — прежние скрипты могут это допускать, но для Phase 1 fixture должна быть точной копией. Finding severity `high`.
- `defaultConfig.accelerators` в fixture — массив; проверяется именно **первый** элемент (`[0] == "gpu"`), Decision T8 явно: «захардкожен GPU + CPU fallback».

**Implementation hints:**
- Использовать Glob для поиска тест-файлов: `**/*Test.kt`, `**/*Tests.kt`, `**/*Spec.kt` под `core-runtime/` и `app/`.
- Для побайтного сравнения fixture и prod-ассета — Read обоих файлов + diff в голове (файлы маленькие, 2 модели). Если отличие только в форматировании (whitespace, порядок полей) — это тоже расхождение для строгого аудита, но finding severity `low` (Gson терпит к порядку, но не к содержимому).
- Decision T8 invariant можно проверить либо через чтение fixture (`defaultConfig.accelerators[0]` каждой модели == `"gpu"`), либо через чтение AllowlistLoaderTest на наличие ассерта. Лучше оба — перекрёстная проверка.
- Для проверки «тест и prod через общий parse-метод» — Read `AllowlistLoader.kt` + `AllowlistLoaderTest.kt`, убедиться, что тест вызывает тот же публичный метод (например, `AllowlistLoader.loadFromStream(is)` / `parse(json)`), что и prod-код внутри `loadFromAsset(context)` делегирует. Если тест парсит Gson-ом напрямую, минуя AllowlistLoader — finding severity `medium`.
- Report пишется один раз. Если в процессе аудита обнаружен новый finding — переписать файл целиком, не добавлять инкрементально.
