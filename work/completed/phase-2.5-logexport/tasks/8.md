# Task 8: Code Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-reviewing` — [skills/code-reviewing/SKILL.md](~/.claude/skills/code-reviewing/SKILL.md)

## Description

Комплексный холистический code review всей фичи Phase 2.5 «Экспорт диагностического лога» после реализации задач 1–7. Задача запускается в Audit Wave после завершения всех code-writing задач и работает с уже написанным кодом — никакой новый production-код не пишется, только отчёт о ревью. Ревьюеры в JSON-формате для этой задачи НЕ назначаются: сам code-auditor и есть ревью; его отчёт затем попадает в `decisions.md` фичи.

Фокус аудита — cross-component проблемы, которые не видны при ревью отдельной задачи, а проявляются только на стыке компонентов:

1. **Согласованность shared-file-state между `CrashHandler` (писатель) и `CrashState` (читатель).** Оба класса работают с одним и тем же `filesDir/logs/crash.log` и flag-файлом `crash.log.dismissed` — причём `CrashHandler` живёт в main-процессе (устанавливается один раз при старте, пишет перед `Process.killProcess`), а `CrashState` живёт только в main-процессе как Hilt-singleton. Проверяется: одинаковая семантика путей (оба вычисляют их от `Context.filesDir`, а не от кэша/внешнего хранилища); согласованные правила truncation (head на 100 KB для `crash.log` — единственное место в `CrashHandler`, `CrashState` только читает существование); согласованная логика dismiss-flag (handler удаляет `.dismissed` при новом крэше — Flow B «новый краш сбрасывает скрытость»; `markDismissed` создаёт zero-byte flag; `clear` удаляет оба файла атомарно-enough для пользовательского восприятия). `CrashReportActivity` в процессе `:crash` тоже читает `crash.log` через свой direct-new `LogExportManager` — согласованность путей всех трёх точек (writer, main-reader, crash-reader) проверяется отдельно.

2. **Соблюдение module-boundary правила `:core-runtime` / `:core-settings` — UI-free.** Ни один импорт `androidx.compose.*` или `androidx.activity.*` не должен появиться в исходниках `:core-runtime/src/main` и `:core-settings/src/main`. Фича Phase 2.5 живёт полностью в `:app`, но проверка обязательна — случайный импорт в core-модуле при работе с `Context.filesDir` или стартом activity из handler может ускользнуть. Правило из `patterns.md § Project-Specific Code Patterns: Module boundary`.

3. **Соответствие `patterns.md`:**
   - **Module hygiene** — см. пункт 2 выше + `:core-*` не получает новых зависимостей.
   - **Error-logging conventions** — никакого нового `ErrorLog.e` из `app/.../crash/*.kt` (Decision 10: handler пишет `crash.log` напрямую через `File.writeText`, `ErrorLog.ALLOWED_COMPONENTS` не расширяется). Проверить, что в коде `:app/crash/` нет вызовов `ErrorLog`, `errorLog.`, а в `core-runtime/.../log/ErrorLog.kt` whitelist не изменён.
   - **SafeMarkdown** — фича не рендерит markdown (она пишет plain `.txt` и показывает Compose `Text` без markdown), поэтому SafeMarkdown здесь не должен появиться. Убедиться, что новый код не задел существующие AboutScreen usages SafeMarkdown и не заменил их на прямой `Markdown(...)` при добавлении раздела «Диагностика».
   - **Error-log format** — никаких `Log.i` / `Log.w` / `Log.d` в production-коде фичи. Единственное допустимое исключение — один `android.util.Log.e("CrashHandler", ...)` breadcrumb внутри outer-catch `CrashHandler` (Decision 10 — `ErrorLog` нельзя из uncaught handler, но один Log.e в logcat ради последующего About-export — допустимо и явно описано в tech-spec Flow A шаг 4).

4. **Архитектурная согласованность с существующими Compose-паттернами:**
   - **SnackbarHost в `ModelManagerScreen`** — до Phase 2.5 у этого экрана `Scaffold` с `SnackbarHost` не было (`ChatScreen.kt` — эталонный референс). Задача 7 его вводит. Проверить, что структура `Scaffold { snackbarHost = { SnackbarHost(state) } }` + `rememberCoroutineScope()` + `snackbarHostState.showSnackbar(...)` повторяет паттерн `ChatScreen.kt`, а не вводит новый.
   - **Двухкнопочный паттерн `HeavyChangeDialog.kt`** — `CrashReportActivity` («Сохранить лог» / «Закрыть») и `RestartCrashBanner` («Сохранить лог» / ✕) должны по структуре повторять существующий паттерн двух кнопок (`TextButton` + `TextButton`, отступы, tone). Проверяется не pixel-perfect, а согласованность подхода.
   - **`AlertDialog` для dev-gesture** — «Спровоцировать тест-краш?» строится через `AlertDialog { confirmButton { … } dismissButton { … } }` как в `HeavyChangeDialog`, а не через hand-rolled `Dialog { Card { … } }`.
   - **В `CrashReportActivity`** — несмотря на non-Hilt (Decision 5), Compose-экран должен следовать общему стилю приложения: тема `Theme.Sanctum`, типографика из `Typography`, цвета через `MaterialTheme.colorScheme.*`, а не хардкоднутые hex.

5. **Явное соблюдение Decisions 5 и 10:**
   - **Decision 5** (non-Hilt `CrashReportActivity`) — класс объявлен как `class CrashReportActivity : ComponentActivity()` **без** аннотации `@AndroidEntryPoint`. `LogExportManager` инстанцируется напрямую в `onCreate` через конструктор, не через `@Inject`. Проверяется grep'ом `@AndroidEntryPoint` и `@Inject` в `CrashReportActivity.kt` — ожидается ноль совпадений.
   - **Decision 10** (own writer для `crash.log`; `ErrorLog.ALLOWED_COMPONENTS` untouched) — см. пункт 3 выше. Дополнительно: git-diff `core-runtime/src/main/kotlin/app/sanctum/machina/core/log/ErrorLog.kt` против `main` — ожидается пустой.

Результат задачи — текстовый отчёт (Markdown), сохранённый в `work/phase-2.5-logexport/logs/working/task-8/code-audit-report.md`. Отчёт затем линкуется в `decisions.md` фичи при финализации (задача 11 → done-скилл).

**Edge cases (в аудите уделить внимание):**
- Голый `NullPointerException` из `getProcessName()` до API 28 — не актуально (minSdk = 31), но проверить, что код не перестраховался вручную и не внёс dead-code path.
- `Context.filesDir` может не содержать `logs/` при первом запуске на свежеустановленном APK — убедиться, что `CrashState.refresh()` (fresh-install path) обрабатывает отсутствие папки без NPE (тест `freshInstall_noLogsDir_succeeds` уже это покрывает, но аудит проверяет синхронизацию production-кода с тестом).
- Гонка: пользователь тапает «Сохранить лог» и сразу жмёт back — SAF-launcher должен отработать result-callback даже если Activity уже в `onStop`. Проверить, что `finally { launching = false }` действительно в callback-е, а не в `onResume`.
- `android:taskAffinity=""` в manifest — именно пустая строка, не отсутствие атрибута (разные семантики на разных версиях Android).

**Implementation hints:**
- Скилл `code-reviewing` описывает 11 измерений + процесс; этот скилл уже покрывает методологию, не нужно её пересказывать в отчёте — цитировать по необходимости.
- Отчёт — на русском (по CLAUDE.md); код-фрагменты в отчёте оставлять на английском.
- Не дублировать в отчёт длинные куски кода — ссылаться `file:line` или давать короткий snippet ≤ 5 строк.
- Findings формулировать с глаголом действия: «Переместить `finally` в …», «Заменить `@Inject` на direct-new в onCreate», а не «Код плохой, потому что…».
- Если findings нет — сказать это прямо: «11-dimension review: без findings; межкомпонентная согласованность подтверждена».
- Пустой список findings — легитимный исход. Не выдумывать замечания ради объёма.
