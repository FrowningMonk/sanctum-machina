# Task 11: KV-cache restoration on chat re-entry — replay history via `ConversationConfig.initialMessages`

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Баг найден после Tasks 1–6 (вне Pre-deploy QA Round 1, отдельно от регрессии AC-3.2): persistent-чат теряет контекст при возврате. Сценарий: создал чат A, назвал модели имя, спросил — модель ответила. Создал чат B, поговорил. Вернулся в A, спросил «как меня зовут?» — модель не помнит.

**Причина.** `ChatViewModel.bootstrapChatModelId` (`app/src/main/kotlin/app/sanctum/machina/ui/chat/ChatViewModel.kt:786-834`) на каждый вход в Persistent-чат запускает `observeFirstReadyThenReset(reason)` (line 940), который вызывает `registry.resetConversation(reason = CHAT_SWITCH | DRAFT_COMMIT)`. Это идёт в `LlmChatModelHelper.resetConversation` (`core-runtime/src/main/kotlin/app/sanctum/machina/core/inference/LlmChatModelHelper.kt:151-198`), который закрывает старую `Conversation` и создаёт новую через `engine.createConversation(ConversationConfig(samplerConfig, systemInstruction, tools))`. Новая Conversation **пустая** — только `systemInstruction`; KV-cache очищен. Auto-resume (`observeFirstReadyThenResume`, `:916-921`) поднимает только хвост `ROLE_USER` (DRAFT_COMMIT и kill-mid-stream); при штатном возврате к чату с paired-историей не запускается, и модель отвечает «с нуля».

Это полу-фикс AC-1.1 в Tasks 2–3: изоляция между чатами достигнута (KV не утекает A→B), но возврат к A теряет свой собственный контекст. Корректная пользовательская формулировка — «KV-cache соответствует текущему чату» (изоляция между чатами **+** восстановление в своём).

**API-находка.** Распаковка `litertlm-android-0.10.0-api.jar` из Gradle cache подтверждает (см. § Details → API surface):

```
public ConversationConfig(
    Contents systemInstruction,
    List<Message> initialMessages,           // ← мы туда не передаём ничего
    List<ToolProvider> tools,
    SamplerConfig samplerConfig,
    boolean automaticToolCalling,
    List<Channel> channels)

public Message(Role, Contents, List<ToolCall>, Map<String,String>)
public enum Role { SYSTEM, USER, MODEL, TOOL }
```

`engine.createConversation(...)` прогоняет `initialMessages` prefill'ом, восстанавливает KV-cache, дальше отвечает уже зная историю. Это штатный путь движка, ограничения движка нет — пробел только у нас.

**Фикс.** Прокинуть `initialMessages: List<Message>` через цепочку `LlmModelHelper.resetConversation` → `DefaultModelRegistry.resetConversation` → `ChatViewModel.observeFirstReadyThenReset` (и аналог в `applyLightOverrides`). На стороне `ChatViewModel`:
- Загрузить `messageDao.getByChatId(chatId)` — DAO уже существует (`app/src/main/kotlin/app/sanctum/machina/data/dao/MessageDao.kt:18`).
- Маппинг `MessageEntity` → `litertlm.Message`: `ROLE_USER → Role.USER`, `ROLE_ASSISTANT → Role.MODEL`. Текст + аттачменты (image/audio) — через существующий `MultimodalContentsBuilder.build(...)` + `Contents.of(...)` (тот же путь, что использует `dispatchSend` для текущего turn'а — `LlmChatModelHelper.runInference:282-294`).
- Если хвост — unpaired `USER` (DRAFT_COMMIT / kill mid-stream): **исключить** его из `initialMessages` и оставить для `observeFirstReadyThenResume` как раньше (auto-resume сам пошлёт его как первый USER turn новой Conversation).
- Передать получившийся список в `resetConversation`.

**Когда replay нужен / не нужен** — фиксируем design здесь, а не оставляем «design-вопросом» (см. § Decisions):

| reason | initialMessages |
|--------|-----------------|
| `CHAT_SWITCH` | paired history без unpaired USER tail |
| `LIGHT_OVERRIDE` | paired history (иначе на каждый Apply slider'а контекст теряется — это уже не «бесплатный» refresh sampler'а) |
| `DRAFT_COMMIT` | `emptyList()` (handover из Draft = брэнд-нью контекст; единственная USER row подберётся auto-resume) |
| `SYSTEM_PROMPT` | `emptyList()` (semi-light + чистка UI history — пользователь хочет «свежий чат с новой инструкцией») |
| `USER` (тап ↻) | `emptyList()` (явный явный wipe) |
| `HEAVY` | `emptyList()` (heavy reinit + ещё секунды на prefill = плохой UX; engine swap по сути новая сессия) |

Эта задача также корректирует user-spec AC-1.1 (`work/phase-3.6-bugfix/user-spec.md:45`) и расширяет Сценарий 1 (`:20-23`) — старая формулировка «Conversation пересоздаётся (KV-cache очищен)» зафиксировала только половину контракта.

**Зависимости.** Зависит от Task 2 (`ResetReason` enum + signature `resetConversation(modelName, systemPrompt, reason)`) и Task 3 (`observeFirstReadyThenReset` уже на месте, выбор `DRAFT_COMMIT` vs `CHAT_SWITCH` уже сделан) — оба done.

### Edge cases

- **История содержит сообщения с `text == ""` и одной аттач'ей.** `MultimodalContentsBuilder.build` (`MultimodalContentsBuilder.kt:38`) пропускает blank text — `Contents` будет содержать только image/audio Part. Это корректно: prefill этой entry не упадёт, KV впитает media token'ы.
- **Bitmap re-decode на каждый bootstrap.** `decodeAttachmentsForEntity` идёт через `BitmapFactory.decodeFile` — синхронно, потенциально 50-200 ms на изображение. На chat'е с 30 сообщениями где 5 имеют image → ~250 ms сверх первого Ready. Mitigation: уже есть `attachmentCache` (`ChatViewModel.kt:1514`) — переиспользовать тот же memo. Если кэш ещё пуст (cold-start VM), стоимость одноразовая на чат — приемлемо.
- **Контейнерная защита** (`resolveInsideAttachmentsRoot`): аттачи с broken paths уже логируются и опускаются (`decodeAttachmentsForEntity`). Это значит entry превратится в `Message(role, Contents.of([Content.Text(entity.text)]))` — без media. Для prefill это OK: модель видит текст без media, KV-cache синхронизирован с реальной историей UI (которая тоже без этой media — `toDomainMessageWithAttachments` тот же путь использует).
- **Хвост unpaired USER одновременно с auto-resume:** `dropUnpairedUserTail = (pendingAutoResume != null)` гарантирует, что одна и та же USER row не уходит и в `initialMessages`, и в `runInference` через auto-resume. Тест `bootstrapPersistent_excludesUnpairedUserFromInitialMessages_whenDraftCommit` это пинит.
- **Пустая история (chat row есть, messages нет):** `getByChatId` возвращает `[]` → `initialMessages = []` → reset идёт как обычно с пустым контекстом. Это нормальный сценарий freshly-created persistent: не fail, не warning.
- **Race между `applyLightOverrides` и in-flight inference.** `applyLightOverrides` сейчас не проверяет `isGenerating`; в реальности `dispatchByLevel` вызывается из «Применить»-кнопки sheet'а, а sheet недоступен пока генерация идёт (Task 6 `engineReady && !isGenerating && !reinitInProgress`). Не вводим новую проверку.
- **Quick chat в `applyLightOverrides`:** identity = Quick → `initial = emptyList()`. Light apply теряет контекст текущего turn'а Quick — это уже текущее поведение Quick (нет персистентной истории), не регрессия.
- **HEAVY: `cleanup + initialize`** не использует `resetConversation` — параметр `initialMessages` не задействуется. После heavy reinit Conversation создаётся в `initialize` пустой (системная инструкция + пустые initial). Не меняем `initialize` сигнатуру: HEAVY = explicit fresh start (см. таблицу в Description).
