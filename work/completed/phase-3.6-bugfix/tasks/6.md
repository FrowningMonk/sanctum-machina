# Task 6: engineReady StateFlow + Settings gating switch in ChatScreen (+ stale comment cleanup)

## Required Skills

Перед выполнением задачи загрузи:

## Description

Фикс Bug 2 (user-spec § Сценарий 3, AC-2.1 / AC-2.2 / AC-2.3): в новом persistent-чате (Draft) кнопка Settings в TopAppBar остаётся выключенной даже после прогрева модели — пользователь не может изменить system prompt / temperature до отправки первого сообщения. Корень: `ChatScreen.kt` гейтит Settings IconButton по `topAppBarState is TopAppBarState.Ready`, а `deriveTopAppBarState` для `ChatIdentity.Draft` всегда возвращает `TopAppBarState.Draft` (нужен для работы model picker dropdown), независимо от `ModelInitStatus`. Поэтому Draft × Ready никогда не активирует Settings.

Решение per Decision 6: ввести в `ChatViewModel` отдельный `val engineReady: StateFlow<Boolean>` — `true` iff entry для текущей модели в `_uiState` имеет `initStatus is ModelInitStatus.Ready` И `!warmupInFlight`. `ChatScreen.kt` переключает гейтинг Settings IconButton с `topAppBarState is TopAppBarState.Ready` на `engineReady` (collected as state). Draft и Persistent ветки наблюдают один и тот же сигнал; `deriveTopAppBarState` при этом продолжает возвращать `TopAppBarState.Draft` для Draft (model picker dropdown не ломаем).

Параллельно: убрать в `ChatScreen.kt` устаревший комментарий рядом со строкой ~329 о том, что edge-to-edge зависит от `WindowCompat.setDecorFitsSystemWindows`, выставленного `MainActivity` — после Task 4 это становится неявно правдой через `enableEdgeToEdge()`, и текущая формулировка вводит в заблуждение (см. `code-research.md` § 1 Bug 3 + § 5).

Wave 4 (depends_on=[3]): `ChatViewModel.kt` пишется Task 3 в Wave 3, мы дописываем сверху; пересечений по строкам нет, но wave-сериализация нужна, чтобы избежать race на write-конфликте.

**Edge cases:**

- `pinnedModelId == null` (нет выбранной модели) → engineReady = false. Покрыто Idle-веткой через `entry == null` ⇒ initStatus != Ready.
- Быстрый flip Initializing → Ready во время того же кадра: `engineReady` отдаст последнее значение из combine — Ready (true). OK.
- `warmupInFlight = true` при `initStatus = Ready` (race во время model switch): false. Покрыто параметризованным тестом (Ready + warmup-in-flight → false).
- Несколько моделей в `models`, активна одна: `engineReady` смотрит только на entry для активной модели (по тому же полю, что использует `deriveTopAppBarState` — `pinnedModelId ?: activeModelId`).
- Compose: при использовании `collectAsStateWithLifecycle` нужен `androidx.lifecycle.compose` import — проверить, что библиотека уже на classpath. Если нет — fallback на `collectAsState()` (работает, просто не lifecycle-aware).

**Implementation hints:**

- Не дублировать логику `deriveTopAppBarState`. `engineReady` — узкий булев срез того же source: `(entry?.initStatus is ModelInitStatus.Ready) && !warmupInFlight`. Можно записать как combine из тех же `models / pinnedModelId / warmupInFlight` flows, что питают `topAppBarState`.
- `stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), initialValue = false)` — стандартный паттерн для derived StateFlow.
- Тест: `runTest { ... advanceUntilIdle(); assertEquals(false, vm.engineReady.value) }` — `runTest` уже используется в `ChatViewModelTest.kt`.
- Удаление stale-комментария: Edit-замена 5 строк на пустоту (или на одну минимальную строку). Не пиши новый комментарий "теперь правда" — пиши код, не нуждающийся в пояснениях.
- НЕ вводи Honor-specific код (per memory `manifest_breadth_over_honor_lock`) — `engineReady` is platform-neutral.
