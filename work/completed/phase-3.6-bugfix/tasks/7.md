# Task 7: Code Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-reviewing` — [skills/code-reviewing/SKILL.md](~/.claude/skills/code-reviewing/SKILL.md)

## Description

Полный код-аудит фазы phase-3.6-bugfix после завершения волн 1–3 (Tasks 1–6). Аудитор читает все изменённые в фазе файлы и ищет cross-component проблемы, которые отдельные ревью каждой таски могли пропустить — нарушения инвариантов между модулями, утечку Honor-специфики в primary path, потерю mutex-дисциплины, нарушение UI-free инварианта `:core-runtime`, и общие нарекания по качеству кода (нейминг, сложность, обработка ошибок, структура).

В этой фазе аудитор выступает в роли финального ревьюера — отдельных reviewer-агентов для Task 7 нет. Артефакт задачи — markdown-отчёт `work/phase-3.6-bugfix/logs/audit/code-audit.md` с findings, ранжированными по severity, со ссылками `file:line`. Отчёт читается на этапе `/do-task` для этой таски и используется как вход в Final Wave (Task 10 pre-deploy QA).

## What to do

1. Прочитать все файлы, изменённые в Tasks 1–6 (см. Context Files / Code files), плюс `patterns.md` для сверки утверждений и `tech-spec.md` для критериев Decisions 1–8.
2. Прогнать checklist (см. Acceptance Criteria) по каждому пункту — для каждого зафиксировать результат (PASS / finding со ссылкой `file:line`).
3. Дополнительно осмотреть весь diff фазы на предмет общих качественных проблем:
   - нейминг (имена функций, переменных, enum-значений)
   - сложность (длина методов, вложенность, дублирование)
   - обработка ошибок (что бросается, что ловится, что логируется)
   - структурные нарекания (правильность размещения, разделение ответственности)
4. Сгруппировать findings по severity (BLOCKER / MAJOR / MINOR / NIT). Для каждого: краткое описание, ссылка `file:line`, предлагаемая правка.
5. Записать отчёт в `work/phase-3.6-bugfix/logs/audit/code-audit.md` по структуре:
   - Summary (1–3 предложения: общий вердикт + сколько findings и каких severity).
   - Checklist results (таблица или список — по каждому пункту acceptance criteria).
   - Findings (по severity, каждое с `file:line` и предложенной правкой).
   - Overall verdict: APPROVE / APPROVE WITH FIXES / REJECT.

## Acceptance Criteria

- [ ] **Checklist 1 — `ResetReason` placement:** enum находится в `core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/ResetReason.kt`, не в `:app`. Импорты в `ChatViewModel` и `DefaultModelRegistry` используют этот путь.
- [ ] **Checklist 2 — Mutex discipline preserved:** каждый путь через `DefaultModelRegistry.resetConversation` (success / non-Ready skip / unknown-name) выполняется под `lifecycleMutex.withLock`. Нет ранних return до взятия mutex там, где это меняет наблюдаемое состояние или логирует.
- [ ] **Checklist 3 — No Honor-specific code in primary path:** grep по diff на `Honor`, `EMUI`, `Build.MANUFACTURER`, `Build.BRAND` — 0 совпадений в production-коде. (Допустимо только в комментариях, описывающих контекст бага, и в тестовых файлах.)
- [ ] **Checklist 4 — `:core-runtime` UI-free invariant:** `grep -r "androidx.compose\|androidx.activity" core-runtime/src/main` → 0 hits.
- [ ] **Checklist 5 — Все callers `registry.resetConversation` передают явный `reason`:** ни один вызов не использует default-параметр (его быть не должно — см. Decision 1). Перечислить все call sites в отчёте.
- [ ] **Checklist 6 — `LIGHT_FIELD_LABELS` не содержит `MAX_TOKENS`:** в `ChatViewModel.kt` константа не упоминает `ConfigKeys.MAX_TOKENS.label`.
- [ ] **Checklist 7 — Stale comment в `ChatScreen.kt:329` удалён:** комментарий о MainActivity / `setDecorFitsSystemWindows` отсутствует или переписан.
- [ ] **Checklist 8 — General code-quality review:** нейминг, сложность, обработка ошибок, структурные нарекания — отмечены или явно подтверждены как чистые.
- [ ] Отчёт записан в `work/phase-3.6-bugfix/logs/audit/code-audit.md` с финальным вердиктом.

## Verification Steps

### Smoke

Audit-задача — deliverable есть markdown-отчёт. Smoke-проверки:

- `grep -rn "androidx.compose\|androidx.activity" core-runtime/src/main` → 0 hits.
- `grep -rin "Honor\|EMUI\|Build\.MANUFACTURER\|Build\.BRAND" app/src/main core-runtime/src/main` → 0 hits в production-коде (комментарии и тесты — допустимо, отметить отдельно).
- `git diff main...HEAD -- '*.kt'` — обзорное чтение всего diff фазы для general code-quality review.
- Файл `work/phase-3.6-bugfix/logs/audit/code-audit.md` существует, содержит Summary + Checklist results + Findings (severity-ranked, каждое с `file:line`) + Overall verdict.

**Dependencies:** Tasks 1–6 должны быть `done` (волна Audit запускается после Wave 4).

**Edge cases:**
- Если кто-то из исполнителей Tasks 1–6 отклонился от tech-spec — зафиксировать в отчёте как finding (с пометкой, является ли это осознанным отклонением, задокументированным в `decisions.md`).
- Если Honor/EMUI упоминается в комментариях, описывающих контекст бага — это допустимо, но отметить, что это комментарий, не код (пер memory `manifest_breadth_over_honor_lock`).
- Unknown-name silent skip в `DefaultModelRegistry.resetConversation` — это намеренно (Decision 1). Не флажить как отсутствие логирования.

**Implementation hints:**
- Сверять каждое утверждение `tech-spec.md § Architecture` с фактическим кодом — расхождение есть finding.
- При проверке checklist 5 (явный `reason` для каждого caller) — найти все call sites через grep по `resetConversation(`, перечислить в отчёте, проверить каждый.
- Для checklist 2 (mutex discipline) — особое внимание новым ветвям в `DefaultModelRegistry.resetConversation`: success-path log и non-Ready warning должны находиться внутри `withLock`, иначе порядок логов может расходиться с фактическим порядком операций.
- Severity guideline: BLOCKER = инвариант сломан / merge-блокер; MAJOR = функциональная регрессия или нарушение spec; MINOR = качественное замечание не блокирующее; NIT = вкусовщина.
