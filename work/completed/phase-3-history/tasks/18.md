### B1 — Вложения не рендерятся в истории persistent-чата

**Repro:** Открыть persistent-чат → прикрепить картинку к первому сообщению в drawer-created draft → Send → перейти в persistent-чат → картинки в user-bubble **нет**, хотя модель её обработала (text ответа про картинку).

**Root cause — зафиксировано чтением кода:** `ChatViewModel.toDomainMessage` (строка 1241) конструирует `Message(..., attachments = emptyList())` **hardcoded**, полностью игнорируя `MessageEntity.imagePath` и `MessageEntity.audioPath`. Task 17 закрыл путь disk-write (staging rename → attachments/{chatId}/), но обратный путь **disk → Bitmap → UI** не реализован. `MessageBubble.MessageAttachmentsRow` рендерит `attachment.bitmap` прямым `Image()` — а `attachments` приходит пустым → ничего не рисуется.

**Fix scope:** `toDomainMessage` должен лениво декодировать `imagePath` в `Attachment.Image(bitmap)` через `BitmapFactory.decodeFile(File(filesDir, imagePath))` и `audioPath` — в `Attachment.Audio(pcm, durationMs)` путём обратного WAV→PCM read. Запись в staging кладёт именно `pcmToWav(pcm, SAMPLE_RATE)`, значит чтение — через `wavToPcm` (либо чтение WAV-заголовка + остаток как PCM byte array). Decode **обязательно** на `Dispatchers.IO` и **лениво** — иначе каждый emit `messageDao.observeByChat()` декодит все картинки синхронно в UI-thread. Варианты: (а) кэш `Map<MessageId, Attachment>` в VM, (б) `Flow.map { ... }.flowOn(Dispatchers.IO)` над `messageDao.observeByChat()` до попадания в `combine`. (б) проще, но декодит всё подряд на каждом emit; (а) эффективнее. Выбирает исполнитель по месту — главное чтобы **картинка появлялась в UI после захода в чат с переоткрытия / cold-start / сразу после first-send**.

### B2 — Переинициализация модели между чатами крутится бесконечно

**Repro:** Открыт чат на модели A (Ready) → выйти на Home → drawer «Новый чат» → в model picker TopAppBar выбрать модель B → cross-model dialog → OK → спиннер «Загружается B» **не останавливается**.

**Почему follow-up #4 (`58bb44e`) не решил:** тот коммит починил `DefaultModelRegistry.initialize(B)` для случая «A Ready → B Initializing»: теперь A освобождается перед флипом target'а на B. Но **цена** — между `releaseEngine(A)` и `engine.create(B)` сейчас под `lifecycleMutex` идёт длинная последовательность: native dispose A → native init B. В то же время `WarmupCoordinator.launchWarmup(modelId)` держит `restartMutex`, а `registry.initialize` внутри берёт `lifecycleMutex`. При cross-model switch из открытого чата draft-режим вызывает `warmupCoordinator.cancelAndRestart(B)` через `bootstrapChatModelId` (путь для Draft есть для explicit modelId). Гипотеза: **либо** одна из мьютексных lock-цепочек не отпускает, **либо** `registry.initialize` висит на native dispose старой модели дольше, чем таймаут UI-ожидания, **либо** `isWarmupInProgress` stays true, но `initStatus` не переходит в `Ready` потому что бывшая Ready A не попала в scope `models.filter { Ready && name != target }` в новом коде.

**Диагностический план (в рамках задачи, не отдельно):**
1. Добавить трассировку `errorLog.d`/`errorLog.e` временную на входах/выходах: `WarmupCoordinator.launchWarmup(modelId=X) START/END`, `DefaultModelRegistry.initialize(name=Y) START/END`, `releaseEngine(name=Y) START/END`. Прогнать на Honor 200 A→Home→Drawer→new chat B с model picker.
2. Посмотреть `errors.log` (через `adb shell run-as ... cat files/errors.log` или через About → Диагностика → Сохранить лог).
3. По трассе локализовать зависание — это **один из** трёх сценариев:
   - **Deadlock mutex:** `launchWarmup(B)` держит `restartMutex`, `cancelAndJoin` ждёт прошлый Job, прошлый Job внутри `registry.initialize(A.name)` держит `lifecycleMutex` и блокирует новый `initialize(B.name)` изнутри нового `launchWarmup`.
   - **Slow native dispose:** `releaseEngine(A)` честно отрабатывает 30+ сек и UI всё это время в Loading — фикс через прогресс-индикатор «Выгружаю A…», не bug сам по себе.
   - **Missed release path:** `58bb44e` чистит только перед Initializing-флипом — если B уже был в Initializing из-за предыдущего отменённого warmup, A так и остаётся Ready → inference блокирован. Логика release должна сработать **в каждом случае cross-model initialize**, не только при первом флипе.

**Fix scope** — зависит от того, что покажут логи; в AC фиксируем **поведенческое** требование: «последовательность A Ready → cross-model switch на B → завершается success (B Ready) или failure (`ModelInitStatus.Failed`) за ≤60 сек, а не висит бесконечно». Если окажется, что это scenario 2 (slow native dispose), часть AC — прогресс-индикатор, см. B2-UX ниже. Если scenario 1 — реорг mutex discipline (см. Implementation hints). Если scenario 3 — пересмотр single-engine release path.

**B2-UX (независимо от root cause):** TopAppBar в cross-model switch сейчас показывает `ChatUiState.Loading` spinner без текста. Добавить подпись «Загружаю {B.name}…» + опционально «Выгружаю {A.name}…» — чтобы юзер видел прогресс, а не «висячее колесо». Это отдельный мелкий UX-фикс в рамках этой же задачи.

### B4 — Первое сообщение в новом persistent-чате не обрабатывается моделью

**Repro:** Drawer → «Новый чат» → route `chat/draft` → печатаю «привет» → Send → переход на `chat/{chatId}` → **USER-сообщение видно, ASSISTANT ответа нет, inference не запущен**. Фактически first-send «проедается» commit-ом и всё.

**Root cause — зафиксировано чтением кода:** `ChatViewModel.commitDraft` (строка 774) на `onSuccess` вызывает `_navigation.tryEmit(NavigateToPersistent(chatId))` и **не запускает inference**. Draft-VM живёт до того как NavHost не пересоздаст `ChatScreen` под новый route. Persistent-VM бутстрапится свежим, в `bootstrapChatModelId` (строка 684) для `ChatIdentity.Persistent` читается `chat.modelId` из Room и... всё. Нет логики «если последнее сообщение USER без парного ASSISTANT → запусти inference». Декомпозиция Task 8 покрывала runInferencePersistent для **уже-открытого** persistent-чата при Send, но не покрыла **handover first-send из draft**: Draft VM пишет USER в Room через `commitDraftChat`, умирает, Persistent VM не знает, что надо что-то запускать.

**Fix scope — два варианта, авторский выбор внутри задачи:**

1. **Вариант A (auto-resume в Persistent VM):** в `bootstrapChatModelId` для `ChatIdentity.Persistent` после установки `_chatModelId` и `applyEffectiveConfigToModel` — дополнительно прочитать последнее сообщение (`messageDao.lastByChat(id.id)` — добавить если метода нет); если `role == USER` и нет следующего `ASSISTANT` → enqueue `runInferencePersistent(identity, text=lastMessage.text, pending=emptyList())` после того как engine станет Ready (подписка на `observeEngineState` → при первой `Ready` → resume once, используя флаг `autoResumeAttempted`). **Плюс:** покрывает и first-send-из-draft, **и** AC-R3 kill-mid-stream resume (уже упомянут в project knowledge как паттерн).
2. **Вариант B (inference в Draft VM до navigate):** в `commitDraft.onSuccess` **перед** `tryEmit(NavigateToPersistent)` запустить `runInferencePersistent` внутри Draft-VM на newly-committed chatId. Тогда Draft-VM продолжает жить до завершения inference, а Persistent VM на следующем заходе подхватывает готовую пару USER+ASSISTANT из Room. Риск: Draft-VM технически уже в режиме Persistent (chatId есть), identity при этом не меняется — нужна аккуратная передача identity. **Минус:** double-streaming-bubble risk если юзер вернётся в drawer и снова откроет этот чат пока inference идёт в Draft VM.

**Рекомендуется Вариант A** — он однородно закрывает оба сценария (draft first-send и AC-R3 kill-mid-stream resume) и живёт в одной точке.

На `null`/fail — log `errorLog.e("attachment-read", "failed to decode $imagePath", cause)`, attachment просто не появится (gracefully degrade — текст сообщения сохраняется).

Переиспользовать/создать `wavToPcm(File): Pair<ByteArray, Long>` в `core-runtime/common` — симметрично существующему `pcmToWav`.

- **Containment-check на чтение**: `resolvedPath.canonicalPath.startsWith(File(filesDir, "attachments").canonicalPath + File.separator)` — та же защита что в записи. Path traversal через `../../secret` в `imagePath` (теоретически возможен если Room row подменён) → `SecurityException` → log → attachment не показывается. Это требование **security-auditor** будет проверять.

### Implementation hints

- **B1 `wavToPcm`:** WAV-заголовок фиксированной длины 44 байта для PCM 16-bit mono — `file.readBytes().drop(44).toByteArray()`. Validate RIFF magic, fail-fast с `errorLog.e("attachment-read", "not a WAV: ...")`. `durationMs = (pcm.size / 2) * 1000L / SAMPLE_RATE` (16-bit = 2 bytes per sample).
- **B1 lazy-decode vs eager:** eager проще и корректнее (Attachment — immutable data с bitmap-ом, уже так устроено для Quick-режима). На десяти картинках в истории памяти съестся ~10×(1024×1024×4B) = ~40 MB — допустимо. Если в будущем будет проблема — lazy через `LruCache<MessageId, Attachment>` в VM.
- **B2 mutex хирургия (scenario 1):** вынести `registry.initialize` за пределы `restartMutex.withLock` через pattern: `restartMutex.withLock { cancelPrevJob(); assignNewJob() }` + job делает `initialize` без mutex. `isWarmupInProgress` читается/пишется через `MutableStateFlow` (thread-safe), без mutex. Single-engine invariant обеспечивается тем, что в любой момент есть не более одного Ready entry (release prior Ready внутри `initialize` через `58bb44e`).
- **B2 UI label:** компоновка в `ChatScreen.TopAppBar` — нужен `currentModelName` из `registry.models.firstOrNull { it.model.modelId == _chatModelId.value }?.model?.name`. Форматирование строки: `stringResource(R.string.chat_topbar_loading_model, modelName)` с `<string name="chat_topbar_loading_model">Загружаю %1$s…</string>`.
- **B3 label на Home:** `HomeViewModel.defaultModelName: StateFlow<String?>` через `combine(appSettings.observeDefaultModelId(), registry.models) { id, models -> models.firstOrNull { it.model.modelId == id }?.model?.name }.stateIn(...)`. В HomeScreen показать под `home_title` второй `Text(style = labelSmall, colorScheme.onSurfaceVariant)` с `Modifier.clickable { onOpenModelManager() }`.
- **B4 `autoResumeTarget` / `autoResumeAttempted`:** приватные `var` в `ChatViewModel`, защищены тем что VM single-threaded (Main dispatcher для mutations). Observe Ready→auto-resume через отдельную `launch { observeFirstReadyThenResume() }` в `init`, которая ждёт `_uiState.first { it is ChatUiState.Ready }`, делает проверку и dispatch.
- **B4 `MessageDao.lastByChat`:** если уже есть `getFirstByRoleDesc` или аналог — переиспользовать. Иначе добавить `@Query("SELECT * FROM messages WHERE chat_id = :chatId ORDER BY created_at DESC, id DESC LIMIT 1") suspend fun lastByChat(chatId: Long): MessageEntity?`. Secondary sort по `id DESC` для детерминизма при одинаковом `created_at`.
- **B5 footer layout:** `ModalDrawerSheet` → `Column(fillMaxSize)` → `[DrawerHeader] [Divider] [Box(weight=1f) { scrolling list OR empty state }] [Divider] [Footer]`. `NavigationDrawerItem` из `androidx.compose.material3` имеет стандартный accessibility + ripple. Не самодельный `Row`.
