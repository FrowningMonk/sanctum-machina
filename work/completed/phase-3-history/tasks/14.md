# Task 14: Security Audit

## Required Skills

Before starting, load:

## Description

Full security audit of all source files created and modified in Phase 3. This task is the security gate before the final QA wave (Task 16). The auditor IS the review — there are no separate reviewers; the audit report itself is the deliverable.

Phase 3 introduces significant new attack surface compared to Phase 2.5: a Room database, file-system paths derived from user-controlled data (`chatId`), a quick-chat incognito guarantee backed by filesystem operations, a cross-process crash-state singleton backed by files, and a DataStore migration that rewrites all per-model settings on first run. Any vulnerability in these surfaces could break privacy guarantees (quick-chat leaking to permanent storage), expose the app's data store to other apps, or silently corrupt settings on migration.

Seven specific focus areas are defined in the tech-spec and must each be audited:

1. **File-path injection** — `filesDir/attachments/{chatId}/` path construction in `DefaultChatRepository`. A `chatId` is an autogenerated `INTEGER PRIMARY KEY AUTOINCREMENT` from Room, so it is numeric. However the audit must verify: (a) the path is constructed with `File(filesDir, "attachments/$chatId")` (not string interpolation of an unvalidated user input), (b) there is no way for a caller to pass an arbitrary string where `chatId` is expected (function signature types `Long`, not `String`), and (c) the staging path `.staging-{uuid}` where `uuid` comes from `UUID.randomUUID().toString()` cannot be manipulated.

2. **SQL injection surface** — Room DAOs (`ChatDao`, `MessageDao`) must use only `@Query` annotations with parameterized `?` or `:param` binding. Any raw string concatenation in query strings is a finding. Room's compile-time query validation is a partial mitigation but does not cover runtime-built queries (if any).

3. **Privacy guarantee: quick/ purge** — `SanctumApplication.onCreate` purges `filesDir/quick/` on every startup. The audit must verify: (a) `deleteRecursively()` is called (not just directory-entry removal), (b) the purge is wrapped in `try/catch` with `ErrorLog.e("attachment-save", ...)` on failure — per the tech-spec a silent failure breaks the incognito guarantee, (c) there is no code path that writes quick-chat content outside `filesDir/quick/`.

4. **PII in logs** — `ErrorLog.e(component, description, cause)` entries written from Phase-3 code paths (engine-warmup, history-read, history-write, attachment-save, attachment-read, settings-io) must not include message text, user prompts, image/audio content, or any chat data. The `description` and `cause.message` fields are bounded (500/200 chars per `patterns.md`), but bounding alone doesn't prevent PII inclusion. Check all `ErrorLog.e` call sites added in Phase 3.

5. **Attachment decode security** — when a `MessageEntity.image_path` or `audio_path` is loaded on chat reopen, the file is decoded into a `Bitmap` or PCM buffer. Verify: (a) there is format validation or at least a `try/catch` around decode, (b) a corrupt/crafted file cannot crash the process silently, and (c) the file is read from app-private storage only (no external URI traversal).

6. **Cross-process crash state** — `AppCorruptionState` singleton signals DB corruption via an in-memory flag (`corruptionOccurred: Boolean`). The backup file `sanctum.db.corrupt_{ts}` sits in `filesDir` (app-private, mode 0600, not accessible to other apps). Verify: (a) `AppCorruptionState` is only read/written within the main process (not exported via `ContentProvider` or `Intent`), (b) the corrupt backup file is created with default `Context.openFileOutput` permissions (not world-readable), (c) there is no information disclosure path from crash state to untrusted callers.

7. **DataStore migration atomicity** — `SettingsMigrationHelper.migrateIfNeeded()` reads all `per_model_overrides` keys, remaps `Model.name → Model.modelId`, drops orphan keys, and writes results in a single `DataStore.updateData { ... }`. Verify: (a) the sentinel `settings_keys_migrated` is read before migration and set after in the same `updateData` block (or in a follow-up `updateData` call that is correctly sequenced), (b) a crash mid-`updateData` cannot leave a partial state (DataStore guarantees atomicity per the spec, but verify the implementation does not manually merge partial results), (c) orphan key logging does not write the key value (which could be a model name as user-visible string) beyond what is safe.

## What to do

1. Read all Phase-3 source files: every file listed in the tech-spec "Files to modify" and "Files to create" sections across Tasks 0–12 (the full set is enumerated in the Details section below). Start with the highest-risk files: `DefaultChatRepository.kt`, `ChatDao.kt`, `MessageDao.kt`, `SanctumApplication.kt`, `SettingsMigrationHelper.kt`, `AppCorruptionState.kt`.

2. For each of the seven focus areas, produce a verdict: **PASS**, **FINDING**, or **N/A** with evidence. A FINDING requires: severity (Critical / High / Medium / Low / Info), the exact file and line, a description of the risk, and a concrete fix recommendation.

3. Also audit for OWASP Mobile Top 10 issues not covered by the seven focus areas — specifically: insecure data storage (are any sensitive values written outside `filesDir`?), improper platform usage (Intent extras, exported components), insufficient cryptography (is the HF token stored via `androidx.security.crypto` as required?).

4. Verify the `SafeMarkdown` / `SafeUriHandler` pattern is used everywhere LLM output is rendered — that no new `RichText`/`Markdown` direct calls were added in Phase-3 UI files.

5. Verify the `android:allowBackup="false"` and `dataExtractionRules` manifest entries are unchanged (regression guard from `patterns.md` Privacy hardening section).

6. Write the audit report as the task's deliverable. Report format: executive summary (1–3 sentences), findings table (severity / area / file:line / description / fix), then per-area narrative. Save the report to `work/phase-3-history/logs/working/task-14/security-audit-report.md`.

**Key security invariants from the spec (must hold):**

- `chatId` is always `Long` (autogenerated PK) — not a user-supplied string. Path construction via `File(filesDir, "attachments/$chatId")` with a numeric type is safe, but verify the type is not widened to `String` anywhere.
- `filesDir/quick/` purge must call `deleteRecursively()` — not just `delete()` (which only removes empty directories). If a non-empty `quick/` directory is not wiped, a previous quick-chat's files survive in incognito mode.
- `ErrorLog` description/cause are bounded at 500/200 chars, but the auditor must check that no chat message text is concatenated into those fields, even within bounds.
- `AppCorruptionState` is an in-memory singleton. It must not be exposed via `Intent`, `ContentProvider`, or `BroadcastReceiver`. The `.corrupt_*` backup file must be in `filesDir` (app-private, not readable by other apps without root).
- The `SafeUriHandler` pattern (`SafeMarkdown`) must be applied to all new Compose surfaces that render LLM output — `DrawerContent` (chat titles from Room), `HomeScreen`, any new `MessageBubble` usages. Chat titles are user/LLM-generated and could contain markdown links.
- `SettingsMigrationHelper` sentinel check: the `settings_keys_migrated` flag must be read atomically before deciding to run migration. Verify that a concurrent second startup cannot trigger migration twice.
