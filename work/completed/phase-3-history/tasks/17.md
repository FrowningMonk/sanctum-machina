# Task 17: Attachment staging (Draft→Persistent file atomicity)

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Закрывает Phase-3 **debt 2**, оставленный Task 8 (см. `decisions.md` → Task 8 → Deviations → "Draft-attachment persistence отложен").

Сейчас `ChatViewModel.commitDraft` передаёт `stagingDir = null` в `chatRepository.commitDraftChat(...)`, и вложения, прикреплённые в Draft-режиме, остаются в памяти (`_attachments`) до Send, а после Send теряются вместе со смертью Draft-VM при `NavigateToPersistent`. Для пользователя это **потеря данных**: прикрепил картинку к первому сообщению → после перехода в персистентный чат картинки нет.

Задача реализует модель staging из `tech-spec.md` → **Decision 6** (она же сценарий, который уже поддерживает `DefaultChatRepository.commitDraftChat` — staging path был написан в Task 4, но никем не подаётся):

1. **Прикрепление в Draft** → файл **сразу** пишется в `filesDir/attachments/.staging-{uuid}/` на `Dispatchers.IO`.
2. **`removeAttachment`** → файл удаляется с диска параллельно со снятием из `_attachments`. Нет "полу-состояний" — юзер всегда видит синхронизированное представление.
3. **`commitDraft`** → передаёт `stagingDir` в `commitDraftChat`. Внутри Room-транзакции происходит `rename(.staging-{uuid}, attachments/{chatId}/)` — **атомарная** операция уровня файловой системы: либо все файлы попали под новый `chatId`, либо Room транзакция откатилась и staging-папка удалена.
4. **`MessageEntity.imagePath`/`audioPath`** заполняются финальными относительными путями (`attachments/{chatId}/{filename}`) **внутри** той же транзакции, которая делает rename — Room-строка и диск согласованы on-commit.
5. **Persistent mode** (последующие сообщения в уже-созданном чате) → staging не нужен, пишем прямо в `filesDir/attachments/{chatId}/` через новый `chatRepository.savePersistentAttachment(chatId, attachment)`. Partial-fail тут не критичен: `chatId` стабилен, sweep на startup подметёт orphan файлы.

### Почему variant A (staging), не variant B (in-memory до Send)

Обсуждалось с пользователем. Решающие аргументы:
- **A даёт atomicity**: rename — одна syscall, kill в середине невозможен. B — серия write'ов + update, при kill возможна half-write (UI показывает сообщение без части картинок).
- **A поддерживает "cancel removes file"**: `removeAttachment` удаляет реальный файл, нет призраков ни на диске, ни в памяти. В B при cancel нечего удалять, но при крашах сироты возникают чаще.
- **Гиганты (ТГ, WA) делают А-подобный pattern** — локальный cache + upload queue; мы без сервера, поэтому staging и есть наша "local cache".

## What to do

1. **`ChatRepository` interface**: добавить два метода.
   ```kotlin
   /**
    * Write [attachment] into [stagingDir], creating the dir lazy-if-missing.
    * Returns the filename (relative to `stagingDir`) under which the payload
    * was stored. Caller must remember the `stagingDir` for `commitDraftChat`.
    */
   suspend fun writeAttachmentStaging(
     stagingDir: File,
     attachment: Attachment,
   ): String

   /**
    * Direct write for already-committed persistent chats. Writes into
    * `filesDir/attachments/{chatId}/`, returns `(imagePath, audioPath)` —
    * exactly one non-null per call (mirrors [MessageEntity] schema).
    */
   suspend fun savePersistentAttachment(
     chatId: Long,
     filesDir: File,
     attachment: Attachment,
   ): PersistedAttachment
   ```
   `PersistedAttachment` — тонкий data-класс `data class PersistedAttachment(val imagePath: String? = null, val audioPath: String? = null)` для симметрии с `MessageEntity`.

2. **`DefaultChatRepository`** реализация:
   - `writeAttachmentStaging`: валидирует, что `stagingDir` лежит под `filesDir/attachments/` (тот же containment-check, что уже делает `commitDraftChat` для безопасности); `mkdirs`; имя файла — `img_{N}.png` / `audio_{N}.wav` где `N` — номер файла в папке для предсказуемости (не uuid — удобнее при чтении логов). `Attachment.Image.bitmap` сериализуется через `Bitmap.compress(PNG, 100)`. `Attachment.Audio.pcm` оборачивается через уже существующий `pcmToWav(pcm, SAMPLE_RATE)` — финальный формат .wav, играбельный любым плеером. На сбой write → удалить частично-записанный файл + пробросить `IOException`.
   - `savePersistentAttachment`: создаёт `filesDir/attachments/{chatId}/`, пишет по той же схеме; имя файла — next-index в папке (читаем `list().size`). Возвращает `PersistedAttachment` с относительным путём `attachments/{chatId}/{filename}`.

3. **`ChatViewModel` — Draft flow**:
   - Добавить `private var draftStagingDir: File? = null` — lazy создаётся при первом `addImage*/addAudio`, UUID-based имя: `File(context.filesDir, "attachments/.staging-${UUID.randomUUID()}")`.
   - В `addImages`/`addImageBitmap`/`addAudio`: после того как attachment создан в памяти — запланировать write в staging через `chatRepository.writeAttachmentStaging(draftStagingDir!!, attachment)` на `viewModelScope.launch`. Результат (имя файла) сохранить в новом поле `Attachment.stagedFilename: String?` (добавить в `Attachment.Image`/`Attachment.Audio` sealed class) или параллельный `Map<Attachment, String>` в VM — на усмотрение, но должно однозначно связывать memory-representation с disk-representation.
   - `removeAttachment(idx)`: если `Attachment` имеет `stagedFilename`, удалить `File(draftStagingDir, filename)` на `Dispatchers.IO`; логировать `errorLog.e("attachment-save", ...)` при сбое delete (не блокирующе — snackbar не нужен, юзер уже нажал remove).
   - `commitDraft`: передаёт `draftStagingDir` (если non-null) в `chatRepository.commitDraftChat(modelId, firstMessage, stagingDir = draftStagingDir, filesDir = context.filesDir)`. **Важно**: `firstMessage` нужно расширить — сейчас `MessageEntity` для Draft идёт с `imagePath = null, audioPath = null`. При передаче staging нужно, чтобы `firstMessage` нёс имена файлов в staging (relative к staging dir) или — проще — чтобы `commitDraftChat` сам знал, как пометить message с первым stagingImage/Audio. Самый чистый вариант: `firstMessage` в `commitDraft` строится с путями `attachments/{chatId}/{stagedFilename}` **до** вставки — т.к. `chatId` узнаём только внутри транзакции, это делается через callback/newInsertChain. Альтернатива — перенести построение `firstMessage` внутрь `commitDraftChat` (API evolution): передавать `attachmentFilenames: List<StagedFile>` вместо готового `MessageEntity`.
   - После успешного commit: `draftStagingDir = null`, `_attachments.value = emptyList()`, эмитится `NavigateToPersistent(chatId)` (как сейчас).
   - На cleanup VM (если draft так и не коммитнулся): ничего делать не нужно — `SanctumApplication` orphan-sweep подметёт `.staging-*` на следующем старте (AC-A3 уже в Task 6).

4. **`ChatViewModel` — Persistent flow** (`runInferencePersistent`):
   - Перед `savePersistentMessage(userMsg)` — если `_attachments.value` non-empty, вызвать `chatRepository.savePersistentAttachment(identity.id, context.filesDir, attachment)` для каждого. Собрать путь(и) в одно/несколько `MessageEntity` (user-spec Decision 6: одна message-строка может нести одну картинку ИЛИ одно аудио, не несколько — если больше одного вложения, пишем несколько message-строк с одинаковым `createdAt`). **Уточнение**: вопрос нужен ли многострочный подход или правильнее положить первое вложение в USER message, остальные игнорировать (MVP) — решить по user-spec; предпочтительно MVP = one attachment per message (текущий Attachment-UI в Phase 2 уже так работает фактически — single image + single audio per send).
   - На failure `savePersistentAttachment` → логировать `errorLog.e("attachment-save", ...)`, **прервать inference** (аналогично hard-gate на USER-persist failure, security-auditor-2 s4) — Ready(isGenerating=false), snackbar `chat_load_failed_title`.

5. **`Attachment` API evolution**: в `Attachment.Image` / `Attachment.Audio` добавить `val stagedFilename: String? = null` ИЛИ хранить маппинг в VM (`Map<Attachment, StagedFile>`). Предпочтительно — первый вариант (immutable data class с новым полем), чтобы не тащить параллельную structure. Если `stagedFilename == null` — attachment ещё не записан на диск (ещё идёт async write), Send не должен стартовать: в `send()` добавить gate — если в `_attachments` есть `Attachment.Image`/`Audio` с `stagedFilename == null`, показать snackbar "Вложение ещё сохраняется" и return.

6. **Ошибки write в staging**: если `writeAttachmentStaging` падает (disk full, permission) — убрать attachment из `_attachments` (он не был сохранён), показать snackbar `attachment_save_failed` (новая строка в `strings.xml`), логировать `errorLog.e("attachment-save", ..., cause)`.

### Edge cases

- **Concurrent addImage + send**: юзер прикрепляет → сразу жмёт Send, пока write ещё в полёте. Send gate на `stagedFilename == null` блокирует. Snackbar "Вложение ещё сохраняется".
- **Staging write fails (disk full)**: attachment снимается из `_attachments`, snackbar `attachment_save_failed`. Inference не стартует потому что юзер не успел нажать Send.
- **App killed в Draft**: staging dir сирота. Startup sweep (Task 6) удалит.
- **App killed во время commitDraftChat**: внутри Room `@Transaction` — откатится, rename не произошёл. Staging dir с файлами остаётся, sweep удалит. Chat в Room отсутствует.
- **App killed после commit, до `NavigateToPersistent`**: chat в Room есть, файлы под `{chatId}/`, UI был в Draft → при следующем запуске юзер видит новый persistent chat в drawer с первым сообщением + картинкой. Один из вариантов "honest state".
- **Persistent `savePersistentAttachment` fails**: hard-gate abort — `runInferencePersistent` не делает `savePersistentMessage(USER)` и не стартует инференс. Snackbar, Ready(isGenerating=false).
- **Remove во время concurrent write в staging**: `removeAttachment` удаляет файл; если `writeAttachmentStaging` был в полёте и пишет тот же файл — последний write выиграет, файл может остаться. Принять как edge-case: startup sweep уберёт при следующем запуске. Альтернатива — cancel coroutine job по `stagedFilename`, но усложнение ради редкого race.
- **Audio .wav vs PCM**: пишем .wav (обёрнутый через `pcmToWav`) — файл самодостаточный, играбельный любым плеером. Формат `pcmToWav` — 16-bit PCM, mono, 16kHz (SAMPLE_RATE из core-runtime/common).
- **Filename collision**: следующий index = `list().size` в папке staging. Collision в staging невозможен (папка уникальная на VM). В persistent dir — если два параллельных `savePersistentAttachment` возьмут одинаковый индекс — race; mutex или UUID в filename решают. Для MVP: `System.nanoTime()` в filename — простое решение без блокировок.

### Implementation hints

- `draftStagingDir` создаётся lazy в приватном helper `ensureDraftStagingDir(): File` — только на первый `addImage*/addAudio`. Не создавать если юзер ничего не прикрепил — orphan cleanup меньше.
- Для `Attachment.Image` `Bitmap.compress(PNG, 100, outputStream)` — PNG без потерь, размер стерпит. JPEG quality-85 дал бы ~5× экономию, но Phase 3 не оптимизирует размер — приоритет целостности.
- Для `Attachment.Audio` использовать `pcmToWav(pcm, SAMPLE_RATE)` — получаем ByteArray с RIFF-хедером, пишем в файл одним вызовом.
- `removeAttachment` — delete файла на `viewModelScope.launch(Dispatchers.IO)`; log errorLog.e на сбой, но не блокировать UI. Юзер уже нажал remove — пусть исчезнет из списка сразу.
