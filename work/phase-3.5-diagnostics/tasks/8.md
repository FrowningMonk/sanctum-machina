# Task 8: DiagnosticsScreen + ViewModel + Drawer pin + AboutScreen refactor

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Финальная UI-волна Phase 3.5. Эта задача доставляет фактический пользовательский intake фичи диагностики: новый экран «Диагностика» (две секции — RAM и Логи), пин в drawer'е (третий пункт между «Модели» и «О приложении»), и зачистка экрана «О приложении» от секции диагностики.

Задача состоит из трёх связанных, но логически отдельных кусков:

1. **`DiagnosticsScreen` + `DiagnosticsViewModel` (новые)** — экран, читающий `DiagnosticsState.lastInitSnapshot()` (через `InitDiagnostics` seam — см. Task 4) и `DeviceInfoProvider.availableMemoryBytes()` (Task 7) с 1-секундным тиком в `viewModelScope`, авто-отменяемым в `onCleared()`. Рендер snapshot'а — четыре варианта по Decision 12: `Ok`, `Failed`, `null` («пока не было»), и явный `InProgress` с маркером «инициализация» (без слова «ok»/«ошибка»). SAF-launcher для экспорта `.txt`-лога мигрирует целиком из `AboutScreen` без дублирования (Decision 11).
2. **Drawer pin** — третий `NavigationDrawerItem` «Диагностика» в `DrawerFooter` между «Модели» и «О приложении», новый параметр `onNavigateToDiagnostics`, новая навигационная route `composable("diagnostics")` в `SanctumApp.kt` сразу после `about`-route.
3. **AboutScreen refactor** — удаляется секция «Диагностика» (заголовок + кнопка «Сохранить лог»), удаляется SAF-launcher state и `launching` flag. Сохраняются: `SafeMarkdown`, `AboutFooter`, `tapCounter` + 7-tap dev-crash-dialog (R-5 в interview), `AlertDialog`, `Modifier.clickable` на version-label. `AboutViewModel.buildAndWrite` удаляется, если у `AboutScreen` после рефакторинга больше нет вызовов (если `AboutViewModel` остаётся пустым — оценить, оставлять ли его как future-hook или удалить вместе с DI-биндингом).

**Зависимости:**
- **Task 4** даёт `DiagnosticsState` (singleton с `AtomicReference<InitSnapshot?>`) — это reader-side seam для VM.
- **Task 5** даёт `formatGbFloor` — переиспользуемый GB-floor формат-хелпер для строк `freeRamBytes → "X.X ГБ"` (last-init вариант + текущая free-RAM строка). Wave 2 завершается до wave 3, так что Task 5 готов до старта Task 8.
- **Task 7** стабилизирует `DeviceInfoProvider` — VM использует `availableMemoryBytes()` для 1-секундного тика свободной RAM, и (опционально) `lastInitSnapshot()` через тот же провайдер. Solver может перепроверить — формально VM может читать `DiagnosticsState` напрямую через Hilt, но отдавать оба source'а через `DeviceInfoProvider` единообразнее с `DeviceInfoCollector`. Решение по DI-инъекции — за solver'ом.

Per project memory «Verify UI chain before device smoke» — полный device-smoke на Honor 200 + matrix внешних устройств бундлится в Task 12 (Pre-deploy QA). Здесь — local-debug user-verify на Honor 200 разработчика.

## TDD Anchor

Тесты, которые нужно написать ДО реализации (pure-JVM, hand-rolled fakes — см. `architecture.md § Testing` + `patterns.md § Testing`):

- `app/src/test/kotlin/app/sanctum/machina/ui/diagnostics/DiagnosticsViewModelTest.kt::lastInit_whenSnapshotOk_rendersOkVariant` — fake `DiagnosticsState.lastInitSnapshot()` возвращает `InitSnapshot(modelName="E4B", freeRamBytes=3_500_000_000, atEpochMs=<fixed>, outcome=Ok)` → `state.value.lastInitText` содержит «3.2 ГБ», «<HH:mm от atEpochMs>», «E4B», «ok».
- `...::lastInit_whenSnapshotFailed_rendersErrorVariant` — `outcome=Failed` → строка содержит «ошибка».
- `...::lastInit_whenSnapshotNull_rendersNeverVariant` — `lastInitSnapshot() == null` → строка == «пока не было» (или равна `R.string.diagnostics_last_init_never`).
- `...::lastInit_whenSnapshotInProgress_rendersFourthVariant` — Decision 12 fixture: `Snapshot(modelName="Gemma-4-E4B-it", freeRamBytes=3_500_000_000, atEpochMs=<fixed>, outcome=InProgress)` → строка содержит подстроки «инициализация», «3.2 ГБ», «<HH:mm>», «Gemma-4-E4B-it» **И НЕ** содержит «ok» / «ошибка».
- `...::freeRam_tickEverySecond_refreshesValue` — закрепляем контракт. **Реализация (commit one shape):** `init { refreshFreeRam(); viewModelScope.launch { while (isActive) { delay(1_000); refreshFreeRam() } } }` — один синхронный read в `init`, далее N delay-driven read'ов после `advanceTimeBy(N * 1_000)`. **Тест:** `runTest` со `StandardTestDispatcher`, `viewModel = DiagnosticsViewModel(...)`, `advanceTimeBy(3_000); runCurrent()` → assert `fakeDeviceInfo.availableMemoryBytesCallCount == 1 + 3 == 4` (initial read + 3 tick-driven). Если solver выберет shape «первый read внутри loop'а» (нет initial read), counter == N, тест должен ассертить именно `N` — но реализация и тест должны совпадать. Recommendation: фиксируем shape с initial read (UI не ждёт первую секунду перед показом значения).
- `...::onCleared_cancelsPolling` — после cancellation `viewModelScope` (через `runTest` scope-cancel или прямой `viewModel.viewModelScope.cancel()` где это уместно) дальнейший `advanceTimeBy(5_000)` НЕ приводит к новым вызовам `availableMemoryBytes()` (counter в fake не растёт).
- `...::buildAndWrite_success_returnsResultSuccess` — НОВЫЙ тест в `DiagnosticsViewModelTest` (НЕ миграция — `AboutViewModelTest` в проекте отсутствует, проверено `Glob **/AboutViewModelTest.kt` → no results). Fake `LogExportManager.buildExport`/`writeTo` — happy path: `viewModel.buildAndWrite(uri)` возвращает `Result.success(Unit)`, fake-counter `writeTo`-вызовов == 1.
- `...::buildAndWrite_ioException_returnsResultFailure` — НОВЫЙ тест. Fake `LogExportManager.writeTo` бросает `IOException` → `Result.isFailure == true`, exception сохранён в `Result.exceptionOrNull()`.

Pipeline: написать тесты → запустить `./gradlew :app:test --tests "*DiagnosticsViewModelTest*"` → все падают (классы ещё не существуют) → реализовать VM → все проходят.

**Edge cases:**
- VM создаётся в момент, когда `lastInitSnapshot() == null` (нет завершённых init'ов в сессии) → render «пока не было». Если init происходит ПОКА экран открыт — snapshot обновится в `AtomicReference`, но VM-цикл читает только `freeRamBytes`. Решение: или (a) пуллить snapshot в том же 1-сек тике, или (b) `Flow`-обёртка над state. Solver фиксирует — Decision 12 в spec'е намекает на (a) (read on each composition / tick).
- 1-сек тик может стартовать до того, как Compose subscribe'нулся к `StateFlow` — `StateFlow` хранит latest, проблем не будет.
- Cancellation race в `onCleared`: `viewModelScope` отменяется автоматически Hilt-ViewModel'ью — `while (isActive)` корректно выходит на следующей итерации после cancellation. В тесте — `runTest`-scope cancellation через `cancel()` или `withTimeout`.
- `delay(1_000)` на `Dispatchers.Main` (default `viewModelScope`) — OK; `availableMemoryBytes()` JNI-call ≈ 50 µs (Decision 9 risk-row), не блокирует UI.
- SAF-launcher `null`-uri (юзер отменил picker) — `viewModel.buildAndWrite` не вызывается; `launching = false`. То же поведение, что в текущем `AboutScreen`.
- Если Decision 12 InProgress-рендер использует format-string `R.string.diagnostics_last_init_inprogress`, важно: НЕТ слов «ok» / «ошибка» в шаблоне — тестовый assert-`assertFalse(text.contains("ok"))` это ловит.
- Если иконка `Icons.Outlined.MonitorHeart` отсутствует в текущей версии `material-icons-extended` — fallback на `BugReport`. Solver проверяет компиляцией.
