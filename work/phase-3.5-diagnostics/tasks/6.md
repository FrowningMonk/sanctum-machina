# Task 6: Wire InitDiagnostics into DefaultModelRegistry.initialize

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Подключаем `InitDiagnostics`-seam (создан в Task 4) к единственному writer'у — `DefaultModelRegistry.initialize`. Registry становится источником снимков init-попыток: на старте каждой попытки фиксирует `freeRamBytes` (через `ActivityManager.MemoryInfo.availMem`), `atEpochMs` и `modelName`; на финале — outcome (Ok/Failed). GPU+CPU fallback считается ОДНОЙ попыткой (один `onInitStart`, один `onInitEnd`) — Decision 8.

Это — registry-side plumbing. Снимок пишется в `DiagnosticsState` (Task 4 impl `InitDiagnostics`) и потребляется `DeviceInfoCollector` шапкой `.txt`-экспорта (Task 7) и `DiagnosticsScreen` (Task 8). End-to-end-видимость снимка проверяется в Task 12 (Honor 200 smoke).

Inject-параметр добавляется в `@Inject`-конструктор — Hilt подставит через `@Binds InitDiagnostics: DiagnosticsState`, зарегистрированный в `DiagnosticsModule` (Task 4). Тестам нужен `RecordingInitDiagnostics` test-fake — он записывает все вызовы `onInitStart`/`onInitEnd` в `MutableList`, чтобы `DefaultModelRegistryTest` мог assert'ить call-order на success/CPU-fallback/failure-arm'ах.

## What to do

1. **Добавить параметр в конструктор `DefaultModelRegistry`:** новый `@Inject`-параметр `initDiagnostics: InitDiagnostics` (после `errorLog`, до `@ApplicationContext context` — порядок не критичен, выбери стилистически чистый).
2. **В `initialize(modelName)` под `lifecycleMutex.withLock`:**
   - **`onInitStart`-call:** ПОСЛЕ `releaseEngine(model)` (текущая строка 233) и ДО `try`-блока (текущая строка 242, перед первым `awaitInitialize` на 243): получить `ActivityManager` через `context.getSystemService(Context.ACTIVITY_SERVICE) as ActivityManager`, заполнить `ActivityManager.MemoryInfo()` через `am.getMemoryInfo(memInfo)`, вызвать `initDiagnostics.onInitStart(modelName, memInfo.availMem, System.currentTimeMillis())`.
   - **`onInitEnd(true)` GPU-arm:** ПОСЛЕ `updateEntry(...) { Ready }` и ПЕРЕД `return@withLock Result.success(Unit)` на success-arm'е (текущие 245-246).
   - **`onInitEnd(true)` CPU-fallback-arm:** ПОСЛЕ `updateEntry(...) { Ready }` и ПЕРЕД `return@withLock Result.success(Unit)` на CPU-fallback success-arm'е (текущие 258-259).
   - **`onInitEnd(false)` failure-arm:** ПОСЛЕ `errorLog.e(LOG_TAG_INIT, "CPU init failed: $err2")` (262) и `updateEntry(...) { Failed(err2) }` (263), ПЕРЕД `Result.failure(RuntimeException(...))` (264).
   - **`CancellationException` arm:** НЕ вызывать `onInitEnd` — отмена — это не «попытка завершилась с outcome». `Outcome.InProgress` остаётся в snapshot'е (фиксируем поведение явно — корректно по Decision 8: «один snapshot на init-attempt», cancellation = attempt не дошёл до финала).
3. **Создать `RecordingInitDiagnostics` test-fake:** новый файл `core-runtime/src/test/kotlin/app/sanctum/machina/core/registry/RecordingInitDiagnostics.kt`. Реализует `InitDiagnostics`. Хранит `starts: MutableList<Triple<String, Long, Long>>` и `ends: MutableList<Boolean>`. Методы добавляют записи. Доп. helper `callOrder()` возвращает упорядоченную последовательность типа `["start:modelA", "end:true"]` для assert'ов на порядок.
4. **Расширить `DefaultModelRegistryTest`:** добавить тест-кейсы (TDD anchor ниже) — каждый передаёт `RecordingInitDiagnostics` в конструктор registry, прогоняет `initialize(modelName)` и проверяет содержимое `starts`/`ends`/`callOrder()`. Существующие тесты, не зависящие от call-order, продолжают использовать `NoOpInitDiagnostics` (доступен из Task 4 в test-source-set).

**Edge cases:**
- **Cross-thread visibility:** registry работает на `Dispatchers.Default`; `DiagnosticsState` использует `AtomicReference` (Decision 7). Со стороны registry никакой дополнительной синхронизации не нужно — мы просто вызываем интерфейсный метод; гарантии visibility — обязанность impl'а.
- **`Context.ACTIVITY_SERVICE`:** константа в `android.content.Context`; `am` приведён к non-null `ActivityManager` (cast гарантированно успешен на Android — system service всегда зарегистрирован).
- **`MemoryInfo()` allocation:** аллоцируется один раз на init-attempt; Decision 9 — стоимость ≈50 µs, приемлемо в init-path (init длится 5-30 секунд по комментарию на 196-198).
- **`CancellationException` arm НЕ вызывает `onInitEnd`:** snapshot остаётся в `Outcome.InProgress`. Если юзер немедленно начнёт новый init (например, retry), `onInitStart` следующего attempt'а перезапишет snapshot — фиксированное поведение из Testing Strategy (`onInitStart` → новый `onInitStart` без `onInitEnd` → replace, не «pending»).
- **Нет race на writer-стороне:** `lifecycleMutex.withLock` гарантирует — между `onInitStart` и `onInitEnd` не приедет другой `onInitStart` от того же registry. Concurrency между несколькими registry-инстансами невозможна — `DefaultModelRegistry` помечен `@Singleton`.

**Implementation hints:**
- Чтение `availMem` — короткий блок:
  - получить `ActivityManager` через `context.getSystemService`,
  - создать `ActivityManager.MemoryInfo()`,
  - вызвать `am.getMemoryInfo(memInfo)`,
  - использовать `memInfo.availMem`.
  Не выноси в helper-метод (один call-site).
- Время — `System.currentTimeMillis()` (epoch ms, как в `InitSnapshot.atEpochMs`).
- Используй уже существующий `@ApplicationContext context` (line 105) — никакого нового inject (Decision 9).
- Не путай `availMem` (свободная RAM сейчас) с `totalMem` (вся RAM устройства) — для snapshot'а нужен именно `availMem` (соответствует `freeRamBytes` в `InitSnapshot`).
