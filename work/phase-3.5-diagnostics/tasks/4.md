# Task 4: InitDiagnostics seam + DiagnosticsState + Hilt module

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Создать чистый interface seam между `:core-runtime` (где живёт `DefaultModelRegistry` — будущий writer) и `:app` (где живёт UI-state — будущие readers). Этот seam держит модульную границу: `:core-runtime` не должен импортировать `:app`-пакеты (см. tech-spec «Verification Steps» и `architecture.md § Module graph`).

В этой задаче — только seam и его реализация: интерфейс `InitDiagnostics` в `:core-runtime/registry/`, `data class InitSnapshot` + `enum Outcome` в `:app/diagnostics/`, `@Singleton class DiagnosticsState : InitDiagnostics` с `AtomicReference<InitSnapshot?>` в `:app/diagnostics/`, Hilt-module с `@Binds` в `:app/diagnostics/di/`, `NoOpInitDiagnostics` тестовый фейк в `:core-runtime/src/test/`. Никаких consumer'ов (writer'а из `DefaultModelRegistry` — Task 6; reader'ов из `DeviceInfoCollector` — Task 7; reader'ов из `DiagnosticsViewModel` — Task 8) в этой задаче нет.

Рациональ выбора паттерна — Decision 6 (interface seam, тот же приём что `LogExportModule` использует для `CommandRunner`/`DeviceInfoProvider`); рациональ выбора `AtomicReference` — Decision 7 (нужна атомарность read-modify-write в `onInitEnd`, чтобы reader никогда не видел частично-обновлённый `InitSnapshot`).

## What to do

1. Объявить `interface InitDiagnostics` в `:core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/InitDiagnostics.kt` с двумя методами: `onInitStart(modelName: String, freeRamBytes: Long, atEpochMs: Long)` и `onInitEnd(success: Boolean)`. Без `@Inject`-аннотаций — это interface, биндится в Hilt-графе через `@Binds` в `:app`.
2. Объявить `data class InitSnapshot(val modelName: String, val freeRamBytes: Long, val atEpochMs: Long, val outcome: Outcome)` и `enum class Outcome { InProgress, Ok, Failed }` в новом файле `app/src/main/kotlin/app/sanctum/machina/diagnostics/InitSnapshot.kt`. Точная сигнатура — из tech-spec § Data Models.
3. Реализовать `@Singleton class DiagnosticsState @Inject constructor() : InitDiagnostics` в `app/src/main/kotlin/app/sanctum/machina/diagnostics/DiagnosticsState.kt`. Внутри — `private val ref = AtomicReference<InitSnapshot?>(null)`. `onInitStart(modelName, freeRamBytes, atEpochMs)` делает `ref.set(InitSnapshot(modelName, freeRamBytes, atEpochMs, Outcome.InProgress))`. `onInitEnd(success)` делает `ref.updateAndGet { current -> current?.copy(outcome = if (success) Outcome.Ok else Outcome.Failed) }` (если `current == null` — остаётся null, это контракт «end без start»). Публичный read-метод `fun lastInitSnapshot(): InitSnapshot? = ref.get()`.
4. Создать Hilt-модуль `app/src/main/kotlin/app/sanctum/machina/diagnostics/di/DiagnosticsModule.kt`: `@Module`, `@InstallIn(SingletonComponent::class)`, `abstract class DiagnosticsModule { @Binds abstract fun bindInitDiagnostics(impl: DiagnosticsState): InitDiagnostics }`. Тот же паттерн, что в `LogExportModule.kt` — `@Binds`-метод БЕЗ `@Singleton`-аннотации. Скоуп задаётся `@Singleton` на классе `DiagnosticsState` плюс `@InstallIn(SingletonComponent::class)` на модуле; ставить `@Singleton` на `@Binds`-метод избыточно.
5. Создать тестовый фейк `core-runtime/src/test/kotlin/app/sanctum/machina/core/registry/NoOpInitDiagnostics.kt` — `class NoOpInitDiagnostics : InitDiagnostics` с пустыми реализациями обоих методов. Цель: позволить тестам `:core-runtime/registry` (`DefaultModelRegistryTest`, Task 6) изолироваться от `:app`-зависимостей. Положительные ассерты на call-order — задача отдельного `RecordingInitDiagnostics` в Task 6, не здесь.
6. Написать unit-тесты для `DiagnosticsState` в `app/src/test/kotlin/app/sanctum/machina/diagnostics/DiagnosticsStateTest.kt` — pure-JVM, без Robolectric. Контракты см. ниже в TDD Anchor.
7. Прогнать `./gradlew :core-runtime:test :app:test` — все тесты зелёные. Прогнать `./gradlew :app:assembleDebug` — Hilt-граф собирается, нет KSP/Dagger-ошибок (валидация что `@Binds` корректен и `DiagnosticsState` доступен в `SingletonComponent`).

## TDD Anchor

Все тесты — в `app/src/test/kotlin/app/sanctum/machina/diagnostics/DiagnosticsStateTest.kt`. Pure-JVM, без Robolectric. Hand-rolled, без MockK/Mockito (см. `patterns.md`).

- `DiagnosticsStateTest::initialStateIsNull` — `DiagnosticsState().lastInitSnapshot() == null` сразу после конструктора.
- `DiagnosticsStateTest::onInitStartProducesInProgressSnapshot` — после `onInitStart("model-x", 4_000_000_000L, 1_700_000_000_000L)` снимок не-null, поля совпадают с переданными, `outcome == Outcome.InProgress`.
- `DiagnosticsStateTest::onInitEndTrueProducesOkAndPreservesFields` — после `onInitStart(...)` + `onInitEnd(true)` снимок имеет `outcome == Outcome.Ok`, остальные три поля идентичны переданным в `onInitStart`.
- `DiagnosticsStateTest::onInitEndFalseProducesFailedAndPreservesFields` — после `onInitStart(...)` + `onInitEnd(false)` снимок имеет `outcome == Outcome.Failed`, остальные три поля идентичны.
- `DiagnosticsStateTest::onInitEndWithoutStartIsNoop` — `DiagnosticsState().onInitEnd(true)` оставляет `lastInitSnapshot() == null` (документирует контракт «updateAndGet с null остаётся null»).
- `DiagnosticsStateTest::secondOnInitStartReplacesPreviousAttempt` — `onInitStart("modelA", ramA, tA)` без последующего `onInitEnd`, затем `onInitStart("modelB", ramB, tB)` → снимок содержит `("modelB", ramB, tB, InProgress)`. Поведение: replace, не «pending» (фиксируем в тесте).
- `DiagnosticsStateTest::concurrentWriterReaderNeverSeesMixedState` — атомарность (concrete contract, требование tech-spec). Два потока, ≥10 000 итераций. Writer-thread в цикле чередует attempt-set'ы: `onInitStart("modelA", ramA, tA); onInitEnd(true)` и `onInitStart("modelB", ramB, tB); onInitEnd(false)`. Reader-thread в цикле читает `lastInitSnapshot()` и assert'ит, что прочитанная пара `(modelName, outcome)` принадлежит одному из валидных attempt-set'ов: `("modelA", InProgress|Ok)` либо `("modelB", InProgress|Failed)` либо `null`. Любая «смесь» (`("modelA", Failed)` или `("modelB", Ok)` или `("modelA", outcome)` со значениями `ramB`/`tB`) — fail. Использовать обычные `Thread` + `CountDownLatch` для синхронного старта/финиша; не подтаскивать jcstress/lincheck.

Скопировать структуру `LogExportModule.kt` точно: `@Module`, `@InstallIn(SingletonComponent::class)`, `abstract class DiagnosticsModule { @Binds abstract fun bindInitDiagnostics(impl: DiagnosticsState): InitDiagnostics }`. `@Singleton`-аннотация ставится ТОЛЬКО на класс `DiagnosticsState` (impl); `@Binds`-метод никаких scope-аннотаций не несёт. Цель — зависимость для регистри-тестов, чтобы они не подтаскивали `:app`. RecordingInitDiagnostics (с positive-assertion'ом на call-order) — уже задача Task 6.
- `app/src/test/kotlin/app/sanctum/machina/diagnostics/DiagnosticsStateTest.kt` — package `app.sanctum.machina.diagnostics`. Pure JUnit4 (по конвенции project test infrastructure). 7 тестов из TDD Anchor.

**Dependencies:**
- Никаких новых external packages (см. tech-spec § Dependencies). Используются: Hilt 2.57.1 (уже в `:app`), JDK `java.util.concurrent.atomic.AtomicReference`, JUnit4.
- Зависимостей от других задач Phase 3.5 нет — Wave 1.

**Edge cases:**
- `onInitEnd` без предыдущего `onInitStart` — `updateAndGet` на null оставляет null. Покрыто тестом `onInitEndWithoutStartIsNoop`. Это контракт, не баг: writer (Task 6) всегда вызывает `onInitStart` перед `onInitEnd` под `lifecycleMutex`, но defence-in-depth.
- Второй `onInitStart` без `onInitEnd` между ними — replace previous (см. `secondOnInitStartReplacesPreviousAttempt`). Поведение фиксируется тестом, в Task 6 этот сценарий невозможен под `lifecycleMutex`, но контракт зафиксирован.
- Concurrent reader/writer: `AtomicReference.set`/`updateAndGet` обеспечивают visibility между потоками. Writer (Task 6) — `Dispatchers.Default` под `lifecycleMutex`; reader (Task 7/Task 8) — Main thread через ViewModel state. Visibility покрыта `concurrentWriterReaderNeverSeesMixedState`.

**Implementation hints:**
- `@Binds` обязательно `abstract` (внутри `abstract class`). Если делать `object DiagnosticsModule` с `@Provides` — Hilt отработает, но `@Binds` дешевле runtime'но и идиоматичнее для interface→impl биндинга. Скопируй структуру `LogExportModule` без отклонений — в частности, на `@Binds`-методе НЕ ставь `@Singleton` (скоуп идёт через `@InstallIn(SingletonComponent::class)` модуля + `@Singleton` на классе `DiagnosticsState`).
- `AtomicReference.updateAndGet { current -> ... }` принимает `UnaryOperator<T>` — pure-функция от current value. Не делай side-effects внутри лямбды (logging, etc.) — спецификация JDK позволяет вызывать функцию повторно при CAS-fail.
- В concurrent-тесте `concurrentWriterReaderNeverSeesMixedState`: используй `CountDownLatch(1)` для синхронного старта обоих потоков, `CountDownLatch(2)` для ожидания завершения обоих, и `AtomicReference<AssertionError?>` для пропихивания assertion failure из reader-thread в test-thread (JUnit assert'ы из non-test-thread'а не валят тест автоматически).
- Не используй `runBlocking` или `runTest` в тесте на atomicity — нужны реальные `Thread`-ы для воспроизведения memory-ordering condition'ов; coroutines на одном dispatcher'е не дают того же race-window.
