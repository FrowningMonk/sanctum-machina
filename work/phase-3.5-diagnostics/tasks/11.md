## Description

Полный test-quality аудит фичи Phase 3.5 (Diagnostics). Прочитать все тестовые файлы, добавленные/изменённые в Tasks 1-8, и проверить, что тестовый набор:

1. Покрывает каждый AC из блока «G. Тесты» user-spec'а (AC-T1..AC-T9), один-в-один.
2. Содержит осмысленные assert'ы — никаких `assertEquals(1, 1)`-style тестов и tautology-проверок, которые проходят при любой реализации.
3. Соблюдает test-pyramid проекта (`architecture.md § Testing`): pure-JVM приоритет, Robolectric только там, где требуется `Bitmap` / Compose-resources / Android-only API без extractable seam'а.
4. Не содержит запрещённых mock-фреймворков (`org.mockito.*`, `io.mockk.*`) — по проектной конвенции используются hand-rolled fakes.
5. Изолированы — каждый тест self-contained, нет shared mutable state между тестами одного класса (никаких `companion object`-кэшей, статических singleton'ов, файлов в `/tmp` без cleanup'а).
6. Концентрационный тест в `DiagnosticsStateTest` (атомарность снапшота) реализован корректно: использует `Thread` + `CountDownLatch` (без jcstress/lincheck), ≥10000 итераций, валидная attempt-set assertion (читатель видит только `(modelA, InProgress|Ok)` либо `(modelB, InProgress|Failed)` либо `null` — никакой «смеси»).
7. Fixture-table тесты для pure-функций `formatRamShortage`, `gateAllowsDownload`, `gitVersionParse` действительно покрывают граничные случаи из tech-spec.
8. Список «Existing tests to verify still passing» (из tech-spec § Testing Strategy) выполняется — все упомянутые тесты либо проходят без изменений, либо адаптация ограничивается передачей нового DI-параметра.

Аудит — пассивная проверка. Никаких изменений в тестах задача не делает. Выход — структурированный отчёт.

## Details

**Files:**

**Dependencies:**
- Tasks 1-8 завершены — иначе нечего аудировать.
- Task 9 (Code Review) и Task 10 (Security Audit) идут в той же волне 4 параллельно; пересечений по правкам нет.

**Edge cases:**
- Если какой-то AC-T не покрыт ни одним тестом — это finding **CRITICAL** в отчёте, итоговый verdict NO-GO.
- Если в коде нашёлся `import org.mockito` или `import io.mockk` — finding **MAJOR**, verdict NO-GO до удаления.
- Если concurrency-тест выполнен через jcstress / lincheck / kotlinx-coroutines-test без `Thread`+`CountDownLatch` — это отклонение от tech-spec'а на pure-JDK threading; finding **MAJOR**.
- Если `DiagnosticsStateTest` итераций <10000 — finding **MAJOR** (потеря статистической мощи atomic-проверки).
- Если ни один Robolectric-тест не введён — это нормально (приоритет pure-JVM по tech-spec'у).

**Implementation hints:**
- Test-master skill (`SKILL.md`) даёт чеклист для test-review — пройти по нему явно.
- Hand-rolled fakes конвенция: pattern из `LogcatReader` (injectable `CommandRunner`) — образец, как должны выглядеть fakes в `GitVersionParseTest`.
- Для assertion на pair-set'е в concurrency-тесте корректная форма — explicit `when (modelName) { "modelA" -> assertThat(outcome in setOf(InProgress, Ok)) ... }`. Если автор написал «прочитанный snapshot не равен `(modelA, Failed)`» — это weaker assertion, finding **MINOR**.
- Trivial-assert scanning: regex `assertEquals\(\s*([^,]+)\s*,\s*\1\s*\)` ловит tautology; ручной просмотр — на тесты без `assert*`.
