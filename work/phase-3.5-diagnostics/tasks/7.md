# Task 7: DeviceInfoCollector header expansion

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Расширяем экспортируемую шапку `.txt`-лога тремя дополнительными источниками сигнала (Solution slice 4 техспека):

1. К существующей строке `memory:` добавляются поля `threshold=` и `lowMemory=` (бесплатно — они уже есть в `ActivityManager.MemoryInfo`).
2. Появляется новая строка `process:` с тремя полями: `java=`, `native=`, `totalPss=` (доступно через `Debug.MemoryInfo` + `Runtime`, **в любом процессе**, в т.ч. `:crash`).
3. Появляется новая строка `last init:` (источник — `DiagnosticsState.lastInitSnapshot()`, заполненный Task 4): четыре варианта рендера согласно AC-D6 + Decision 12 (Ok / Failed / null → «пока не было» / InProgress).

Контракт между процессами зафиксирован в Architecture-таблице Shared Resources: `DiagnosticsState` живёт только в main-процессе. В `:crash`-процессе secondary-ctor `AndroidDeviceInfoProvider` возвращает `null` для `lastInitSnapshot()`, что рендерится как «пока не было». При этом `memory:`/`process:` строки рендерятся нормально в обоих процессах — поля доступны через системные API без `DiagnosticsState`. Этот контракт покрывается AC-H4.

Задача чисто per-process — не трогает гейт, registry, UI, strings.xml. Зависит от Task 4 (даёт `DiagnosticsState.lastInitSnapshot(): InitSnapshot?` и `InitSnapshot` модель).

## What to do

1. Расширить интерфейс `DeviceInfoProvider` шестью новыми методами:
   - `thresholdMemoryBytes(): Long` — `MemoryInfo.threshold`
   - `isLowMemory(): Boolean` — `MemoryInfo.lowMemory`
   - `processJavaHeapBytes(): Long` — `Runtime.getRuntime().totalMemory() - freeMemory()`
   - `processNativeHeapBytes(): Long` — `Debug.getNativeHeapAllocatedSize()`
   - `processTotalPssBytes(): Long` — сумма `Debug.MemoryInfo` PSS-полей (через `Debug.MemoryInfo()` + `Debug.getMemoryInfo(info)` + `info.totalPss * 1024L`)
   - `lastInitSnapshot(): InitSnapshot?` — снимок из `DiagnosticsState`, либо `null`
2. В `AndroidDeviceInfoProvider` (структура трёх конструкторов — см. также «Details»):
   - **Private primary-ctor** (уже существует) — расширить его дополнительным thunk'ом `lastInitSnapshotProvider: () -> InitSnapshot?` по аналогии с уже существующим `entriesProvider`. `lastInitSnapshot()` реализуется как `lastInitSnapshotProvider()`.
   - **`@Inject` secondary-ctor** (Hilt-entry-point, main-процесс) — принимает `DiagnosticsState` в качестве параметра и форвардит thunk в primary: `: this(context, { registry.models.value }, { state.lastInitSnapshot() })`.
   - **Не-Hilt secondary `:crash`-ctor `(Context)`** — `DiagnosticsState` НЕ принимает; форвардит `{ null }` в primary.
   - Все 6 методов реализованы в primary; первые 5 — через системные API, доступны в обоих процессах. `lastInitSnapshot()` в `:crash` возвращает `null` за счёт thunk-паттерна.
   - Извлечь `Debug.MemoryInfo` чтение в private helper по аналогии с существующим `memoryInfo(): ActivityManager.MemoryInfo`. На любую ошибку чтения (catch + fallback) — возвращать sentinel-значение, по которому `buildHeader` отрендерит `n/a` для конкретного поля (см. ниже).
3. В `DeviceInfoCollector.buildHeader`:
   - Расширить строку `memory:` так, чтобы она содержала `total=`, `available=`, `threshold=`, `lowMemory=` (формат: `total=11.7 GB, available=6.2 GB, threshold=0.4 GB, lowMemory=false`).
   - Между строками `memory:` и `active model:` вставить две новые строки:
     - `process: java=<X.X> GB, native=<X.X> GB, totalPss=<X.X> GB` (с `n/a` для отдельного поля при ошибке источника, остальные ОК).
     - `last init: <отформатированная строка>` — четыре варианта согласно AC-D6 + Decision 12:
       - `Ok` → «`<X.X> ГБ RAM · HH:mm · <modelName> · ok`».
       - `Failed` → «`<X.X> ГБ RAM · HH:mm · <modelName> · ошибка`».
       - `null` → «`пока не было`».
       - `InProgress` → «`Идёт инициализация: <X.X> ГБ RAM · HH:mm · <modelName>`» (точная wording — за solver'ом, но строка ДОЛЖНА содержать маркер «инициализация», `freeRamBytes`, `atEpochMs`, `modelName` и **НЕ ДОЛЖНА** содержать «ok»/«ошибка»).
4. Сохранить весь существующий формат шапки (строки `=== … ===`, `exported:`, `applicationId:`, `version:`, `device:`, `active model:`, `downloaded models:`) бит-в-бит — `headerFormatting_deterministicFromStub` не должен сломаться по существующей части.
5. В `DeviceInfoCollectorTest`:
   - Расширить `StubDeviceInfoProvider` шестью новыми полями + override'ами (с дефолтами, чтобы существующие тесты не ломались).
   - В `headerFormatting_deterministicFromStub` дополнить `expected`-blob двумя новыми строками + расширенной `memory:`.
   - Добавить тесты на отдельные ветки (см. TDD Anchor).
6. В `LogExportManagerTest.headerContainsRequiredFields` — расширить список substring'ов до `process:` и `last init:` (текущий список уже включает `memory:`, его не трогаем, формат уже изменён).
7. В `stubProvider()` `LogExportManagerTest`'а — добавить override'ы новых интерфейсных методов (тривиальные дефолты).
8. **Discover-and-extend all hand-rolled `DeviceInfoProvider` stubs in test source-set.** Расширение интерфейса (6 новых методов) ломает компиляцию ЛЮБОГО hand-rolled стаба, у которого не будет override'ов. Перед тем как считать задачу готовой:
   - Запустить `Grep` по `app/src/test` с паттерном `DeviceInfoProvider`, `output_mode: files_with_matches`. На момент планирования задачи (snapshot wave-2) совпадают только `DeviceInfoCollectorTest.kt` и `LogExportManagerTest.kt` — они уже покрыты пунктами 5 + 7. **Edge cases:**

- Per-field `n/a`: solver выбирает sentinel-механизм (например, `Long.MIN_VALUE` или Result-обёртка). Контракт — `buildHeader` рендерит `n/a` для конкретного поля при ошибке источника, остальные поля ОК. Тест `processLine_perFieldNaOnSourceError` это покрывает.
- `:crash`-process: secondary-ctor `AndroidDeviceInfoProvider(Context)` существует и используется в `LogExportManager(context)` (см. `nonHiltConstruction_instantiatesWithoutInjection` в `LogExportManagerTest`). Обязательно: secondary-ctor НЕ должен требовать `DiagnosticsState`.
- `InProgress`-рендер: формулировка строки за solver'ом, но обязательные substring'и зафиксированы в TDD Anchor (маркер «инициализация», RAM, время, modelName; **запрещены** «ok»/«ошибка»). Decision 12 явно расширяет AC-D6 четвёртой веткой.
- Форматирование RAM в `last init:` — оставлять `X.X ГБ` (русский), а не `X.X GB` (английский) — Decision 12 фиксирует именно так. `memory:`/`process:` — английский `GB` (формат `formatGb`). Это разные форматтеры; не путать.
- Времена в `last init:` — `HH:mm` (Decision 12). Локально: `OffsetDateTime.ofInstant(Instant.ofEpochMilli(atEpochMs), ZoneId.systemDefault()).format(DateTimeFormatter.ofPattern("HH:mm"))` или эквивалент.

**Implementation hints:**

- Существующий приватный helper `formatGb(bytes)` подходит для `memory:`/`process:` (английский `GB`). Для `last init:` понадобится отдельный helper `formatGbRu(bytes): String` («`X.X ГБ`»).
- Для `last init:` имеет смысл вынести рендер 4 веток в pure-helper рядом с `DeviceInfoCollector` (или приватный) — это упрощает pure-JVM unit-тесты на 4 ветки без дёргания `buildHeader` целиком. Сигнатура — на усмотрение solver'а (например, может ли helper брать `ZoneId`/`Clock` для тестируемости — solver решит сам); главное — `buildHeader` использует тот же helper и возвращает его результат внутри строки `last init:`.
- Для AC-H4 теста — конструировать `StubDeviceInfoProvider` с `lastInitSnapshot = null` и валидными остальными полями. Никакого Robolectric не нужно — `StubDeviceInfoProvider` уже в JUnit-source-set.
- `processTotalPssBytes()`: `Debug.MemoryInfo.totalPss` возвращает PSS в KB, умножить на `1024L` чтобы получить байты для единого `formatGb`.
- При выборе механики передачи `DiagnosticsState` в `AndroidDeviceInfoProvider`: можно использовать тот же thunk-паттерн что для `entriesProvider` (`snapshotProvider: () -> InitSnapshot? = { state.lastInitSnapshot() }`), либо инжектить интерфейс напрямую в primary-ctor — solver выбирает по simplicity. Главное — secondary-ctor должен возвращать `null` без живого `DiagnosticsState`.
