# Task 10: Security Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:security-auditor` — [skills/security-auditor/SKILL.md](~/.claude/skills/security-auditor/SKILL.md)

## Description

Полный security-аудит фичи Phase 3.5 Diagnostics. Аудитор читает все файлы, изменённые/созданные в Tasks 1-8, и проходит OWASP Top 10 в контексте мобильного Android-приложения. Цель — убедиться, что нововведения фазы (allowlist parser с recalibration, git-describe versionName, расширенный `*:W` logcat, `InitDiagnostics` seam + `DiagnosticsState`, pre-flight gate UI, расширенная шапка `.txt`-экспорта, новый DiagnosticsScreen с SAF-экспортом) не вносят уязвимостей: input validation, отсутствие path traversal / command injection, отсутствие сенситивных данных в экспортируемом `.txt`, корректная thread-safety записи snapshot'а, корректная degradation на `:crash`-процессе.

Это финальная аудит-волна перед Pre-deploy QA: задача параллельная Tasks 9 (Code Audit) и 11 (Test Audit), пишет независимый отчёт без reviewer-субагентов.

## What to do

1. Прочитать все файлы, изменённые/созданные в Tasks 1-8 (см. tech-spec § Implementation Tasks → Files to modify, decisions.md). Особое внимание:
   - Task 1: `core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/AllowlistLoader.kt` (regex `MODEL_ID_REGEX`, range-check `minDeviceMemoryInGb in 1..64`, JSON-shape rejection — все три проверки находятся именно здесь, в `parse()` и его companion'е), `core-runtime/src/main/kotlin/app/sanctum/machina/core/data/ModelAllowlist.kt` (только data class `AllowedModel` — input-validation там нет), `core-runtime/src/main/assets/model_allowlist.json` (значения порогов, отсутствие путей).
   - Task 2: `build-logic/.../GitVersionPlugin.kt`, `GitVersionParse.kt` — argv-форма exec, отсутствие eval/shell.
   - Task 3: `app/src/main/kotlin/app/sanctum/machina/logexport/LogcatReader.kt` — расширение `*:E` → `*:W` (потенциальное расширение поверхности утечки в логах).
   - Task 4, 6, 7: `DiagnosticsState`, `InitSnapshot`, `DefaultModelRegistry.initialize`, `DeviceInfoCollector.buildHeader` — содержимое snapshot'а, thread-safety, `:crash`-degradation.
   - Task 5, 8: `ModelManagerScreen`, `DiagnosticsScreen`, `DiagnosticsViewModel` — пользовательский ввод (нет), SAF-экспорт.
2. Пройти OWASP Top 10 в Android-контексте (A01-A10), для каждой категории — verdict (ok / n/a / finding) + обоснование.
3. Точечно проверить риски, перечисленные в tech-spec и Decisions:
   - **Input validation в allowlist-парсере (Decision 5):** все три проверки сидят в `core-runtime/.../core/registry/AllowlistLoader.kt` — `MODEL_ID_REGEX` (объявлен в companion'е, применяется в `parse()`) отвергает мусорные id; `require(minDeviceMemoryInGb in 1..64)` блокирует negative/oversized значения; broken JSON shape ловится в `parse()` через `IllegalArgumentException`, логируется через `errorLog.e("download", …)` в `load()`. `ModelAllowlist.kt` — только data class, валидации не содержит.
   - **Path traversal через `model_allowlist.json`:** проверить, что URL/path-поля моделей валидируются (если есть), что `Model.getPath(context)` не позволяет выйти за пределы app-private storage, что parser не интерпретирует относительные пути.
   - **Command injection в `gitVersionName()` (Decision 10):** `providers.exec { commandLine("git", "describe", "--tags", "--always", "--dirty=-dev") }` — argv-list, не shell-string. Pure-функция `gitVersionParse(stdout, exitCode)` не делает `eval`/`Runtime.exec`/`ProcessBuilder` — только парсит уже captured stdout.
   - **Сенситивные данные в экспортируемом `.txt` (`DeviceInfoCollector.buildHeader`):** проверить отсутствие PII (имя пользователя, email), токенов (auth, API keys, HF tokens), путей с username (`/data/user/0/<package>/...` ok, `/Users/<name>/...` или `C:\Users\<name>\...` — не ok), package-private state. Phase 3.5 добавляет `process:` (java/native/totalPss bytes — non-sensitive) и `last init:` (modelName из allowlist, freeRamBytes, atEpochMs — non-sensitive). Подтвердить, что новые поля не содержат ничего специфичного для пользователя.
   - **Thread-safety `DiagnosticsState` (Decision 7):** `AtomicReference.updateAndGet` гарантирует CAS-обновление; единственный writer под `lifecycleMutex` — race по построению нет, но visibility между Dispatchers.Default writer и Main reader обеспечена `AtomicReference`. Подтвердить, что `onInitEnd` делает `updateAndGet { current -> current?.copy(outcome = …) }`, не `set`.
   - **`:crash`-process degradation (AC-H4):** secondary-ctor `AndroidDeviceInfoProvider` возвращает `null` для `lastInitSnapshot()`; `DeviceInfoCollector.buildHeader` рендерит «пока не было» при `null`-snapshot'е. Никакого NPE / падения из-за отсутствия `DiagnosticsState` (singleton живёт только в main-process).
   - **`MemoryInfo.availMem` через `@ApplicationContext` (Decision 9):** read-only JNI-call в `ActivityManager`, не требует runtime permissions, не повышает privilege.
4. Зафиксировать findings (severity: critical / high / medium / low / info) с конкретными файлами и строками, либо явно указать «no findings» для категории.
5. Написать отчёт в `work/phase-3.5-diagnostics/logs/audit/security-audit.md`.

## Details

**Files:**
- `work/phase-3.5-diagnostics/logs/audit/security-audit.md` (new) — структура отчёта:
  1. Summary (1-3 предложения: total findings по severity, общий verdict).
  2. Scope (список файлов, прочитанных по Tasks 1-8).
  3. OWASP Top 10 (A01-A10): для каждой категории — verdict + 1-2 предложения обоснования.
  4. Targeted checks (7 пунктов из «What to do»): для каждого — verdict + обоснование с ссылками на конкретные файлы/строки.
  5. Findings (если есть): severity, location, description, recommendation.
  6. Recommendations (если уместны low/info-замечания, не блокирующие deploy).

**Dependencies:** Tasks 1-8 завершены (все `Files to modify` существуют в их финальной форме). Параллельно Tasks 9, 11 в той же Audit Wave.

**Edge cases:**
- `:crash`-process не имеет `DiagnosticsState`-singleton'а — secondary-ctor `AndroidDeviceInfoProvider` должен возвращать `null` для `lastInitSnapshot`. Проверить, что отсутствие Hilt-graph в `:crash`-процессе не приводит к падению при чтении/построении header'а.
- `MODEL_ID_REGEX` (если присутствует) — проверить, что не подвержен ReDoS (catastrophic backtracking) на pathological input'е.
- `gitVersionParse(stdout, exitCode)` — что делает с пустым stdout, non-zero exit, multi-line output, неожиданными символами.
- SAF-экспорт `.txt` — проверить, что `Uri` от `CreateDocument("text/plain")` не позволяет писать вне разрешённого SAF-scope (Android handles this, но verify usage pattern).

**Implementation hints:**
- Подход — белый ящик: читаем код, проверяем surface'ы, не запускаем dynamic-сканеры.
- Severity-rubric: `critical` (RCE / sensitive data leak / privilege escalation в production-flow), `high` (exploitable при специфичных условиях), `medium` (defense-in-depth gap), `low` (best-practice deviation), `info` (наблюдение для future hardening).
- Если категория OWASP неприменима в Android-on-device-LLM-context (например A02 Cryptographic Failures для фичи без криптографии) — явно «n/a» с обоснованием в одну строку.
- Ссылки на код: формат `path/to/file.kt:LL` или `path/to/file.kt:LL-LM`.
- Ссылки на decisions: `[Decision N](../decisions.md)` или `tech-spec § Decision N`.
