# Task 10: Test Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:test-master` — [skills/test-master/SKILL.md](../../../.claude/skills/test-master/SKILL.md) (особенно references/test-quality-review.md и references/unit-tests.md)

## Description

Аудит качества всех тестов, написанных в Phase 2.5 Tasks 2–5 (волны 1–2): `CrashHandlerTest`, `CrashStateTest`, `LogExportManagerTest`, `DeviceInfoCollectorTest`, `LogcatReaderTest`, `TapCounterTest`, `SanctumApplicationTest`. Проверка, что тесты действительно покрывают поведенческие критерии из user-spec (а не только «классы и функции существуют»), что ассерты содержательны (конкретные байты/маркеры/направления, а не `assertNotNull`), что стиль консистентен с каноничным `ErrorLogTest` (Robolectric + hand-rolled fakes, без Mockito/MockK — см. code-research §9.3), и что баланс пирамиды правильный (size M → только unit-тесты; отсутствие integration/E2E обосновано code-research §9.1: `androidTest` не сконфигурирован).

Это audit-задача, а не задача написания кода. Аудитор сам является ревью — никаких дополнительных ревьюеров. Выход — письменный отчёт.

## What to do

1. Прочитать все 8 тестовых файлов из Context Files (7 новых + существующий `ErrorLogTest.kt` как эталон стиля) и сам tech-spec Testing Strategy.
2. Для каждого измерения покрытия (список ниже) — явно оценить, покрывается ли оно, и какими именно тестами (с указанием имён тестов). Если не покрывается — пометить как finding.
3. Написать отчёт по структуре ниже. Отчёт — единственный deliverable задачи.

**Измерения покрытия, которые должны быть явно оценены:**

- **Поведенческое покрытие user-spec AC:**
  - Поля header'а присутствуют (`DeviceInfoCollectorTest`).
  - `[empty]` плейсхолдеры при отсутствующих источниках (`LogExportManagerTest`).
  - Порядок секций `crash → errors → errors.1 → logcat` проверяется через сравнение индексов строк (`LogExportManagerTest.allSectionsPresent_orderedCorrectly`), а не через жёстко вбитый regex целого файла.
  - Направление усечения правильное: `crash.log` — head-truncated (сохраняются тип исключения + верхние фреймы), `logcat` — tail-truncated (сохраняются последние строки). Это отдельные тесты с направленными ассертами, а не только проверка длины.
  - Логика видимости баннера: `crash.log` ∧ ¬`.dismissed` — все четыре ячейки истинностной таблицы (`CrashStateTest`).
  - Границы 7-tap: inclusive на 2-с, exclusive выше 2-с (`TapCounterTest`).
  - Overwrite-семантика: `repeatedCrashes_overwriteNotAppend` в `CrashHandlerTest`.
  - Кросс-компонентный сброс `.dismissed`: `dismissedThenNewCrash_reappears` в `CrashStateTest` плюс удаление `.dismissed` в `CrashHandlerTest` — обе стороны контракта покрыты.
- **Содержательность ассертов:** тесты утверждают конкретные байты/маркеры/направления, а не `assertNotNull`. Найти любые «сахарные» тесты и перечислить их как findings.
- **Консистентность с `ErrorLogTest` по стилю:** Robolectric + hand-rolled fakes; отсутствие Mockito/MockK (code-research §9.3); `runTest` для suspend; seed/teardown `filesDir/logs/` в `@Before`/`@After`.
- **Баланс пирамиды:** size M → unit-only корректно. Отсутствие integration/E2E обосновано code-research §9.1 (нет `androidTest`, `compose-ui-test` не на classpath). SAF-диалог, `:crash`-граница процессов, dispatch `Thread.setDefaultUncaughtExceptionHandler` — заявлены как требующие ручной user-верификации.
- **Покрытие seam-ов:** `Killer`, `CommandRunner`, `DeviceInfoCollector`'s provider interface — все имеют тестовые дублёры.
- **Негативные пути:** `writeTo_ioException_surfaces`, `writeTo_nullOutputStream_surfacesAsIoException`, ветки `[logcat unavailable: *]`, `handlerInternalFailure_killsOnce`.
- **Reflection-инварианты:** `handlerShape_noSuspendCallsNoCoroutineImports` (удерживает Decision 10 без хрупкого совпадения по тексту исходника).
- **Argv-shape ассерт в `LogcatReaderTest`:** защищает Decision 8 + OWASP A03 (нет shell-wrapping, фиксированные аргументы + `Process.myPid()` как integer).

## Details

**Files:** пишем только отчёт. Исходные тесты не модифицируем — рекомендации по правкам формулируем как findings для следующего раунда (или для task-creator в fix-режиме).

**Рекомендуемый путь отчёта:** `work/phase-2.5-logexport/logs/working/task-10/test-audit-report.md`.

**Структура отчёта:**

1. **Summary** — 1–3 предложения: общая оценка и ключевые findings.
2. **Coverage Matrix** — таблица: измерение × покрывающий тест × статус (covered / partial / missing). Все измерения из списка выше должны быть в таблице.
3. **Assertion Quality** — выборочный перечень ассертов, подтверждающих содержательность (конкретные байты/маркеры/направления). Отдельно — findings по «сахарным» тестам (если есть).
4. **Style Consistency** — сравнение с `ErrorLogTest`: Robolectric runner, `@Config(sdk = [33])`, `runTest` для suspend, hand-rolled fakes, seed/teardown `filesDir/logs/`. Отклонения перечислить.
5. **Pyramid Balance** — обоснование unit-only для size M со ссылкой на code-research §9.1; перечисление того, что явно отнесено к ручной user-верификации (SAF, `:crash` граница, `setDefaultUncaughtExceptionHandler` dispatch).
6. **Seam Coverage** — `Killer`, `CommandRunner`, `DeviceInfoCollector`'s provider interface: все ли имеют тест-дублёры и осмысленные fake-реализации (`RecordingKiller`, не-Mockito).
7. **Negative Paths** — явный чек перечисленных негативных тестов.
8. **Reflection / Argv Invariants** — `handlerShape_noSuspendCallsNoCoroutineImports` и argv-shape в `LogcatReaderTest`.
9. **Findings** — список с severity и recommended fix. Если findings нет — явно написать «no findings».
10. **Verdict** — approve / approve-with-fixes / reject + что должно случиться для разблокировки Final Wave.

**Dependencies:** задачи 1–7 завершены, тестовые файлы существуют, suite зелёный.

**Edge cases аудита:**
- Тест существует, но пустой/TODO — finding critical.
- Тест-имя обещает проверку `overwriteNotAppend`, но ассертит только длину — finding major (misleading coverage).
- Mockito/MockK импорты — finding major (нарушение §9.3 консистентности и потенциальное добавление зависимости не в `libs.versions.toml`).
- Отсутствие `runTest` на suspend путях → flaky тесты — finding major.
- `assertNotNull` вместо конкретного совпадения — finding minor (сахар).

**Implementation hints:**
- Читать тестовые файлы по очереди, для каждого заполнять соответствующие ячейки Coverage Matrix.
- Грепать по тестовым файлам `org.mockito|io.mockk|MockK|Mockito` — ожидаем zero hits.
- Грепать по тестовым файлам `assertNotNull` — каждое попадание проверить вручную (иногда уместно, иногда сахар).
- Подтянуть `ErrorLogTest.kt` как эталон при оценке style consistency — сравнивать `@Before`/`@After`/`@Config(sdk = [33])`/`runTest` использование.
