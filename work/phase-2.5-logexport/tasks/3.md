# Task 3: LogExportManager + DeviceInfoCollector + LogcatReader + TapCounter

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Реализовать четыре класса под `app/src/main/kotlin/app/sanctum/machina/logexport/` — ядро сборки диагностического `.txt`-экспорта для фазы 2.5:

1. **`LogExportManager`** (`@Singleton`) — собирает финальный текстовый отчёт (header + `crash.log` + `errors.log` + `errors.log.1` + `logcat`) в одну `String` на `Dispatchers.IO` и умеет писать этот `String` в выбранный пользователем SAF-URI. Каждая секция имеет свой cap согласно Decision 7 (`crash.log` head-truncated ≤100 KB, `logcat` tail-truncated ≤100 KB, `errors.log`/`errors.log.1` ≤2 MB каждая — уже гарантировано ротацией `ErrorLog`, header ≤~1 KB). Если секция обрезана — дописывается маркер `[truncated at NNN KB]`. Отсутствующие/пустые источники рендерятся как `[empty]`; отсутствующий `errors.log.1` — секция опускается целиком (без пустого `=== errors.log.1 ===` разделителя).
2. **`DeviceInfoCollector`** — формирует текстовый header по фиксированному шаблону (applicationId, version + debug flag, manufacturer/model, Android version + API level, total/available RAM, active model id либо `none`, downloaded models list либо `(none)`). Зависимости (`PackageInfo`, `ActivityManager`, `Build`-константы) берутся через собственный интерфейс-провайдер, чтобы юнит-тесты могли подсовывать детерминированный stub.
3. **`LogcatReader`** — читает собственный лог процесса через `ProcessBuilder("logcat", "-d", "-v", "threadtime", "--pid=${Process.myPid()}", "*:E").redirectErrorStream(true)`. Чтобы обойти 64-KB deadlock pipe-буфера, stdout вычитывается в отдельном worker-потоке параллельно с `waitFor(2, SECONDS)`; на таймаут — `destroy()`. Возвращает raw-байты (обрезку выполняет `LogExportManager`) либо placeholder `[logcat unavailable: <reason>]`. Для тестов — seam `CommandRunner`.
4. **`TapCounter`** — pure JVM (никаких Android-зависимостей), 7-tap детектор: 7 касаний с промежутками ≤2 с подряд → срабатывает ровно один раз на 7-м касании; разрыв >2 с → счётчик сбрасывается в 1. Время берётся через функциональный параметр `nowNanos: () -> Long` для тестирования без моков.

Задача — ядро Wave 1 фичи фазы 2.5. Wave 2 (CrashReportActivity, AboutScreen, ModelManagerScreen banner) напрямую зависит от этих четырёх классов. Относится к Decisions 1 (single `.txt`), 7 (section caps + direction), 8 (logcat source-aware), 11 (no filtering/redaction).

## What to do

1. **Создать package `app/src/main/kotlin/app/sanctum/machina/logexport/`** — четыре новых файла + их тестовые двойники под `app/src/test/kotlin/app/sanctum/machina/logexport/`.

2. **`TapCounter.kt`** — pure-JVM класс без Android-импортов. Публичный API: конструктор принимает `nowNanos: () -> Long`; метод `tap(): Boolean` возвращает `true` только на 7-м касании в пределах допустимых окон. Граничное условие — inclusive ≤2 с (ровно 2_000_000_000 нс от предыдущего касания считается «в окне»), 2 с + 1 нс — reset. После срабатывания (7-й tap → `true`) счётчик обнуляется: 8-й tap возвращает `false` и начинает новый цикл с 1.

3. **`LogcatReader.kt`** — класс + `interface CommandRunner` с методом, принимающим `argv: List<String>` и `timeoutMs: Long` и возвращающим структуру с `exitCode: Int?`, `stdout: ByteArray`, `timedOut: Boolean`. Прод-реализация обёрнута в `DefaultCommandRunner`, стартует `ProcessBuilder` с ровно шестью аргументами `["logcat", "-d", "-v", "threadtime", "--pid=${Process.myPid()}", "*:E"]`, `redirectErrorStream(true)`, читает stdout в фоновом `Thread`/executor'е, `process.waitFor(2, TimeUnit.SECONDS)` — при `false` → `process.destroy()` → timeout. `LogcatReader.read(): String` возвращает: raw UTF-8 stdout → если stdout пустой → `[logcat unavailable: empty]`; timeout → `[logcat unavailable: timeout]`; non-zero exit → `[logcat unavailable: exit=N]`; иначе — raw output дословно (обрезку делает `LogExportManager`).

4. **`DeviceInfoCollector.kt`** — класс + `interface DeviceInfoProvider` (методы: `applicationId()`, `versionName()`, `versionCode()`, `isDebug()`, `manufacturer()`, `model()`, `androidRelease()`, `apiLevel()`, `totalMemoryBytes()`, `availableMemoryBytes()`, `activeModelId(): String?`, `downloadedModels(): List<Pair<String, Long>>` — id + size bytes, либо аналогичный чистый snapshot). `DeviceInfoCollector.buildHeader(): String` рендерит блок по шаблону из tech-spec «Data Models». `DeviceInfoCollector` — `public class DeviceInfoCollector @Inject constructor(private val provider: DeviceInfoProvider)`; конструктор обязан быть `public` и вызываемым напрямую (не `internal`, не `@Inject`-only). Прод-реализация `AndroidDeviceInfoProvider` — `public class AndroidDeviceInfoProvider @Inject constructor(@ApplicationContext context: Context) : DeviceInfoProvider`; конструктор тоже `public` и вызываемый без Hilt через `AndroidDeviceInfoProvider(context)` (аннотация `@ApplicationContext` — это label для Hilt-графа, runtime-препятствий для manual-instantiation не создаёт). Использует `PackageInfo` + `ActivityManager.MemoryInfo` + `Build` (без статических обращений изнутри коллектора, чтобы тесты могли подставить stub без Robolectric).

5. **`LogExportManager.kt`** — `@Singleton class LogExportManager @Inject constructor(@ApplicationContext private val context: Context, private val deviceInfo: DeviceInfoCollector, private val logcat: LogcatReader)` — первичный Hilt-friendly конструктор с DI-seam'ами для тестов. **Обязательный secondary constructor `LogExportManager(context: Context) : this(context, DeviceInfoCollector(AndroidDeviceInfoProvider(context)), LogcatReader(DefaultCommandRunner()))`** — нужен для Task 4 (Decision 5: `:crash` процесс конструирует менеджер вручную без Hilt). Secondary ДОЛЖЕН быть `public` и должен принимать только `Context` — все остальные зависимости поднимаются из `Context` внутри. `AndroidDeviceInfoProvider` и `DefaultCommandRunner` обязаны иметь `public` конструктор, вызываемый без Hilt (`AndroidDeviceInfoProvider(context: Context)` и `DefaultCommandRunner()` без аргументов). API:
    - `enum class ExportSource { About, CrashReport }`.
    - `suspend fun buildExport(source: ExportSource): String` — всё на `withContext(Dispatchers.IO)`, собирает в `StringBuilder` header (без cap, формально ≤1 KB) → `\n\n=== crash.log ===\n` → content/`[empty]` → `\n\n=== errors.log ===\n` → content/`[empty]` → `\n\n=== errors.log.1 ===\n` → content (или пропустить блок если файла нет) → `\n\n=== logcat ===\n` → `CrashReport` даёт `[logcat available only via About export]` (Decision 8), `About` — `logcat.read()` с tail-truncate до 100 KB.
    - `suspend fun writeTo(uri: Uri, content: String)` — `context.contentResolver.openOutputStream(uri).use { out -> ... }`; если `openOutputStream` вернул `null` — явно `throw IOException("openOutputStream returned null")`. `IOException` пробрасывается наружу (вызывающий код в Wave 2 показывает Toast/Snackbar).
    - **Truncation:** `crash.log` — head-truncate (берём первые 100 KB — стектрейс + тип исключения в начале), в конец дописываем `\n[truncated at 100 KB]`. `logcat` — tail-truncate (берём последние 100 KB — наиболее свежие строки), в начало дописываем `[truncated: head NNN bytes]\n`. Маркеры появляются **только** если обрезка реально сработала.
    - **Fresh-install guard:** если `filesDir/logs/` не существует — не бросать, все секции рендерятся как `[empty]`/пропущенные без `FileNotFoundException`.

6. **Тесты** — mirror Robolectric-паттерн из `core-runtime/src/test/kotlin/app/sanctum/machina/core/log/ErrorLogTest.kt` (JUnit4 + `@RunWith(RobolectricTestRunner::class)` + `@Config(sdk = [33])` + `ApplicationProvider.getApplicationContext()` + явные `@Before setUp()` / `@After tearDown()` с `logFile.parentFile?.deleteRecursively()`). `TapCounterTest` — без Robolectric, обычный JUnit4.

- `app/src/main/kotlin/app/sanctum/machina/logexport/LogExportManager.kt` — **создать**. `@Singleton class LogExportManager @Inject constructor` (Context via `@ApplicationContext`, `DeviceInfoCollector`, `LogcatReader`). Методы `suspend buildExport(source: ExportSource): String` и `suspend writeTo(uri: Uri, content: String)`. Enum `ExportSource { About, CrashReport }` — в том же файле или рядом top-level. **Edge cases:**
- `filesDir/logs/` не существует (fresh install или после `CrashState.clear()` в параллельной сессии) — `buildExport` не бросает, все секции `[empty]`, без `FileNotFoundException`.
- `crash.log` / `errors.log` нулевого размера (файл есть, но пустой) — рендерится как `[empty]`, а не как пустая секция с голым разделителем.
- `errors.log.1` отсутствует — **секция пропускается целиком** (ни разделителя, ни placeholder'а) — отличается от `crash.log` / `errors.log`, которые всегда рендерятся с `[empty]`. Это явное требование tech-spec.
- Очень большой `crash.log` (>100 KB) — head-truncate до 100 KB + маркер. Очень большой logcat → tail-truncate до 100 KB + маркер. `errors.log` уже ≤2 MB по инварианту ротации в `ErrorLog`, доп-защиту не ставим — доверяем инварианту (tech-spec Decision 7).
- `ProcessBuilder("logcat", ...).start()` кидает `IOException` на некоторых OEM (paranoid mode) — `DefaultCommandRunner` должен это поймать и вернуть `LogcatResult(stdout=ByteArray(0), exitCode=null, timedOut=false)` → в итоге `[logcat unavailable: empty]` (или отдельный reason — **не пробрасывать IOException** наружу).
- Pipe-buffer deadlock: stdout >64 KB до `waitFor` — если читать **после** `waitFor`, процесс заблокируется на write. Поэтому чтение — **в отдельном потоке**, стартует **до** `waitFor`.
- SAF URI может указывать на удалённый провайдер, который закроет `OutputStream` асинхронно — `.use { }` гарантирует корректный cleanup, `writeTo` не ловит `IOException` (передаёт вызывающему).
- `Process.myPid()` в `:crash` процессе даёт pid нового процесса → logcat для него будет пустым (Decision 8: поэтому `CrashReport` ветка вообще не вызывает logcat).
- Boundary тест `TapCounter`: 2 с ровно — inclusive, 2 с + 1 нс — exclusive. Гарантирует стабильность семантики к возможному будущему переходу на ms.
- **Non-Hilt construction от `:crash` процесса (Task 4):** `LogExportManager(applicationContext)` вызывается из `CrashReportActivity.onCreate` БЕЗ какой-либо Hilt-инъекции. Это означает: (а) `DeviceInfoCollector`, `LogcatReader`, `AndroidDeviceInfoProvider`, `DefaultCommandRunner` — все `public` с `public` конструкторами, (б) secondary конструктор `LogExportManager(context: Context)` делегирует в первичный, не дублирует логику, (в) на пустом `filesDir` (fresh `:crash` процесс) `buildExport(ExportSource.CrashReport)` не бросает. Покрыто `LogExportManagerTest::nonHiltConstruction_instantiatesWithoutInjection`.
- **Hilt `LogExportModule` duplicate-binding риск:** `LogExportManager`, `CrashState`, `DeviceInfoCollector`, `LogcatReader` имеют `@Singleton @Inject constructor` — этого достаточно для Hilt-графа. Добавление `@Provides fun provideLogExportManager(...)` в `LogExportModule` приведёт к compile-error `DuplicateBindings`. Гвард в AC + grep-проверке.

**Implementation hints:**
- Формат header'а — см. tech-spec «Data Models» (строки `applicationId: ...`, `version: 0.1.0 (1), debug=true`, `device: Honor / HONOR 200 / Android 14 (API 34)`, `memory: total=11.7 GB, available=6.2 GB`, список моделей с отступом `  - <id> (<size>)`). Размер памяти форматировать в GB с одним знаком после точки через ОДНУ helper-функцию, чтобы `DeviceInfoCollectorTest` мог проверить exact-match.
- Section dividers: **ровно** `=== crash.log ===` (без лидирующих/trailing символов), между секциями — одна пустая строка (`"\n\n"`).
- Tail-truncate logcat: работаем в байтах до последнего шага; для byte→String декодирования при обрезке посреди UTF-8 multibyte-символа использовать `String(bytes, Charsets.UTF_8)` (JVM заменит битые начальные байты на `?`) или `CharsetDecoder` с `CodingErrorAction.REPLACE` — задокументировать выбор в KDoc.
- Head-truncate `crash.log`: аналогично — `bytes.copyOfRange(0, 100 * 1024)` + декодирование в UTF-8.
- `@Singleton` + `@Inject constructor` — Hilt автоматически подхватит class-level bindings; `@Provides`/`@Binds`-модуль нужен ТОЛЬКО для интерфейса `CommandRunner` (прод-биндинг на `DefaultCommandRunner`) и опционально для `DeviceInfoProvider` (если вынесен в интерфейс). `LogExportModule` — минимальный, в том же package. **Критично:** `LogExportManager`, `CrashState`, `DeviceInfoCollector`, `LogcatReader` НЕ перечисляются в `@Provides`/`@Binds` — класс-уровневые `@Singleton @Inject constructor` уже дают Hilt-биндинг; дублирование → `DuplicateBindings` compile-error Hilt'а.
- `CommandRunner` — обычный `interface`, не `fun interface`, чтобы `RecordingCommandRunner` мог хранить state (`var lastArgv: List<String>?`).
- Следуй стилю `ErrorLog.kt` — `suspend` + `withContext(Dispatchers.IO)` вокруг ВСЕХ файловых операций, никаких блокирующих операций на caller-потоке.

**Security (OWASP A03 гвард):**
- `ProcessBuilder` принимает **list** аргументов, не строку — shell-интерполяция невозможна by design. Тест `argvShape_exactlySixArgs_knownPositions` закрепляет эту инвариантность.
- `--pid=${Process.myPid()}` — `myPid()` возвращает `Int`, интерполяция безопасна; формально `argv[4].matches(Regex("^--pid=\\d+$"))` это и ловит.
- `LogExportManager.writeTo(uri, content)` не логирует content и не валидирует URI — это контракт SAF, URI выбирает пользователь (Decision 11 явно принимает non-redaction).
