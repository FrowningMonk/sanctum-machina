# Task 2: CrashHandler + CrashState file primitives

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Фундамент для захвата и «памяти» аварийных падений в Phase 2.5. На этой таске реализуется невидимая для пользователя плотва — три файла в `app/.../crash/`: сам `UncaughtExceptionHandler`, seam-интерфейс `Killer` для юнит-тестов без смерти JVM, и `@Singleton` `CrashState` с `StateFlow<Boolean>`, который по состоянию файловой системы сообщает, что «прошлый запуск упал и баннер ещё не скрыт».

Почему именно этот набор и именно этой таской:

- `CrashHandler` пишет `crash.log` (overwrite, head-truncated ≤100 КБ), удаляет `crash.log.dismissed`, стартует `CrashReportActivity` explicit-интентом (`FLAG_ACTIVITY_NEW_TASK or FLAG_ACTIVITY_CLEAR_TASK`) и убивает главный процесс через инъектируемый `Killer`. Внешний `try { … } catch (Throwable)` короткозамыкает на `Process.killProcess` через тот же `Killer` при любом внутреннем сбое и оставляет след в logcat через `android.util.Log.e` — это защита от бутстрэп-петли (Risks row 1, Decision 5).
- `CrashState` — однотонный Hilt-сервис главного процесса, который знает, виден баннер рестарт-сценария на `ModelManagerScreen` или нет. Состояние filesystem-only: `crash.log` существует ∧ `crash.log.dismissed` не существует (Decision 6). Без DataStore, без SharedPreferences — через диск главный и `:crash` процесс уже и так общаются.
- `Killer` — мелкий интерфейс (`fun kill(pid: Int)`) с дефолтной реализацией-синглтоном `Killer.Default` (companion object внутри интерфейса), делегирующей в `Process.killProcess(pid)`. В продакшен-коде `CrashHandler` получает `Killer.Default` по умолчанию, в тестах — `RecordingKiller`, который внутри `kill()` читает `crash.log` и сохраняет байты, чтобы тест мог утверждать «файл лежал на диске к моменту смерти процесса». Task 5 будет вызывать `Killer.Default` по тому же имени — без двусмысленности.

В этой таске нет UI, нет манифеста и нет установки хэндлера в `SanctumApplication` — это Task 4/5/7. Здесь только file/handler-прослойка с юнит-тестами на Robolectric, повторяющими стиль `ErrorLogTest.kt` (`RobolectricTestRunner`, `@Config(sdk = [33])`, чистка `filesDir/logs/` в `@Before`/`@After`, no Mockito/MockK — handrolled fakes, code-research §9.3). Decisions, которые нужно соблюсти буквально: D4 (process-guard — НЕ здесь, в Task 5), D6 (filesystem-only state), D10 (не звать `ErrorLog.e`, не расширять `ALLOWED_COMPONENTS`).

- `app/src/main/kotlin/app/sanctum/machina/crash/CrashHandler.kt` — НОВЫЙ. Класс `CrashHandler` реализует `Thread.UncaughtExceptionHandler`. Конструктор принимает `Context` и `Killer` (с дефолтом `Killer.Default`). **Edge cases:**
- Исключение без сообщения (`RuntimeException()` без `message`) — шапка должна записать пустую строку для `message:`, не падать на `null`.
- Исключение с chained `cause` — `printStackTrace(PrintWriter)` само печатает `Caused by: ...`, ничего дополнительно не делаем.
- `filesDir/logs/` не существует — первый же `mkdirs()` создаст, стандартная практика.
- `crash.log` уже существует (предыдущий несохранённый краш) — overwrite-семантика (`writeText`, не `appendText`), спецификация явно требует (Decision 7).
- `crash.log.dismissed` не существует на момент удаления — `File.delete()` возвращает `false`, не бросает; игнорируем результат.
- На Robolectric `filesDir` — временный каталог, между тестами чистится руками через `parentFile?.deleteRecursively()` в `@Before`/`@After` (зеркало `ErrorLogTest`).
- Head-truncation на границе UTF-8 code-point: работаем на байтах (`toByteArray(Charsets.UTF_8)`). `String(bytes, 0, cutoff, Charsets.UTF_8)` корректно игнорирует неполные хвосты — это стандартное поведение JVM. Затем конкатенируем маркер. Тест `stacktraceOver100KB_truncationMarkerAtEnd` проверяет итоговый размер ≤ 100 КБ + маркер в конце.
- Internal failure — в тесте используй конструкторный test-seam (форма на усмотрение имплементатора: single-method interface, high-order function, что-то ещё — см. Files to create). Тест подставляет реализацию, кидающую `IOException("test")` на записи `crash.log`. Это detrministic `handlerInternalFailure_killsOnce` без подмены прав файловой системы.

**Implementation hints (НЕ псевдокод):**
- Conventions: `CrashHandler` — верхний слой `:app`, НЕ `:core-runtime` (имеет прямые Android-зависимости вроде `Intent`). Модульная граница `:core-*` без Compose/Activity сохранена (см. patterns.md § Module boundary).
- Hilt: `CrashState` — обычный `@Singleton class` с `@Inject constructor(@ApplicationContext)`. Модуль отдельный НЕ нужен: Hilt автогенерит provider для `@Singleton class` с `@Inject` конструктором. Если в проекте есть центральный `LogExportModule` (появится в Task 3) — НЕ добавляй туда `@Provides CrashState`, это избыточно и источник расхождения с реальным графом.
- `Killer` — `companion object Default : Killer` внутри интерфейса достаточно для Task 2; в Task 5 `SanctumApplication.onCreate` вызовет `Thread.setDefaultUncaughtExceptionHandler(CrashHandler(this, Killer.Default))`.
- `CrashReportActivity` ссылку оставь через строковый FQN (`Intent().setClassName(context.packageName, "app.sanctum.machina.crash.CrashReportActivity")`) с TODO-комментарием на Task 4. Это развязывает wave 1 от wave 2 и позволяет юнит-тестам компилироваться. Tests сами `startActivity` не триггерят (Robolectric его просто логирует в `ShadowApplication.getNextStartedActivity`; при желании ассертим факт вызова).
- Test-seam для internal failure: добавь `internal` (не `private`) параметр конструктора, через который `handlerInternalFailure_killsOnce` может уронить путь записи `crash.log` детерминированно (`IOException("test")`). Форма — выбор имплементатора (single-method interface, high-order function `(File, String) -> Unit`, и т.д.); главное — совместимость с production-путём через дефолт и возможность теста подставить свой падучий экземпляр.
- Стиль тестов: НЕТ `runTest` (нет suspend), НЕТ Mockito/MockK (не подключены в `libs.versions.toml`). Handrolled `RecordingKiller` — это и есть проектный паттерн (code-research §9.3).
- `CrashState.refresh()` может внутри быть `@Synchronized` (или `synchronized(this)`) для защиты от одновременных вызовов из `LaunchedEffect` и `clear()` на одном инстансе; но это обычный JVM lock, НЕ coroutine mutex — не тащи `kotlinx.coroutines.sync.Mutex` сюда без нужды (`ErrorLog` использует его из-за IO+suspend, здесь обе стороны синхронные).
- НЕ выноси файловые пути-константы в публичный API. Это внутренняя деталь класса; Task 3 `LogExportManager` читает ровно те же файлы — но пути строит по тем же именам («`logs`/`crash.log`/`crash.log.dismissed`») независимо. Если захочется центровать — это Task 8 (Code Audit) решит, не Task 2.
