# Task 9: Security Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:security-auditor` — [skills/security-auditor/SKILL.md](../../../.claude/skills/security-auditor/SKILL.md)

## Description

Full-feature security audit всех файлов, созданных и изменённых в задачах 1–7 phase-2.5-logexport, против OWASP Top 10 (редакция 2021+). Аудитор выступает финальной ревью-инстанцией Audit Wave для измерения безопасности — сам себе reviewer. На выходе — письменный отчёт с перечислением всех проверенных OWASP-осей, найденных уязвимостей (classified by severity) или явного подтверждения отсутствия, и рекомендованных фиксов, если они есть.

Контекст: фича добавляет трёхпутевой экспорт диагностических логов — `CrashHandler` в основном процессе пишет `crash.log` и запускает `CrashReportActivity` в отдельном OS-процессе `:crash`; `LogExportManager` собирает `.txt` из header + `crash.log` + `errors.log` + `errors.log.1` + own-pid logcat и пишет его в SAF URI; About-экран и ModelManagerScreen вызывают SAF через `CreateDocument("text/plain")`. Ключевые архитектурные решения (Decisions 1–12) перечислены в `tech-spec.md`. Аудит проверяет, что эти решения действительно реализованы в коде и что принятые residual-риски (особенно Decision 11 — нефильтрованный `.txt` на Phase 2.5) документированы как осознанный trade-off, а не как blind spot.

Задача не включает написание кода или тестов — только чтение и вынесение вердикта. Если найдены уязвимости критической или высокой степени, они документируются в отчёте; фикс планируется отдельной задачей (не в рамках Audit Wave).

## What to do

1. Прочитать всю feature-спецификацию: `user-spec.md`, `tech-spec.md`, `decisions.md` в папке фичи.
2. Прочитать все исходные файлы, созданные и изменённые задачами 1–7 (список — в секции `Details.Files`).
3. Пройтись по OWASP Top 10 dimensions ниже — по каждой вынести вердикт: **pass / finding / not applicable**. Для finding'ов указать severity (Critical / High / Medium / Low), attack vector, impact, actionable fix.
4. Отдельно сверить, что Decision 11 (нефильтрованный экспорт) явно задокументирован как осознанный residual risk и ссылается на user-spec "Ограничения"; отметить в отчёте как **accepted residual risk**, а не как уязвимость.
5. Написать отчёт в `work/phase-2.5-logexport/logs/working/task-9/security-audit-report.md` с разделами: Summary, OWASP Top 10 Evaluation (по каждой оси), Accepted Residual Risks (Decision 11 + SIGSEGV fallback), Findings (если есть), Recommendations (если есть).
6. Если найдены Critical/High — явно пометить их в Summary и рекомендовать завести фикс-задачу перед деплоем.

## Acceptance Criteria

- [ ] Отчёт написан в `work/phase-2.5-logexport/logs/working/task-9/security-audit-report.md`.
- [ ] OWASP A01 (Broken Access Control) явно оценён: `android:exported="false"` на `CrashReportActivity`, `taskAffinity=""` для изоляции таска, отсутствие внешних caller'ов, проверено по `AndroidManifest.xml`.
- [ ] OWASP A03 (Injection) явно оценён: `ProcessBuilder("logcat", …)` получает массив строковых литералов + `Process.myPid()` integer — shell-аргументы не интерполируются из user-controlled input. Проверено по `LogcatReader.kt` и по argv-shape тесту в `LogcatReaderTest`.
- [ ] OWASP A05 (Security Misconfiguration) явно оценён: `:crash` process permissions (наследуются от родителя, никаких дополнительных `<uses-permission>`), resolution темы `Theme.Sanctum`, отсутствие `FileProvider` (Decision 2 — keeps attack surface small).
- [ ] OWASP A08 (Software and Data Integrity Failures) явно оценён: что физически попадает в экспортный `.txt`. Отдельно зафиксировано: SELinux-context / hardware-identifier leakage из logcat (Decision 11) и полные stacktrace'ы в `crash.log` (Decision 11) — classified as accepted residual risks for Phase 2.5 closed alpha.
- [ ] OWASP A09 (Security Logging and Monitoring Failures) явно оценён: outer try/catch → `android.util.Log.e` breadcrumb → `Process.killProcess` в `CrashHandler` предотвращает bootstrap loop; SAF IOException path даёт user-visible feedback (Toast/Snackbar) и сохраняет `crash.log` для retry.
- [ ] Secrets/credentials exposure в экспортном `.txt` проверен: `DeviceInfoCollector` output и формат `errors.log` / `crash.log` / logcat осмотрены на предмет API-ключей, токенов в model-download URL, путей с device-owner идентификаторами.
- [ ] SAF URI permission scope проверен: `CreateDocument` возвращает write-only URI; в коде нет `openInputStream` на экспортный URI и нет `takePersistableUriPermission` / `ContentResolver.takePersistableUriPermission` calls.
- [ ] Decision 11 явно отмечен в отчёте как **accepted residual risk** (не как blind spot) со ссылкой на user-spec "Ограничения" "Приватность: на фазе закрытого тестирования фильтрация содержимого не применяется".
- [ ] Для каждого finding'а указаны severity, file:line, attack vector, impact, actionable fix. Отсутствие finding'ов для оси — тоже валидный результат, фиксируется явно.

## Verification Steps

### Automated

Не применимо. Аудит не запускает тесты — он их читает. Дополнительно аудитор может запустить structural greps для подтверждения инвариантов:

- `grep -n "android:exported=\"true\"" app/src/main/AndroidManifest.xml` → 0 matches на `CrashReportActivity` (должно быть `false` или отсутствовать, так как activity не имеет intent-filter).
- `grep -rEn "openInputStream|takePersistableUriPermission" app/src/main/kotlin/app/sanctum/machina/logexport app/src/main/kotlin/app/sanctum/machina/crash` → 0 matches (SAF URI — write-only, permission не persist'ится).
- `grep -rEn "Runtime.getRuntime\\(\\).exec|sh -c|ProcessBuilder\\(\"/bin" app/src/main/kotlin/app/sanctum/machina/` → 0 matches (единственный ProcessBuilder — `LogcatReader` с фиксированным argv).
- `grep -rEn "BuildConfig\\.DEBUG" app/src/main/kotlin/app/sanctum/machina/ui/about/AboutScreen.kt` → 0 matches на dev-gesture (Decision 12 — жест активен в release на Phase 2.5, это осознанный выбор из user-spec).

Отсутствует. Tech-spec для Task 9 не задаёт `Verify-smoke` — аудит это статический анализ кода и спецификаций, а не executable check.

### User

## Details

| Файл | Что проверять | OWASP оси |
|------|---------------|-----------|
| `AndroidManifest.xml` | `android:exported="false"`, `taskAffinity=""`, `excludeFromRecents="true"`, отсутствие `<uses-permission>` под `:crash`, ровно один `android:process=":crash"` | A01, A05 |
| `SanctumApplication.kt` | Process-name guard стоит до install handler'а; install — до `DefaultDownloadRepository.mainActivityFqn` | A05, A09 |
| `CrashHandler.kt` | Outer `try { … } catch(Throwable) { Log.e(…); Process.killProcess(myPid) }`; не вызывает `ErrorLog.e` (Decision 10); head-truncation ≤100 KB; удаляет `.dismissed` до записи | A09 |
| `CrashState.kt` | Пути к `crash.log` / `.dismissed` собираются из `filesDir` (internal storage, private) — не `externalCacheDir` / `externalFilesDir` | A01, A05 |
| `CrashReportActivity.kt` | Plain `ComponentActivity`, NOT `@AndroidEntryPoint` (Decision 5); в `onCreate` нет dynamic resource lookups по user input; SAF-launcher имеет in-flight guard | A01, A05 |
| `LogExportManager.kt` | `writeTo` использует `openOutputStream(uri).use { … }`; null-stream trap (`IOException`); нет вызовов `openInputStream` на экспортный URI; `buildExport` целиком in-memory и head/tail-truncates per Decision 7 | A01, A08 |
| `DeviceInfoCollector.kt` | Header содержит только то, что перечислено в tech-spec Data Models (manufacturer, model, API level, memory, active model id, downloaded model ids). Никаких путей с device-owner (`/data/data/<uid>`), никаких tokens из model download URL (model id ≠ URL), никакого IMEI/ANDROID_ID/BSSID | A02, A08 |
| `LogcatReader.kt` | `ProcessBuilder("logcat", "-d", "-v", "threadtime", "--pid=${Process.myPid()}", "*:E")` — 6 аргументов, все литералы кроме `--pid=<int>`; нет shell wrapping; `redirectErrorStream(true)`; concurrent read thread + `waitFor(2, SECONDS)` предотвращает pipe-buffer deadlock; `destroy()` на timeout | A03, A05, A09 |
| `TapCounter.kt` | Pure JVM; нет Android-API calls; детектор — чистая функция от `nowNanos: () -> Long`. Security-irrelevant, но подтвердить отсутствие сетевого / файлового side-effects | — |
| `AboutScreen.kt` | Dev-gesture: `throw RuntimeException(...)` из onClick в release build (Decision 12 — осознанно). 7-tap детектор не wrap'ится в `BuildConfig.DEBUG` на Phase 2.5 — это accepted trade-off, зафиксировать в отчёте как residual risk с ссылкой на Phase 5 backlog | A05 |

**OWASP Top 10 (2021) — explicit evaluation axes:**

- **A01 Broken Access Control** — фокус: доступ к `CrashReportActivity` только из системного крэш-сценария (не exported); `:crash` taskAffinity изолирует таск в recents; SAF URI — user-picked destination (не произвольная запись в чужой package).
- **A02 Cryptographic Failures** — N/A (нет шифрования, нет паролей, нет TLS — но подтвердить отсутствие случайного секрета в header'е).
- **A03 Injection** — фокус: `ProcessBuilder` argv не содержит интерполяции user input; `Process.myPid()` возвращает int; нет shell wrapping; аргумент `--pid=<int>` безопасен даже если бы он был user-controlled (int cast режет shell metacharacters).
- **A04 Insecure Design** — фокус: архитектурные решения (Decisions 1–12) не создают новых attack surface'ов vs. базовой Android-модели. Отсутствие FileProvider (Decision 2) — явный security-positive выбор.
- **A05 Security Misconfiguration** — фокус: `:crash` process не получает `<uses-permission>` сверх того, что у main; theme `@style/Theme.Sanctum` существует; manifest-атрибуты `exported`, `excludeFromRecents`, `taskAffinity` выставлены явно.
- **A06 Vulnerable and Outdated Components** — tech-spec Dependencies section говорит "No new external dependencies". Подтвердить чтением `app/build.gradle.kts` и `gradle/libs.versions.toml` (diff относительно main должен быть пуст в dep-секциях). Существующие зависимости под CVE-проверку не попадают — это не scope Task 9 (separate concern, обычно отдельная задача зависимостей).
- **A07 Identification and Authentication Failures** — N/A (аутентификации в фиче нет).
- **A08 Software and Data Integrity Failures** — фокус: содержимое экспортного `.txt`. Явно задокументировать, что:
  - SELinux-контексты и hardware-идентификаторы из logcat МОГУТ появляться в `.txt` (Decision 11) — accepted residual risk на Phase 2.5 closed alpha.
  - Полные stacktrace'ы из `crash.log` идут verbatim без сокращения (Decision 11) — accepted residual risk.
  - Model download URL'ы с токенами (если они есть в `errors.log`) — **проверить отдельно** чтением формата `ErrorLog` и списка вещей, которые туда попадают в текущем codebase.
- **A09 Security Logging and Monitoring Failures** — фокус: bootstrap-loop guard в `CrashHandler` (outer try/catch), SAF-IOException user feedback (Toast/Snackbar), preservation `crash.log` на неудачную запись (retry path).
- **A10 Server-Side Request Forgery** — N/A (приложение мобильное, серверных запросов в фиче нет).

**Secrets scan:**
- `DeviceInfoCollector` output — перечислить все поля, убедиться, что среди них нет API-ключей, session-токенов, download-URL.
- `errors.log` формат — прочитать `ErrorLog.kt`, проверить, какие cause'ы туда пишутся и не могут ли они случайно содержать Authorization-header (например, если HTTP-клиент логирует failed request).
- `logcat *:E` — по сути ловит третьепартийные error-лог записи; это residual risk, но зафиксировать явно.

**Dependencies:** Task 9 зависит от Tasks 1–7 (все feature-файлы должны быть готовы). Аудитор IS the review — отдельные reviewer'ы не запускаются.

**Edge cases:**
- Если аудит находит Critical/High finding — это может блокировать переход в Final Wave (Task 11). Явно флагнуть в Summary отчёта.
- Если аудит находит Medium — рекомендовать фикс, но не блокировать (по договорённости с project lead'ом, но в рамках closed alpha — допустимо).
- Если аудит подтверждает что Decision 11 — осознанный trade-off и он документирован в user-spec "Ограничения" и tech-spec Decision 11 — это не finding, а accepted residual risk.

**Implementation hints:**
- Читать файлы целиком, не по snippet'ам — security-контекст часто в импортированной функции или в call-graph'е вверх.
- Для каждой OWASP оси вынести explicit verdict — даже "N/A, не применимо потому что X" лучше, чем пропуск.
- Для finding'ов — attack scenario должен быть реалистичен. "Теоретически возможно если X" с пятью предпосылками — обычно Low severity.
- Отчёт — markdown, человекочитаемый, без JSON. Финальный output этой задачи — путь к отчёту.
