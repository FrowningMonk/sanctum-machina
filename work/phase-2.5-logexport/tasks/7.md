# Task 7: ModelManagerScreen — restart banner wiring

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Вшиваем в `ModelManagerScreen` баннер о прошлом аварийном завершении (Flow B из tech-spec). Это финальная UI-поверхность Phase 2.5: при холодном старте после краха пользователь видит карточку над списком моделей с двумя действиями — «Сохранить лог» и ✕ («скрыть»). «Сохранить лог» открывает SAF-диалог, собирает экспорт через `LogExportManager.buildExport(About)` (Decision 8 — логкэт из живого главного процесса доступен, в отличие от пути через `:crash`), пишет на выбранный URI, по успеху вызывает `CrashState.clear()` (удаляет оба файла, `hasUnresolvedCrash` становится `false`, баннер пропадает) и показывает Snackbar «Лог сохранён». ✕ вызывает `CrashState.markDismissed()` — создаёт `crash.log.dismissed`, баннер скрывается; следующий краш удалит `.dismissed` в обработчике (Flow B шаг 6) и баннер вернётся.

Видимость баннера полностью продиктована файловой системой (Decision 6 — один источник правды между `:crash` и main). На входе в экран дергаем `crashState.refresh()` через `LaunchedEffect(Unit)`, чтобы перечитать диск после внешних изменений (новый краш между сессиями, ручное удаление файлов).

Сейчас на экране нет `SnackbarHost` (code-research §3) — добавляем его, иначе показывать Snackbar некуда. Паттерн — как в `ChatScreen.kt` (см. `snackbarHostState = remember { SnackbarHostState() }` + `rememberCoroutineScope` + `Scaffold(snackbarHost = { SnackbarHost(hostState = ...) })`). Примечание: `ChatScreen` разбит на две композабл-функции — состояние создаётся в внешнем композабле (`ChatScreen.kt:68`) и потребляется во внутреннем private композабле, владеющем `Scaffold` (`ChatScreen.kt:258`); `ModelManagerScreen` — одна композабл-функция, поэтому `snackbarHostState` создаётся и потребляется inline в той же функции.

Зависимости: `CrashState` и `LogExportManager` инжектируются Hilt (оба `@Singleton` в главной Hilt-графе — см. tech-spec Shared resources). Инжектить проще всего через существующий `ModelManagerViewModel` (добавить поля в `@Inject`-конструктор + проэкспозить наружу) либо через `hiltViewModel()`-owned wrapper. Решение по точке инъекции оставляем исполнителю — важно, что это главный процесс, и Hilt-граф доступен полностью.

## What to do

1. Добавить `SnackbarHost` в `Scaffold` (по паттерну `ChatScreen.kt`): держать `SnackbarHostState` через `remember`, создать `rememberCoroutineScope()` для запуска `showSnackbar`, передать `snackbarHost = { SnackbarHost(hostState = ...) }` в `Scaffold`.
2. Получить ссылки на `CrashState` и `LogExportManager` (через `ModelManagerViewModel` или через отдельный `hiltViewModel()`-wrapper — см. Implementation hints). Для главной процессной Hilt-графы оба доступны как `@Singleton`.
3. Собирать `CrashState.hasUnresolvedCrash` через `collectAsStateWithLifecycle()`.
4. На входе в экран: `LaunchedEffect(Unit) { crashState.refresh() }` — Decision 6, фс-источник правды.
5. Поднять локальное состояние in-flight guard: `val inFlight = remember { mutableStateOf(false) }` — передавать в `RestartCrashBanner` (эту ручку композабл ожидает из Task 4); повторные нажатия на «Сохранить лог» до возврата из SAF игнорируются.
6. Подготовить SAF-лончер через `rememberLauncherForActivityResult(ActivityResultContracts.CreateDocument("text/plain"))`. Suggested filename формировать на момент клика как `sanctum-log-YYYYMMDD-HHmm.txt` (текущее локальное время, `Locale.US`).
7. В колбэке SAF-лончера (в `finally` сбрасывать `inFlight.value = false` независимо от ветки):
   - `uri != null` → на корутинной области вызвать `LogExportManager.buildExport(ExportSource.About)` → `LogExportManager.writeTo(uri, content)`. Успех: `crashState.clear()` + `snackbarHostState.showSnackbar("Лог сохранён")`. `IOException`: `snackbarHostState.showSnackbar("Не удалось сохранить лог")` + **НЕ** вызывать `clear()` — баннер остаётся, пользователь может повторить.
   - `uri == null` (Cancel): ничего не показываем, `clear()` не вызываем — баннер остаётся.
8. Отрендерить `RestartCrashBanner` (из Task 4) над `ModelList` **только когда** `hasUnresolvedCrash == true`. Прокинуть:
   - `onSaveClick` → `inFlight.value = true; launcher.launch(filename)`;
   - `onDismissClick` → `crashState.markDismissed()` (баннер уйдёт из-за перехода флоу в `false`);
   - `inFlight` state для блокировки повторных нажатий.
9. Структурно: обернуть `ModelList` + баннер в `Column` внутри `Scaffold` content-слота; баннер идёт первым, `ModelList` — вторым. `innerPadding` применить к внешнему `Column`, `ModelList` получит `PaddingValues(0.dp)` чтобы не дублировать отступы. **Важно:** также заменить существующий аргумент `contentPadding = innerPadding` на вызове `ModelList` в `ModelManagerScreen.kt:72` на `contentPadding = PaddingValues(0.dp)` — без этой правки отступ удвоится.
10. Строки только через `R.string.*` (ключи из Task 1) — **не** хардкодить русский текст (lint `HardcodedText`). `ErrorLog.e(...)` в этом файле не появляется (общий принцип — ошибки SAF видит пользователь через Snackbar).

## Details

**Files:**
- `app/src/main/kotlin/app/sanctum/machina/ui/modelmanager/ModelManagerScreen.kt` — текущее состояние (после прочтения): функциональный `@Composable` с `Scaffold(topBar = {...}) { innerPadding -> ModelList(...) }`, `SnackbarHost` отсутствует, инъекция только через `hiltViewModel()` → `ModelManagerViewModel` (сейчас только `models: StateFlow<List<ModelEntry>>`, `navEvents`, и колбэки `onDownload`/`onCancel`/`onLoad`). Что добавить: `SnackbarHost`, `rememberCoroutineScope`, SAF-лончер, `LaunchedEffect(Unit) { crashState.refresh() }`, условный рендер `RestartCrashBanner`, обёртка содержимого в `Column { Banner; ModelList }`. Точка инъекции `CrashState`/`LogExportManager` — через расширение `ModelManagerViewModel` (добавить в `@HiltViewModel`-конструктор и вынести геттеры) ИЛИ через отдельный `@HiltViewModel` crash-панель wrapper. Исполнитель выбирает — оба допустимы.

**Dependencies:**
- Task 1 — строки (особенно «Лог сохранён», «Не удалось сохранить лог», строки баннера).
- Task 2 — `CrashState` (Hilt `@Singleton`, API `hasUnresolvedCrash`, `refresh`, `markDismissed`, `clear`).
- Task 3 — `LogExportManager.buildExport(ExportSource.About)`, `LogExportManager.writeTo(uri, content)`, enum `ExportSource`.
- Task 4 — `RestartCrashBanner` composable (сигнатура: `onSaveClick`, `onDismissClick`, in-flight guard state, иконки Warning/Close).
- Task 5 — `CrashHandler` установлен в `SanctumApplication` (иначе никакого `crash.log` не появится — невозможно проверить user-verify).
- Task 6 — dev-gesture в `AboutScreen`, нужен чтобы физически сгенерировать краш для ручной проверки.

**Edge cases:**
- Холодный старт без `logs/` папки (freshInstall): `crashState.refresh()` должен молча вернуть `false` и не падать — гарантировано тестами `CrashStateTest.freshInstall_noLogsDir_succeeds`.
- Пользователь успел убить процесс между `buildExport` и `writeTo`: обработка как при обычной корутинной отмене — Snackbar не появляется, `clear()` не вызывается, `crash.log` сохранён, баннер покажется снова.
- `writeTo` вернул `IOException("openOutputStream returned null")` (системная SAF-ошибка от провайдера): покрывается той же веткой «IOException» → Snackbar «Не удалось сохранить лог», баннер остаётся.
- Ротация экрана во время SAF-диалога: SAF-лончер (`rememberLauncherForActivityResult`) переживает пересоздание; `ActivityResultRegistry` доставит колбэк, в `finally` guard всё равно сбросится. `remember { mutableStateOf(false) }` на ротации сбросится — но это приемлемо, т.к. SAF всё равно открыт и видим пользователю, повторный клик уже невозможен.
- Двойной тап «Сохранить лог» до открытия SAF — второй тап игнорируется из-за `inFlight.value == true`.
- Тап «Сохранить лог» при наличии только `crash.log` без `errors.log`: валидный кейс — `buildExport(About)` вернёт заголовок + секцию crash.log + `[empty]` для errors, `logcat` живой → файл пишется успешно, `clear()` удалит `crash.log`.

**Implementation hints:**
- Лёгкий путь для инъекции — расширить `ModelManagerViewModel` полями `crashState: CrashState` и `logExportManager: LogExportManager` (оба `@Inject` через уже имеющийся `@HiltViewModel`-конструктор), выставить наружу как `val crashState get() = _crashState` или функции-прокси. Тогда из composable-экрана вызовы идут через `viewModel.*` — в одном стиле с остальным экраном.
- Альтернатива — отдельный `@HiltViewModel` crash-панель либо Hilt entryPoint. ViewModel-расширение проще.
- Корутины: SAF-колбэк синхронный, но `buildExport` и `writeTo` — `suspend`. Запускать через `rememberCoroutineScope().launch { try { val content = logExportManager.buildExport(ExportSource.About); logExportManager.writeTo(uri, content); crashState.clear(); snackbarHostState.showSnackbar(...) } catch (e: IOException) { snackbarHostState.showSnackbar(...) } finally { inFlight.value = false } }`.
- Формат filename: `SimpleDateFormat("'sanctum-log-'yyyyMMdd-HHmm'.txt'", Locale.US).format(Date())` → передать в `launcher.launch(filename)`. Зафиксировать `Locale.US` чтобы цифры ASCII-только.
- SnackbarHost подключать в `Scaffold(snackbarHost = { SnackbarHost(hostState = snackbarHostState) }, topBar = {...}) { ... }` — `ChatScreen.kt:258` как референс, `ChatScreen.kt:68-74` — создание + collect.
- Порядок композиции внутри `Scaffold` content: `Column(Modifier.padding(innerPadding)) { if (hasUnresolvedCrash) { RestartCrashBanner(...); }; ModelList(contentPadding = PaddingValues(0.dp), ...) }` — чтобы `ModelList` не наследовал лишний `innerPadding` дважды.
- НЕ вызывать `markDismissed()` после успешного save — Dismissed и Saved это разные состояния: Save → `clear()` (удаляет оба файла, «всё чисто»), Dismiss → только touches `.dismissed` (crash.log остаётся на диске — полезно если пользователь потом передумал и пошёл в About делать proactive export).
- Не добавлять `ErrorLog.e(...)` — ошибка SAF уже видна пользователю через Snackbar; это consumer-facing UI, диагностика уходит в сам экспорт.
