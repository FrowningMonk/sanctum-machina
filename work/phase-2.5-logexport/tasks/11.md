# Task 11: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Финальная приёмка фазы 2.5 перед ручной передачей APK. Агент прогоняет все автоматические гейты (Gradle test / lint / assemble), проверяет структурные инварианты грепами по коду/манифесту, сопоставляет реализацию с каждым критерием приёмки из user-spec и tech-spec, и формирует hand-off список для on-device проверки пользователем (SAF-диалог, `:crash` процесс, баннер на cold-start, 7-тап жест — всё то, что требует инструментации и физического устройства Honor 200).

Это последняя задача фазы: после неё пользователь вручную собирает debug-APK, переносит на устройство и проходит сценарии US-A/B/C. Деплой-задачи нет (мобильный проект, ручной перенос APK по `deployment.md`). Post-deploy проверки нет (on-device checks — пользовательские, MCP-верифицируемой поверхности нет).

Эта задача НЕ пишет код и НЕ имеет TDD-цикла — QA сам по себе является верификацией.

2. **Сверить каждый критерий приёмки** из user-spec "Критерии приёмки" и tech-spec "Acceptance Criteria" читая код + вывод тестов. Сформировать чеклист: AC → PASS / FAIL / DEFERRED. DEFERRED — для AC, которые требуют on-device проверки (SAF-диалог, `:crash` переживает смерть главного процесса, баннер на cold-start, 7-тап жест).

3. **Прогнать структурные грепы** (все должны выполниться):
   - `grep -rEn "androidx.compose|androidx.activity" core-runtime/src/main core-settings/src/main` → 0 совпадений (правило module-boundary из `patterns.md`).
   - `grep -rEn "ErrorLog|errorLog" app/src/main/kotlin/app/sanctum/machina/crash/` → 0 совпадений (Decision 10).

4. **Собрать QA-отчёт** с разделами: (a) Automated Gates (Gradle выводы + PASS/FAIL), (b) Structural Greps (каждая команда + результат), (c) AC Coverage (таблица user-spec/tech-spec AC → PASS/FAIL/DEFERRED), (d) Hand-off Checklist для пользователя (US-A/B/C шаги для on-device проверки на Honor 200). Сохранить в `work/phase-2.5-logexport/logs/working/task-11/pre-deploy-qa-report.md`.

5. **Передать пользователю** hand-off чеклист с шагами on-device проверки по user-spec "Как проверить / Пользователь проверяет":
   - **US-A** (crash path): test-crash через 7-тап в "О программе" → `CrashReportActivity` появляется → "Сохранить лог" → SAF → сохранённый `.txt` содержит все секции → "Отмена" в SAF возвращает на экран без сообщений.
   - **US-B** (restart banner): test-crash → "Закрыть" на CrashReportActivity → перезапуск → баннер виден → "Сохранить лог" ИЛИ ✕ "Скрыть" → корректное поведение на следующих cold-start.
   - **US-C** (proactive export): "О программе" → "Сохранить лог" → сохранённый `.txt` содержит непустую секцию `logcat` (не placeholder).

### User

После того как QA-отчёт готов, агент передаёт пользователю чеклист on-device проверки на Honor 200. Пользователь устанавливает свежий `app-debug.apk` поверх предыдущей версии (если signing conflict — сначала `adb uninstall`) и проходит:

- **US-C / pipeline check:** "О программе" → "Сохранить лог" → SAF → Downloads → `sanctum-log-YYYYMMDD-HHmm.txt` → открыть в файловом менеджере → убедиться, что есть хедер с device info и секции `crash.log` (`[empty]`), `errors.log`, `logcat` (непустой).
- **US-A / crash path:** "О программе" → 7 тапов по строке версии (паузы ≤2 с) → диалог "Спровоцировать тест-краш?" → "Да" → приложение падает → появляется `CrashReportActivity` с двумя кнопками → "Сохранить лог" → SAF → открыть `.txt` → секция `=== crash.log ===` содержит `RuntimeException: test crash from About` и стэктрейс.
- **US-B / restart banner:** снова test-crash → "Закрыть" на CrashReportActivity → запустить приложение → на `ModelManagerScreen` над списком моделей виден баннер "Прошлый запуск завершился аварийно." → "Скрыть" → баннер пропал, cold-start снова — баннер не вернулся → test-crash ещё раз → cold-start — баннер опять виден.
- **Negative path:** CrashReportActivity или AboutScreen → "Сохранить лог" → в SAF нажать системную "Отмена" → Activity/экран остаётся без сообщений, кнопка кликабельна.

## Details

**Report location:** `work/phase-2.5-logexport/logs/working/task-11/pre-deploy-qa-report.md`.

**Report structure (рекомендуемая):**
1. Summary — итоговый статус (READY / BLOCKED), дата, commit.
2. Automated Gates — команда, stdout/stderr (обрезанный хвост), PASS/FAIL.
3. APK size delta — текущий размер vs baseline, дельта.
4. Structural Greps — команда → expected → actual → match.
5. AC Coverage — таблица: AC (источник: user-spec раздел / tech-spec) | Status (PASS / FAIL / DEFERRED) | Evidence (файл:строка или тест-класс).
6. Hand-off Checklist — 4 сценария (US-A / US-B / US-C / Negative) в виде чеклиста с шагами и ожидаемым результатом.
7. Blockers / Open questions — если что-то не сошлось.

**Как мапить AC на PASS / FAIL / DEFERRED:**
- PASS — подтверждается кодом (чтение файлов) или выводом тестов. Пример: "Пойманный uncaught exception приводит к записи crash.log до смерти процесса" → PASS (подтверждено `CrashHandlerTest` c `RecordingKiller`).
- FAIL — обнаружено расхождение. Блокирует передачу пользователю; завести issue / вернуть назад в соответствующую задачу фазы 2.5.
- DEFERRED — требует физического устройства. Примеры: SAF-диалог, `:crash` переживает `Process.killProcess`, баннер на cold-start, 7-тап жест. Эти AC передаются пользователю в hand-off чеклисте.

**APK size baseline:**
- Ожидаемое место: `app/build/outputs/apk/debug/app-debug.apk` после `:app:assembleDebug`.
- Сравнение: последний известный размер APK main. Если хранится — сравнить байты; если нет — задокументировать абсолютный размер собранного APK и пометить baseline "unknown (pre-phase-2.5 APK не сохранён)".

**Dependencies:** Tasks 8, 9, 10 (их отчёты должны существовать как контекст; их findings уже разрешены к моменту старта Task 11 — если что-то открыто, это блокер).

**Edge cases:**
- Тест виснет / падает на CI флaky — перезапустить один раз; если повторяется — FAIL.
- `:app:assembleDebug` не собрался — блокер, возврат в Task 5 / Task 2.
- Grep выдал неожиданное количество совпадений — детально зафиксировать в отчёте, не глотать.
- Критерий user-spec неоднозначен (требует физустройства И подтверждается частично через тесты, например "баннер появляется только если crash.log есть ∧ .dismissed отсутствует") — статус PASS (partial: file-state logic verified in `CrashStateTest`; on-screen render deferred to user) + явная ссылка на тест + добавить в hand-off.

**Implementation hints:**
- Скилл `pre-deploy-qa` содержит методологию формирования AVP-отчёта — следовать ей.
- Бежать грепы через `Grep` tool, а не `bash grep` — избегает проблем с permissions и ripgrep-оптимизацией.
- `git diff` лучше запустить через `Bash` tool: `git diff --stat` сначала для обзора, потом нацеленные diff по конкретным файлам.
- Stdout Gradle-команд стоит сохранить (хотя бы последние ~50 строк) в отчёт — это источник истины для AC.
- Не переиспользовать отчёты Task 8/9/10 как свой — только ссылаться: Task 11 — про verification, не про re-audit.
