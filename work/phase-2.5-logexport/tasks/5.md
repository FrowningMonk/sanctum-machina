# Task 5: SanctumApplication handler install + AndroidManifest activity

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Подключаем машинерию краш-репортинга из Task 2 и Task 4 в два места, где её видит ОС: в `AndroidManifest.xml` (чтобы `CrashReportActivity` можно было запустить в отдельном процессе `:crash`) и в `SanctumApplication.onCreate` (чтобы `CrashHandler` был установлен как `Thread.setDefaultUncaughtExceptionHandler` до того, как основной процесс успеет упасть).

Задача маленькая по объёму кода, но критичная для Flow A из tech-spec (краш → CrashReportActivity → SAF). Без корректной записи в манифесте ОС не сможет поднять активити после смерти основного процесса, а без process-name guard'а `CrashHandler` поставится и в процессе `:crash`, что даёт рекурсию при повторном падении (Decision 4).

Есть тонкость про порядок: `installCrashHandler()` должен идти **до** строки `DefaultDownloadRepository.mainActivityFqn = …`. Это единственный оператор в текущем `onCreate` — если он сам бросит `Throwable` (теоретически возможно при ClassLoader-проблемах), handler уже должен быть живой, иначе краш утонет в системном диалоге без записи в `crash.log`.

Для активити используем уже существующую тему `Theme.Sanctum` (из `app/src/main/res/values/themes.xml`), флаги защиты из Decision 3 (`exported=false`, `excludeFromRecents=true`, `taskAffinity=""`), и процесс `:crash` по Decision 3.

`SanctumApplicationTest` на Robolectric закрывает оба бранча guard'а (main vs `:crash` по имени процесса), страхуя от молчаливой регрессии Decision 4 при будущих рефакторингах `onCreate`.

## What to do

1. **AndroidManifest.xml** — внутри существующего `<application>`-блока (после существующего `<activity android:name=".MainActivity">` и `<service>`-объявления) добавить декларацию `<activity>` для `CrashReportActivity` с атрибутами из Decision 3: `android:name=".crash.CrashReportActivity"`, `android:process=":crash"`, `android:exported="false"`, `android:excludeFromRecents="true"`, `android:taskAffinity=""`, `android:theme="@style/Theme.Sanctum"`. Без `<intent-filter>` — активити запускается только по явному `Intent` из `CrashHandler`.

2. **SanctumApplication.kt** — добавить приватный метод `installCrashHandler()`, который вызывает `Thread.setDefaultUncaughtExceptionHandler(CrashHandler(applicationContext, Killer.Default))`. В `onCreate`, **до** существующей строки `DefaultDownloadRepository.mainActivityFqn = "app.sanctum.machina.MainActivity"`, добавить guard: `if (getProcessName() == packageName) { installCrashHandler() }`. Никаких try/catch вокруг guard'а — внутренние сбои `CrashHandler.uncaughtException` обрабатывает сам handler, а `getProcessName()` доступен с API 28 (minSdk=31 > 28).

3. **SanctumApplicationTest.kt** — новый тест на Robolectric в зеркальном стиле `ErrorLogTest.kt` (`@RunWith(RobolectricTestRunner::class)`, `@Config(sdk = [33])`). Перед каждым тестом сохранять текущий `Thread.getDefaultUncaughtExceptionHandler()`, после — восстанавливать (static state, иначе пачкаем соседние Robolectric-классы). Два теста:
   - happy path: процесс = `packageName` → после `onCreate` `Thread.getDefaultUncaughtExceptionHandler() is CrashHandler`;
   - guard path: процесс = `"${packageName}:crash"` → handler не меняется (остаётся тем, что был до `onCreate`, т.е. `is CrashHandler` должно быть `false`, либо идентичность сохранена).

   Способ стабировать имя процесса: Robolectric предоставляет `ShadowApplication` / рефлексию `Application.getProcessName()` — выбрать механизм, работающий на Robolectric 4.12 без новых зависимостей. Допустимо: (а) подставить свой подкласс `SanctumApplication`, переопределяющий `getProcessName()`; (б) использовать shadow API. Выбор оставлен реализующему, но **не** модифицировать `SanctumApplication` ради тестируемости — класс остаётся как есть.

## Details

### Files

**`app/src/main/AndroidManifest.xml`** — текущее состояние: внутри `<application>` объявлены только `<uses-native-library>`, `<activity android:name=".MainActivity">` (с `LAUNCHER`-фильтром) и `<service android:name="androidx.work.impl.foreground.SystemForegroundService" tools:node="merge">`. Добавить новую `<activity>` рядом (рекомендуется — после `MainActivity`, до `<service>`, чтобы активити сгруппировались). Без `<intent-filter>` и без `<meta-data>`.

**`app/src/main/kotlin/app/sanctum/machina/SanctumApplication.kt`** — текущее состояние:
```
@HiltAndroidApp
class SanctumApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        DefaultDownloadRepository.mainActivityFqn = "app.sanctum.machina.MainActivity"
    }
}
```
После изменения: `super.onCreate()` → process-name guard с `installCrashHandler()` → существующая строка с `mainActivityFqn`. Приватный `installCrashHandler()` — один оператор, `Thread.setDefaultUncaughtExceptionHandler(...)`. Импорты: `app.sanctum.machina.crash.CrashHandler`, `app.sanctum.machina.crash.Killer`.

**`app/src/test/kotlin/app/sanctum/machina/SanctumApplicationTest.kt`** — новый файл. Пакет `app.sanctum.machina`, аннотации `@RunWith(RobolectricTestRunner::class)`, `@Config(sdk = [33])`. Поля: `private lateinit var previousHandler: Thread.UncaughtExceptionHandler?`. `@Before` сохраняет текущий global handler и, при необходимости, сбрасывает его в null/default перед тестом. `@After` восстанавливает через `Thread.setDefaultUncaughtExceptionHandler(previousHandler)` — критично, потому что Robolectric не изолирует JVM-global state между тест-классами.

### Dependencies

- Task 2 (`CrashHandler`, `Killer`) — импортируем `CrashHandler` и `Killer.Default` (companion object внутри интерфейса `Killer`).
- Task 4 (`CrashReportActivity`) — FQN `app.sanctum.machina.crash.CrashReportActivity` должен существовать, иначе `:app:assembleDebug` упадёт из-за `<activity android:name>`, указывающего на несуществующий класс.
- Никаких новых gradle-зависимостей: `robolectric` 4.12, `androidx.test.core` 1.6.1, `junit` 4.13.2 уже на тестовом classpath (tech-spec Dependencies).
- **Wave 2 — это фазовая метка, а не строгий параллелизм.** Tech-spec группирует задачи 4/5/6/7 в Wave 2 как «зависит от Wave 1», но порядок внутри самой Wave 2 задаёт `depends_on`. Для Task 5 это `[2, 4]`: Task 2 даёт `CrashHandler` + `Killer.Default`, Task 4 даёт `CrashReportActivity` (FQN нужен манифесту на этапе `:app:assembleDebug`). Ставить Task 5 раньше, чем Task 4 замерджен, бессмысленно — сборка упадёт.

### Edge cases

- **Handler установился, но MainActivity FQN нет.** Если `installCrashHandler()` отработал, а строка с `mainActivityFqn = …` бросила (теоретически — ClassLoader-проблемы, OOM в статическом инициализаторе `DefaultDownloadRepository`), `CrashHandler` перехватит падение и запишет `crash.log`. Это и есть причина порядка "сначала handler, потом FQN".
- **`:crash`-процесс re-entry в `onCreate`.** При запуске `CrashReportActivity` ОС поднимает свой `SanctumApplication.onCreate` заново в процессе `:crash`. Guard должен исключить установку handler'а — иначе падение внутри `CrashReportActivity` запустит `CrashHandler`, который попытается открыть `CrashReportActivity` ещё раз → рекурсия/второй `:crash`-процесс (Decision 4).
- **`getProcessName()` теоретически возвращает null.** По документации Android это не так на API 28+, но для паранойи guard `if (getProcessName() == packageName)` всё равно корректен — `null == packageName` → false → handler не ставится. Бросать исключение из guard'а не нужно.
- **Несколько `<application>`-манифест-мёрджеров.** `:core-runtime/src/main/AndroidManifest.xml` собственный манифест уже объявляет `<application>` с `tools:node="merge"`-ребёнком. Мёрджер объединит — новая `<activity>` останется ровно одна (grep-проверка это гарантирует).
- **Тесты меняют global `Thread` state.** Без симметричного `@After`-восстановления следующий Robolectric-класс в том же прогоне увидит наш handler и может падать рандомно. `ErrorLogTest` — эталон симметричной очистки, он чистит `logFile.parentFile`; мы чистим `Thread.setDefaultUncaughtExceptionHandler`.

### Implementation hints

- Порядок в `onCreate`: `super.onCreate()` → guard с `installCrashHandler()` → `DefaultDownloadRepository.mainActivityFqn = …`. **Не** оборачивать весь `onCreate` в try/catch — `CrashHandler` это и делает за нас через `Thread.setDefaultUncaughtExceptionHandler`.
- `Killer.Default` — из Task 2; не создавать свой `Killer` здесь, используем продакшн-реализацию, которая действительно убивает процесс.
- FQN `".crash.CrashReportActivity"` в `android:name` — относительный путь от `package` атрибута, который на современных AGP берётся из `namespace` в `build.gradle.kts` (`app.sanctum.machina`) → итого `app.sanctum.machina.crash.CrashReportActivity`. Сверить с фактическим FQN файла, созданного в Task 4.
- Для стаба `getProcessName()` в тесте самый простой путь — локальный подкласс `SanctumApplication` внутри тестового файла, переопределяющий `getProcessName()`: `class TestApplicationMainProcess : SanctumApplication() { override fun getProcessName() = packageName }` / `class TestApplicationCrashProcess : SanctumApplication() { override fun getProcessName() = "$packageName:crash" }`. В тесте вручную инстанцировать и вызывать `onCreate()` — не требует Robolectric-манифест-аннотаций. Если Robolectric-контейнер в `@Config(application = ...)` проще, допустимо и это.
- Проверка "handler установлен" — `assertTrue(Thread.getDefaultUncaughtExceptionHandler() is CrashHandler)`. Проверка "не установлен" — `assertFalse(Thread.getDefaultUncaughtExceptionHandler() is CrashHandler)` (учитывая, что до `onCreate` мы явно выставили non-CrashHandler sentinel или null).
- Не импортировать Compose/Activity-классы в `SanctumApplication` — модуль-бордер правила `patterns.md` здесь не нарушаются, но и новых violation'ов создавать не нужно. `CrashHandler` уже лежит в `:app`, импорт из своего модуля — ок.
