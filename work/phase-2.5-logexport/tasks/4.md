# Task 4: CrashReportActivity (non-Hilt) + RestartCrashBanner composable

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Создаём два новых Compose-файла в `app/.../crash/`, закрывающих UI-половину Flow A (Crash → экран отчёта → SAF) и поставляющих переиспользуемый баннер для Flow B (банер на `ModelManagerScreen`, подключаемый в Task 7).

**`CrashReportActivity`** — standalone `ComponentActivity` **без** `@AndroidEntryPoint` (Decision 5). Живёт в `:crash`-процессе (манифест ставится в Task 5). Два Compose-кнопки («Сохранить лог» / «Закрыть»), SAF через `ActivityResultContracts.CreateDocument("text/plain")` с suggested-name `sanctum-log-YYYYMMDD-HHmm.txt`. `LogExportManager` инстанциируется напрямую через non-Hilt конструктор в `onCreate` — никаких инъекций (вся суть Decision 5: Hilt-граф `:crash`-процесса свежий, ломаться на recovery-пути нельзя).

**`RestartCrashBanner`** — чистый Compose-composable (`Card { Row { Icon, Text, TextButton, IconButton } }`) с callbacks `onSaveClick` / `onDismissClick` и внешним `launching: Boolean` флагом. Никакого hosting-state внутри — всем владеет вызывающий экран (Task 7). Иконки: `Icons.Outlined.Warning`, `Icons.Outlined.Close`. Все строки — из `strings.xml`, добавленные в Task 1.

**In-flight guard flag** (`var launching: Boolean` в `CrashReportActivity`; `remember { mutableStateOf(false) }` в хостинге баннера) — защита от двойного тапа, пока SAF-диалог ещё не вернул результат. Очистка в `finally` result-callback **независимо от ветки** (success / IOException / cancel).

Критично: этот таск ничего не wire-ит. Активность станет достижимой только после Task 5 (манифест) + Task 6 (dev-gesture, чтобы произвести тестовый краш); баннер попадает на экран только в Task 7. Поэтому end-to-end user-верификация отложена до Task 6 и Task 7 — здесь только smoke: компиляция + lint чистые.

## What to do

1. Создать `app/src/main/kotlin/app/sanctum/machina/crash/CrashReportActivity.kt`:
   - Объявить `class CrashReportActivity : ComponentActivity()` (НЕ `@AndroidEntryPoint`).
   - В `onCreate`: инстанциировать `LogExportManager` напрямую через non-Hilt конструктор (передать `applicationContext`, `PackageInfo` через `packageManager.getPackageInfo(...)`, `ActivityManager` через `getSystemService(...)`, всё, что требует `LogExportManager` из Task 3).
   - Объявить in-flight guard: `var launching = false`.
   - Подготовить SAF launcher через `registerForActivityResult(ActivityResultContracts.CreateDocument("text/plain")) { uri -> ... }`. Внутри callback обернуть всю логику в `try { ... } finally { launching = false }`.
   - На `uri != null`: запустить в `lifecycleScope.launch` (или любом coroutine-scope активности) `val content = logExportManager.buildExport(ExportSource.CrashReport)` → `logExportManager.writeTo(uri, content)`. На успехе: удалить `filesDir/logs/crash.log`, показать `Toast.makeText(..., R.string.log_saved, Toast.LENGTH_SHORT).show()`, `finish()`. На `IOException` (включая обёрнутый `openOutputStream returned null`): `Toast.makeText(..., R.string.log_save_failed, Toast.LENGTH_SHORT).show()`, `crash.log` НЕ удалять, кнопки остаются кликабельными (guard очистится в `finally`).
   - На `uri == null` (user cancelled): guard очищается в `finally`, ничего не показываем.
   - `setContent { SanctumTheme { /* two-button screen */ } }`. Компоновка — как в `HeavyChangeDialog.kt` (референс двухкнопочного паттерна), но на уровне полноэкранного layout (`Scaffold` / `Column` с заголовком, пояснением, двумя `TextButton`). Кнопки disable-ятся пока `launching == true`.
   - «Сохранить лог» onClick: `if (!launching) { launching = true; launcher.launch("sanctum-log-${formattedTimestamp()}.txt") }`. Формат имени: `yyyyMMdd-HHmm`.
   - «Закрыть» onClick: `finish()` (`crash.log` остаётся → на следующем холодном старте сработает баннер через Flow B).
   - Все строки — через `stringResource(R.string....)`, ключи из Task 1 (`strings.xml`): заголовок, пояснение, «Сохранить лог», «Закрыть», Toast «Лог сохранён», Toast «Не удалось сохранить лог».

2. Создать `app/src/main/kotlin/app/sanctum/machina/crash/RestartCrashBanner.kt`:
   - `@Composable fun RestartCrashBanner(launching: Boolean, onSaveClick: () -> Unit, onDismissClick: () -> Unit, modifier: Modifier = Modifier)`.
   - Структура: `Card(modifier = modifier.fillMaxWidth().padding(...)) { Row(verticalAlignment = CenterVertically, horizontalArrangement = ...) { Icon(Icons.Outlined.Warning, contentDescription = null); Text(stringResource(R.string.restart_banner_body)); TextButton(onClick = onSaveClick, enabled = !launching) { Text(stringResource(R.string.btn_save_log)) }; IconButton(onClick = onDismissClick) { Icon(Icons.Outlined.Close, contentDescription = stringResource(R.string.restart_banner_dismiss_cd)) } } }`.
   - `launching` флаг только disable-ит «Сохранить лог» (IconButton dismiss остаётся активным — отдельный экшн, не конфликтует с SAF).
   - Никакого internal state (`remember { mutableStateOf(...) }`) — всё прокидывается извне. Хостинг в Task 7 будет владеть `var launching by remember { mutableStateOf(false) }` и очищать его в SAF-callback.

3. Убедиться, что оба файла компилируются в составе `:app`-модуля (`./gradlew :app:assembleDebug`), и что lint не жалуется на новые warnings/errors (`./gradlew :app:lintDebug`).

Импорты примерно: `androidx.compose.foundation.layout.*`, `androidx.compose.material.icons.Icons`, `androidx.compose.material.icons.outlined.Close`, `androidx.compose.material.icons.outlined.Warning`, `androidx.compose.material3.Card`, `androidx.compose.material3.Icon`, `androidx.compose.material3.IconButton`, `androidx.compose.material3.Text`, `androidx.compose.material3.TextButton`, `androidx.compose.runtime.Composable`, `androidx.compose.ui.Alignment`, `androidx.compose.ui.Modifier`, `androidx.compose.ui.res.stringResource`, `androidx.compose.ui.unit.dp`, `app.sanctum.machina.R`.

**Dependencies:**

- **Task 1** (`strings.xml` ключи) — должен быть `done`: `crash_screen_title`, `crash_screen_body`, `btn_save_log`, `btn_close`, `log_saved`, `log_save_failed`, `restart_banner_body`, `restart_banner_dismiss_cd`. Если ключ отсутствует — падает в lint как `MissingResource`. Точный набор имён ключей — как зафиксировано в Task 1.
- **Task 2** (`CrashHandler.kt` + `CrashState.kt` + `Killer.kt`) — `CrashHandler` запускает `CrashReportActivity` через `Intent(ctx, CrashReportActivity::class.java).addFlags(FLAG_ACTIVITY_NEW_TASK or FLAG_ACTIVITY_CLEAR_TASK)`, ссылка на класс требует присутствия `CrashReportActivity.kt` в classpath. Компиляция `:app` упадёт, если Task 2 зафиксирован, а Task 4 ещё не начат: это ожидаемая последовательность в Wave 2 после Wave 1.
- **Task 3** (`LogExportManager.kt`) — нужен публично доступный конструктор (или secondary non-Hilt-конструктор) — иначе в `onCreate` нельзя собрать граф руками. Сверить сигнатуру после Task 3 и не ограничивать видимость `internal`/`@Inject`-only. Также нужен `ExportSource.CrashReport` enum-value.
- Material3 `Card`, `TextButton`, `IconButton`, `Icon` уже на classpath (BOM 2026.03.00, tech-spec Dependencies). `Icons.Outlined.*` — из `androidx.compose.material.icons.extended` (1.7.8).

**Edge cases:**

- **Двойной тап «Сохранить лог» до открытия SAF:** `var launching` защитит — второй клик пропускается. Подтверждение поведения — в Task 6/7 ручной (см. user-spec AC «Повторное нажатие "Сохранить лог"...»).
- **`openOutputStream(uri)` вернул `null`:** `LogExportManager.writeTo` (Task 3) оборачивает в `IOException("openOutputStream returned null")`. `catch (e: IOException)` в activity ловит оба случая одинаково. Guard очистится в `finally`.
- **Удаление `crash.log` после успешного сохранения может вернуть `false`** (race / permissions): `File(...).delete()` silent-false — допустимо, success-Toast всё равно показан. Папка `crash/` по Decision 10 не должна импортировать `ErrorLog` — если хочется breadcrumb, `android.util.Log.e("CrashReportActivity", ...)`.
- **Configuration change** (поворот экрана пока SAF открыт): `registerForActivityResult` + `ActivityResultContracts` переживают rotation. `var launching` — поле активности, пересоздаётся; guard сбросится. На crash-экране это приемлемо (редкий кейс).
- **System back на SAF-диалоге:** == cancel, `uri == null`, guard очищается, остаёмся на экране.
- **`ExportSource.CrashReport` → logcat placeholder:** обеспечивает `LogExportManager` (Decision 8), от активности не требуется ничего дополнительного.

**Implementation hints:**

- `HeavyChangeDialog.kt` — лучший пример в кодовой базе для tone двухкнопочного UI через material3 + stringResource. Основной layout `CrashReportActivity` — не AlertDialog, а full-screen `Scaffold` или `Column`, но структура кнопок и подача строк идентична.
- Suggested filename: `SimpleDateFormat("yyyyMMdd-HHmm", Locale.ROOT).format(Date())` — `Locale.ROOT` важен, иначе на не-ISO локалях формат плывёт.
- Non-Hilt конструктор `LogExportManager` из Task 3: точный набор параметров сверить перед имплементацией. Все зависимости вытащить в `onCreate` через `applicationContext`, `packageManager.getPackageInfo(...)`, `getSystemService(ActivityManager::class.java)` и т.д.
- Coroutine-контекст для `buildExport` + `writeTo`: `lifecycleScope.launch { ... }` — оба suspend уйдут на `Dispatchers.IO` изнутри `LogExportManager`. `withContext(Dispatchers.Main)` для Toast и `finish()` не требуется — `lifecycleScope` по умолчанию `Dispatchers.Main.immediate`.
- Guard flag (`var launching`) — поле активности, НЕ `rememberSaveable`: переживать rotation не требуется, SAF-launcher сам сохранит своё состояние через `ActivityResultRegistry`.
- `RestartCrashBanner` НЕ оборачивает свой контент в `SanctumTheme` — это делает хостинг (`ModelManagerScreen` в Task 7). Banner работает в любом material3 ColorScheme.
- `Icons.Outlined.Close` → `contentDescription` обязателен — это интерактивный `IconButton` (a11y). Для `Icons.Outlined.Warning` — не обязателен (декоративный, передавать `null`).
