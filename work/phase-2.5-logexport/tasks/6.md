# Task 6: AboutScreen — Diagnostics section + 7-tap dev-gesture

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Добавляем в существующий `AboutScreen.kt` две пользовательские поверхности из фазы 2.5, Flow C и Dev-gesture:

1. **Раздел «Диагностика»** с одной кнопкой «Сохранить лог» под блоком `SafeMarkdown`. Кнопка поднимает системный SAF-диалог (`ActivityResultContracts.CreateDocument("text/plain")`) с предложенным именем `sanctum-log-YYYYMMDD-HHmm.txt`, на URI собирает экспорт через `LogExportManager.buildExport(ExportSource.About)` → `writeTo(uri, content)`, а затем показывает Snackbar «Лог сохранён» при успехе или «Не удалось сохранить лог» при `IOException`. Это штатный проактивный экспорт из US-C — `crash.log` и флаг `.dismissed` в этом пути не трогаем.

2. **Жест «7 тапов по строке версии»** в `AboutFooter`: оборачиваем `Text` с версией `Modifier.clickable`, который скармливает события чистому `TapCounter` из Task 3. На срабатывании (7 тапов подряд с паузами ≤2 с) показываем `AlertDialog` «Спровоцировать тест-краш?» с кнопками «Да» / «Отмена». На «Да» прямо из `onClick` главного потока кидаем `RuntimeException("test crash from About")` — не через `Handler.post`, не через `Thread { … }.start()`, не через `scope.launch { … }` (Decision 9: нужно воспроизвести именно UI-callback путь, а не обходной фон). Исключение поднимается в уже установленный `CrashHandler` (Flow A).

Задача принадлежит Wave 2: зависит от Task 1 (строки), Task 3 (`LogExportManager`, `TapCounter`), Task 4 (сам `CrashReportActivity` — UI-экран, который должен появиться после теста-краша) и Task 5 (установленный `CrashHandler` + манифест-запись `:crash`-активити — без них конечная user-verification «Да → app dies → CrashReportActivity appears» не может быть пройдена). Жест активен и в debug-, и в release-сборке на Phase 2.5 (Decision 12) — обёртка в `BuildConfig.DEBUG` отложена на Phase 5.

**Scope delimiter (что НЕ входит в эту задачу):** UI из Task 6 — это раздел «Диагностика» в `AboutScreen` + `AlertDialog` «Спровоцировать тест-краш?» в `AboutFooter`. Это **не** `RestartCrashBanner` — баннер «сохранить/отклонить» при перезапуске после краша живёт в Task 4 (компонент в `:crash` процессе) и Task 7 (показ баннера при возврате в основное приложение) и к `AboutScreen` отношения не имеет. Если при реализации увидишь необходимость тронуть логику баннера — это сигнал остановиться и свериться с tech-spec / Task 4 / Task 7.

**Edge cases:**
- SAF возвращает `uri = null` (пользователь нажал системную «Отмена»): тихо сбросить `launching` в `finally`, Snackbar не показывать.
- Двойной тап «Сохранить лог» до возврата SAF-диалога: второй тап игнорируется `enabled = !launching`.
- `LogExportManager.writeTo` бросает `IOException` (недостаточно места, права на URI): Snackbar «Не удалось сохранить лог», `crash.log` не трогаем.
- Жест на неактивной стадии: пользователь тапает 6 раз → пауза >2 с → диалог не появляется. Проверка «8-й тап не срабатывает повторно» — ответственность `TapCounter`, покрыта в Task 3.
- `throw` из `AlertDialog.confirmButton.onClick` на главном потоке (Decision 9): Compose-рекомпозиция в этот момент не успеет завершиться — это ожидаемо, JVM отдаёт исключение `Thread.uncaughtExceptionHandler`, `CrashHandler` из Task 2 его ловит.
- Если `CrashHandler` ещё не установлен (например, Task 5 не применён) — бросок приведёт к стандартному «Приложение остановлено» Android, а не к `CrashReportActivity`. Это обоснование `depends_on: [5]`.
- `AboutScreen` открывается во время пересоздания Activity (rotation): `launching`-флаг в `remember` сбросится. Это допустимо — ротацию во время SAF-диалога на Honor 200 не проверяем, и в реальном use-case UI заблокирован системным диалогом.
- Timestamp для suggested filename берётся на **клик**, а не на композицию экрана (иначе минуты устареют если юзер долго читает `about.md`).

**Implementation hints:**
- `coroutineScope = rememberCoroutineScope()` на уровне `AboutScreen`; `scope.launch { ... }` для вызова `suspend`-метода ViewModel'а из SAF-коллбэка (сам коллбэк не `suspend`).
- Для in-flight флага используй `var launching by remember { mutableStateOf(false) }` (delegated property). Сброс — в `finally` внутри `try { ... } catch (e: IOException) { ... } finally { launching = false }` в теле `launch { }` coroutine'ы.
- SAF-лаунчер создаётся на уровне composable: `val launcher = rememberLauncherForActivityResult(ActivityResultContracts.CreateDocument("text/plain")) { uri -> ... }`. Запуск: `launcher.launch("sanctum-log-$timestamp.txt")`.
- `TapCounter` (сигнатура из Task 3 — `class TapCounter(nowNanos: () -> Long, maxGapNanos: Long = 2_000_000_000L, threshold: Int = 7)`, метод `fun tap(): Boolean`) сконструировать через `remember { TapCounter(nowNanos = { System.nanoTime() }) }`. **У `TapCounter` нет параметра `onTrigger` / коллбэка** — семантика срабатывания возвращается через boolean из `tap()`: 7-й тап в окне → `true`, остальные → `false`. Дефолтные значения `maxGapNanos` и `threshold` из Task 3 не переопределять. **Важно:** именно лямбда `{ System.nanoTime() }`, не `::System.nanoTime` — bound reference к статическому Java-методу не компилируется в Kotlin.
- Для отсутствия индикации ripple на строке версии: `Modifier.clickable(interactionSource = remember { MutableInteractionSource() }, indication = null) { if (tapCounter.tap()) showDialog = true }`. Решение «показать AlertDialog» принимается composable в обработчике клика; `TapCounter` — чистый детектор, состояние UI не трогает.
- НЕ использовать `@AndroidEntryPoint` — у composable нет этого атрибута. `hiltViewModel()` автоматически подхватит ViewModel через `LocalViewModelStoreOwner` который основная `MainActivity` (помеченная `@AndroidEntryPoint`) предоставляет. Это контраст с `CrashReportActivity` (Task 4), которая Hilt-free по Decision 5.
- Decision 10 / 11 тут НЕ применяются — задача не пишет ничего в `crash.log`, не логирует через `ErrorLog`, и не фильтрует содержимое экспорта. Всё это делают Task 2 и Task 3 соответственно.
- Не забудь: AlertDialog держится на `showDialog == true`, сбрасывается `onDismissRequest`, dismissButton и **перед** `throw` в confirmButton. Порядок `showDialog = false; throw ...` важен — если поменять местами, Compose не успеет выйти из composition, и после рестарта `:crash` процесса может моргнуть старое состояние.
- `LogExportManager.buildExport` — `suspend` и внутри себя переключается на `Dispatchers.IO` (tech-spec §Solution); вызывать его можно с `Dispatchers.Main` корутины без доп. обёртки `withContext(Dispatchers.IO) { ... }`.
