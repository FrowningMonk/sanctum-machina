## Description

Финальная интеграция чат-экрана: реализация системы настроек вывода (inference settings) с трёхуровневой классификацией изменений (light / systemPromptDefault / heavy), диалоги подтверждения и прогресса для heavy-изменений, чистая функция слияния конфига `EffectiveConfig.merge`, стейт-машины ChatViewModel, автоскролл, TopAppBar с кнопками Settings / Reset / Back, полный сброс диалога через `resetConversation()`.

Архитектурные решения:
- **D15** — три уровня классификации изменений настроек (light, systemPromptDefault, heavy)
- **D16** — `EffectiveConfig` — чистая pure-функция слияния дефолтов и оверрайдов, type-safe
- **D12** — диалог подтверждения heavy-изменений с ReinitProgressDialog
- **D21** — reinit движка при heavy-изменениях: stop → cleanup → initialize
- **D23** — `resetConversation()` очищает сообщения, аттачменты и контекст движка с systemPrompt

## What to do

1. Создать `EffectiveConfig.kt` — pure-функцию `merge(defaults, overrides)`: type-safe, null/empty/partial overrides → возвращает дефолты; не мутирует defaults.
2. Создать `InferenceSettingsBottomSheet.kt` — 7 полей настроек, поле `enableThinking` показывается условно (только если `llmSupportThinking == true`). Читает `repository.observePerModelSettings(modelId)`, показывает effective-значения. Кнопки «Применить» и «По умолчанию».
3. Реализовать логику кнопки «Применить» по классификации D15:
   - **light** → save + `applyLightOverrides` (без reinit)
   - **systemPromptDefault** → save + `applySystemPromptAndReset` + snackbar
   - **heavy** → показать `HeavyChangeDialog` → при подтверждении показать `ReinitProgressDialog` → `applyHeavySetting` (stop + cleanup + initialize)
4. Реализовать кнопку «По умолчанию»: сбросить оверрайды; если сброшенное поле было heavy — запустить reinit flow.
5. Создать `HeavyChangeDialog.kt` — диалог подтверждения тяжёлых изменений.

8. Обновить `ChatScreen.kt`:

## TDD Anchor

Написать тесты ДО реализации. Запустить → убедиться что падают → реализовать → убедиться что проходят.

**EffectiveConfigTest** (`app/src/test/kotlin/app/sanctum/machina/ui/chat/EffectiveConfigTest.kt`):
- `overridesNull_returnsDefaults` — при overrides = null возвращает дефолты без изменений
- `partialOverrides_mergedCorrectly` — частичный оверрайд: заполненные поля применяются, незаполненные берутся из дефолтов
- `boolOverride_explicitFalse_overridesTrue` — explicit false в оверрайде должен перекрыть true в дефолтах
- `typeSafety_protoFloatToKotlinFloat` — proto float корректно маппится в Kotlin Float без потери точности
- `pureFunction_defaultsNotMutated` — после вызова merge объект defaults не изменён
- `emptyEqualsNull_defaultInstance` — пустой override-объект (default instance) эквивалентен null — возвращает дефолты

**ChatViewModelTest** (`app/src/test/kotlin/app/sanctum/machina/ui/chat/ChatViewModelTest.kt`):
- `addImage_updatesAttachments` — добавление изображения обновляет список аттачментов
- `addImage_11thRejected_snackbarFlag` — при добавлении 11-го изображения флаг snackbar выставляется, аттачмент не добавляется
- `removeAttachment_removesCorrectIndex` — удаление по индексу удаляет правильный элемент
- `send_emptyTextEmptyAttachments_noInference` — отправка пустого текста без аттачментов не запускает inference
- `send_thinkingEnabled_accumulates` — при enableThinking=true thinking-токены накапливаются отдельно
- `send_thinkingDisabled_skips` — при enableThinking=false thinking-токены пропускаются
- `send_llmSupportThinkingFalse_skips` — при llmSupportThinking=false thinking пропускается независимо от enableThinking
- `applyLightOverrides_updatesConfigValues_noCleanup` — light apply: конфиг обновляется, cleanup/initialize не вызываются
- `applyHeavySetting_sequencing_stopCleanupInitialize` — heavy apply: порядок stop → cleanup → initialize строго соблюдён
- `applyHeavySetting_initCrash_failedState` — при краше initialize ViewModel переходит в состояние failed
- `applySystemPromptAndReset_resetsWithPrompt` — применение systemPromptDefault сбрасывает диалог с новым промптом
- `resetConversation_clearsAll` — сброс очищает сообщения, аттачменты и контекст движка

**Edge cases:**
- `boolOverride_explicitFalse_overridesTrue` — explicit false не должен восприниматься как «не задан» (proto default)
- 11-й аттачмент: snackbar flag выставляется, список не мутирует
- heavy apply с крашем initialize: ViewModel должен перейти в failed-состояние, не зависнуть в progress
- `emptyEqualsNull` — proto default instance семантически эквивалентен null в merge

**Implementation hints:**
- `EffectiveConfig.merge` — использовать proto `hasField` или kotlin sealed для различения explicit false от отсутствия значения
- Порядок reinit строго: stop() → cleanup() → initialize() — использовать `withContext` + sequential корутины
- Автоскролл: `LaunchedEffect(messages.size, lastMessageLength)` — два ключа чтобы триггериться и на новое сообщение, и на стриминг последнего
- InferenceSettingsBottomSheet читает через `observePerModelSettings(modelId).collectAsState()`, показывает effective через `EffectiveConfig.merge(defaults, overrides)`
