# Task 2: Создать `:core-settings` + version catalog entries

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Создать новый Gradle-модуль `:core-settings`, который инкапсулирует хранение пользовательских настроек через Jetpack DataStore (Proto). Модуль реализует proto-схему `app_settings.proto` с маппингом per-model настроек, сериализатор, репозиторий с Hilt DI и обработкой ошибок через `ErrorLog`. Цель — изолировать слой настроек от остальных модулей (D1) и подготовить инфраструктуру для per-model конфигурации (temperatura, maxTokens и т.д.).

## What to do

1. Добавить версии и зависимости в `gradle/libs.versions.toml`: protobuf, datastore-proto, protobuf-javalite.
2. Зарегистрировать alias плагина protobuf в корневом `build.gradle.kts` (`apply false`).
3. Создать `core-settings/build.gradle.kts` с плагинами (com.android.library, kotlin.android, hilt, protobuf 0.9.5), подключить datastore-proto 1.1.7 и hilt.
4. Создать `core-settings/src/main/AndroidManifest.xml` — library hygiene, без `<application>`-атрибутов.
5. Создать `core-settings/src/main/proto/app_settings.proto` по схеме из tech-spec: `PerModelSettings` (7 optional-полей), `AppSettings` (map `per_model_overrides`), Java-пакет `app.sanctum.machina.core.settings.proto` (D2).
6. Создать `AppSettingsSerializer.kt` — реализация `Serializer<AppSettings>` для DataStore.
7. Создать `AppSettingsRepository.kt` — интерфейс с `observePerModelSettings`, `savePerModelSettings`, `resetPerModelSettings`.
8. Создать `DefaultAppSettingsRepository.kt` — реализация интерфейса; перехватывать `IOException` и `CorruptionException`, логировать через `ErrorLog.e("settings-io", ...)` (D27, R13).
9. Создать `di/CoreSettingsModule.kt` — Hilt-модуль, предоставляет `DataStore<AppSettings>` как singleton, биндит репозиторий.
10. Зарегистрировать `:core-settings` в `settings.gradle.kts`.
11. Добавить зависимость `implementation(project(":core-settings"))` в `app/build.gradle.kts`.
12. Написать `AppSettingsRepositoryTest.kt` с TDD Anchor тестами (см. раздел TDD Anchor) до реализации.

## TDD Anchor

Тесты пишем ДО реализации. Запускаем → убеждаемся что падают → пишем код → убеждаемся что проходят.

- `AppSettingsRepositoryTest::save_observe_roundTrip` — сохранить PerModelSettings для модели, прочитать через Flow, убедиться что данные совпадают
- `AppSettingsRepositoryTest::save_multipleModels_observeIndependently` — сохранить настройки для двух разных modelId, убедиться что Flow для каждой модели независим
- `AppSettingsRepositoryTest::reset_observeNull_mapNoKey` — сбросить настройки модели, убедиться что Flow вернул null и ключ отсутствует в map
- `AppSettingsRepositoryTest::mergeOptional_saveOnlyTemperature_readPartial` — сохранить только temperature (остальные поля не заданы), убедиться что `hasTemperature()` == true, остальные `hasXxx()` == false
- `AppSettingsRepositoryTest::concurrent_twoSavesSequential_lastWins` — два последовательных save для одной модели, убедиться что последний выигрывает
- `AppSettingsRepositoryTest::ioException_observeDoesNotCrash` — повреждённый файл DataStore → observe возвращает default/null, не падает (R13, D27)

**Tech-spec decisions:**
- D1 — module isolation: `:core-settings` не зависит от `:app` или других feature-модулей
- D2 — proto path: Java-пакет `app.sanctum.machina.core.settings.proto`, proto-файл в `core-settings/src/main/proto/`
- D3 — modelId key: ключ в map = modelId (строка)
- D4 — optional fields: все поля `PerModelSettings` объявлены как `optional` для корректной работы `hasXxx()`
- D27 — ErrorLog settings-io: тег логирования `"settings-io"` строго фиксирован

**Proto schema (tech-spec):**
```proto
syntax = "proto3";
option java_package = "app.sanctum.machina.core.settings.proto";
option java_multiple_files = true;

message PerModelSettings {
  optional int32 max_tokens = 1;
  optional int32 top_k = 2;
  optional float top_p = 3;
  optional float temperature = 4;
  optional bool enable_thinking = 5;
  optional string accelerator = 6;              // "GPU" | "CPU"
  optional string system_prompt_default = 7;
}

message AppSettings {
  map<string, PerModelSettings> per_model_overrides = 1;
}
```

**Repository public API:**
```kotlin
interface AppSettingsRepository {
    fun observePerModelSettings(modelId: String): Flow<PerModelSettings?>
    suspend fun savePerModelSettings(modelId: String, settings: PerModelSettings)
    suspend fun resetPerModelSettings(modelId: String)
}
```

**Edge cases:**
- `CorruptionException` при десериализации — DataStore должен не упасть, а залогировать и вернуть дефолтный `AppSettings.getDefaultInstance()`
- `IOException` — аналогично: логировать, не пробрасывать наверх (или пробрасывать — уточнить в R13)
- `resetPerModelSettings` — удалить ключ из map, не писать пустой `PerModelSettings`
- Тест на optional-поля: сохранить только `temperature`, остальные `hasXxx()` должны возвращать false (не 0!)

**Implementation hints:**
- Для unit-тестов репозитория использовать `TestScope` + `UnconfinedTestDispatcher` и in-memory DataStore (через `DataStoreFactory.create` с `ByteArrayOutputStream`)
- Protobuf-плагин в `build.gradle.kts` настраивается через блок `protobuf { ... }` с указанием `artifact` для `protoc` и `javalite` генератора
- Hilt: DataStore провайдится с `@ApplicationContext`, именуется через `@Named("app_settings")` если нужно избежать конфликта
