# Task 8: Security Audit

## Required Skills

## Description

Полнофункциональный security-аудит фазы 3.6. Аудитор читает все файлы, изменённые в Tasks 1–6, проводит OWASP Top 10 ревью и проверяет специфические для этой фазы инвариант-точки, которые tech-spec выделил как критичные:

1. **Log-injection через конструкцию reset-сообщения** — текст `status=...` в `DefaultModelRegistry.resetConversation` интерполирует `Throwable.message` (для `ModelInitStatus.Failed`) и имя `ResetReason`. Все эти строки ОБЯЗАНЫ проходить через существующий `sanitize()` пайплайн в новом приватном `write(level, ...)` хелпере `ErrorLog`. Параллельных форматтеров быть не должно — это как раз то, чем грозила Decision 5.
2. **`enableEdgeToEdge()` не должен затирать `FLAG_SECURE`** в `CrashReportActivity`. Tech-spec прямо требует, чтобы `enableEdgeToEdge()` шёл ПОСЛЕ `setFlags(FLAG_SECURE, FLAG_SECURE)` (line 83 текущего файла), и FLAG_SECURE оставался эффективным после вызова.
3. **Никаких новых permission в `AndroidManifest.xml`** — фаза задекларирована как pure-bugfix, разрешений добавляться не должно.
4. **Length-bounding по-прежнему действует для новых уровней `i`/`w`** — единственный приватный `write(level, ...)` хелпер обязан применять 500/200-char границы для ВСЕХ уровней (e/i/w), без параллельной форматирующей ветки.
5. **Closed-whitelist invariant** для `ErrorLog.ALLOWED_COMPONENTS` сохранён — неизвестные компоненты отвергаются (`IllegalArgumentException`) на ВСЕХ трёх уровнях, не только на `e()`.
6. **OWASP Top 10** ревью всех модифицированных файлов (A01 Broken Access Control, A03 Injection, A04 Insecure Design, A05 Security Misconfiguration, A09 Security Logging Failures, и т.д.) — где применимо.

Аудитор сам по себе ЯВЛЯЕТСЯ ревью этой задачи: `reviewers: []`. Дальнейшие финдинги попадают в `pre-deploy-qa` (Task 10).

**Specific check anchors (from tech-spec):**

1. **Log-injection (CWE-117 Improper Output Neutralization for Logs).** В `DefaultModelRegistry.resetConversation` non-Ready путь будет логировать `"skipped reason=$reason status=$initStatus"`. `ModelInitStatus.Failed(err)` несёт строковое сообщение, потенциально из native LiteRT-LM (произвольный текст с `\n`/`\r`/`\t`). Audit must confirm: (a) формирование description-строки идёт через `errorLog.w("inference-reset", "...")`, не через `Log.w` или `print`; (b) приватный `write(level, ...)` хелпер прогоняет через `sanitize()` (collapse `[\n\r\t]` → space) И через `take(MAX_DESCRIPTION_LEN=500)`; (c) для `cause`-параметра — через `take(MAX_CAUSE_MESSAGE_LEN=200)`. Параллельный форматтер для `i`/`w` = automatic high-severity finding.

2. **FLAG_SECURE preservation (CWE-200 Exposure of Sensitive Information).** Текущий `CrashReportActivity.kt:83`:
   ```
   window.setFlags(WindowManager.LayoutParams.FLAG_SECURE, WindowManager.LayoutParams.FLAG_SECURE)
   ```
   Tech-spec Decision 7: `enableEdgeToEdge()` идёт после этой строки. Ожидание: после Task 4 строка `enableEdgeToEdge()` стоит между `setFlags(FLAG_SECURE, ...)` и `setContent { ... }`. AndroidX `enableEdgeToEdge()` НЕ трогает FLAG_SECURE (он управляет только `decorFitsSystemWindows` + `SystemBarStyle`), но порядок важен для будущей читабельности. Если порядок инвертирован — medium-severity finding (не нарушение, но нарушает явное решение).

3. **No new permission.** `git diff` по `AndroidManifest.xml` — список `<uses-permission>` идентичен pre-3.6. Если что-то добавлено — high-severity finding (фаза заявлена как bugfix).

4. **Length-bounding for i/w.** В новом `write(level, component, description, cause)` приватном хелпере: единый путь должен вызывать `sanitize(description, MAX_DESCRIPTION_LEN)` и `sanitize(cause.message, MAX_CAUSE_MESSAGE_LEN)`. `e()`, `i()`, `w()` — все три должны делегировать сюда, без альтернативных форматтеров. Если найден отдельный путь для `i`/`w`, который НЕ применяет `take(...)` — high-severity (drift risk, явный предмет Decision 5).

5. **Closed-whitelist invariant on all levels.** В существующем коде (`ErrorLog.kt:77`): `require(component in ALLOWED_COMPONENTS)` стоит в `e()`. После Task 1 рефактора `i()` и `w()` должны иметь ровно тот же `require`-чек (через общий `write(...)` или дублированно — но обязательно). Если `i("unknown-component", ...)` не throws — high-severity finding (closed-whitelist invariant нарушен).

6. **General OWASP Top 10 sweep:**
   - A01 (Broken Access Control) — затронут? Нет нового access-surface.
   - A03 (Injection) — main concern: log-injection (см. #1).
   - A04 (Insecure Design) — `engineReady` StateFlow: race conditions при rapid chat-switch.
   - A05 (Security Misconfiguration) — edge-to-edge может ли скрыть чувствительный UI? Нет, диалоги — `Dialog`, не `Activity`-fullscreen.
   - A07 (Identification & Auth) — N/A.
   - A09 (Security Logging Failures) — главный фокус: новые `i`/`w` уровни, length-bounding, whitelist (см. #1, #4, #5).
   - A10 (SSRF) — N/A.

**Edge cases to specifically probe:**
- `Throwable.message` с многострочной stack-trace в составе — должен collapse'ить в одну строку.
- `Throwable.message == null` — buildLine'у даёт `""`, не должно крашить.
- `ResetReason.toString()` — это enum `name`, неинтерполируемый user input, но всё равно должен пройти через ту же sanitize-цепочку.
- В non-Ready ветке статус `ModelInitStatus.Failed(err: String)` — параметр `err` идёт от `LlmModelHelper`, может содержать native error text → highest log-injection risk.

**Implementation hints:**
- Использовать структуру отчёта: `# Security Audit — Phase 3.6 Bugfix` → `## Summary` (1-3 предложения) → `## Findings` (severity-ranked, с file:line + CWE) → `## Specific Check Verdicts` (шесть пунктов с pass/fail) → `## OWASP Top 10 Coverage` (таблица или маркированный список).
- Для каждого finding: ссылка на конкретную строку (`DefaultModelRegistry.kt:309`), не диапазон.
- Severity: `critical` (немедленно блокирует merge), `high` (надо фиксить до merge), `medium` (надо фиксить, но можно follow-up'ом), `low` (полезно, но не обязательно), `info` (наблюдение, не finding).
