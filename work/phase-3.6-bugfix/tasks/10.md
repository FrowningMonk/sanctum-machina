## Description

Финальная приёмочная верификация phase-3.6-bugfix перед merge в `main`. Прогоняем полный набор Gradle-проверок (unit-тесты `:app` и `:core-runtime`, lint, debug APK), проверяем структурные инварианты (модульная граница `:core-runtime` без UI-зависимостей, обновлённый `LIGHT_FIELD_LABELS`, whitelist `inference-reset`, переписанный D15 в `patterns.md`), и систематически проходим каждый AC из user-spec (AC-1.1 … AC-3.3) и из § Acceptance Criteria tech-spec.

Часть AC требует физический Honor 200 (визуальный smoke, мульти-чат сценарий, repetition-loop reproduction, max_tokens HeavyDialog) — для них агент формулирует и отправляет пользователю единый блок Verify-user шагов, ждёт ответы и записывает результаты.

Итог — агрегированный отчёт `work/phase-3.6-bugfix/logs/qa/pre-deploy-qa.md` с per-AC статусом (pass / fail / deferred) и ссылками на источники (gradle output, аудиты Tasks 7/8/9, ответ пользователя). Отчёт служит входом для решения о merge.

Деплоя нет — мобильное приложение, CI/CD не настроен (см. CLAUDE.md). Post-deploy QA не выполняется в этой фазе.

### Edge cases

- **Один из аудитов недоступен:** если файл `logs/audit/{name}.md` отсутствует — записать в отчёт «Audit X not found» и поставить вердикт BLOCKED, не выдумывать findings.
- **Пользователь не отвечает или отвечает частично:** AC, требующие user smoke, помечаются `deferred`; вердикт — READY WITH NOTES (если все agent-проверки pass и нет CRITICAL findings) либо BLOCKED. Решение по deferred остаётся за пользователем.
- **Lint падает на не-затронутых файлах:** не наша зона ответственности — отметить в отчёте, но не блокировать (фаза 3.6 — bugfix, не широкий cleanup). Если падает на затронутых файлах — fail.
- **`adb logcat` недоступен у пользователя (нет USB-debug):** AC-1.5 — отметить deferred, основная диагностика через Diagnostics-экран если он экспортирует ErrorLog.
- **APK build падает локально из-за окружения (JDK/SDK):** не наша вина по коду — записать ошибку, попросить пользователя подтвердить сборку у себя; AC отметить deferred до подтверждения.
- **Discrepancy между unit-тестами и user smoke:** unit зелёный, юзер ловит баг — приоритет user, AC fail, вердикт BLOCKED.

### Implementation hints

- Verify-user блок отправлять **одним сообщением** — не разбивать по AC. Пользователь проходит сценарий целиком и присылает ответы списком — так быстрее и не путается порядок.
- Перед запуском Gradle — `./gradlew --stop` чтобы убить зависшие демоны, если предыдущие задачи оставили мусор.
- На Windows работаем через PowerShell; для grep'а из § Smoke использовать инструмент `Grep` (а не shell-grep) — он переживёт абсолютные пути с двоеточиями.
- Stdout Gradle сохранять усечённо (последние 30 строк, summary `BUILD SUCCESSFUL` + тестовая сводка) — полные логи раздуют отчёт без пользы.
- Структурные проверки делать через `Grep`/`Read` инструменты, не через bash-pipe — детерминированнее и без shell-quoting.
- Если открыты CRITICAL/HIGH из аудитов и они НЕ закрыты в этой фазе — это **блокер**, даже если все Gradle и user-smoke прошли. Merge запрещён.
- В заключении отчёта явный вердикт обязателен — следующая итерация (или пользователь) принимает решение по нему.
