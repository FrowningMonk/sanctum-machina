# Task 2: ResetReason enum + ModelRegistry.resetConversation signature + DefaultModelRegistry logging

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Вторая волна Phase 3.6 — фундамент для всех вызовов `resetConversation` с диагностикой. Задача вводит новый тип `ResetReason` (enum) в модуле `:core-runtime`, расширяет интерфейс `ModelRegistry.resetConversation` обязательным параметром `reason: ResetReason` (без дефолта — каждый caller обязан осознанно выбрать причину) и добавляет в `DefaultModelRegistry.resetConversation` диагностическое логирование через новые `errorLog.i` / `errorLog.w` уровни (Task 1).

Сейчас (до правки) `DefaultModelRegistry.resetConversation` молча пропускает вызов, если engine не в состоянии Ready, что не даёт диагностировать причины потери KV-cache в багах вроде Bug 1 phase-3.6 (грязный контекст между чатами). После правки:
- Успешный путь (Ready) — `errorLog.i("inference-reset", "reason=<NAME>")`.
- Skip non-Ready (Idle / Initializing / Failed) — `errorLog.w("inference-reset", "skipped reason=<NAME> status=<STATUS>")`. Текст cause/status проходит через ту же приватную `write(level, ...)` (length-bound + `sanitize()`) — никаких параллельных форматтеров (Decision 5).
- Skip unknown model name — остаётся silent (Decision 1: race с `WarmupCoordinator` производил бы log spam без диагностической ценности).

Mutex discipline сохраняется без изменений: все пути проходят под `lifecycleMutex`. Это backend-only задача — никакого UI; полная end-to-end проверка пользователя откладывается до Task 3, который соединит callers (`ChatViewModel`) с этим API.

Task поддерживает (частично) AC-1.4 и AC-1.5 из user-spec — полное удовлетворение наступит после Task 3.

## What to do

1. Создать новый файл `core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/ResetReason.kt`. В нём — `enum class ResetReason` со значениями `CHAT_SWITCH`, `DRAFT_COMMIT`, `LIGHT_OVERRIDE`, `SYSTEM_PROMPT`, `HEAVY`, `USER`. Файл лежит рядом с `ModelRegistry.kt`, в том же пакете.
2. В `ModelRegistry.kt` (interface) расширить сигнатуру `resetConversation` параметром `reason: ResetReason` — БЕЗ default-значения. Это сломает сборку у всех callers; в этой задаче callers не правятся (это работа Task 3 / Task 6) — в рамках Task 2 ломаем интерфейс осознанно и фиксим только impl + тесты модуля `:core-runtime`.
3. В `DefaultModelRegistry.kt` (impl) под существующим `lifecycleMutex.withLock`:
   - lookup entry по `modelName`; если entry отсутствует (unknown name) — silent return (поведение сохраняется).
   - Если entry есть, но `initStatus !is Ready` — заменить текущий silent skip на `errorLog.w("inference-reset", "skipped reason=<reason.name> status=<status>")`. Status string должен включать имя состояния (`Idle`, `Initializing`, `Failed`). Если у Failed есть cause/message — он попадает в description целиком и проходит через приватный `write(level, ...)` (sanitize + length-bound из Task 1) — никакой ручной интерполяции `Throwable.message` без `sanitize`.
   - Success path (Ready): после успешного вызова `llmHelper.resetConversation(...)` — `errorLog.i("inference-reset", "reason=<reason.name>")`.
   - Если helper бросает исключение — `lifecycleMutex` отпускается через `withLock` (это уже гарантирует contract); тест `resetConversation_propagatesHelperException` это пиннит.
4. Обновить `DefaultModelRegistryTest.kt` — добавить семь новых тестов (см. TDD Anchor) и адаптировать существующие, если они вызывают `resetConversation` без `reason` (передать нейтральный `reason = USER`).
5. Запустить `./gradlew :core-runtime:testDebugUnitTest` — все тесты модуля зелёные.

## TDD Anchor

Тесты пишем ДО реализации; запускаем — должны падать; затем имплементируем; затем зелёные.

- `core-runtime/src/test/kotlin/app/sanctum/machina/core/registry/DefaultModelRegistryTest.kt::resetConversation_skipsAndLogsWarning_whenEngineIdle` — seed entry с `ModelInitStatus.Idle`; вызов `resetConversation(name, systemPrompt, reason = CHAT_SWITCH)`; assert: helper НЕ вызывался; `errorLog.w("inference-reset", description)` ровно один раз; description содержит `"skipped"`, `"CHAT_SWITCH"`, `"Idle"`.
- `...::resetConversation_skipsAndLogsWarning_whenEngineInitializing` — seed `Initializing`; те же ассерты, status string содержит `"Initializing"`.
- `...::resetConversation_skipsAndLogsWarning_whenEngineFailed` — seed `Failed(cause)`; те же ассерты, status string содержит `"Failed"`. Failure cause text проходит через `sanitize()` + length-bound (общий пайплайн `write(level, ...)` из Task 1) — НЕТ параллельного форматтера.
- `...::resetConversation_dispatchesAndLogsInfo_whenReady` — seed `Ready`; вызов `resetConversation(name, prompt, reason = LIGHT_OVERRIDE)`; assert: helper вызван ровно один раз с `model.configValues` мерджем и переданным `prompt`; `errorLog.i("inference-reset", description)` один раз; description содержит `"LIGHT_OVERRIDE"`.
- `...::resetConversation_skipsSilently_whenModelMissing` — registry без entry для `unknown-name`; вызов `resetConversation("unknown-name", prompt, reason = USER)`; assert: helper НЕ вызывался; ни `errorLog.i`, ни `errorLog.w`, ни `errorLog.e` НЕ вызывались.
- `...::resetConversation_isSerializedByLifecycleMutex` — `runTest(StandardTestDispatcher())` + delay-injected helper (helper фиксирует sequence number через переданный `AtomicInteger`, делает `delay(50)`); запустить два конкурентных `resetConversation` через `launch`; `advanceUntilIdle()`; assert: записанная sequence строго `[1, 2]` (не interleaved). **Важно: использовать `StandardTestDispatcher`, НЕ `UnconfinedTestDispatcher`** — Unconfined запускает корутины eagerly до первого suspension, что даст ложно-зелёный результат, не отражающий реальную сериализацию. Также **НЕЛЬЗЯ** использовать `runBlocking` (как в существующих тестах файла) — `delay(50)` на реальном диспатчере замедлит тест на 100мс и не даст контроля над scheduling. Только `runTest` + `StandardTestDispatcher` дают virtual-time сериализацию, которая пиннит работу `lifecycleMutex`, а не артефакт диспатчера.
- `...::resetConversation_propagatesHelperException` — helper бросает `IllegalStateException("boom")`; первый `resetConversation` бросает то же исключение наружу; второй последующий `resetConversation` (на уже Ready entry) проходит успешно — пиннит, что `lifecycleMutex` отпускается после throw.

## Details

**Files:**

- `core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/ResetReason.kt` — НОВЫЙ файл. `enum class ResetReason { CHAT_SWITCH, DRAFT_COMMIT, LIGHT_OVERRIDE, SYSTEM_PROMPT, HEAVY, USER }`. Пакет `app.sanctum.machina.core.registry`. Никаких полей, никакого `companion object` — простой enum. Lives next to `ModelRegistry.kt` interface (Decision 1: placement в `:core-runtime` сохраняет cross-module API типобезопасным; вариант `String` reason rejected как typo-prone, вариант placement в `:app/ui/chat/` rejected — `:core-runtime` потерял бы type safety).

- `core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/ModelRegistry.kt` — текущая сигнатура (verified):
  ```kotlin
  suspend fun resetConversation(modelName: String, systemPrompt: String? = null)
  ```
  `systemPrompt` — nullable с default `null` (в `DefaultModelRegistry` существующие callers `ChatViewModel` строки 333, 461 полагаются на это). Расширить параметром `reason: ResetReason` БЕЗ дефолта; параметр последний; **обязательно сохранить nullable + default у `systemPrompt`**, чтобы не сломать существующих callers, которые в Task 2 не правятся:
  ```kotlin
  suspend fun resetConversation(
      modelName: String,
      systemPrompt: String? = null,
      reason: ResetReason
  )
  ```
  Импл `DefaultModelRegistry.resetConversation` сейчас имеет сигнатуру `override suspend fun resetConversation(modelName: String, systemPrompt: String?)` (строка 302) — там тоже добавить `reason: ResetReason` последним параметром (без дефолта на override-уровне — Kotlin запрещает default на override, дефолт живёт на интерфейсе для `systemPrompt`).

- `core-runtime/src/main/kotlin/app/sanctum/machina/core/registry/DefaultModelRegistry.kt` — текущий импл под `lifecycleMutex.withLock`: lookup entry → если null silent return → если `initStatus !is Ready` silent skip. Изменить:
  - lookup null → silent return (без изменений).
  - non-Ready → `errorLog.w("inference-reset", "skipped reason=${reason.name} status=${formatStatus(initStatus)}")`. `formatStatus` — приватный helper в файле, возвращающий `"Idle"`, `"Initializing"`, `"Failed"` (или `"Failed: <message>"` — но передавать в `errorLog.w` целиком, не интерполировать sanitize вручную; `errorLog.w` сам прогонит description через `write(level, ...)`).
  - Ready (success path) → после `llmHelper.resetConversation(...)` (или эквивалентного вызова) → `errorLog.i("inference-reset", "reason=${reason.name}")`.

- `core-runtime/src/test/kotlin/app/sanctum/machina/core/registry/DefaultModelRegistryTest.kt` — добавить 7 новых тестов (см. TDD Anchor). Если в файле есть существующие тесты с вызовом `resetConversation` без `reason` — добавить `reason = ResetReason.USER` (нейтральный выбор для тестов, не относящихся к конкретной причине).

**Dependencies:**
- Task 1 — должен landed: `ErrorLog.kt` имеет публичные `i()` и `w()`, `"inference-reset"` в `ALLOWED_COMPONENTS`, общая `write(level, ...)` приватная функция с length-bound + sanitize.
- Никаких новых пакетов — существующая инфра тестов (JUnit 4.13.2 + Robolectric 4.12 + kotlinx-coroutines-test для `runTest`).

**Edge cases:**
- Helper выбрасывает исключение посредине success path → `lifecycleMutex.withLock` отпускает блокировку → `errorLog.i` НЕ должен вызываться (исключение бросается раньше). Тест `resetConversation_propagatesHelperException`.
- `Failed(cause)` с очень длинным `cause.message` (>500 символов) → должен быть обрезан до 500 через `write(level, ...)` пайплайн (пиннится в Task 1 тестом `iAndW_lengthBoundingMatchesE`, косвенно проверяется и здесь — failed-skip тест не должен крашиться на длинной строке).
- Конкурентные `resetConversation` (например, быстрое переключение чатов + Light apply) → сериализуются через `lifecycleMutex`. Тест `resetConversation_isSerializedByLifecycleMutex`.
- `cause.message` содержит spec-символы (newline, tab) → `sanitize()` свернёт whitespace в одиночные пробелы (поведение `write(level, ...)` из Task 1).

**Implementation hints:**
- Description для `errorLog.w` — единая форма `"skipped reason=${reason.name} status=${statusFormatted}"`; тесты ассертят подстроки `"skipped"`, `"<REASON_NAME>"`, `"<Status>"` — формат не должен от этого отклоняться.
- Не интерполировать `Throwable.message` напрямую в format string без подачи через `errorLog.w`. Пайплайн `write(level, ...)` уже делает sanitize — повторно не санитизировать.
- Для теста `resetConversation_isSerializedByLifecycleMutex` использовать паттерн из существующих init/cleanup serialisation тестов (если они есть в файле — повторить структуру). Schema: `AtomicInteger sequence`, `helper { val n = sequence.incrementAndGet(); delay(50); recorded += n }`.
- **Существующий `DefaultModelRegistryTest.kt` использует `runBlocking`** — для нового serialisation теста этого недостаточно. Использовать `runTest(StandardTestDispatcher())` (импорт `kotlinx.coroutines.test.StandardTestDispatcher` + `runTest`), не `UnconfinedTestDispatcher` (eager execution скрывает race) и не `runBlocking` (нет virtual time). После `launch { ... }` для двух конкурентных вызовов — `advanceUntilIdle()`.
- Если у `DefaultModelRegistry` есть зависимость от диспатчера в конструкторе — пробросить тестовый диспатчер; если нет — мьютекс всё равно сериализует на любом диспатчере, но именно `StandardTestDispatcher` даёт детерминированный планировщик, который пиннит порядок суспеншенов.
- Не добавлять новый component string в `ALLOWED_COMPONENTS` — `"inference-reset"` уже добавлен Task 1.
- Не править callers `resetConversation` в `:app` — это явная зона Task 3 / Task 6. Сборка `:app` будет красной до завершения Task 3 — это ожидаемо в рамках wave 2.
