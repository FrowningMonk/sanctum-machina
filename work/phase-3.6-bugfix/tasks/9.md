## Description

Полный аудит тестового покрытия фазы 3.6. После того, как Tasks 1–6 завершены и все unit-тесты зелёные, нужна независимая проверка КАЧЕСТВА тестов (не факта их прохождения):

1. Каждый AC из user-spec (AC-1.1 … AC-3.3) и из tech-spec § Acceptance Criteria либо имеет покрывающий unit-тест, либо явно отнесён к user-only verification (AC-3.2 / AC-3.3 — только Honor 200 smoke).
2. Тестовая пирамида соответствует выбранной стратегии (size S → unit-only): не появилось спонтанных integration / E2E тестов, не появилось мок-БД (этот проект не использует Room для логики, проверка скорее на дисциплину).
3. Ассерты осмысленные — проверяют поведение, а не «вызвался ли вообще». Особое внимание трём конкретным тестам, перечисленным в чеклисте.
4. Нет утечки test-only поведения в production-код (например, `@VisibleForTesting` пути не подменяют реальную логику; никаких `if (testMode) ...` в production).
5. Регрешн-гарды из старых тестов сохранены — конкретно no-cleanup/no-init guard из исходного `applyLightOverrides_updatesConfigValues_noCleanup`, который Task 3 заменил на `applyLightOverrides_callsResetConversation_withLightOverrideReason` (per tech-spec § Testing Strategy).
6. Качество recording-fakes (`FakeModelRegistry`) достаточно для всех ассертов: `lastResetReason`, `cleanupCalls`, `initializeCalls`, упорядоченный `sharedCalls` для проверки последовательности диалогов в Heavy-пути.

Аудитор — это и есть ревью; задача не имеет отдельных reviewers. Результат — отчёт с AC→test mapping, выявленными пробелами и качественными findings, который будет потом потреблён Task 10 (Pre-deploy QA).

**Audit checklist (developing the report):**

1. **AC coverage** — построить таблицу AC → test. Эталон маппинга (минимум, из tech-spec):
   - AC-1.1 (chat switch → Conversation reset) → `bootstrapPersistent_chatSwitchReset_waitsForReady`.
   - AC-1.2 (commit draft → reset с reason DRAFT_COMMIT) → `bootstrapPersistent_emitsDraftCommitReset_whenLastIsUnpairedUser`.
   - AC-1.3a (Light tier для temperature/topK/topP) → `applyLightOverrides_callsResetConversation_withLightOverrideReason` + `classifyApplyLevel_returnsLight_forTemperature`.
   - AC-1.3b (max_tokens via Heavy) → `classifyApplyLevel_returnsHeavy_forMaxTokens` + `applyMaxTokens_followsHeavyDialogSequence`.
   - AC-1.4 (non-Ready skip = warning, не silent) → `resetConversation_skipsAndLogsWarning_whenEngine{Idle,Initializing,Failed}` (×3).
   - AC-1.5 (каждый успешный reset логируется с тегом причины) → `resetConversation_dispatchesAndLogsInfo_whenReady` + `applySystemPromptAndReset_passesSystemPromptReason` + `userTapReset_passesUserReason`. Также Whitelist: `whitelistCount_is15`, `inferenceResetComponent_acceptedByAllLevels`, `unknownComponent_stillRejected_negative`.
   - AC-2.1 (Settings active в draft после Ready) → `engineReady_combinatorics` (Ready + no-warmup → true).
   - AC-2.2 (изменения до первого сообщения применяются к нему) — проверить, есть ли отдельный test или это implicit behaviour (per tech-spec § User-Spec Deviations: "satisfied by existing data flow"). Если только implicit — это допустимо, но должно быть явно отмечено в отчёте.
   - AC-2.3 (Settings заблокированы при non-Ready) → `engineReady_combinatorics` (Idle / Initializing / Failed → false; Ready+warmup-in-flight → false).
   - AC-3.1 (`enableEdgeToEdge()` вызван в обеих activity) — code-only assertion; обычно не покрывается unit-тестом (Robolectric для Activity onCreate в этом проекте не настроен — проверить факт). Если нет unit-теста — отметить как USER-ONLY с явным указанием на Honor 200 smoke + grep по файлам Activity как pre-deploy QA gate.
   - AC-3.2 / AC-3.3 → USER-ONLY (per user-spec § Как проверить → Пользователь проверяет → Bug 3).

2. **Test pyramid sanity** — Glob по `**/test/**/*.kt` в обоих модулях, убедиться, что новых файлов с suffix `IntegrationTest` / `E2ETest` / `*AndroidTest*` (в `:app/src/androidTest/`) нет. Если есть — это deviation от tech-spec § Testing Strategy.

3. **Meaningful assertions check (3 critical tests):**
   - `applyMaxTokens_followsHeavyDialogSequence` — должен ассертить ОРДЕРНУЮ последовательность `sharedCalls == ["stopResponse", "cleanup", "initialize"]` (соответствует базовому `applyHeavySetting_sequencing_stopCleanupInitialize`, см. Task 3 TDD Anchor). Если ассерт только `cleanupCalls == 1 && initializeCalls == 1` без проверки ordered `sharedCalls` — finding (major): тест не пинит heavy reinit path. Дополнительно (supplementary, НЕ в `sharedCalls`): аудитор может проверить, что dialog-gating StateFlow в `ChatViewModel` (например, `_heavyChangeDialogVisible` или аналог — сверь имя по существующему тестовому паттерну) эмитил `true`, и `reinitInProgress` прошёл `true → false`. Эти ассерты — на StateFlow ChatViewModel, а не на список `sharedCalls` recording-fake'а.
   - `bootstrapPersistent_chatSwitchReset_waitsForReady` — должен быть двухфазный: (a) сначала Initializing → assert `resetReasons.isEmpty()`, (b) потом Ready → assert `resetReasons == [CHAT_SWITCH]`. Если single-phase — finding (major): не покрыт race cold-start vs reset.
   - `iAndW_lengthBoundingMatchesE` — должен передавать input ВЫШЕ лимита (600+ chars description, 300+ chars cause) и ассертить truncation до 500/200. Если только `assertDoesNotThrow` без проверки реальной длины записи — finding (major): не доказывает, что `i`/`w` идут через тот же `write()` helper.

4. **No test-only behaviour leaks** — Grep по production-источникам (`core-runtime/src/main`, `app/src/main`):
   - `@VisibleForTesting` — допустимо, но проверить, что аннотированные методы/поля не добавили альтернативных code-paths в production (например, `internal fun resetForTest()` который обходит `lifecycleMutex`).
   - `if (BuildConfig.DEBUG)` или `if (testMode)` ветки в новых изменениях — должно быть 0.
   - Никаких новых `open` модификаторов на ранее `final` классах ради моков (kotlin.allopen plugin отсутствует, проверить, что не появился).

5. **Regression guards preserved** — В `applyLightOverrides_callsResetConversation_withLightOverrideReason` должны быть КАК ассерты на новое поведение (`lastResetReason == LIGHT_OVERRIDE`), ТАК и сохранённые из старого теста: `assertEquals(0, fakeRegistry.cleanupCalls)`, `assertEquals(0, fakeRegistry.initializeCalls)`. Если no-cleanup/no-init guard потерян — finding (blocker): регрессия защиты от случайного hard-reset на Light path.

6. **Recording-fake quality** — `FakeModelRegistry` (или как он там называется в коде, проверить точное имя):
   - Должен записывать: `lastResetReason: ResetReason?`, `lastResetSystemPrompt: String?`, `cleanupCalls: Int`, `initializeCalls: Int`, упорядоченный `sharedCalls: List<String>`, `resetReasons: List<ResetReason>` (для multi-reset тестов вроде bootstrap two-phase).
   - Никаких mock-фреймворков (mockk / mockito) для core поведения registry — должен быть hand-written fake (per project pattern). Если используется mock — отметить (minor) и проверить, что ассерты всё ещё behavioural.

**Edge cases для аудита:**
- Если фактическое число тестов в файле НЕ совпадает с tech-spec (например, `ErrorLogTest` имеет 5 а не 4) — это либо bonus coverage (хорошо), либо deviation (плохо). Описать в отчёте.
- Если тест существует, но имя отличается от tech-spec — это deviation, но не обязательно проблема (имена могли уточниться при имплементации). Отметить, не блокировать.
- AC-2.2 — implicit behaviour, не имеет dedicated теста. Это OK по tech-spec § User-Spec Deviations, но в отчёте должно быть явно зафиксировано «covered by existing data flow per tech-spec deviation note», не потеряно.

**Implementation hints:**
- Используй Grep для подсчёта тестов: `Grep pattern="@Test" path="<test file>" output_mode="count"`.
- Используй Grep для извлечения имён тестов: `Grep pattern="fun \w+\(\)" path="<test file>" output_mode="content"` — даст список fun-сигнатур, дальше отфильтровать `@Test`-аннотированные.
- Severity guideline для findings:
  - **blocker** — потерян regression guard или AC без покрытия и без USER-ONLY-метки.
  - **major** — ассерт не проверяет заявленное поведение (smoke вместо behavioural).
  - **minor** — naming deviation от tech-spec, лишний бонус-тест без AC-привязки.
  - **nit** — stylistic, не влияет на корректность.
