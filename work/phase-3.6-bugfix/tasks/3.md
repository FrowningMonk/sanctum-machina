# Task 3: Wire reset reasons across ChatViewModel and reclassify MAX_TOKENS to Heavy

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Combined Bug-1 fix в `ChatViewModel`. Закрывает корневую причину утечки KV-cache между чатами и игнорирования Light-tier sliders, плюс устраняет misclassification `MAX_TOKENS` (физически baked в `EngineConfig`, не `ConversationConfig`).

После завершения этой задачи:
- Persistent-чат при загрузке всегда стартует с чистым `Conversation` (новый KV-cache), а log-tag отличает «переключение чата» от «коммит draft».
- Применение temperature/topK/topP из bottom-sheet реально перечитывается движком (через `resetConversation`) — больше не silent no-op.
- `applySystemPromptAndReset` и user-tap reset передают явный `reason` для diagnostic logging.
- `max_tokens` теперь идёт через Heavy-путь (`HeavyChangeDialog → ReinitProgressDialog → cleanup + initialize`), потому что только пересоздание engine применяет новый лимит.

Опирается на enum `ResetReason` и расширенную сигнатуру `ModelRegistry.resetConversation(reason: ResetReason)` из Task 2 (Wave 2). Этот таск не трогает `deriveTopAppBarState` и не добавляет `engineReady` StateFlow — это работа Task 6 (Wave 4) над тем же файлом, разделённая по волнам, чтобы избежать параллельных правок.

## What to do

1. **Persistent branch of `bootstrapChatModelId`** — после существующего `lastMsg = messageDao.lastByChat(id.id)` дождись первого `Ready` сигнала через тот же паттерн, что используется в `observeFirstReadyThenResume` (line 832), и вызови `registry.resetConversation(modelName, systemPrompt = effective, reason)`. Reason выбирается эвристикой:
   - `lastMsg?.role == ROLE_USER` → `ResetReason.DRAFT_COMMIT` (handover из draft commit, последний пользовательский ход без ответа модели)
   - иначе → `ResetReason.CHAT_SWITCH`
   Эвристика повторно использует значение `lastMsg`, уже считанное для AC-R3 auto-resume — отдельного похода в DAO не делаем.

2. **`applyLightOverrides`** — после существующей мутации `model.configValues = EffectiveConfig.merge(defaults, overrides)` вызови `registry.resetConversation(model.name, systemPrompt = effective, reason = ResetReason.LIGHT_OVERRIDE)`. UI history (`_messages`, `_streamingMessage`, `_attachments`) НЕ трогается — Light-tier это sampler refresh, не context wipe.

3. **`applySystemPromptAndReset`** — у существующего вызова `registry.resetConversation(...)` передай `reason = ResetReason.SYSTEM_PROMPT`.

4. **User-tap reset** (callsite, который сейчас покрыт тестом `resetConversation_clearsAll`) — передай `reason = ResetReason.USER`.

5. **`LIGHT_FIELD_LABELS`** (`ChatViewModel.kt:1518`) — удалить запись `ConfigKeys.MAX_TOKENS.label` из множества.

6. **`classifyApplyLevel`** — расширить условие HEAVY: возвращать `HEAVY` если `acceleratorChanged || maxTokensChanged` (раньше только `acceleratorChanged`). Сравнение `maxTokensChanged` берётся из снапшота defaults vs. предложенных значений по тому же паттерну, что уже использует `acceleratorChanged`.

7. **`deriveTopAppBarState` НЕ ТРОГАТЬ** — Bug 2 (Settings disabled в Draft) фиксится отдельно в Task 6 через новый `engineReady: StateFlow<Boolean>`. Этот таск только wiring reasons + max_tokens reclassification.

8. Все вызовы `registry.resetConversation` теперь должны передавать explicit `reason` — нет `default`-параметра в интерфейсе после Task 2.

**Edge cases:**

- Cold-start race: `bootstrapChatModelId` вызывает `applyEffectiveConfigToModel` которая может сама стартовать warmup. Reset должен ждать первого `Ready`, иначе попадёт на `Initializing` и улетит в `errorLog.w("inference-reset", "skipped ... Initializing")` без эффекта (KV останется грязным). Тест `bootstrapPersistent_chatSwitchReset_waitsForReady` фиксирует именно этот сценарий.
- Rapid drawer-switch: пользователь дёргает между чатами быстрее, чем engine успевает дойти до Ready. `WarmupCoordinator.cancelAndRestart` отменяет предыдущую инициализацию; новый VM поднимается с новым chatId. Reset из предыдущего VM мог не успеть → новый VM запускает свой reset на свежем Ready. Поведение корректно: каждый VM отвечает за свой chatId.
- `lastByChat` возвращает `null` (новый persistent чат без сообщений вообще): не USER, значит → `CHAT_SWITCH`. Корректно.
- `applyLightOverrides` при не-Ready engine: `DefaultModelRegistry.resetConversation` залогирует `errorLog.w("inference-reset", "skipped reason=LIGHT_OVERRIDE status=...")` (Task 2). UI history не очищается — пользователь продолжает видеть переписку.
- `max_tokens` Heavy-путь: HeavyChangeDialog требует подтверждения. Если пользователь нажал «Cancel» — состояние не должно остаться «зависшим» в `applyHeavySetting`. Существующий путь акселератора уже обрабатывает cancel; повторно использовать его — никаких новых ветвей.

**Implementation hints:**

- Эвристика `lastMsg?.role == ROLE_USER` уже вычислена для AC-R3 auto-resume в `bootstrapChatModelId` Persistent branch — переиспользовать это значение, не делать второй вызов в DAO.
- Pattern для ожидания Ready: смотреть `observeFirstReadyThenResume` (line 832) — там уже есть pattern `_uiState.map { ... }.filter { it is Ready }.first()` (или аналогичный); тот же подход для reset.
- `effective` (system prompt) — берётся из того же merge, что уже использует существующий `applyEffectiveConfigToModel` или `applySystemPromptAndReset`. Не строить заново — переиспользовать.
- `classifyApplyLevel`: посмотри как сейчас определяется `acceleratorChanged` (видимо, через сравнение текущих `model.configValues[ACCELERATOR]` vs proposed). По тому же паттерну добавь `maxTokensChanged`.
- НЕ добавлять новый StateFlow / новый callback / новые поля в `_uiState` — это работа Task 6. Этот таск чисто wiring + reclassify.
