## Description

Phase 3.6 implementation invalidates two existing claims in `.claude/skills/project-knowledge/references/patterns.md`. PK is the system of record for project methodology — leaving stale rules in place means future agents and humans will follow them and re-introduce the bugs this phase fixes.

Two textual updates required:

1. **§ Three-tier settings application classification (D15) — Light bullet (currently around line 62).** The line states Light tier `"applies from next send() without engine touch"`. Code research (§ 1, § 4 of `code-research.md`) proved this false: `LlmChatModelHelper.initialize` bakes `SamplerConfig(topK, topP, temperature)` into `ConversationConfig` once at conversation creation. Mutating `model.configValues` after the fact has zero effect on the live `Conversation`. Phase 3.6 Tasks 2 + 3 fix the runtime by adding a `registry.resetConversation(reason = LIGHT_OVERRIDE)` call after the map mutation. Additionally, `MAX_TOKENS` is reclassified from Light to Heavy (Decision 4) because `maxNumTokens` is a field of `EngineConfig` (set at `engine.create`), not `ConversationConfig` — even with Conversation recreation, the slider was a no-op for `max_tokens`.

2. **§ ErrorLog component strings (currently around line 16).** The whitelist enumeration must grow from 14 entries to 15 with `"inference-reset"`. Phase 3.6 Tasks 1 + 2 add `errorLog.i("inference-reset", ...)` (success path) and `errorLog.w("inference-reset", ...)` (non-Ready skip) inside `DefaultModelRegistry.resetConversation`. The whitelist is enforced at runtime (`IllegalArgumentException` on unknown component) — code in `ErrorLog.kt` and the patterns.md prose must stay in lockstep.

This is a doc-only task with no production-code touch. Wave 3 places it in parallel with Tasks 3 (ChatViewModel wiring) and 4 (enableEdgeToEdge) because the file sets are disjoint.
