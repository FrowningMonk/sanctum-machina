# Task 1: ErrorLog refactor — shared `write(level, ...)` helper + add `i()` / `w()` + whitelist `"inference-reset"`

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Сейчас `ErrorLog` (в `:core-runtime`) выставляет наружу только `e(component, description, cause)`. Этот метод владеет всем входным конвейером: проверка whitelist'а компонента, `sanitize()` (свёртка `\n\r\t` в пробел), длиновое усечение (`description ≤ 500`, `cause.message ≤ 200`), запись с `\n`, ротация при превышении 2 MB.

В phase 3.6 (Decision 5 tech-spec'а) `DefaultModelRegistry.resetConversation` начнёт писать диагностические события двух уровней — `i` (успешный сброс) и `w` (skip когда engine не Ready). Эти уровни должны проходить через ровно тот же входной конвейер: иначе появится drift (security-finding из tech-spec — пример: `status=...` в warning может попытаться интерполировать `Throwable.message` без `sanitize()` / без длинового усечения).

Решение: рефакторинг `ErrorLog` так, чтобы все три уровня (`e`/`i`/`w`) ходили через один частный helper `write(level, component, description, cause)`. `e()`, `i()`, `w()` становятся тонкими публичными обёртками. Длиновое усечение, `sanitize()`, ротация, mutex и I/O — внутри helper'а. Формат строки расширяется на префикс уровня (`ERROR`/`INFO`/`WARN`).

Дополнительно в `ALLOWED_COMPONENTS` добавляется новая запись `"inference-reset"` (был size 14 → станет 15) — Task 2 использует её в `errorLog.i("inference-reset", ...)` / `errorLog.w("inference-reset", ...)`. Closed-whitelist invariant сохраняется: неизвестные компоненты по-прежнему бросают `IllegalArgumentException`.

Эта задача — prerequisite для Task 2 (Wave 2): пока `i`/`w` нет, `DefaultModelRegistry` не может писать reset-логи, требуемые AC-1.4 / AC-1.5.

## What to do

1. Рефакторинг `ErrorLog.kt`:
   - Извлечь существующее тело `e()` (whitelist check + mutex + `sanitize`/truncate + `appendText` + ротация) в private `suspend fun write(level: Level, component: String, description: String, cause: Throwable?)`.
   - Ввести private `enum class Level { ERROR, INFO, WARN }` (или эквивалент) — печатается в начале строки вместо хард-кода `"ERROR "`.
   - `e(component, description, cause)` остаётся публичным API, тело сводится к `write(Level.ERROR, component, description, cause)`.
   - Добавить публичные `suspend fun i(component, description, cause = null)` → `write(Level.INFO, ...)` и `suspend fun w(component, description, cause = null)` → `write(Level.WARN, ...)`. Сигнатура `cause: Throwable?` опциональна — same shape as `e()`.
   - `buildLine` принимает `level`, печатает `"<LEVEL> [component] desc"` плюс `:: CauseClass: causeMsg` если cause != null. `LEVEL` строкой = `level.name`.
   - Whitelist check (`require(component in ALLOWED_COMPONENTS)`) делается ОДИН раз в `write` — не дублируется в `e`/`i`/`w`. `IllegalArgumentException` бросается из `write`, до открытия файла.

2. Добавить `"inference-reset"` в `ALLOWED_COMPONENTS` после существующих Phase-3 элементов. Размер 14 → 15.

3. Обновить KDoc верхнего уровня: упомянуть, что есть три уровня (`e`/`i`/`w`), что format-line начинается с `ERROR`/`INFO`/`WARN`, и что rotation/sanitize/length-bounding общие. Ссылку на `patterns.md § Error logging conventions` сохранить.

4. Расширить `ErrorLogTest.kt`:
   - Заменить хардкоженный `assertEquals("expected 14 allowed components", 14, allowed.size)` на 15. Добавить `"inference-reset"` в список `allowed` в существующем тесте `knownComponents_accepted`.
   - Новый тест `whitelistCount_is15` явно проверяет `assertEquals(15, ALLOWED_COMPONENTS.size)`.
   - Новый тест `inferenceResetComponent_acceptedByAllLevels` — `e("inference-reset", "x")`, `i("inference-reset", "x")`, `w("inference-reset", "x")` все три проходят и пишут по строке. После трёх вызовов в файле — 3 строки; первая стартует с `ERROR `, вторая с `INFO `, третья с `WARN `.
   - Новый тест `unknownComponent_stillRejected_negative` — для каждого из `e`/`i`/`w` вызов с неизвестным `"inference-reset-x"` бросает `IllegalArgumentException` и не создаёт файл.
   - Новый тест `iAndW_lengthBoundingMatchesE` — `i("inference-reset", "a".repeat(600), RuntimeException("x".repeat(300)))`, симметрично для `w`. Проверить: description в строке ровно 500 символов, cause.message ровно 200, prefix `INFO [inference-reset] ` / `WARN [inference-reset] ` корректный, `:: RuntimeException: ` присутствует.

5. Не вносить ни одного нового compose/activity import. `:core-runtime` остаётся UI-free.

## Acceptance Criteria

- [ ] `ErrorLog` имеет публичные `suspend fun e/i/w` с одинаковой сигнатурой `(component, description, cause = null)`.
- [ ] Все три уровня внутри проходят через единственный private `write(level, component, description, cause)` — нет дублирования whitelist-check / `sanitize` / truncation / mutex / rotation.
- [ ] Формат строки начинается с `ERROR `/`INFO `/`WARN ` соответственно; остальная структура (`[component] description :: CauseType: message`) идентична существующей.
- [ ] `ALLOWED_COMPONENTS.size == 15`; новая запись — `"inference-reset"`.
- [ ] Closed-whitelist invariant: `e/i/w` с любым компонентом не из whitelist'а бросают `IllegalArgumentException` ДО открытия файла; файл не создаётся.
- [ ] Length-bounding одинаков для всех уровней: `description` усекается до 500, `cause.message` — до 200 (TAC-15 inherited).
- [ ] `:core-runtime` остаётся UI-free: новых `androidx.compose|androidx.activity` импортов в `core-runtime/src/main` нет.
- [ ] Все тесты в `ErrorLogTest.kt` зелёные, в т.ч. четыре новых (`whitelistCount_is15`, `inferenceResetComponent_acceptedByAllLevels`, `unknownComponent_stillRejected_negative`, `iAndW_lengthBoundingMatchesE`).

## Details

**Files:**
- `core-runtime/src/main/kotlin/app/sanctum/machina/core/log/ErrorLog.kt` —
  - Текущее состояние: один публичный `suspend fun e(component, description, cause)`. Private `buildLine(component, description, cause)`. Private `sanitize(raw, maxLen)`. Константы `MAX_DESCRIPTION_LEN = 500`, `MAX_CAUSE_MESSAGE_LEN = 200`, `MAX_LOG_BYTES = 2 MB`, `LOG_DIR = "logs"`, `LOG_FILE = "errors.log"`, `ROTATED_FILE = "errors.log.1"`, `CONTROL_WS = Regex("[\\n\\r\\t]")`. Mutex и `Dispatchers.IO` инкапсулированы в `e()`.
  - Что меняем: тело `e()` извлекается в `write(level, ...)`. `e()`/`i()`/`w()` — тонкие обёртки. `buildLine` принимает `level` (или принимает строковый префикс — выбор по вкусу автора). `ALLOWED_COMPONENTS` дополняется `"inference-reset"`.
- `core-runtime/src/test/kotlin/app/sanctum/machina/core/log/ErrorLogTest.kt` —
  - Текущее состояние: 9 `@Test` методов; `knownComponents_accepted` явно ассертит size 14; `unknownComponent_*` — два варианта; `causeChainBounding_*` — два варианта; `descriptionTruncation_cappedAt500`; `descriptionSanitization_*`; `nullCauseMessage_*`. Все тесты исключительно для `e()`.
  - Что меняем: 14 → 15 в `knownComponents_accepted` и добавить `"inference-reset"` в список. Добавить четыре новых теста (`whitelistCount_is15`, `inferenceResetComponent_acceptedByAllLevels`, `unknownComponent_stillRejected_negative`, `iAndW_lengthBoundingMatchesE`).

**Dependencies:** Нет блокирующих зависимостей (Wave 1, depends_on пуст). `kotlinx.coroutines`, Robolectric, JUnit 4.13.2 — уже на classpath.

**Edge cases:**
- `i`/`w` с `cause = null` (default) — должны печатать ровно `INFO [component] desc` без хвоста `:: ...` (mirror existing `e()` behaviour, см. `nullCauseMessage_emitsEmptyTail` для эталона; добавь зеркальный тест если уровень доверия требуется).
- `description` с control-whitespace (`\n\r\t`) — должен пройти `sanitize` для `i`/`w` так же, как для `e()`. Существующий `descriptionSanitization_controlWhitespaceReplaced` тест на `e()` достаточен по референсу.
- Concurrent `e/i/w` от разных корутин — single mutex `mutex` в `ErrorLog` гарантирует сериализацию записи (тот же mutex что и сейчас). Это уже покрыто тем фактом, что mutex живёт в helper'е.
- Ротация: helper отрабатывает её ровно один раз на запись, для всех уровней одинаково. Существующих тестов на ротацию в `ErrorLogTest.kt` нет (это ОК — она покрыта вручную / в Phase-1 интеграционно); добавлять не требуется.
- `IllegalArgumentException` от whitelist — должен бросаться ДО открытия файла. Тест `unknownComponent_stillRejected_negative` именно это и проверяет (через `assertFalse(logFile.exists())` после catch).

**Implementation hints:**
- Минимальный diff: оставить `MAX_DESCRIPTION_LEN`/`MAX_CAUSE_MESSAGE_LEN`/etc. константы как есть. Не менять формат строки кроме первого слова. Не менять имя файла, dir, mutex API.
- `Level` enum — private, лежит в том же файле. Имя enum-значений `ERROR`/`INFO`/`WARN` совпадает с печатаемой строкой → `level.name` достаточно для `buildLine`.
- `e()` оставить публичным (есть существующие caller'ы в Phase 1/2/3 — `download`, `inference-init` etc.). НЕ переименовывать, НЕ deprecate'ить.
- `i()`/`w()` сигнатура: `suspend fun i(component: String, description: String, cause: Throwable? = null)`. Симметрично для `w`.
- В тесте `inferenceResetComponent_acceptedByAllLevels` после трёх вызовов проверь `logFile.readLines()` → `lines.size == 3`, `lines[0].startsWith("ERROR [inference-reset]")`, `lines[1].startsWith("INFO [inference-reset]")`, `lines[2].startsWith("WARN [inference-reset]")`.
- В тесте `iAndW_lengthBoundingMatchesE` — после `i("inference-reset", "a".repeat(600), RuntimeException("x".repeat(300)))` распарсить строку: `prefix = "INFO [inference-reset] "`, `desc = ...substringAfter(prefix).substringBefore(" :: ")`, `causeTail = ...substringAfter(":: RuntimeException: ").trimEnd('\n')`. Asserts: `desc.length == 500`, `desc.all { it == 'a' }`, `causeTail.length == 200`, `causeTail.all { it == 'x' }`. Симметрично для `w`.
- НЕ менять `ALLOWED_COMPONENTS` visibility (`internal`) — Task 2 будет передавать строковый литерал, runtime check останется единственным барьером, как и сейчас.
