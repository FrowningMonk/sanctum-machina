# Gallery Fork Analysis — варианты подхода

Дата: 2026-04-14
Источник: `work/research/code-research.md` (отчёт code-researcher по `gallery-source/`)
Цель документа: описать три варианта создания PhoneWrap (полный форк / ядро + своя оболочка / голый старт) с плюсами, минусами и развилками — **без принятия решения**.

---

## Исходные данные (кратко из code-research.md)

| Факт | Значение |
|---|---|
| Размер Gallery (Android) | 170 Kotlin-файлов, ~37 447 строк |
| Чистое ядро инференса + data-model | ~2500 строк (~5-7% кодбазы) |
| Ядро + менеджмент моделей | ~4500-5000 строк (~15% кодбазы) |
| Остальное | UI, customtasks, onboarding, ModelManagerViewModel, promo — ~85% |
| Тесты | Нет вообще (ни unit, ни instrumentation) |
| Лицензия | Apache 2.0 (форк разрешён, нужно сохранять атрибуцию) |
| Стек | Kotlin 2.2, Compose, Hilt, Navigation, WorkManager, Proto DataStore, litertlm 0.10.0, ML Kit GenAI (AICore) |
| Абстракция движка | `LlmModelHelper` интерфейс + 2 реализации (LiteRT-LM и AICore) — **изолирована от UI** |
| Persistence чатов | **Отсутствует** (`ChatUiState.messagesByModel` — только in-memory) |
| UI AI Chat / Ask Image / Ask Audio | Один `ChatView` + тонкие обёртки с флагами `showImagePicker`/`showAudioPicker` — уже унифицирован |
| Prompt Lab | Отдельный UI (`VerticalSplitView`) — не ChatView |
| Agent Skills | Тот же `ChatView` + слой tools/skills сверху |
| Plugin API | `CustomTask` через Hilt `@IntoSet` — можно добавлять задачи без правки ядра |
| Firebase, AICore, Mobile Actions, Tiny Garden | Опциональны, отключаются удалением модулей |
| God-object | `ModelManagerViewModel` = 1411 строк (скачивание + HF auth + init движка + allowlist + theme + история) |
| Upstream churn | Высокий (файлы с Copyright 2026, `#noinspection GradleDependency` на litertlm, много `@OptIn(ExperimentalApi::class)`) |

**Три вывода, критичных для сравнения вариантов:**

1. **Ядро отделимо.** Runtime-слой (`LlmModelHelper` + `LlmChatModelHelper`) не зависит от Compose/ViewModel. Вытащить его в модуль/библиотеку — 2500 строк плюс рефакторинг `ModelManagerViewModel`.
2. **UI для чата уже унифицирован.** «Разделять» AI Chat / Ask Image / Ask Audio нечего — это один Composable с флагами. Работа будет по добавлению истории/проектов, а не по разделению экранов.
3. **Нет слоя персистентности чатов.** Любой вариант форка обязан добавить новую БД (Room или новый proto) + сущности Chat/Project/Message. Это не «косметика», а новая функциональность.

## Вариант A. Полный форк

**Определение:** форк всего репозитория Google AI Edge Gallery как стартовой точки. Переименование пакета/приложения, удаление ненужных customtasks (Mobile Actions, Tiny Garden, Prompt Lab, Benchmark, Agent Skills), удаление Firebase, AICore-ветки, промо-экранов. Добавление слоя персистентности чатов и сущности Project поверх существующих ChatViewModel / ChatView. Периодическое подтягивание upstream.

### Плюсы
- **Самый быстрый до первого рабочего билда.** Всё готово: скачивание моделей, HF OAuth, UI чата, обработка изображений/аудио, WorkManager с foreground-уведомлением, deep links. Не надо это писать — только чистить.
- **Ядро и обёртки уже интегрированы.** `ModelManagerViewModel`, `ChatViewModel`, `LlmChatModelHelper` уже общаются между собой, вся «сантехника» работает.
- **Есть `CustomTask` plugin API** — новые задачи (например, «Project Chat») можно добавить без правок ядра.
- **Возможность тянуть upstream-апдейты LiteRT и UX-улучшений** — пока код не разошёлся сильно, merge-конфликты терпимы. Особенно ценно для `litertlm` (API ещё не стабилен).

### Минусы
- **Нужно тянуть upstream или смириться с отставанием.** Google активно обновляет Gallery (файлы с Copyright 2026). В одиночку разрулить merge-конфликты в ~30 000 строк UI/customtasks — реалистично только если ты трогаешь мало файлов. Как только начнёшь переделывать `ModelManagerViewModel` или `ChatView` — каждый upstream-merge будет болью.
- **Таскать ~85% чужого кода, из которого пилить свой UX.** Плитки, промо, скиллы, benchmark — всё это надо либо удалить (и создать merge-конфликты), либо оставить (и таскать мёртвый вес).
- **God-object `ModelManagerViewModel` (1411 строк)** придётся либо рефакторить (merge-ад), либо жить с ним.
- **Много глобального состояния в Gallery** (`ExperimentalFlags.enableConversationConstrainedDecoding`, статические `cleanUpListeners`, `firebaseAnalytics`) — унаследуешь как есть.
- **Attribution-требования Apache 2.0** — сохранять копирайт-хедеры, отмечать изменения в изменённых файлах.
- **Риск «ты форкал не то».** Если через полгода Google выпустит v2 с полной переработкой (новый UI, новый модуль инференса, iOS), твой форк окажется на мёртвой ветке.

### Когда это правильный выбор
- Цель — максимально быстро получить рабочий PhoneWrap, обратно совместимый с моделями из Gallery allowlist.
- Готов к боли от upstream-синхронизации (или решил её игнорировать и сидеть на замороженной версии).
- Планируешь **минимальные** изменения UI — только добавить историю и проекты, не переделывать внешний вид.

### Оценка усилий (грубо)
- Старт: 1-2 дня на чистку (удаление Firebase, customtasks, переименование пакета).
- MVP с историей чатов и проектами: ~2-3 недели (Room + миграция ChatUiState + UI для списка чатов/проектов).
- Поддержка upstream: постоянный налог — оценить сложно, зависит от того, что Google меняет.

## Вариант B. Ядро + своя оболочка

**Определение:** вытащить из Gallery **только ядро инференса** (`runtime/`, `data/Model.kt`, `data/Config.kt`, `DownloadRepository`, `DownloadWorker`, часть `ModelManagerViewModel`) как модуль/локальную библиотеку. Написать свой UI (Compose), свою навигацию, свой слой персистентности (Room) и сущности Project/Chat с нуля. Ядро подтягивать как snapshot, обновлять вручную по мере выхода важных релизов litertlm.

### Плюсы
- **Чистый UX-слой под свои цели.** PhoneWrap сразу строится вокруг понятий Project и Chat, а не вокруг Task (плитки Gallery). Никакого наследства плиток / промо / benchmark.
- **Ядро — маленькое, изолированное, с низким churn.** Runtime-слой Gallery стабилен (его редко переделывают), обновлять ~2500 строк гораздо проще, чем 30 000.
- **Отказ от god-object.** `ModelManagerViewModel` разбивается на честные сервисы (ModelRegistry, ModelInitService, DownloadService, ModelAuthService) — один раз, без merge-конфликтов.
- **Возможность добавить тесты** (в Gallery их нет) — при написании своего UI проще сразу покрыть.
- **Можно сразу переделать `Model.instance: Any?`** в чистую модель (`ModelDefinition` + `ModelRuntimeHandle`).
- **Возможность заменить Hilt** на что-то попроще (или оставить).

### Минусы
- **Дольше до MVP.** UI чата, сообщения, рендеринг markdown, image picker, audio recorder, model manager экран — всё это есть в Gallery, но ты пишешь свой. ~30 000 строк UI не появятся сами.
- **Потенциально теряешь тонкости litertlm-интеграции.** `LlmChatModelHelper` и `AICoreModelHelper` вместе — 687 строк; там много edge-cases (NPU vs GPU, sampler=null, thinking-channel, cleanup race conditions). Легко пропустить что-то неочевидное при вытаскивании.
- **Подтягивать upstream ядра — вручную.** Нет автоматического merge, только следить за diff в `runtime/` и переносить правки. На ~2500 строк с низким churn это посильно, но требует дисциплины.
- **Ядро всё равно привязано к Android** (`Context`, `Bitmap`, `nativeLibraryDir`) — это не «чистый kotlin». Переиспользовать ядро в другом приложении можно, но не в Desktop/Server Kotlin.
- **Риск неправильно нарезать границы ядра.** Если граница пройдёт не там, будешь переносить всё больше кода, пока не получится деградация к варианту A.

### Когда это правильный выбор
- Готов вложиться в 2-3 месяца работы над UI.
- Хочешь сам контролировать архитектуру (свой data-слой, свой navigation, своё разделение сервисов) и не готов жить с god-object.
- Планируешь **существенные** изменения UX: свой главный экран, свой чат-экран, свой способ управления моделями.
- Цель — долгоиграющий проект, где upstream-налог большего масштаба неприемлем.

### Оценка усилий (грубо)
- Извлечение и рефакторинг ядра: 1-2 недели (runtime + data + model management + рефакторинг god-object).
- Свой UI (чат + список чатов + проекты + менеджер моделей + settings): 4-8 недель.
- Миграция на новые версии litertlm: несколько часов раз в 1-2 месяца.

### Развилка внутри варианта B
- **B1.** Ядро как отдельный gradle-модуль в этом же проекте (`:core-runtime`). Проще поддерживать, быстрее билдить.
- **B2.** Ядро как внешняя локальная Maven-публикация / submodule. Избыточно для соло-проекта.

По умолчанию — B1.

## Вариант C. Голый старт

**Определение:** свой проект с нуля, litertlm подключается как gradle-зависимость напрямую, без заимствования кода Gallery. Своя модель данных, свой UI, свой способ скачивания моделей, своя аутентификация HF.

### Плюсы
- **Полная свобода.** Ни строчки чужого кода, никаких наследственных проблем, никакого attribution.
- **Код ровно столько, сколько нужно.** Нет мёртвого веса.
- **Самый понятный для своей поддержки.** Всё написано своими руками → дебаггинг очевиднее.
- **Можно выбрать другие архитектурные решения** (не Compose, не Hilt, не MVVM — если ты этого хочешь).

### Минусы
- **Очень долго до MVP.** Нужно заново написать:
  - Интеграцию с litertlm (streaming, cancel, thinking-channel, NPU/GPU/CPU backends, sampler config, session reset).
  - Скачивание моделей с прогрессом (WorkManager + foreground notification).
  - HuggingFace OAuth через AppAuth.
  - Model allowlist и сохранение импортированных моделей.
  - Chat UI (markdown, stream, image preview, audio playback).
  - Image picker + camera (CameraX).
  - Audio recorder.
  - Settings (theme, secrets, feature flags).
  - Проверки on-device requirements (free storage, RAM, NPU availability).
- **Легко сделать хуже, чем в Gallery.** Gallery написана инженерами Google, у которых есть опыт с LiteRT и мобильной оптимизацией. Дубляж с нуля даст первую наивную реализацию с багами, которые в Gallery уже пойманы.
- **Кривая обучения litertlm по документации.** Gallery — лучший референс того, как пользоваться этим SDK, и если не смотреть в код Gallery, надо учиться по docs (которых мало).
- **Теряется attribution-преимущество.** Разработка Gallery — это open source, не использовать её = тратить своё время на то же, что уже сделано бесплатно.

### Когда это правильный выбор
- Цель — учебная (хочу изучить LiteRT с нуля, написать каждую строку).
- Планируешь **радикально** другой UX — голосовой assistant, специализированное приложение, не похожее на Gallery вообще.
- Хочешь не-Compose UI (XML views, Flutter, KMP).
- Считаешь Apache 2.0 attribution-требования неприемлемыми.

### Оценка усилий (грубо)
- MVP: 3-6 месяцев соло.
- Риск бросить на полпути: высокий.

## Уточняющие вопросы (до принятия решения)

1. **Срок и коммитмент.** Это проект «сделать за месяц и использовать» или «развивать полгода-год»? От этого зависит, стоит ли инвестировать в архитектуру (B) или схватить готовое (A).

2. **Насколько сильно хочется свой UX?**
   - Если нужен примерно Gallery-UX + история + проекты → вариант A.
   - Если хочется экран «проекты → чаты → сообщения» как главный навигационный паттерн (а не «плитки задач → модели → чат») → вариант B.

3. **Отношение к upstream.** Готов ли ты раз в 1-2 месяца разбирать diff upstream и решать, что тянуть? Если нет, вариант A теряет смысл (тогда это просто замороженный снапшот, который хуже, чем B).

4. **Нужны ли customtasks Gallery** (Agent Skills, Mobile Actions, Tiny Garden, Prompt Lab, Benchmark)? Если нужен хотя бы один — вариант A выигрывает. Если все не нужны — их мёртвый вес съедает ценность A.

5. **Нужна ли AICore-ветка** (ML Kit GenAI, серверный on-device LLM от Google)? Или только litertlm (модели скачиваются сами)? AICore требует отдельной permission + работает только на некоторых устройствах. Для Honor 200 непонятно — нужно проверять.

6. **Проекты с общим контекстом — что именно это значит?**
   - Один system prompt на группу чатов?
   - Общая «память» (vector store / retrieval)?
   - Общие файлы/документы как контекст?
   - Первый вариант тривиально ложится на существующий `systemInstruction` в litertlm. Второй и третий — это серьёзная новая функциональность, требующая отдельного спека.

7. **История чатов — масштаб.** Сотни сообщений на юзера или тысячи? Если тысячи — Room обязателен, DataStore не потянет. Если сотни — оба подхода работают.

8. **Нужна ли поддержка мультимодальности** (image, audio) с первого MVP, или только текст? Если только текст — можно проще и урезать наследство из Gallery.

9. **Готов ли делать переименование пакета + свою иконку + свою тему до первого запуска?** (Нужно для обоих вариантов A и B.)

10. **Что с iOS?** Если когда-нибудь понадобится — ни один из вариантов не поможет напрямую (litertlm есть и на iOS, но Gallery-iOS-часть не исследовалась, её нет в чекауте). Влияет на выбор архитектуры: вариант B с чистым ядром даёт шанс поделиться data-моделью через KMP в будущем.

## Принятое решение (2026-04-14)

**Вариант B: ядро + своя оболочка.**

### Обоснование
- Цель — долгоиграющий инструмент, живущий на устройстве годами, работающий с локальной моделью без зависимостей от внешних сервисов.
- UI — «как у гигантов» (ChatGPT/Claude/Qwen): проекты → чаты → сообщения. Структурно несовместим с Gallery (задачи → модели → чат), поэтому свой UI.
- Архитектура — под масштаб, с чистыми сервисами вместо god-object.
- iOS в будущем — оставляем шанс через KMP, вытаскивая чистое ядро.

### Где «магия скорости» (для справки при реализации)
- В ядре, не в оболочке. Ключевое: выбор backend (CPU/GPU/NPU), per-model конфиги (`maxNumTokens`, `supportedBackends`, sampler), `resetConversation` vs full re-init, правильный cleanup.
- В Compose-оболочке ускорителей нет — это рендер. Скорость такая же, как у Gallery, если вытащить ядро с его per-model конфигами.

### Что вытаскиваем в `:core-runtime` (gradle-модуль)
- `runtime/LlmModelHelper.kt`, `runtime/ModelHelperExt.kt`
- `ui/llmchat/LlmChatModelHelper.kt` → переезжает в `runtime/`
- `data/Model.kt` (рефакторинг: `ModelDefinition` + `ModelRuntimeHandle`)
- `data/Config.kt`, `ConfigValue.kt`, `Consts.kt`, `Tasks.kt` (только модельная часть)
- `data/DownloadRepository.kt`, `worker/DownloadWorker.kt`
- Allowlist с per-model конфигами (свой файл или форк Gallery)
- `ModelManagerViewModel` разбиваем на сервисы: `ModelRegistry`, `ModelInitService`, `DownloadService`, `ModelAuthService`

### Что НЕ берём
Compose-UI Gallery, customtasks (Agent Skills, Mobile Actions, Tiny Garden, Prompt Lab, Benchmark), Firebase, AICore (`AICoreModelHelper` + ML Kit GenAI), промо-экраны.

### Что пишем своё в `:app`
- Navigation: ProjectsScreen → ChatsScreen → ChatScreen.
- Room: `projects`, `chats`, `messages`, `project_files`.
- Свой `ChatScreen` (референс — `ChatPanel.kt`, но писать заново под новую data-модель).
- Мультимодальный ввод: image picker + CameraX, audio recorder.
- Settings, model manager, HF-авторизация.

### Отложено
- iOS.
- Общие файлы проекта (RAG) — только каркас под это (структура `project_files`), фичу пилим позже.
- `CustomTask` plugin API — не переносим, свой plugin-слой не нужен на MVP.
