# Task 16: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Final Wave acceptance testing for Phase 3 — Chat History. This task runs after all code tasks (Tasks 0–11), the docs update (Task 12), and all three audits (Tasks 13–15 code, security, test). Its job is to confirm that the entire Phase 3 feature is production-ready before merge into `main` and tagging `v0.3-history`.

Phase 3 introduces Room persistence (SanctumDatabase v1), application-scope model warmup (WarmupCoordinator), navigation overhaul (HomeScreen, ModalNavigationDrawer, typed chat routes), and several bundled fixes (Model.name→modelId DataStore migration, ErrorLog whitelist expansion, DeviceInfoCollector real data, imePadding). QA validates that all of these subsystems compile, pass tests, and meet the acceptance criteria stated in both the user-spec and tech-spec — without a live deployment environment. All checks are executable locally via Gradle.

This task also surfaces any remaining issues from the audit reports (Tasks 13–15). If audit findings were accepted but not yet fixed, they become blocking items for QA to document in the final report.

## What to do

1. Run the full Gradle build and confirm `BUILD SUCCESSFUL` with zero compile errors.
2. Run all unit test suites across all three modules (`:app`, `:core-runtime`, `:core-settings`) and confirm all pass, including all Phase-3 test classes added in Tasks 1–8.
3. Run instrumented DAO tests on a connected device (Honor 200) and confirm all pass, including ON DELETE CASCADE and schema JSON snapshot test.
4. Run `lintDebug` and confirm no critical errors.
5. Build the debug APK and confirm `app-debug.apk` is produced in `app/build/outputs/apk/debug/`.
6. Verify the Room schema JSON file exists at `app/schemas/app.sanctum.machina.data.SanctumDatabase/1.json` and is committed to git.
7. Static-grep `ErrorLog.kt` to confirm all 14 expected `ALLOWED_COMPONENTS` entries are present (8 pre-Phase-3 + 6 new Phase-3 components).
8. Static-grep `app_settings.proto` to confirm all three new `optional` fields are present (`default_model_id`, `last_used_model_id`, `settings_keys_migrated`).
9. Verify the `ModelRegistry` interface exposes `val activeModelName: StateFlow<String?>`.
10. Verify `Model.kt` has `val modelId: String` field.
11. Verify `ChatViewModel.onCleared()` contains no `registry.cleanup()` call.
12. Verify `SanctumApplication.onCreate` contains staging-dir orphan cleanup (`attachments/.staging-*`).
13. Check `AndroidManifest.xml` privacy regression guard: `allowBackup=false` and `dataExtractionRules` unchanged.
14. Review decisions.md entries from Tasks 13–15 audits; flag any unresolved findings as blocking.
15. Write the QA report — list each verification step, its result (pass/fail), and any blocking issues discovered. If all pass, declare the feature ready for merge.

## Details

**Dependencies:** Tasks 13 (Code Audit), 14 (Security Audit), 15 (Test Audit) must be complete before starting. Read their reports in decisions.md — any unresolved blocking findings must be fixed before QA can pass.

**Device requirement:** Step 5 (`connectedAndroidTest`) requires a physical Android device connected via USB (Honor 200 is the reference). If no device is available, document this as a blocker and report what was verified without it.

**ErrorLog component count:** The tech-spec states exactly 14 components in `ALLOWED_COMPONENTS`. The 8 pre-Phase-3 components are: `"download"`, `"inference-init"`, `"inference"`, `"inference-cleanup"`, `"settings-io"`, `"camera"`, `"audio"`, `"attachment-decode"`. The 6 new Phase-3 components are: `"model"`, `"engine-warmup"`, `"history-read"`, `"history-write"`, `"attachment-save"`, `"attachment-read"`. Grep for all 14 in `ErrorLog.kt` individually to confirm none is missing.

**Proto fields check:** `app_settings.proto` must have 3 new fields (`default_model_id = 2`, `last_used_model_id = 3`, `settings_keys_migrated = 4`) all declared as `optional`. Note: `settings_keys_migrated` is a bool field — it may not appear in the same grep as the string fields; check separately.

**Schema JSON git status:** `ls` confirms the file exists on disk. Also run `git status app/schemas/` to confirm it is tracked (not in `.gitignore` or untracked). An untracked schema JSON means the Room schema was never committed — this is AC-R5 violation and a hard blocker.

**`ChatViewModel.onCleared()` check:** grep for `cleanup` in `ChatViewModel.kt`. The expected result is zero matches inside `onCleared`. If `cleanup` appears elsewhere in the file (e.g., in a comment or in a different method for a different purpose), verify by context that it is not inside `onCleared`. Decision 5 explicitly requires this call to be deleted entirely.

**`aapt` availability:** `aapt` is part of the Android SDK build-tools. If not on PATH, use the full path: `$ANDROID_HOME/build-tools/<version>/aapt`. Alternatively run `./gradlew :app:assembleDebug` first, then inspect the manifest using `aapt`.

**Audit report format:** Write the QA report as a structured pass/fail table matching the 11-step Agent Verification Plan from the tech-spec, plus additional static-check results. For each step record: command run, actual output summary, result (PASS / FAIL / BLOCKED). Conclude with overall status: READY FOR MERGE or list of blocking issues.

**On partial failure:** If any step fails, report the exact failure output in decisions.md. Do not mark the feature ready. The implementing agent for the failing task must fix the issue; QA re-runs that step after the fix is committed.
