## Description

Full Phase-3 code quality audit. This is the Audit Wave task: the auditor reads every file created or modified during Phase 3 and produces a holistic quality report. Unlike a per-task code review, this task looks across all implementation tasks at once — checking that invariants are maintained consistently, that branches are complete, and that no cross-cutting issue was missed by individual task reviewers.

The audit covers 7 focus areas defined in the tech-spec:

1. **Single-engine invariant** — only one model in `ModelInitStatus.Ready` at a time; `WarmupCoordinator.cancelAndRestart` cancels the in-flight `Job` BEFORE calling `registry.initialize()` for the new model.
2. **`ChatIdentity` branch coverage** — all three modes (`Quick`, `Draft`, `Persistent`) are handled correctly and completely in `ChatViewModel`; no mode falls through to a wrong code path.
3. **Staging-dir cleanup completeness** — staging dirs cleaned on both success and failure paths; `SanctumApplication.onCreate` sweeps orphan `.staging-*` dirs on startup; no disk leak on any code path.
4. **No blocking calls on Main thread** — all file I/O, Room writes, and Bitmap operations run on `Dispatchers.IO`; no `runBlocking` on Main; no unguarded `File.*` calls in composables or in ViewModel `init {}`.
5. **Hilt scope correctness** — `@Singleton` applied to the correct components (`WarmupCoordinator`, `SanctumDatabase`, `ChatRepository`, `AppCorruptionState`); no scope leaks (e.g., a `@Singleton` holding a reference to an `Activity`-scoped object).
6. **No `Model.name` vs `Model.modelId` mix** — after the `SettingsMigrationHelper` migration, all Room rows use `modelId` (stable HF id), all DataStore keys use `modelId`; no code path silently falls back to `name` for persistence.
7. **`onCleared()` in `ChatViewModel` must NOT call `registry.cleanup()`** — this was an explicit architectural decision (Decision 5, AC-E6); audit verifies it was not accidentally re-added.

The auditor writes a structured audit report covering findings (severity: Critical / Major / Minor / Suggestion), affected files, and recommended fixes. The report is the deliverable — no code changes are made in this task.

## What to do

1. Read `work/phase-3-history/decisions.md` to understand what was actually implemented across all tasks and any deviations from the tech-spec.

2. Read all new and modified source files listed in the "All files to audit" section of the Details below. Read them in logical groups: data layer first (entities, DAOs, DB, repository), then engine layer (WarmupCoordinator, AppCorruptionState), then DI (AppModule), then application startup (SanctumApplication), then UI layer (ViewModels and screens), then core-runtime changes (ModelRegistry, Model, ErrorLog), then settings layer (proto, AppSettingsRepository, SettingsMigrationHelper), then diagnostics (DeviceInfoCollector), then the test file.

3. For each of the 7 focus areas, systematically check every relevant file and record findings. Use the acceptance criteria from the tech-spec as the checklist baseline.

4. Additionally check: coroutine dispatcher discipline (`.IO` for I/O), Flow collection lifecycle (no `GlobalScope` leaks), `@Transaction` annotations on multi-step DAO operations, Room entity field names match SQL column names, `ON DELETE CASCADE` present on `messages.chat_id`, schema version matches exported JSON.

5. Write the audit report as the final output of this task. Report structure:
   - **Summary** — overall quality assessment (1 paragraph).
   - **Critical findings** — issues that must be fixed before merge (blocking correctness, data loss, crashes).
   - **Major findings** — significant quality issues (performance, missing error handling, invariant violations).
   - **Minor findings** — style, naming, incomplete edge-case coverage that won't cause failures in normal use.
   - **Suggestions** — optional improvements.
   - **Per-focus-area verdict** — one line per area: PASS / PASS WITH NOTES / FAIL.
   - **Files reviewed** — list of all files read.

6. Record the audit report in `decisions.md` under Task 13 (summary + link to report JSON if saved to logs).

## Details

**Focus area reference — what to look for in each:**

**1. Single-engine invariant (`WarmupCoordinator`):**
- `cancelAndRestart(modelId)`: verify the existing warmup `Job` is `.cancel()`-ed and `.join()`-ed (or at minimum cancelled) BEFORE `registry.initialize(modelId)` is called. A race where both `initialize` calls overlap violates the single-engine invariant.
- Verify that `SanctumApplication.onCreate` launches `warmupCoordinator.warmupDefault()` inside the `getProcessName() == packageName` guard, AFTER `installCrashHandler()`.
- Verify Task 4 added AC-F3 observer in `WarmupCoordinator`: collects `registry.models`, detects first `ModelEntry.isDownloaded=true` entry (field added by Task 2), sets `default_model_id` only when it is currently empty.

**2. `ChatIdentity` branch coverage (`ChatViewModel`):**
- `ChatIdentity.Quick`: engine state observed reactively (no `registry.initialize` call), in-memory message list only (zero `MessageDao` calls), `onCleared` does NOT call `registry.cleanup()`.
- `ChatIdentity.Draft`: draft→committed transition via `chatRepository.commitDraftChat()`, navigation `popUpTo("chat/draft"){inclusive=true}`, staging dir logic delegated to repository.
- `ChatIdentity.Persistent`: Room-backed `messageDao.observeByChat(chatId)` combined with in-memory `_streamingMessage`, `cancelAndRestart` called on "Загрузить" tap.
- The `onCleared()` method must be absent or explicitly empty — no `registry.cleanup()` call anywhere.

**3. Staging-dir cleanup (`DefaultChatRepository`, `SanctumApplication`):**
- `commitDraftChat()` success path: staging dir renamed to final `{chatId}/` path atomically.
- `commitDraftChat()` failure paths: (a) IOException during staging file write → `deleteRecursively()` on staging dir before throwing; (b) Room `@Transaction` fails after staging write → `deleteRecursively()` on staging dir.
- `SanctumApplication.onCreate`: orphan `.staging-*` dirs under `filesDir/attachments/` enumerated and deleted with `try/catch + ErrorLog.e("attachment-save", ...)`.
- `filesDir/quick/` purged on every startup with `try/catch + ErrorLog.e("attachment-save", ...)` (failure is not silent — it breaks incognito guarantee).

**4. No blocking on Main thread:**
- All `File.*` operations (read, write, deleteRecursively, rename, listFiles) are inside `withContext(Dispatchers.IO)` blocks.
- All Room DAO calls that are `suspend fun` are called from a coroutine; `Flow` collectors use `flowOn(Dispatchers.IO)` or the DAO itself specifies it.
- `Bitmap.compress(PNG, ...)` calls (attachment write) confirmed on `Dispatchers.IO`.
- No `runBlocking` anywhere in production code paths (only in tests if any).

**5. Hilt scope correctness:**
- `AppModule` (`@InstallIn(SingletonComponent::class)`): `@Singleton @Provides` on `SanctumDatabase`, `ChatDao`, `MessageDao`, `DefaultChatRepository` (bound as `ChatRepository`), `WarmupCoordinator`, `AppCorruptionState`.
- `ChatViewModel`: `@HiltViewModel` (ViewModel scope — correct, not singleton).
- `DrawerViewModel`, `HomeViewModel`: `@HiltViewModel` (ViewModel scope — correct).
- Check that `SanctumDatabase` is NOT created multiple times (DB builder called only in `AppModule`).

**6. `Model.name` vs `Model.modelId`:**
- Verify Task 5 added `ChatEntity.modelId` field (`TEXT NOT NULL`) populated from `Model.modelId` (added by Task 2), not from `Model.name`.
- `DefaultChatRepository.commitDraftChat(modelId, ...)` receives and stores `modelId`.
- `WarmupCoordinator` reads `AppSettings.default_model_id` / `last_used_model_id` and passes `model.modelId` (the stable HF id from Task 2) as the value to `registry.initialize(modelName)` — the parameter is still named `modelName` but the value passed must be `model.modelId`, not `model.name`.
- `SettingsMigrationHelper` remaps `per_model_overrides` keys from name → id, not id → name.
- `DrawerViewModel` and `ChatViewModel` use `chat.modelId` for registry lookups, not `chat.modelName` or similar.

**7. `onCleared()` in `ChatViewModel`:**
- Grep the file for `registry.cleanup` — must be absent.
- Grep for `onCleared` — if present, verify it only cancels the ViewModel's own `viewModelScope` (implicit) and does not call any registry method.

**Additional checks:**
- `@Transaction` annotation present on `ChatDao` or `DefaultChatRepository` method that performs the atomic INSERT chats + INSERT messages operation.
- `ErrorLog.ALLOWED_COMPONENTS` contains exactly 14 entries from the tech-spec AC criteria: `"download"`, `"inference-init"`, `"inference"`, `"inference-cleanup"`, `"settings-io"`, `"camera"`, `"audio"`, `"attachment-decode"`, `"model"`, `"engine-warmup"`, `"history-read"`, `"history-write"`, `"attachment-save"`, `"attachment-read"`.
- `app_settings.proto`: verify Task 2 added `optional string default_model_id = 2`, `optional string last_used_model_id = 3`, `optional bool settings_keys_migrated = 4`.
- `ModelRegistry` interface: verify Task 3 added `val activeModelName: StateFlow<String?>`.
- `Model.kt`: verify Task 2 added `val modelId: String` field.
- `SanctumDatabase` uses `exportSchema = true`, `version = 1`, `entities = [ChatEntity::class, MessageEntity::class]`.
- `messages.chat_id` FK: `@ForeignKey(entity = ChatEntity::class, parentColumns = ["id"], childColumns = ["chat_id"], onDelete = ForeignKey.CASCADE)` in `MessageEntity`.
- `chats.project_id` nullable (no `NOT NULL` annotation, no `@ForeignKey` on it).
- No `project_files` table entity in `SanctumDatabase.entities`.
- Module boundary: `core-runtime` and `core-settings` files contain no Compose / Activity / ViewModel imports. Quick grep: `grep -rEn "androidx.compose|androidx.activity|androidx.lifecycle.viewmodel" core-runtime/src/main core-settings/src/main`.
- `DeviceInfoCollector` / `AndroidDeviceInfoProvider`: primary constructor reads from `registry.models.value`; secondary no-arg constructor returns null/emptyList stubs.
- `SanctumApp.kt`: old `chat/{modelName}` route removed; new routes `chat/quick`, `chat/draft`, `chat/{chatId}` present; `home` is the start destination.
- DB corruption handler in `AppModule` (or `SanctumApplication`): `Room.databaseBuilder.build()` wrapped in `try/catch`; on exception renames DB file, creates fresh DB, sets `AppCorruptionState.corruptionOccurred = true`, logs `ErrorLog.e("history-read", ...)`.
