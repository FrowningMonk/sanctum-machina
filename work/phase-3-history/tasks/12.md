## Description

Phase 3 implementation diverged from several things that `architecture.md` and `patterns.md` described as the plan. This task brings those two project-knowledge files up to date so that any new agent loading PK gets an accurate picture of the codebase, not a snapshot from before Phase 3 was finalized.

The two files contain three categories of stale information:

1. **`architecture.md` — Data Model section.** The pre-Phase-3 description says `chats.project_id` is `NOT NULL` with a FK to `projects`, and lists a `project_files` table as "created in v1 schema". Both are superseded by tech-spec Decision 1 (AC-R4, AC-R7): `project_id` is nullable with no FK in v1, and `project_files` is not created until Phase 4 `Migration(1,2)`.

2. **`architecture.md` — Navigation routes and SanctumApplication.** The nav section still shows the old `chat/{modelName}` route. `SanctumApplication` description lacks warmup wiring and housekeeping details introduced in Phase 3.

3. **`patterns.md` — ErrorLog whitelist, Model lifecycle, and model-switch business rule.** The whitelist entry lists only 8 components; Phase 3 adds 6 more (and `settings-io` already existed). The "Model lifecycle" pattern still describes `onCleared()` calling cleanup — this was explicitly deleted in Decision 5 (AC-E6). A new "model-switch in draft mode" business rule exists but is not yet documented.

This task is pure documentation editing — no code is written or changed.

**`.claude/skills/project-knowledge/references/patterns.md`** — current state and what to change:

- **ErrorLog component strings whitelist** (line ~16): Currently lists 8 components: `"download"`, `"inference-init"`, `"inference"`, `"inference-cleanup"`, `"settings-io"`, `"camera"`, `"audio"`, `"attachment-decode"`. Update the "Adding a new component" sentence to match.

- **Model lifecycle** (line ~28): Currently describes `ModelInitService.initialize(model)` called on first chat open, result held as application-scoped singleton, `onCleared` implies cleanup. Rewrite to Phase 3 reality:
  - Engine lifecycle is owned by `WarmupCoordinator` (`:app`-level `@Singleton`), not by `ChatViewModel`.
  - `SanctumApplication.onCreate` triggers `WarmupCoordinator.warmupDefault()` — initializes the default model on a background coroutine.
  - `ChatViewModel.onCleared()` does NOT call `registry.cleanup()` — that call was deleted entirely (Decision 5, AC-E6). The engine outlives any single `NavBackStackEntry`.
  - Cleanup happens only in two situations: (1) explicit cross-model switch via `warmupCoordinator.cancelAndRestart(modelId)`, (2) process death.
  - Same-model navigation (chat list → another chat with same `model_id`) reuses the warm engine; `ChatViewModel` observes registry status reactively.

- **Business Rules → Model switching mid-chat** (line ~130): Currently says "Deferred. MVP: a chat is bound to one `model_id` for its lifetime. If the user wants to try a different model on the same topic, they start a new chat." Replace with Phase 3 rule:
  - A committed chat's `model_id` is immutable (set at `commitDraftChat` time).
  - **Draft mode:** TopAppBar shows a model-picker `DropdownMenu` listing downloaded models. Selecting the currently warm model is a no-op. Selecting a different model triggers a cross-model `AlertDialog` (confirm to switch); on confirm → `warmupCoordinator.cancelAndRestart(modelId)`.
  - **Persistent chat with non-active model:** `ChatScreen` TopAppBar shows "Загрузить" button (when engine status for `chat.model_id` is `Idle` or `Failed`). Tap → `warmupCoordinator.cancelAndRestart(chat.model_id)`. During re-init: spinner + "Модель перезагружается…" label, Send disabled.
  - **Persistent chat with same model as warm engine:** history loads instantly from Room; Send is enabled immediately (engine already `Ready`).
