## Description

This task creates the application-level engine warmup coordinator and the Hilt DI module that wires together the Room database, DAOs, repository, and the coordinator itself.

**WarmupCoordinator** is the `:app`-level singleton that owns the engine lifecycle on behalf of `SanctumApplication`. It injects both `ModelRegistry` and `AppSettingsRepository`, which is only possible at the `:app` layer — `:core-runtime` cannot depend on `:core-settings` (Decision 3). The coordinator resolves the default model using the three-step rule (AC-F5): `default_model_id` → `last_used_model_id` → no-op; calls `registry.initialize(modelId)` in a cancellable coroutine; on success writes `last_used_model_id` (AC-F6); on failure logs `ErrorLog.e("engine-warmup", ...)` (AC-D4). It also owns the AC-F3 observer: a background coroutine collecting `registry.models` that auto-sets `default_model_id` the first time any model's `downloadStatus.status` becomes `ModelDownloadStatusType.SUCCEEDED` while the setting is still empty.

`cancelAndRestart(modelId: String)` cancels the in-flight warmup `Job` before starting a new `initialize()` call — this is the mechanism that prevents a 50–60 s double-wait when the user taps "Load" for a different model while the default warmup is still running (see user-spec Risk about `lifecycleMutex` serialisation).

**AppModule** is the `@InstallIn(SingletonComponent::class)` Hilt module in `app/di/`. It provides: `SanctumDatabase` (built with `Room.databaseBuilder`, `exportSchema = true`), `ChatDao` and `MessageDao` (extracted from the database instance), `ChatRepository` (bound to `DefaultChatRepository`), and `WarmupCoordinator`.

Tasks 1, 2, 3, and 4 must be complete before this task starts: Task 1 produces `SanctumDatabase`, `ChatDao`, `MessageDao`; Task 2 adds the proto fields and new `AppSettingsRepository` methods; Task 3 adds `activeModelName` to `ModelRegistry`; Task 4 produces `ChatRepository` and `DefaultChatRepository` which `AppModule` binds.

## What to do

1. Create `app/src/main/kotlin/app/sanctum/machina/engine/WarmupCoordinator.kt` as a `@Singleton` class injecting `ModelRegistry`, `AppSettingsRepository`, and `ErrorLog`. It must hold a `Job?` field for the current warmup coroutine and a `CoroutineScope` on `Dispatchers.Default + SupervisorJob`.

2. Implement `warmupDefault()`: reads `appSettings.getDefaultModelId()` (Task 2 method); if blank, reads `appSettings.getLastUsedModelId()`; if both blank, returns without launching anything; otherwise launches a new `Job` calling `registry.initialize(modelId)` — on `onSuccess` calls `appSettings.setLastUsedModelId(modelId)`, on `onFailure` calls `errorLog.e("engine-warmup", ...)`.

3. Implement `cancelAndRestart(modelId: String)`: cancels the current `Job` and `join()`s it (so `lifecycleMutex` is released before the new `initialize` starts), then launches a new Job calling `registry.initialize(modelId)` with the same success/failure handling.

4. Implement the AC-F3 observer as a private `launch` block started in the coordinator's `init {}`: collect `registry.models`, for each emission check if any `ModelEntry` has `entry.downloadStatus.status == ModelDownloadStatusType.SUCCEEDED`; if yes and `appSettings.getDefaultModelId()` is empty, call `appSettings.setDefaultModelId(entry.model.modelId)` and break the loop (one-shot).

5. Create `app/src/main/kotlin/app/sanctum/machina/di/AppModule.kt` as `@Module @InstallIn(SingletonComponent::class)`. Provide `SanctumDatabase` via `Room.databaseBuilder(context, SanctumDatabase::class.java, "sanctum.db").build()`. Provide `ChatDao` and `MessageDao` by calling the corresponding methods on the `SanctumDatabase` instance. Provide `ChatRepository` via `@Binds` pointing to `DefaultChatRepository`. Provide `WarmupCoordinator` (Hilt resolves the `@Singleton` constructor automatically — no explicit `@Provides` needed beyond `@Binds` if an interface is involved; since `WarmupCoordinator` is a concrete class, just ensure `@Singleton @Inject constructor` is on it).

6. Write `WarmupCoordinatorTest` (JUnit 4 + coroutines test) covering all five test cases listed under TDD Anchor. Use hand-rolled fakes for `ModelRegistry`, `AppSettingsRepository`, and `ErrorLog` — no Mockito/MockK per patterns.md.

## Details

### Files

**`app/src/main/kotlin/app/sanctum/machina/engine/WarmupCoordinator.kt`** — new file, does not exist yet.
- `@Singleton class WarmupCoordinator @Inject constructor(...)` with: `ModelRegistry`, `AppSettingsRepository`, `ErrorLog`. Do not include `@ApplicationContext Context` — the coordinator performs no context-dependent work (no file I/O, no system services); injecting it would be dead weight.
- Needs its own `CoroutineScope(SupervisorJob() + Dispatchers.Default)` — not `viewModelScope`. The class is not a ViewModel; it lives as long as the process.
- The `warmupDefault()` and `cancelAndRestart()` methods are public; the AC-F3 observer is private, started in `init {}`.
- `cancelAndRestart` must ensure the in-flight coroutine is fully stopped before launching the new one — not just signalled for cancellation. Use `Job.cancelAndJoin()` (see kotlinx.coroutines structured concurrency docs); plain `cancel()` is insufficient because the mutex may still be held when the new `initialize()` starts, causing the 50–60 s double-wait described in user-spec Risks.
- Model resolution in `warmupDefault()`: call `appSettings.getDefaultModelId()` (from Task 2). The `ModelRegistry` does not need to be consulted for file-existence here — `registry.initialize(modelId)` will fail if the model file is gone, and the failure path logs via `ErrorLog.e("engine-warmup", ...)` which is the appropriate outcome.
- For AC-F3: check `entry.downloadStatus.status == ModelDownloadStatusType.SUCCEEDED` — `ModelEntry` does not have an `isDownloaded` field; the download state is represented by `entry.downloadStatus: ModelDownloadStatus`. Collect `registry.models` via `Flow.collectLatest` or a plain `collect`; iterate entries; apply the predicate. Once `setDefaultModelId` is called, break out of the collection loop by cancelling its `Job`.
- `"engine-warmup"` must already be in `ErrorLog.ALLOWED_COMPONENTS` (added by Task 2) — the call will throw `IllegalArgumentException` at runtime if that task is not done first.

### Edge cases

- `warmupDefault()` called multiple times before first warmup completes: each call should cancel the previous Job and start a new one (same logic as `cancelAndRestart`). Consider making `warmupDefault()` delegate to `cancelAndRestart(resolvedModelId)` internally to avoid duplication.
- `cancelAndRestart` called with the same `modelId` as the currently warming model: should still cancel-and-restart (idempotent behaviour, consistent with "cancel before starting").
- AC-F3 observer: if multiple models have `downloadStatus.status == ModelDownloadStatusType.SUCCEEDED` in the same emission, pick the first one (iteration order of the list). Do not call `setDefaultModelId` more than once.
- If `registry.models` emits an empty list on startup (no models downloaded), the observer should not call `setDefaultModelId`; it waits for a future emission with at least one downloaded entry.

### Implementation hints

- `cancelAndJoin()` is a Kotlin Coroutines extension — use it directly on the `Job`.
- The download status is available via `entry.downloadStatus.status` — compare to `ModelDownloadStatusType.SUCCEEDED`. This value is populated by `DefaultModelRegistry` from the local file scan at construction time (confirmed in code-research K1 and tech-spec Architecture § Cold start sequence step 5). No additional file-existence check is needed in WarmupCoordinator.
- The `AppSettingsRepository` interface (as of Task 2) will have `suspend fun getDefaultModelId(): String` and `suspend fun setDefaultModelId(modelId: String)` etc. Read those method signatures from the Task 2 output before writing WarmupCoordinator.
- `ModelRegistry.models: StateFlow<List<ModelEntry>>` — `StateFlow` can be collected as a hot flow; the AC-F3 observer should `cancel()` its own Job after calling `setDefaultModelId` (one-shot). A clean way: wrap the collect in a `Job` saved to a field, then cancel it inside the collect lambda after the first match.
- For `AppModule`: the abstract class + companion object pattern is idiomatic for mixing `@Binds` and `@Provides` in one Hilt module. See the existing `CoreRuntimeModule` for reference.
