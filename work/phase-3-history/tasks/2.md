## Description

This task is pure plumbing â€” no UI composables. It touches five distinct areas that must land together because they are mutually dependent foundations for every Wave 2+ task.

**1. Proto schema extension.** `app_settings.proto` gets three new `optional` fields on `AppSettings`: `default_model_id = 2`, `last_used_model_id = 3`, `settings_keys_migrated = 4`. These fields don't exist yet; the current schema has only `per_model_overrides = 1`. Without them, `WarmupCoordinator` (Task 5) has nowhere to read/write the default model, and `SettingsMigrationHelper` has no sentinel.

**2. Repository methods.** `AppSettingsRepository` (interface) and `DefaultAppSettingsRepository` (impl) gain eight new methods: `getDefaultModelId`, `setDefaultModelId`, `observeDefaultModelId`, `getLastUsedModelId`, `setLastUsedModelId`, `observeLastUsedModelId`, `isSettingsMigrated`, `markSettingsMigrated`. Currently the interface only exposes per-model override CRUD. The new suspend/flow methods follow the existing `IOException` catch + `ErrorLog.e("settings-io", ...)` pattern. The two `observe*` methods return `Flow<String>` backed by `dataStore.data.map { ... }`, consistent with the existing `observePerModelSettings(): Flow<Map<String, PerModelSettings>>` pattern â€” Task 11 depends on `observeDefaultModelId()` to reactively drive the star indicator in Model Manager.

**3. `Model.modelId` field.** `Model` data class does not have a `modelId: String` field today (confirmed by reading the file). `AllowedModel` has it, but `toModel()` uses it for URL construction and discards it. Adding `val modelId: String = ""` to `Model` and passing `modelId = modelId` in `AllowedModel.toModel()` is a one-line fix (code-research K2). This unlocks Room's `chats.model_id` column using the stable HF repo id rather than the human-readable name. The empty default means all non-allowlist `Model` construction sites compile without change.

**4. `SettingsMigrationHelper`.** New class in `:app`. On first Phase 3 boot, reads all `per_model_overrides` map keys (currently written as `Model.name`, e.g., `"Gemma-4-E4B-it-litert-lm"`), matches each to its `AllowedModel.modelId` via `ModelRegistry.models.value`, rewrites the map keys from name to modelId, drops orphan keys (no matching allowlist entry â€” logs them via `ErrorLog.e("settings-io", ...)`), and writes the result with a single atomic `DataStore.updateData {}`. Sets `settings_keys_migrated = true` after. Subsequent starts check the sentinel and skip entirely. Decision 8.

**5. `ErrorLog` whitelist expansion.** Six new component strings added to `ALLOWED_COMPONENTS`: `"model"`, `"engine-warmup"`, `"history-read"`, `"history-write"`, `"attachment-save"`, `"attachment-read"`. Without these, any call to `ErrorLog.e()` with these components throws `IllegalArgumentException` at runtime. `ErrorLogTest` must be extended to verify all 6 new strings pass.

**6. `DeviceInfoCollector` stub fix (AC-D1).** `AndroidDeviceInfoProvider.activeModelId()` and `downloadedModels()` currently return `null` / `emptyList()` hardcoded (Phase 2.5 TODO). They need to be wired to `ModelRegistry`. The primary `@Inject constructor` gains a `ModelRegistry` parameter; the two methods read from `registry.models.value`. The secondary no-arg constructor used by `LogExportManager(context)` in the `:crash` process stays as-is â€” it keeps null stubs, because the crash process has no running engine (Decision 10). The fix is: add a second constructor `constructor(@ApplicationContext context: Context) : this(context, null)` and make `registry` nullable, or overload with separate crash-process stub. The crash path `LogExportModule` binding uses the Hilt constructor; the `:crash` hand-instantiation path uses the no-arg secondary constructor.
