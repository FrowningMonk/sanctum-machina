## Description

Создаём data-layer поверх Room DAOs из Task 1: интерфейс `ChatRepository` и его реализацию `DefaultChatRepository`, плюс pure-функцию `AutoTitleGenerator`.

`DefaultChatRepository` оркестрирует все операции с чатами и сообщениями. Ключевая сложность — атомарная запись вложений при первом Send в draft-чате (Decision 6): файлы сначала идут в staging-директорию `filesDir/attachments/.staging-{uuid}/`, затем Room `@Transaction` INSERT chats + messages, затем `rename()` всей директории в `filesDir/attachments/{chatId}/`. Если на любом шаге падает исключение — staging удаляется, Room row (если успел вставиться) откатывается. Это единственный способ гарантировать «all or nothing» на Android-файловой системе без транзакционных FS-примитивов.

Остальные методы: `savePersistentMessage()` — INSERT одного сообщения (USER сразу при Send, ASSISTANT на `done=true`); `updateChatLastMessage()` — UPDATE `chats.last_message_at` и опционально title; `updateChatTitle()` — ручное переименование; `deleteChat()` — CASCADE DELETE через Room + `File.deleteRecursively()` на `filesDir/attachments/{chatId}/`; `observeChats()` / `observeMessages()` — проброс Flow из DAOs; `sweepZombieChats()` — удаляет чаты с 0 сообщений и отсутствующей директорией вложений (zombie после краша посреди `commitDraftChat`).

`AutoTitleGenerator` — pure-функция без Android-зависимостей: принимает первое USER-сообщение (текст + timestamp), возвращает строку-заголовок по алгоритму AC-U2.

Весь файловый I/O выполняется в `Dispatchers.IO`. Этот репозиторий потребляется `ChatViewModel` (Task 8), `DrawerViewModel` (Task 9) и `SanctumApplication` (Task 6).

## What to do

1. Создать `ChatRepository.kt` — интерфейс с методами:
   - `suspend fun commitDraftChat(modelId: String, firstMessage: MessageEntity, stagingDir: File): Long` — возвращает `chatId`
   - `suspend fun savePersistentMessage(message: MessageEntity)`
   - `suspend fun updateChatLastMessage(chatId: Long, timestampMs: Long)`
   - `suspend fun updateChatTitle(chatId: Long, title: String, isManuallyTitled: Boolean)`
   - `suspend fun deleteChat(chatId: Long, filesDir: File)`
   - `fun observeChats(): Flow<List<ChatEntity>>`
   - `fun observeMessages(chatId: Long): Flow<List<MessageEntity>>`
   - `suspend fun sweepZombieChats(filesDir: File)`

2. Создать `DefaultChatRepository.kt` — реализацию интерфейса, инжектирующую `ChatDao`, `MessageDao`, `ErrorLog` и `CoroutineDispatcher` (для тестируемости IO-диспетчера):
   - `commitDraftChat`: создать `ChatEntity`, INSERT через `@Transaction`-метод DAO, после успеха rename staging → `filesDir/attachments/{chatId}/`; при падении rename — DELETE chat row из Room и `deleteRecursively` staging
   - `deleteChat`: вызвать `chatDao.deleteById(chatId)` (CASCADE удалит messages), затем `File(filesDir, "attachments/$chatId").deleteRecursively()` на `Dispatchers.IO`
   - `sweepZombieChats`: получить все чаты с 0 сообщениями через `messageDao.countByChat()`, проверить отсутствие `filesDir/attachments/{chatId}/`, удалить строку + логировать `ErrorLog.e("history-write", "zombie chat swept: id=...", null)`
   - все IO-операции с файлами — `withContext(ioDispatcher)`

3. Создать `AutoTitleGenerator.kt` — object или top-level функцию `fun generateTitle(firstUserText: String?, createdAtMs: Long): String`:
   - Если `firstUserText` null или пустой после trim → fallback `"Чат от DD.MM HH:mm"` (из `createdAtMs`, форматировать `Locale.getDefault()`)
   - Иначе: trim → collapse whitespace (все `\s+` → один пробел) → если результат ≤ 20 символов → вернуть как есть; иначе найти последний пробел в позиции ≤ 20, обрезать, добавить `"…"`; если пробела нет → обрезать hard на 20 + `"…"`

4. Написать `AutoTitleGeneratorTest.kt` со всеми AC-U2 кейсами (до написания реализации).

5. Написать `ChatRepositoryTest.kt` со всеми сценариями из tech-spec (до написания реализации).

6. Убедиться, что `DefaultChatRepository` помечен `@Singleton`-scope через Hilt (провайдер добавить в Task 5 `AppModule`, здесь только аннотация `@Inject constructor`).

## Details

### Files

**`ChatRepository.kt`** — создать с нуля. Kotlin interface. Suspend + Flow методы только, никаких blocking. Сигнатура `commitDraftChat` принимает `stagingDir: File` (staging-директория уже наполнена вызывающей стороной — `ChatViewModel` пишет туда файлы до вызова репозитория).

**`DefaultChatRepository.kt`** — создать с нуля. `@Inject constructor(private val chatDao: ChatDao, private val messageDao: MessageDao, private val errorLog: ErrorLog, private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO)`. Аннотация `@Singleton` НЕ здесь — провайдер будет в `AppModule` (Task 5). Инжект через конструктор позволяет подменять `ioDispatcher` в тестах. `ErrorLog` нужен для вызовов `ErrorLog.e()` в `sweepZombieChats` и прочих error-путях — это suspend-функция, поэтому зависимость обязательна для компиляции.

**`AutoTitleGenerator.kt`** — создать как `object AutoTitleGenerator` с единственным public методом `fun generateTitle(firstUserText: String?, createdAtMs: Long): String`. Никаких Android-зависимостей — только `java.text.SimpleDateFormat` и `java.util.Date`. Locale для формата даты: `Locale.getDefault()` (отображается пользователю).

**`AutoTitleGeneratorTest.kt`** — JUnit 4, чистый JVM без Robolectric (нет Android API). Использовать `assertEquals` / `assertTrue`.

**`ChatRepositoryTest.kt`** — JUnit 4 + kotlinx.coroutines.test (`runTest`). DAO-фейки реализовать как inner-классы теста (`FakeChatDao`, `FakeMessageDao`) — они возвращают заготовленные данные, позволяют проверять вызовы. `File` передавать через `@Rule TemporaryFolder` или через `java.io.tmpdir` — создавать реальные директории на диске. Для проверки `rename()` использовать реальные `File` objects (не mock), так как `File.renameTo` — platform call.

### Edge cases

- **rename() возвращает false**: на Android `File.renameTo()` может вернуть `false` вместо исключения при неудаче. Реализация должна проверять возвращаемое значение и бросать `IOException("rename failed: ...")` явно.
- **stagingDir не существует при `commitDraftChat`**: если нет вложений, staging-директория не создаётся вызывающей стороной. `commitDraftChat` должен работать с `stagingDir = null` или caller передаёт `null` когда вложений нет. Нужно определить сигнатуру — рекомендуется `stagingDir: File?` с проверкой перед rename.
- **deleteChat с несуществующей директорией вложений**: `deleteRecursively()` на несуществующей директории не падает (возвращает `false`). Это нормально — чаты могут существовать без вложений.
- **sweepZombieChats и чаты без директории**: директория `filesDir/attachments/{chatId}/` может не существовать у чатов без вложений — это НЕ признак zombie, если у чата есть сообщения. Sweep удаляет только 0-message чаты с отсутствующей директорией.
- **Конкурентный вызов sweepZombieChats**: вызывается только из `SanctumApplication.onCreate` в одном фоновом корутине, конкурентности нет.

### Implementation hints

- `@Transaction`-метод DAO для `commitDraftChat` должен быть определён в `ChatDao` (Task 1), принимать и `ChatEntity`, и `MessageEntity`, выполнять оба INSERT в одной транзакции, возвращать `Long` (chatId). `DefaultChatRepository.commitDraftChat` вызывает этот DAO-метод.
- После успешного Room INSERT и перед `rename()` chatId уже известен — директория назначения `filesDir/attachments/$chatId/`.
- В тестах `ChatRepositoryTest` не нужен Robolectric: `TemporaryFolder` даёт реальную temp-директорию на JVM, `FakeChatDao` имитирует Room. Проверка rename происходит через `File.exists()` на ожидаемых путях.
- `ErrorLog.e` — suspend-функция; в `sweepZombieChats` вызывать внутри `withContext(ioDispatcher)` или в coroutine scope репозитория.
- Для fallback даты в `AutoTitleGenerator` использовать: `SimpleDateFormat("dd.MM HH:mm", Locale.getDefault()).format(Date(createdAtMs))`.
