# Task 8: ChatViewModel rework

## Description

`ChatViewModel` — центральный класс для всех режимов чата (Quick / Draft / Persistent). Сейчас он тесно завязан на `modelName` как идентификатор и самостоятельно вызывает `registry.initialize(modelName)` при инициализации. В Phase 3 это неприемлемо по двум причинам:

1. **Прогрев engine — ответственность `WarmupCoordinator`**, а не VM. Если VM вызывает `initialize` в `init {}`, то открытие чата уничтожает смысл фонового warmup и создаёт гонки с `WarmupCoordinator`.
2. **`onCleared()` убивает прогретый движок** — вызов `registry.cleanup(modelName)` при каждом Back означает, что warmup умирает при первом же уходе из чата (AC-E6, Decision 5).

Задача: переписать `ChatViewModel` вокруг `ChatIdentity` (sealed class Quick / Draft / Persistent), заменить imperative `init {}` на реактивный `registry.models.collect {}`, добавить Room-backed message flow для persistent-чатов, реализовать draft→committed atomic transition, и удалить `onCleared()` cleanup.

Quick и Draft режимы не получают `modelName` из nav arg — вместо этого они читают активную модель из `registry.activeModelName.value` (StateFlow, поддерживается Task 3). Это та модель, которую `WarmupCoordinator` уже прогрел и которую пользователь видит в `TopAppBar`.

`ChatViewModelTest` необходим для проверки ключевых инвариантов до интеграции с UI-слоем (Tasks 9–11). UI-level верификация поведения VM деферирована на Tasks 9–11, поэтому `verify: []`.

## What to do

1. **Введи `ChatIdentity` sealed class** в пакете `app.sanctum.machina.ui.chat` (или `app.sanctum.machina.ui.model` — на усмотрение, но рядом с `ChatViewModel`):
   - `object Quick` — route `chat/quick`
   - `object Draft` — route `chat/draft`
   - `data class Persistent(val id: Long)` — route `chat/{chatId}`

2. **Замени `modelName` на `ChatIdentity`** как primary identity в `ChatViewModel`. Читается из `SavedStateHandle`: если есть `chatId: Long` nav arg → `Persistent(id)`; если есть `kind = "draft"` → `Draft`; иначе → `Quick`. Константы nav args переименуй: `NAV_ARG_MODEL_NAME` → `NAV_ARG_CHAT_ID`, `NAV_ARG_KIND`.

3. **Замени `init {} registry.initialize(...)` на реактивный observer** `registry.models.collect {}`:
   - Для каждого `ModelEntry` подходящей модели (`chat.model_id` для Persistent, текущая прогретая для Quick/Draft) смотри `initStatus` и обновляй `_uiState` реактивно.
   - `ModelInitStatus.Ready` → `ChatUiState.Ready(isGenerating = false)` (сохраняя `isGenerating` если уже идёт стрим)
   - `ModelInitStatus.Initializing` → `ChatUiState.Loading`
   - `ModelInitStatus.Failed(msg)` → `ChatUiState.Failed(msg)`
   - `ModelInitStatus.Idle` / null → `ChatUiState.Loading` (ожидание; пользователь нажмёт «Загрузить»)
   - VM не вызывает `registry.initialize(...)` автоматически — warmup — задача `WarmupCoordinator`.

4. **Добавь Room-backed message flow** для `Persistent` режима:
   - `persistedMessages: Flow<List<MessageEntity>>` = `messageDao.observeByChat(chatId)`, коллектируется в `viewModelScope`.
   - `_streamingMessage: MutableStateFlow<Message?>` — in-memory streaming ASSISTANT bubble.
   - `displayMessages: StateFlow<List<Message>>` = `combine(persistedMessages, _streamingMessage)` через `.stateIn(viewModelScope, SharingStarted.Eagerly, emptyList())`.
   - **Atomic handover при `done=true`**: когда `persistedMessages` первый раз эмитит список с новой ASSISTANT-строкой (той, что только что была записана в Room) — сбрасывай `_streamingMessage = null` в том же `mapLatest`/`transformLatest` блоке. Цель: не допустить emission где и in-memory, и persisted ASSISTANT видны одновременно.
   - Quick и Draft режимы: `persistedMessages` не используется, `_messages` (in-memory) остаётся как в Phase 2.

5. **Реализуй draft→committed atomic transition**:
   - При первом `send()` в `Draft` режиме: вызвать `chatRepository.commitDraftChat(modelId, firstMessage, stagingDir)`.
   - После успешного commit: `navController.navigate("chat/$chatId") { popUpTo("chat/draft") { inclusive = true } }`.
   - При failure: rollback уже описан в `ChatRepository` (Task 4) — VM остаётся на draft, вложения в `_attachments` сохраняются для retry.

6. **Удали `onCleared()` cleanup** — весь блок `CoroutineScope(SupervisorJob() + Dispatchers.IO).launch { registry.cleanup(modelName) }`. После удаления `onCleared` сводится к `super.onCleared()` или вовсе отсутствует.

7. **Напиши `ChatViewModelTest`** в `app/src/test/kotlin/app/sanctum/machina/ui/chat/ChatViewModelTest.kt` (создай файл):
   - Quick lifecycle: VM не вызывает `MessageDao` ни разу.
   - Draft→Persistent promotion: при первом send в Draft вызывается `chatRepository.commitDraftChat(...)`.
   - `onCleared` не вызывает `registry.cleanup(...)`.
   - Engine state transitions: `registry.models` emitting `Ready` → `uiState` becomes `Ready`; `Failed` → `uiState` becomes `Failed`.
   - Double-bubble handover: нет emission где и `_streamingMessage`, и persisted ASSISTANT присутствуют одновременно.

## Details

### Files

**`app/src/main/kotlin/app/sanctum/machina/ui/chat/ChatViewModel.kt`** — полная переработка:

*Текущее состояние:*
- `modelName: String` читается из `SavedStateHandle["modelName"]`; выбрасывает `error()` если отсутствует.
- `init {}` запускает `applyEffectiveConfigToModel()` затем `registry.initialize(modelName)` — imperative, блокирует на 5–30 с.
- `_messages: MutableStateFlow<List<Message>>` — всё in-memory, теряется при смерти процесса.
- `onCleared()` вызывает `registry.cleanup(modelName)` — убивает прогретый движок.
- `NAV_ARG_MODEL_NAME = "modelName"` — устаревший nav arg.

*Что изменить:*
- Ввести `ChatIdentity` sealed class; заменить `modelName` как primary identity; переключить nav args на `chatId: Long` и `kind: String`.
- Заменить imperative `registry.initialize(...)` в `init {}` на реактивный collect по `registry.models` — паттерн аналогичен `ModelManagerViewModel` в этой же кодовой базе.
- Для `Persistent` режима подключить Room Flow через `messageDao.observeByChat(chatId)` и объединить с in-memory streaming bubble с помощью `combine()` из `kotlinx.coroutines.flow` — см. паттерн слияния двух Flow в `patterns.md`.
- **Quick и Draft режимы получают активную модель из `registry.activeModelName.value` (StateFlow из Task 3)** — это модель, которую прогрел `WarmupCoordinator` и которую пользователь видит в `TopAppBar`. Не читать nav arg `modelName` для этих режимов.
- Атомарный handover при завершении стрима: сбросить in-memory streaming bubble в том же emission-цикле, где Room Flow впервые возвращает строку нового ASSISTANT-сообщения. Цель — не допустить состояния, когда оба источника видны одновременно.
- Удалить вызов `registry.cleanup(...)` из `onCleared()`.

### Edge cases

- **Draft + Quick mode**: `chatIdentity` не содержит `chatId` → `messageDao.observeByChat(...)` не вызывается; `_persistedMessages` остаётся пустым.
- **Persistent, process death mid-stream**: USER уже в Room (записан до `runInference`), ASSISTANT не записан (done=true не наступил) → при переоткрытии Room возвращает только USER, пользователь видит незавершённый разговор и может перезадать (AC-R3).
- **Rotation во время Draft send**: если `commitDraftChat` ещё не завершился и процесс ротируется — `SavedStateHandle` содержит ещё `kind=draft`; VM пересоздаётся в Draft режиме. Текст и вложения должны быть в `SavedStateHandle` или в памяти ViewModel (ViewModels переживают rotation).
- **Double-bubble race**: persisted ASSISTANT row приходит из Room, но `_streamingMessage` ещё не null → нужно сбросить `_streamingMessage` атомарно в том же `transformLatest` блоке, не дожидаясь следующего emit.
- **`registry.models` emit перед тем как Room-DAO готов** (для Persistent): VM должен быть в Loading пока не получит и engine Ready, и начальный snapshot из Room.
- **`applyEffectiveConfigToModel()` и модель ещё не в registry**: метод вызывает `preInitModel()` который может вернуть null → silent no-op, продолжаем. Initialize-вызов теперь не наш — WarmupCoordinator разберётся.
- **`send()` в Draft режиме при `reinitInProgress`**: блокируем как и прежде — если идёт cross-model reinit через TopAppBar, Send недоступен.

### Implementation hints

- `ChatIdentity` читается из `SavedStateHandle` в `init` — не в ленивом свойстве, чтобы ошибка "nav arg missing" всплывала сразу.
- Quick и Draft режимы не имеют `chatId` в nav args → активная модель берётся из `registry.activeModelName.value` (StateFlow из Task 3). Это та модель, которую прогрел `WarmupCoordinator` и которую пользователь видит в `TopAppBar`.
- Для слияния Room Flow и in-memory streaming bubble используй `combine()` из `kotlinx.coroutines.flow` — см. `ModelManagerViewModel` для примера паттерна combine в этой кодовой базе.
- Атомарный handover требует, чтобы сброс streaming state и возврат persisted state происходили в одном emission — без промежуточного состояния, где видны оба. Паттерн: обработай оба источника внутри одного оператора трансформации потока.
- Тестирование `onCleared`: вызывать через рефлексию (`getDeclaredMethod("onCleared").apply { isAccessible = true }.invoke(viewModel)`) — затем проверить `FakeModelRegistry.cleanupCalls.isEmpty()`.
- Внутри `collect {}` обновляй `_uiState` только при реальном изменении статуса — избегай лишних recompositions при повторном emit того же состояния.
- `viewModelScope` использует `Dispatchers.Main.immediate` по умолчанию — в тестах подменяй через `Dispatchers.setMain(UnconfinedTestDispatcher())`.
