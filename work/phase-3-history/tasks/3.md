# Task 3: ModelRegistry — activeModelName StateFlow

## Required Skills

Before starting, load:
- `/skill:code-writing` — [SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

This task adds `val activeModelName: StateFlow<String?>` to the `ModelRegistry` interface and implements it in `DefaultModelRegistry` as a derived StateFlow that emits the `model.name` of whichever `ModelEntry` currently has `initStatus === ModelInitStatus.Ready`, or `null` when no model is Ready.

**Why this matters.** Phase 3 has multiple consumers that need to know "which model is warm right now" without doing a `.find {}` scan on `registry.models` on every read:
- `DrawerContent` — before navigating to a chat on another model, checks `activeModelName.value` to decide whether to show a cross-model dialog.
- `ChatViewModel` — TopAppBar state machine (same model ready → read-only label; different model → "Загрузить" button).
- `WarmupCoordinator.cancelAndRestart()` — needs to know which model to `cleanup()` before initialising the new one.

A dedicated `StateFlow<String?>` derived from `models` is the minimal, correct abstraction: it leverages the existing `lifecycleMutex`-guarded `_models` MutableStateFlow, costs nothing at runtime (Kotlin's `map + stateIn` on a hot StateFlow), and eliminates scattered `.find { it.initStatus === Ready }` call sites across the codebase.

**Single-engine invariant.** Only one `ModelEntry` can ever be in `ModelInitStatus.Ready` at a time (enforced by `lifecycleMutex` in `DefaultModelRegistry`). The derived flow exploits this: `firstOrNull { it.initStatus === ModelInitStatus.Ready }?.model?.name` always produces the right answer and never needs a tie-breaking rule.

**Decision 7** from the tech-spec formally codifies this: `activeModelName: StateFlow<String?>` is the single source of truth for engine state across all consumers.

## What to do

1. **Extend `ModelRegistry` interface** — add `val activeModelName: StateFlow<String?>` as a new property declaration below `val models`. Add KDoc: emits the `model.name` of the first `ModelEntry` with `ModelInitStatus.Ready`, or `null` if none is Ready.

2. **Implement in `DefaultModelRegistry`** — add `override val activeModelName: StateFlow<String?>` as a derived StateFlow from `_models`. Use `map { list -> list.firstOrNull { it.initStatus === ModelInitStatus.Ready }?.model?.name }` combined with `stateIn(scope, SharingStarted.Eagerly, null)`. The initial value must be `null` (no model is Ready at construction time).

3. **Create the test file** `core-runtime/src/test/kotlin/app/sanctum/machina/core/registry/ModelRegistryActiveModelTest.kt` — write unit tests per the TDD Anchor below before implementing step 2.

4. **Verify the build** — run `./gradlew :core-runtime:testDebugUnitTest` and confirm all new tests pass with zero compilation errors.

**Dependencies:**

- `kotlinx-coroutines-test` must be present in `core-runtime/build.gradle.kts` as a `testImplementation`. Confirm before writing tests; add if missing (it is needed by Task 3 and later tasks).
- No new production dependencies — `StateFlow`, `map`, `stateIn`, `SharingStarted` are all from `kotlinx-coroutines-core`, already on the classpath.

**Edge cases:**

- Multiple models in the list — only the first one with `initStatus === Ready` is emitted. The single-engine invariant means there will never actually be two Ready entries simultaneously, but the `firstOrNull` is still correct and safe.
- `ModelInitStatus.Failed(message)` — the flow must emit `null`, not the failed model's name. `firstOrNull { it.initStatus === Ready }` naturally handles this because `Failed !== Ready`.
- `ModelInitStatus.Initializing` — likewise emits `null`. The flow correctly withholds the name until the engine is actually warm.
- Empty model list (before `refreshAllowlist` completes) — emits `null`. `firstOrNull` on an empty list returns null.
- Transition from Ready back to Idle (via `cleanup`) — `_models.update` replaces the Ready entry with an Idle one; the derived `map` fires immediately with the new snapshot, emitting `null`.

**Implementation hints:**

- `SharingStarted.Eagerly` is correct here: the flow must start collecting from `_models` immediately at construction, not lazily on first subscriber. The `scope` already has `SupervisorJob() + Dispatchers.Default` — same scope used for the startup scan.
- `===` (referential equality) is used throughout `DefaultModelRegistry` for `ModelInitStatus` comparisons (e.g., line 149: `entry.initStatus !== ModelInitStatus.Idle`). Use `===` consistently in both the production code and the tests for `ModelInitStatus.Ready` (it is an `object`, so referential equality is correct and slightly more explicit about intent than `is`).
- The test for the J5 concurrency hazard does not need real thread interleaving. `UnconfinedTestDispatcher` + sequential mutations of a `MutableStateFlow` within a `TestScope` is sufficient to verify that every emitted value of `activeModelName` corresponds to a valid `_models` snapshot — no actual race is introduced by the `map + stateIn` derivation since Kotlin StateFlow's `map` is atomic with respect to each emission.
