# Task 15: Test Audit

## Required Skills

## Description

Full Phase-3 test quality audit. All nine test files created in this feature are read and analyzed holistically. The auditor is not writing or fixing code — this task produces a written audit report documenting the quality, coverage gaps, and structural issues found across the test suite.

The audit targets:

1. **Atomic transition coverage** — does `ChatRepositoryTest` cover both the success and failure paths of the draft→committed transition (staging write → Room `@Transaction` → rename)? Is the rollback branch (rename throws after successful Room INSERT) explicitly tested?
2. **Concurrency scenarios** — does `WarmupCoordinatorTest` verify that the in-flight `Job` is cancelled *before* a new `initialize()` call starts during `cancelAndRestart` (not just the final observable state)? Does `ModelRegistryActiveModelTest` exercise concurrent reads during `initialize`?
3. **Error paths** — are `IOException` scenarios for `ChatRepository` staging writes covered? Is warmup failure (`ErrorLog.e("engine-warmup", ...)`) tested? Is DB corruption handling tested anywhere?
4. **Test pyramid balance** — 6 unit tests vs 3 instrumented tests. Is the split appropriate given what each layer can verify? Are there unit tests that belong in instrumented or vice versa?
5. **Meaningful assertions** — do tests assert actual behavior (state transitions, side effects, emitted values, call counts) or merely that no exception was thrown?
6. **Double-bubble handover** — does `ChatViewModelTest` include a test that verifies no emission where both the in-memory `_streamingMessage` AND a persisted ASSISTANT row co-exist simultaneously (the flicker scenario from tech-spec Risks)?
7. **TDD compliance** — do the test files have the structural marks of TDD (tests designed around spec AC items, not reverse-engineered from implementation)? Git history can be checked to verify test files precede implementation files.

This audit task is Wave 7 (Audit Wave). The auditor IS the reviewer — no separate reviewers. The output is a written report committed to `decisions.md` plus any findings that should be raised as follow-on issues.

## What to do

1. Read all nine test files listed in Context Files below.

2. For each test file, record:
   - Which acceptance criteria it maps to (from tech-spec Testing Strategy section)
   - Whether all relevant AC branches are exercised (not just the happy path)
   - Whether assertions check actual behavior or are trivially "did not throw"

3. Audit atomic transition coverage in `ChatRepositoryTest`:
   - Happy path: staging write → Room `@Transaction` INSERT → rename → `chatId` returned
   - Failure path A: `IOException` on staging write → no Room row, staging dir cleaned up
   - Failure path B: Room `@Transaction` succeeds but `rename()` throws → staging dir cleaned up AND Room row deleted
   - Confirm both failure paths are present as independent test cases

4. Audit concurrency in `WarmupCoordinatorTest`:
   - Does `cancelAndRestart` test verify the in-flight Job is cancelled BEFORE new `initialize()` call starts? (Observable via mock call ordering or a `TestCoroutineScheduler` advancement, not just checking final state)
   - Is the AC-F3 observer test (auto-set `default_model_id` on first downloaded model) present and testing the "previously empty" precondition?

5. Audit concurrency in `ModelRegistryActiveModelTest`:
   - Are concurrent reads during `initialize` tested (the J5 hazard mentioned in tech-spec Task 3)?
   - Does `activeModelName` correctly return null when Idle/Initializing/Failed?

6. Audit `ChatViewModelTest` for double-bubble:
   - Is there an explicit test that emits a Room ASSISTANT row and a `_streamingMessage` simultaneously and verifies that no single `displayMessages` emission contains BOTH?
   - Is Quick mode verified to never touch `MessageDao`?
   - Is `onCleared` verified to not call `registry.cleanup`?

7. Audit error paths:
   - `WarmupCoordinatorTest`: warmup failure → `ErrorLog.e("engine-warmup", ...)` — is this tested with a mock `ErrorLog` or log capture?
   - `ChatRepositoryTest`: `deleteChat()` — are both Room DELETE and `File.deleteRecursively()` asserted (not just one of them)?
   - `ChatRepositoryTest`: zombie sweep — is the "0-message chat with missing dir → deleted" case tested?

8. Audit test pyramid balance:
   - Are there any tests in `ChatDaoTest` / `MessageDaoTest` / `SanctumDatabaseTest` that test pure logic that could be unit-tested without a real SQLite engine?
   - Are there any unit tests that require Room semantics (e.g., Flow emissions from DAO) that are awkwardly mocked instead of running instrumented?

9. Check `AutoTitleGeneratorTest` completeness against AC-U2:
   - trim → whitespace collapse → cut at last space ≤ 20 → "…" appended
   - fallback "Чат от DD.MM HH:mm" when first message has no text
   - manual-title flag suppresses auto-title

10. Check `SettingsMigrationHelperTest` completeness against AC-R8 and Decision 8:
    - Single `updateData` call per run (atomicity)
    - Orphan key dropped + logged
    - Sentinel prevents re-run on second call

11. Verify TDD compliance via git log:
    - Run `git log --follow --diff-filter=A --name-only --format="%H %ad %s" -- app/src/test/... app/src/androidTest/...` (or equivalent) to check commit timestamps of test files vs implementation files for each task wave.
    - Note any test files that were committed after their corresponding implementation files (TDD violation).

12. Write and record the audit report:
    - Summary: overall test quality (pass/borderline/fail per dimension)
    - Findings per test file: what is well-covered, what is missing

    - Recommendations: specific test cases to add (if any)

## Details

**This is an audit task — no code is written.** The output is a written report only. If the auditor finds gaps, they are documented as recommendations; fixing those gaps is a separate follow-on task.

**Audit scope — what each test file should cover:**

`AutoTitleGeneratorTest` (pure function, no mocks needed):
- trim → whitespace collapse → cut at last space ≤ 20 → "…" suffix
- No cut needed when text is exactly 20 chars
- Cut falls on a space boundary (not mid-word)
- Fallback "Чат от DD.MM HH:mm" when no text in first message
- `isManuallyTitled = true` → auto-title generator not invoked

`SettingsMigrationHelperTest` (DataStore mock):
- Happy path: all `per_model_overrides` keys rekeyed from `Model.name` to `Model.modelId` in single `updateData` call
- Orphan key (no matching allowlist entry) dropped + `ErrorLog.e("settings-io", ...)` called
- Sentinel `settings_keys_migrated = true` prevents second run (`updateData` not called again)
- Single `updateData` call per run (atomicity — not multiple separate writes)

`ChatRepositoryTest` (faked DAOs + mock File):
- `commitDraftChat()` happy path: staging files written → Room `@Transaction` INSERT → staging dir renamed
- Failure A: `IOException` on staging write → no Room row created, staging dir cleaned up
- Failure B: Room `@Transaction` succeeds but `rename()` throws → staging dir cleaned up AND Room row deleted
- `deleteChat()`: both Room DELETE and `File.deleteRecursively()` asserted (not just one)
- Zombie sweep: 0-message chat with missing attachment dir → deleted; chat with messages → not touched
- Auto-title triggered on successful commit

`WarmupCoordinatorTest` (mock ModelRegistry + AppSettingsRepository):
- Model resolution order: (a) `default_model_id` set + file exists → use it; (b) fallback `last_used_model_id`; (c) neither → no warmup
- `cancelAndRestart`: in-flight Job is cancelled AND its cancellation completes BEFORE `registry.initialize(newModelId)` is called — verified via mock call ordering or `TestCoroutineScheduler`, not just checking final result
- Warmup failure → `ErrorLog.e("engine-warmup", ...)` called with correct component string
- `last_used_model_id` updated on success (not on failure)
- AC-F3 observer: when `registry.models` emits a model with `isDownloaded=true` and `appSettings.getDefaultModelId()` returns empty → `setDefaultModelId()` called; if already non-empty → not called

`ModelRegistryActiveModelTest` (real `DefaultModelRegistry` or minimal fake):
- `activeModelName` emits model name when `ModelInitStatus.Ready`
- `activeModelName` emits `null` when `Idle`, `Initializing`, `Failed`
- Concurrent reads during `initialize` produce consistent values (J5 hazard — no race-window where `activeModelName` emits a stale/intermediate value)

`ChatViewModelTest` (TestCoroutineScope, fake DAOs, mock WarmupCoordinator):
- Draft mode: `commitDraftChat()` called on first Send → nav to `chat/{chatId}`
- Quick mode: `MessageDao` never called for any operation (room never touched)
- `onCleared()`: `registry.cleanup()` not called
- Engine-state transitions drive TopAppBar state (Idle → "Загрузить", Initializing → spinner, Ready → label, Failed → "Загрузить")
- Double-bubble: emit persisted ASSISTANT row + non-null `_streamingMessage` simultaneously → verify no single `displayMessages` emission contains both (the flicker scenario)

`ChatDaoTest` / `MessageDaoTest` (instrumented, real Room in-memory):
- CRUD: insert / query by id / update / delete
- `observeByChat` Flow emits on insert/update
- CASCADE: delete chat row → message rows gone (verified by querying messages after chat delete)
- `observeByChat` sorted by `created_at`

`SanctumDatabaseTest` (instrumented):
- Fresh DB opens without error
- Schema JSON (exported to `app/schemas/...SanctumDatabase/1.json`) matches `@Entity` definitions
- `messages.chat_id` FK with `ON DELETE CASCADE` fires correctly

**Test infrastructure context (from `patterns.md`):**
- JUnit 4.13.2 + Robolectric 4.12 for unit tests; hand-rolled fakes, no Mockito/MockK
- `androidx.room:room-testing` for instrumented DAO tests
- Compose UI tests are explicitly out of scope

**Dependency on Task 12:** Task 12 updates `patterns.md` with the expanded `ErrorLog` whitelist (6 new Phase-3 components). The auditor must check that `WarmupCoordinatorTest` and `ChatRepositoryTest` use the updated component strings (`"engine-warmup"`, `"history-write"`, `"attachment-save"` etc.), not any stale pre-Phase-3 strings. `depends_on: [12]` ensures this task runs after docs are accurate.

**Double-bubble flicker — background (tech-spec Risks):**
The risk is: at `done=true`, `ChatViewModel` does Room INSERT for the ASSISTANT message (persisted list emits new row), then clears `_streamingMessage`. If there's any emit window where the combined `displayMessages` contains both the new persisted row AND the still-non-null `_streamingMessage`, the UI briefly shows two ASSISTANT bubbles. The mitigation is that `_streamingMessage` is cleared inside the same `mapLatest` block that observes the new persisted emission. The test must verify this timing constraint — not just that the final state is correct.
