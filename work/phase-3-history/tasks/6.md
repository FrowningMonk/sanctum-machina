## Description

This task wires together the Phase 3 infrastructure into `SanctumApplication.onCreate` and hardens `AppModule` against DB corruption. After Tasks 4 and 5 delivered `ChatRepository`, `WarmupCoordinator`, `SettingsMigrationHelper`, and `AppModule` (with a bare `Room.databaseBuilder.build()`), this task:

1. Extends `SanctumApplication` to `@Inject` both `WarmupCoordinator` and `ChatRepository`, then drives the cold-start sequence: `warmupCoordinator.warmupDefault()` + `settingsMigrationHelper.migrateIfNeeded()` + three housekeeping coroutines — all behind the existing `getProcessName() == packageName` guard.
2. Wraps `Room.databaseBuilder.build()` inside `AppModule` with a corruption handler (rename + fresh DB + in-memory signal).
3. Creates `AppCorruptionState` — a simple Hilt `@Singleton` boolean flag; `HomeScreen` reads it in Task 10 to show a one-time banner.

The cold-start sequence after this task is fully operational: `super.onCreate()` → Hilt injection → `installCrashHandler()` → (packageName guard) warmup launched → housekeeping launched. No part of this touches UI — the task is purely plumbing in `:app`.

## What to do

1. **Create `AppCorruptionState.kt`** — a Hilt `@Singleton` with a single `var corruptionOccurred: Boolean = false` property. No persistence; the flag lives only for the duration of the current process.

2. **Extend `AppModule.databaseBuilder` block** — wrap `Room.databaseBuilder(...).build()` in a `try/catch(Exception)`. On exception: rename `sanctum.db` to `sanctum.db.corrupt_{YYYYMMDD-HHmmss}` (format with `Locale.ROOT`), call `Room.databaseBuilder(...).build()` again for a fresh empty DB, set `appCorruptionState.corruptionOccurred = true`, log `ErrorLog.e("history-read", "db open failed — renamed to sanctum.db.corrupt_{ts}", cause)`. Inject `AppCorruptionState` and `ErrorLog` into the provider method (or the module constructor). The rename target is inside `context.getDatabasePath("sanctum.db").parentFile`.

3. **Extend `SanctumApplication`** with `@Inject lateinit var warmupCoordinator: WarmupCoordinator` and `@Inject lateinit var chatRepository: ChatRepository`. No `AppSettingsRepository` injection needed — `WarmupCoordinator` already owns that.

4. **Update `SanctumApplication.onCreate`** — inside the existing `if (getProcessName() == packageName)` block, after `installCrashHandler()`, add:
   - Launch a `CoroutineScope(SupervisorJob() + Dispatchers.Default)` coroutine that calls `warmupCoordinator.warmupDefault()`.
   - Launch a separate coroutine (same or new scope) for `SettingsMigrationHelper.migrateIfNeeded()`.
   - Launch a third coroutine (on `Dispatchers.IO`) for housekeeping: (a) `filesDir/quick/` purge — wrapped in `try/catch` + `ErrorLog.e("attachment-save", ...)` on failure; (b) orphan `.staging-*` cleanup inside `filesDir/attachments/` — same `try/catch + ErrorLog.e("attachment-save", ...)`; (c) `chatRepository.sweepZombieChats()`.

5. **Inject `SettingsMigrationHelper`** — add `@Inject lateinit var settingsMigrationHelper: SettingsMigrationHelper` to `SanctumApplication`. `SettingsMigrationHelper` has an `@Inject constructor` (created in Task 2) and is resolved by Hilt automatically — no explicit `@Provides` method in `AppModule` is needed or expected.

6. **Write unit tests** in `SanctumApplicationTest` covering: (a) housekeeping coroutine does NOT run when `getProcessName() != packageName`; (b) `filesDir/quick/` purge failure calls `ErrorLog.e("attachment-save", ...)`; (c) orphan staging-dir cleanup failure calls `ErrorLog.e("attachment-save", ...)`.
   Write tests for `AppCorruptionState` itself (trivially: flag starts false, can be set true).
   Write **instrumented** tests for the DB corruption path in `AppModule` (see TDD Anchor): pre-create a corrupt `sanctum.db` file in `filesDir`, invoke `provideDatabase`, verify rename + fresh DB + `corruptionOccurred == true`. These go in `app/src/androidTest/` because intercepting `Room.databaseBuilder` is not feasible in JUnit 4 unit tests without PowerMock/MockK static mocking (not in this project).

- `app/src/test/.../SanctumApplicationTest::testQuickPurgeFailureLogsAttachmentSave` — when deleting `filesDir/quick/` throws an `IOException`, `ErrorLog.e("attachment-save", ...)` is called; no exception propagates.

- `app/src/test/.../AppCorruptionStateTest::testCanBeSetTrue` — setting `corruptionOccurred = true` persists within the same instance.
- `app/src/androidTest/.../AppModuleCorruptionTest::testCorruptDbRenamedAndFreshDbCreated` — **instrumented test** (runs on device/emulator). Before constructing the Room database, write garbage bytes to `context.getDatabasePath("sanctum.db")` so the file exists but is not a valid SQLite file. Then invoke `AppModule().provideDatabase(context, appCorruptionState, errorLog)`. Verify: (a) a file matching `sanctum.db.corrupt_*` exists in the same directory, (b) the returned database is open and operational (e.g. `db.isOpen == true`), (c) `appCorruptionState.corruptionOccurred == true`, (d) `ErrorLog.e("history-read", ...)` was called.
- `app/src/androidTest/.../AppModuleCorruptionTest::testNormalDbOpenNoCorruptionFlag` — instrumented test. Do not pre-create any corrupt file; invoke `provideDatabase(...)` normally. Verify `appCorruptionState.corruptionOccurred` remains `false` and no `sanctum.db.corrupt_*` file was created.

## Details

**Files:**

- `SanctumApplication.kt` — currently 26 lines: `@HiltAndroidApp`, `onCreate` with `super.onCreate()`, the `if (getProcessName() == packageName)` block calling only `installCrashHandler()`, and `DefaultDownloadRepository.mainActivityFqn` assignment. The `packageName` guard is already in place from Phase 2.5. New `@Inject lateinit var` fields sit at class level; all new logic goes inside the existing guard, after `installCrashHandler()`.
- `AppModule.kt` — created in Task 5; provides `SanctumDatabase`, `ChatDao`, `MessageDao`, `ChatRepository`, `WarmupCoordinator`. The `@Provides @Singleton fun provideDatabase(...)` method needs `appCorruptionState: AppCorruptionState` and `errorLog: ErrorLog` added to its parameters, and the `build()` call wrapped in `try/catch`.
- `AppCorruptionState.kt` (new) — three lines: package declaration, `@Singleton class AppCorruptionState @Inject constructor()`, `var corruptionOccurred: Boolean = false`. No need for `StateFlow` here — `HomeScreen` reads it once in `LaunchedEffect` (that's Task 10's concern).

**Dependencies:**
- Task 4 must be done: `ChatRepository` interface with `sweepZombieChats()`.
- Task 5 must be done: `WarmupCoordinator` with `warmupDefault()`, `AppModule` base, `SettingsMigrationHelper` with `migrateIfNeeded()`.
- `ErrorLog` whitelist must include `"attachment-save"` and `"history-read"` (added in Task 2).

**Edge cases:**
- `filesDir/quick/` may not exist on first launch — `File.deleteRecursively()` on a non-existent file returns `false` without throwing; this is fine. Re-create the directory only if explicitly needed downstream (not needed here — the incognito guarantee is "purge what exists").
- Orphan staging dirs: `filesDir/attachments/` may not exist either. Glob via `File.listFiles { _, name -> name.startsWith(".staging-") }` — if `listFiles` returns `null` (dir doesn't exist), skip the cleanup loop silently.
- Corruption timestamp format: use `java.time.LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMdd-HHmmss"))` with `Locale.ROOT`. The rename target file name must be deterministic for the log entry to match.
- `Room.databaseBuilder.build()` can throw `SQLiteException`, `IllegalStateException`, or `RuntimeException` — catch `Exception` broadly, not a specific subtype.
- The second `Room.databaseBuilder.build()` call (after rename) should NOT itself be in a try/catch — if a fresh DB cannot be opened, something is fundamentally broken and the uncaught exception should propagate to `CrashHandler` (which is already installed by this point).

**Hilt wiring note:**
- `SanctumApplication` is annotated `@HiltAndroidApp`, so its fields annotated `@Inject lateinit var` are injected automatically by Hilt before `onCreate()` is called — no `EntryPointAccessors` or manual entry-point injection needed. This is simpler than the pattern described in code-research section A5 (which was written before the `@HiltAndroidApp` injection behaviour was fully explored).
- `AppCorruptionState` is a class-annotated singleton. It is available for injection in `AppModule` without any explicit `@Provides` method — Hilt resolves `@Singleton @Inject constructor()` classes directly.

**Process-guard correctness:**
- The `DefaultDownloadRepository.mainActivityFqn` assignment (currently line 20 of `SanctumApplication.kt`) is OUTSIDE the `packageName` guard and must remain so — the download worker runs in the background process and needs this FQN. Do not accidentally move it inside the guard.

**Coroutine scope:**
- Use `CoroutineScope(SupervisorJob() + Dispatchers.Default)` for the warmup and migration coroutines (fire-and-forget; scope outlives `onCreate`). Use `Dispatchers.IO` for the housekeeping coroutine (file operations).
- Do not use `GlobalScope` (lint-discouraged per `patterns.md`). A local `val appScope = CoroutineScope(SupervisorJob() + Dispatchers.Default)` defined in the `packageName` block is acceptable — it lives for the process lifetime.
